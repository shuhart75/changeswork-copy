# Заметки к scope prototype

## Цель
Согласовать MVP-матрицу ролей и product scope так, чтобы соседние features опирались на единый источник истины по доступам.

## Какой идеальный сценарий показываем
- пользователь входит в систему и получает доступ только к допустимым продуктам и разделам;
- UI скрывает недоступные действия на host screens;
- backend отклоняет недопустимые запросы даже при ошибке UI;
- одна и та же ролевая модель одинаково трактуется в approvals, packages, pilots, deployments и scorecards.

## Какой MVP cut показываем отдельно
- роли `prm`, `methodologist`, `approver`, `ratifier`, `admin`;
- product scope и множественные роли на пользователя;
- базовая матрица видимости разделов и действий;
- интеграция с существующим контуром `rscon-sudir` без отдельного admin UI.

## Какие planning stories обсуждаем через этот прототип
- `imported-rbac-scope`
- `STORY-ROLES-001` — контрольный feature-level story для проверки требований

## Mock data
- 1 пользователь с одной ролью в одном продукте;
- 1 пользователь с несколькими ролями в разных продуктах;
- примеры скрытого action на FE и отклонённого запроса на BE;
- таблица видимости разделов `Pilots`, `Deployments`, `Approvals`, `Packages`, `Scorecards`.

## Открытые вопросы
- как разрешаются конфликты при нескольких ролях и product scopes;
- какие denied actions должны уходить в аудит уже в MVP;
- нужно ли отдельно фиксировать системные служебные роли, не видимые в пользовательском UI.
