## Бизнес-требования

### Цель

Реализовать серверный API для управления скоркартами как отдельным доменным агрегатом, который:
- создаётся из версии шаблона скоркарты;
- поддерживает версионирование через `ScorecardVersion`;
- хранит конфигурацию и финансовые эффекты;
- отслеживает происхождение данных из `Simulation`, если скоркарта создавалась с частичным предзаполнением;
- связывается с версиями `Pilot` и `Deployment` и хранит источники типа `Simulation`;
- вычисляет свой статус по наличию связей с доменными элементами;
- принимает рассчитанную на фронте критичность.

### Источник правды

Основным источником правил для этой спецификации является `final-spec/spec_domain_model.md`.  
При расхождениях с устаревшими задачами приоритет имеет доменная модель версии 3.2.

### Определения

- `Скоркарта` — самостоятельная предметная сущность, описывающая набор настроек и финансовых эффектов для оценки изменения в продукте.
- `Версия скоркарты` — неизменяемый снимок содержимого скоркарты на конкретный момент времени. Именно версия скоркарты связывается с версиями доменных элементов.
- `Версия шаблона скоркарты` — зафиксированная структура полей, значений по умолчанию и порогов расчёта критичности, по которой строится форма и валидируются данные скоркарты.
- `Родительский доменный элемент` (`parent_context`) — доменный элемент, из формы которого пользователь создаёт скоркарту. В MVP это только `Pilot` или `Deployment`.
- `Источник данных скоркарты` (`source_refs`) — ссылка на сущность, данные которой использовались для частичного предзаполнения скоркарты. В MVP поддерживается только `Simulation`.
- `Использование скоркарты` — факт связи конкретной версии скоркарты с конкретной версией `Pilot` или `Deployment`.
- `Обязательная lineage-скоркарта внедрения` — скоркарта, которая для внедрения типа `simulation_based` зафиксирована в `Deployment.required_scorecard_id` и не может быть удалена из состава такого внедрения.

### Бизнес-правила

#### БП-1. Создание скоркарты
- Скоркарта создаётся только на основе валидной `ScorecardTemplateVersion`.
- В MVP создание скоркарты инициируется только из контекста доменного элемента `Pilot` или `Deployment`.
- Родительский доменный элемент принадлежит ровно одному продукту, поэтому продуктовый контекст заранее известен и должен быть согласован с выбранной `template_version`.
- При создании формируется агрегат `Scorecard` и первая запись `ScorecardVersion` с номером версии `1`.
- При создании одновременно формируется первая связь с версией доменного элемента из `parent_context`.
- Так как создание первой связи меняет состав связанных сущностей, для доменного элемента из `parent_context` создаётся новая версия по правилам его доменного контекста.
- Скоркарта может быть:
  - ручной (`is_manual = true`);
  - созданной с частичным предзаполнением по данным `Simulation`.
- Если скоркарта не ручная, у неё должен быть минимум один источник.
- Название скоркарты обязательно.
- Название скоркарты во всех сценариях создания формируется автоматически и не редактируется пользователем ни при создании, ни при последующем редактировании скоркарты.
- Если скоркарта создаётся вручную из контекста `Pilot`, фронт должен автоматически передать название в формате `Скоркарта {scorecardId} для пилота {pilotId}: {pilotName}.`.
- Если скоркарта создаётся вручную из контекста `Deployment`, фронт должен автоматически передать название в формате `Скоркарта {scorecardId} для внедрения {deploymentId}: {deploymentName}.`.
- Если скоркарта создаётся с частичным предзаполнением по `Simulation`, фронт должен автоматически передать название в формате `Скоркарта {scorecardId} по симуляции {simulationId}: {simulationName}.`; ручное редактирование этого поля на фронте недоступно.
- На момент отправки формы значение `{scorecardId}`, используемое в автоназвании, должно быть известно форме создания.

#### БП-2. Версионирование
- Любое содержательное изменение скоркарты создаёт новую `ScorecardVersion`.
- Номера версий последовательные, целочисленные, без пропусков.
- Поле `name` является системным и не может изменяться через API редактирования скоркарты после создания.
- В `Scorecard` хранятся агрегатные метаданные, в `ScorecardVersion` хранятся:
  - `configuration`;
  - `financial_effects`;
  - рассчитанная критичность;
  - ссылка на автора версии;
  - дата создания версии.
- Версия скоркарты всегда жёстко ассоциирована с версиями доменных элементов, с которыми она связана.
- Создание новой версии скоркарты обязательно влечёт создание новых версий всех связанных доменных элементов.
- Если версия связанного доменного элемента находилась в цикле согласования, при создании новой версии этот цикл должен прерываться по правилам жизненного цикла соответствующей сущности.
- Связи `ScorecardVersion <-> DomainElementVersion` фиксируются отдельными M:N-таблицами по типам доменных элементов.

#### БП-3. Связи и область применения
- Скоркарта принадлежит одному продукту через `template_version -> template -> product`.
- В MVP подпродукты для скоркарт не используются.
- Скоркарта может иметь много источников типа `Simulation` через `scorecard_source`.
- Скоркарта может быть связана со многими `Pilot` и `Deployment`.
- `Pilot` и `Deployment` могут быть связаны со многими скоркартами.
- Удаление или создание новой связи скоркарты с доменным элементом создаёт новую версию только этого доменного элемента, а не всех связанных.
- Изменение состава связей может выполняться только через API соответствующего доменного элемента.
- API редактирования скоркарты не должно принимать операции добавления или удаления связей с доменными элементами, так как такие изменения могут требовать отдельного согласования или утверждения по правилам этих сущностей.
- Если при изменении состава связей удаляется или добавляется связь только с одним доменным элементом, новая версия создаётся только для этого доменного элемента.
- Если при изменении состава связей доменный элемент `Pilot` или `Deployment` теряет связь со своей последней скоркартой, система обязана автоматически создать для этого доменного элемента новую скоркарту из актуальной версии шаблона его продукта со значениями по умолчанию и сразу связать её с новой версией этого доменного элемента.
- Такое автосоздание замещающей скоркарты выполняется в рамках того же бизнес-сценария изменения связей доменного элемента и не требует отдельного пользовательского действия.
- Если скоркарта зафиксирована как `Deployment.required_scorecard_id` для внедрения типа `simulation_based`, удаление её связи с этим внедрением запрещено.

#### БП-4. Источники и использование
- Источники скоркарты фиксируются через `scorecard_source`.
- Допустимые типы источника в MVP:
  - `simulation`;
- Для деталки должны быть доступны:
  - источники формирования скоркарты;
  - использование скоркарты в пилотах и внедрениях.

#### БП-5. Жизненный цикл и статус
- Жизненный цикл скоркарты в MVP включает только два статуса:
  - `active`;
  - `archive`.
- `active` устанавливается, если со скоркартой связана хотя бы одна версия доменного элемента, не считая источников.
- `archive` устанавливается, если удаляется последняя связь скоркарты с версиями доменных элементов, не считая источников.
- Источники из `scorecard_source` не участвуют в вычислении статуса скоркарты.
- Скоркарта не проходит согласование как самостоятельная сущность.
- Скоркарта в статусе `archive` не редактируется.

#### БП-6. Права доступа
- Просмотр скоркарт доступен всем ролям, имеющим доступ к доменной модели.
- Создание и редактирование доступны:
  - `prm` своего продукта;
  - `methodologist` своего продукта;
  - `admin` без ограничения по продукту.
- `approver` и `ratifier` не создают и не редактируют скоркарты.

## Границы MVP

### Входит в MVP
- Создание ручной скоркарты.
- Создание скоркарты из контекста `Pilot` или `Deployment`.
- Частичное предзаполнение данных скоркарты по результатам `Simulation` с передачей `source_refs`.
- Редактирование скоркарты с созданием новой версии.
- Детальная карточка скоркарты.
- Единый блок источников и использования скоркарты.
- Фиксация источников скоркарты через `scorecard_source` и M:N-связей использования между версиями скоркарты и версиями `Pilot` / `Deployment`.
- API списка выбора существующих скоркарт для управления связями с доменными элементами.
- API только для чтения по версиям шаблонов скоркарт.
- Вычисление статуса скоркарты по наличию связей с доменными элементами.
- Передача рассчитанной на фронте критичности в create/update API.

### Не входит в MVP
- Отдельная пользовательская страница списка скоркарт с расширенной таблицей.
- Экспорт списка скоркарт в Excel.
- Управление колонками, расширенные пользовательские фильтры, настройка плотности строк.
- Автоматическое наполнение фактических метрик из внешних систем.

## Пользовательские сценарии

### Сценарий 1. Ручное создание скоркарты
1. Пользователь с ролью `prm`, `methodologist` или `admin` открывает форму создания из контекста `Pilot` или `Deployment`.
2. Продуктовый контекст определяется родительским доменным элементом.
3. Фронт автоматически определяет и загружает актуальную версию шаблона для продукта.
4. Фронт автоматически заполняет `name` по контексту создания:
   - `Скоркарта {scorecardId} для пилота {pilotId}: {pilotName}.`
   - `Скоркарта {scorecardId} для внедрения {deploymentId}: {deploymentName}.`
5. Пользователь заполняет структурированные поля шаблона и эффекты.
6. Фронт рассчитывает критичность.
7. Система создаёт `Scorecard`, `ScorecardVersion v1` и новую версию доменного элемента из `parent_context`.
8. Между новой версией доменного элемента и `ScorecardVersion v1` фиксируется связь.
9. Новая скоркарта сохраняется со статусом `active`, так как при создании сразу появляется связь с родительским доменным элементом.

### Сценарий 2. Создание скоркарты с частичным предзаполнением по `Simulation`
1. Пользователь инициирует создание скоркарты из контекста `Pilot` или `Deployment`, используя данные `Simulation` как источник частичного предзаполнения.
2. Форма получает prefill конфигурации и/или финансовых эффектов из `Simulation`.
3. Фронт автоматически заполняет `name` в формате `Скоркарта {scorecardId} по симуляции {simulationId}: {simulationName}.` и не позволяет пользователю изменить его вручную.
4. Backend создаёт скоркарту, новую версию родительского доменного элемента и записывает источник в `scorecard_source`.

### Сценарий 3. Редактирование скоркарты
1. Пользователь открывает существующую скоркарту, доступную на редактирование.
2. Меняет конфигурацию, финансовые эффекты или метаданные.
3. Система создаёт новую `ScorecardVersion`.
4. Система создаёт новые версии всех связанных доменных элементов.
5. Предыдущие версии остаются доступны в истории.

### Сценарий 3.1. Изменение связей скоркарты
1. Пользователь добавляет или удаляет связь скоркарты из формы соответствующего доменного элемента.
2. Система создаёт новую версию только этого доменного элемента.
3. Если после удаления связи у этого доменного элемента не остаётся ни одной связанной скоркарты, система автоматически создаёт новую скоркарту из актуальной версии шаблона продукта со значениями по умолчанию и связывает её с новой версией доменного элемента.
4. Остальные связанные элементы и версия скоркарты не перевыпускаются только из-за изменения связи.

### Сценарий 4. Просмотр деталки
1. Пользователь открывает скоркарту по прямой ссылке или из связанных сущностей.
2. Получает основную информацию, текущую версию, источники, использование и агрегированные счётчики.

### Сценарий 5. Выбор скоркарты в доменной форме
1. Пользователь открывает форму доменного элемента, где допускается управление связями со скоркартами.
2. Сервер возвращает доступные скоркарты для сценария выбора.
3. Форма сохраняет только допустимые для продукта сущности.

## Критерии приемки

### Критерий 1. CRUD и versioning
- [ ] `POST /api/v1/scorecards` создаёт скоркарту и первую версию
- [ ] `POST /api/v1/scorecards` поддерживает ручной сценарий и сценарий с `source_refs`
- [ ] `POST /api/v1/scorecards` принимает `parent_context.entity_type` только из `pilot` или `deployment`
- [ ] `POST /api/v1/scorecards` принимает `source_refs` только для частичного предзаполнения по `simulation`
- [ ] `POST /api/v1/scorecards` создаёт новую версию доменного элемента из `parent_context` и связь с `ScorecardVersion v1`
- [ ] `PUT /api/v1/scorecards/{id}` создаёт новую версию вместо in-place обновления
- [ ] `POST /api/v1/scorecards` и `PUT /api/v1/scorecards/{id}` принимают рассчитанную на фронте критичность
- [ ] Номер версии инкрементируется последовательно
- [ ] История версий не теряется

### Критерий 2. Деталка и связанные данные
- [ ] `GET /api/v1/scorecards/{id}` возвращает полную деталку
- [ ] `GET /api/v1/scorecards/{id}/relations` одним ответом возвращает и источники, и использование скоркарты
- [ ] Деталка скоркарты содержит полные сведения по источникам и использованию: тип, название, статус и ссылку

### Критерий 3. Lookup и шаблоны
- [ ] `GET /api/v1/scorecards` поддерживает сценарий выбора скоркарт в формах
- [ ] `GET /api/v1/scorecard-template-versions` возвращает технический список версий шаблонов для автоматического выбора актуальной версии по продукту
- [ ] `GET /api/v1/scorecard-template-versions/{id}/default-config` возвращает техническую конфигурацию шаблона для построения формы и расчёта критичности

### Критерий 4. Валидация и доступ
- [ ] Поле `name` обязательно
- [ ] При ручном создании из `pilot` frontend передаёт автоматически сформированное название в формате `Скоркарта {scorecardId} для пилота {pilotId}: {pilotName}.`
- [ ] При ручном создании из `deployment` frontend передаёт автоматически сформированное название в формате `Скоркарта {scorecardId} для внедрения {deploymentId}: {deploymentName}.`
- [ ] При создании с `source_refs` типа `simulation` frontend передаёт автоматически сформированное название в формате `Скоркарта {scorecardId} по симуляции {simulationId}: {simulationName}.`
- [ ] Поле `name` не редактируется пользователем ни в create, ни в edit сценарии
- [ ] `PUT /api/v1/scorecards/{id}` не допускает изменение `name`
- [ ] Валидация JSON-конфигурации работает
- [ ] Нельзя сохранить несуществующий `template_version_id`
- [ ] Нельзя редактировать скоркарту в статусе `archive`
- [ ] Права по ролям и продукту соблюдаются

### Критерий 5. Статус и связи
- [ ] При наличии хотя бы одной связи с доменным элементом, не считая источников, скоркарта имеет статус `active`
- [ ] Источники из `scorecard_source` не влияют на статус скоркарты
- [ ] При создании новой версии скоркарты создаются новые версии всех связанных доменных элементов
- [ ] При добавлении или удалении связи перевыпускается только затронутый доменный элемент
- [ ] Если доменный элемент теряет связь со своей последней скоркартой, backend в том же сценарии создаёт новую скоркарту из актуального шаблона продукта со значениями по умолчанию и привязывает её к новой версии этого доменного элемента
- [ ] При удалении последней связи скоркарта автоматически получает статус `archive`
- [ ] Если скоркарта является `required_scorecard_id` внедрения типа `simulation_based`, backend запрещает удаление этой связи из внедрения

### Критерий 6. Миграция данных
- [ ] Для каждого существующего пилота создана скоркарта с дефолтными значениями индикаторов
- [ ] Финансовые метрики из legacy-полей пилота корректно скопированы в `financial_effects` скоркарты
- [ ] Все созданные скоркарты имеют статус `active` и версию `1`
- [ ] Для каждого мигрированного пилота создана новая версия с увеличенным номером
- [ ] Связи `scorecard_version_pilot_version` созданы корректно
- [ ] Критичность скоркарт рассчитана по алгоритму из шаблона
- [ ] Миграция идемпотентна (повторный запуск не создаёт дубликаты)

### ФТ-4. API создания

**Маршрут API:** `POST /api/v1/scorecards`

Запрос:

```json
{
  "name": "Application scorecard Q2",
  "criticality": "high",
  "template_version_id": "uuid",
  "parent_context": {
    "entity_type": "pilot",
    "entity_id": "uuid"
  },
  "configuration": {
    "launch_new_credit_product": false,
    "new_decision_model": true
  },
  "financial_effects": {
    "ar_delta_pp": 0,
    "issuance_volume_pct": 0.02,
    "rc_pct": 0.08,
    "npv_pct": 0.12,
    "npl30_pct": 0,
    "f_rorwa_pct": 0
  },
  "description": "Описание скоркарты",
  "source_refs": [
    {
      "source_type": "simulation",
      "source_id": "uuid"
    }
  ]
}
```

Правила:
- `name`, `criticality`, `template_version_id`, `configuration` обязательны;
- при создании с `source_refs` типа `simulation` поле `name` заполняется фронтом автоматически по формату `Скоркарта {scorecardId} по симуляции {simulationId}: {simulationName}.`;
- значение `{scorecardId}` должно быть известно форме на момент отправки запроса;
- `criticality` приходит с фронта как результат вычисления по полям шаблона;
- `parent_context` обязателен при создании;
- продукт не передаётся отдельно, а должен быть консистентен между `parent_context`, выбранной `template_version` и всеми `source_refs`;
- `parent_context` описывает родительский доменный элемент, из которого открыта форма создания;
- `parent_context.entity_type` в MVP допускает только `pilot` или `deployment`;
- `source_refs` описывает происхождение данных скоркарты и не обязан совпадать с `parent_context`;
- в MVP `source_refs` допускает только ссылки на `simulation`;
- `source_refs` обязательны только если `is_manual = false`;
- если `source_refs` не переданы, скоркарта считается ручной;
- успешное создание создаёт новую версию доменного элемента из `parent_context`, потому что появляется новая связь со скоркартой.
- успешное создание устанавливает статус скоркарты `active`, так как возникает связь с родительским доменным элементом.
- если `parent_context.entity_type = deployment`, созданная по `Simulation` скоркарта может быть позже зафиксирована API внедрений как обязательная lineage-скоркарта для `deployment_type = simulation_based`.

### Принцип проектирования операций-действий

В текущем MVP отдельные action-операции по самой скоркарте не выделяются в API.

Если в рамках расширения объёма MVP появятся явные действия над скоркартой, которые не являются обычным чтением или обновлением ресурса, они должны оформляться через единый маршрут API:
- `POST /api/v1/scorecards/{id}/action`

Тело такого запроса должно содержать поле `action`, а не требовать отдельного маршрута API на каждое действие.

## Описание модели данных

### Таблица `scorecard`

Назначение:
- Хранит сам агрегат скоркарты: его стабильный идентификатор, жизненный цикл и общие поля, не зависящие от номера версии.
- Нужна для того, чтобы все версии одной и той же скоркарты жили под одним `scorecard_id`.

| Поле | Тип | Обязательность | Описание |
|---|---|---:|---|
| `id` | UUID | да | PK скоркарты |
| `template_version_id` | UUID | да | FK на `scorecard_template_version.id`; определяет исходный шаблон и продукт |
| `name` | VARCHAR(255) | да | Бизнес-название скоркарты |
| `status` | VARCHAR(32) | да | Текущий вычисленный статус скоркарты: `active` или `archive` |
| `description` | TEXT | нет | Описание назначения и особенностей скоркарты |
| `is_manual` | BOOLEAN | да | Признак ручного создания; если `false`, должен быть хотя бы один источник |
| `created_at` | TIMESTAMP WITH TIME ZONE | да | Дата и время создания агрегата |
| `created_by` | UUID | да | Пользователь, создавший агрегат |
| `updated_at` | TIMESTAMP WITH TIME ZONE | да | Дата и время последнего изменения агрегата |
| `updated_by` | UUID | да | Пользователь, внёсший последнее изменение агрегата |

Примечания:
- `display_id` должен отдаваться в API как человекочитаемый идентификатор скоркарты; он не является отдельной колонкой таблицы `scorecard`, а способ его хранения или вычисления определяется общим проектным механизмом и не детализируется в рамках этой задачи.
- `product_id` не хранится отдельно, а определяется через `template_version_id`.
- `criticality` не хранится в таблице `scorecard`; в API она отдается как критичность текущей версии из `scorecard_version`.

### Таблица `scorecard_version`

Назначение:
- Хранит версионное содержимое скоркарты.
- Нужна для фиксации каждого изменения конфигурации, финансовых эффектов и критичности без перезаписи истории.

| Поле | Тип | Обязательность | Описание |
|---|---|---:|---|
| `id` | UUID | да | PK версии скоркарты |
| `scorecard_id` | UUID | да | FK на `scorecard.id` |
| `version` | INTEGER | да | Порядковый номер версии, начиная с `1` |
| `config` | JSONB | да | Конфигурация скоркарты; в API маппится в поле `configuration` |
| `financial_effects` | JSONB | нет | Финансовые эффекты версии |
| `criticality` | VARCHAR(16) | да | Рассчитанная критичность версии, переданная с фронта; для MVP допустимы `low` и `high` |
| `created_at` | TIMESTAMP WITH TIME ZONE | да | Дата и время создания версии |
| `created_by` | UUID | да | Пользователь, создавший версию |

Примечания:
- Для одной скоркарты комбинация `(scorecard_id, version)` должна быть уникальной.
- Текущая версия определяется по максимальному `version` или явно через сервисный слой.

### Таблица `scorecard_source`

Назначение:
- Хранит происхождение данных конкретной версии скоркарты.
- Нужна для отображения блока `Источники` в деталке и для фиксации, из какой `Simulation` было выполнено частичное предзаполнение.

| Поле | Тип | Обязательность | Описание |
|---|---|---:|---|
| `scorecard_version_id` | UUID | да | FK на `scorecard_version.id` |
| `source_type` | VARCHAR(16) | да | Тип источника; в MVP допустимо только значение `simulation` |
| `source_id` | UUID | да | Идентификатор сущности-источника; в MVP это `simulation.id` |
| `created_at` | TIMESTAMP WITH TIME ZONE | да | Дата фиксации связи источника |

Примечания:
- Композитный PK: `(scorecard_version_id, source_type, source_id)`.
- Для MVP строгий FK на разные source-таблицы не обязателен; валидируется бизнес-логикой.
- В текущей доменной модели `Simulation` не версионируется, поэтому `scorecard_source` хранит ссылку на `simulation.id`, а не на отдельную версию симуляции.

### Таблица `scorecard_version_pilot_version`

Назначение:
- Хранит фактическое использование версии скоркарты в конкретной версии `Pilot`.
- Нужна для блока `Используется в`, для вычисления статуса скоркарты и для правил перевыпуска связанных доменных элементов.

| Поле | Тип | Обязательность | Описание |
|---|---|---:|---|
| `scorecard_version_id` | UUID | да | FK на `scorecard_version.id` |
| `pilot_version_id` | UUID | да | FK на `pilot_version.id` |
| `linked_at` | TIMESTAMP WITH TIME ZONE | да | Дата и время фиксации связи |
| `linked_by` | UUID | да | Пользователь или сервис, создавший связь |

Примечания:
- Таблица фиксирует именно связь версий, а не агрегатов.
- Композитный PK: `(scorecard_version_id, pilot_version_id)`.

### Таблица `scorecard_version_deployment_version`

Назначение:
- Хранит фактическое использование версии скоркарты в конкретной версии `Deployment`.
- Нужна для блока `Используется в`, для вычисления статуса скоркарты и для правил перевыпуска связанных доменных элементов.

| Поле | Тип | Обязательность | Описание |
|---|---|---:|---|
| `scorecard_version_id` | UUID | да | FK на `scorecard_version.id` |
| `deployment_version_id` | UUID | да | FK на `deployment_version.id` |
| `linked_at` | TIMESTAMP WITH TIME ZONE | да | Дата и время фиксации связи |
| `linked_by` | UUID | да | Пользователь или сервис, создавший связь |

Примечания:
- Таблица фиксирует именно связь версий, а не агрегатов.
- Композитный PK: `(scorecard_version_id, deployment_version_id)`.

### Таблица `scorecard_template` (используется только для чтения)

Назначение:
- Хранит агрегат шаблона скоркарты как сущность верхнего уровня.
- Нужна для привязки шаблона к продукту и для группировки нескольких `scorecard_template_version` в один шаблон.

| Поле | Тип | Обязательность | Описание |
|---|---|---:|---|
| `id` | UUID | да | PK шаблона скоркарты |
| `product_id` | UUID | да | FK на `product.id`; определяет продукт, к которому относится шаблон |
| `name` | VARCHAR(255) | да | Бизнес-название шаблона |
| `created_at` | TIMESTAMP WITH TIME ZONE | да | Дата и время создания шаблона |
| `created_by` | UUID | да | Пользователь, создавший шаблон |
| `updated_at` | TIMESTAMP WITH TIME ZONE | да | Дата и время последнего изменения шаблона |
| `updated_by` | UUID | да | Пользователь, внёсший последнее изменение шаблона |

Примечания:
- Таблица относится к соседнему доменному контексту шаблонов и в рамках текущей задачи используется только для чтения.
- Именно через `scorecard_template.product_id` обеспечивается продуктовая принадлежность `scorecard_template_version` и, как следствие, создаваемых скоркарт.
- Поле `name` используется для заполнения `template_name` в API списка версий шаблонов.

### Таблица `scorecard_template_version` (используется только для чтения)

Назначение:
- Даёт скоркарте исходную структуру полей, значения по умолчанию и пороги расчёта критичности.
- Нужна только для чтения при создании и редактировании скоркарты; запись в неё в рамках этой задачи не рассматривается.

| Поле | Тип | Обязательность | Описание |
|---|---|---:|---|
| `id` | UUID | да | PK версии шаблона |
| `template_id` | UUID | да | FK на `scorecard_template.id` |
| `version` | VARCHAR(32) | да | Бизнес-версия шаблона |
| `default_config` | JSONB | да | Конфигурация по умолчанию |
| `valid_from` | DATE | нет | Дата начала действия версии шаблона |
| `valid_to` | DATE | нет | Дата окончания действия версии шаблона |

Примечания:
- Таблица уже относится к соседнему доменному контексту шаблонов, но обязательна для реализации выбора шаблона в скоркарте.
- Продукт версии шаблона определяется не напрямую, а через связь `scorecard_template_version.template_id -> scorecard_template.product_id`.
- Для одного шаблона комбинация `(template_id, version)` должна быть уникальной.

### Решение по структуре связей скоркарты

В MVP для скоркарты используются отдельные таблицы `scorecard_source`, `scorecard_version_pilot_version` и `scorecard_version_deployment_version`, а не одна общая полиморфная таблица связей.

Обоснование:
- `Источник` и `Использование` имеют разный доменный смысл:
  - источник отвечает только за происхождение данных;
  - использование отвечает за фактическое применение скоркарты в доменных элементах.
- Только связи использования участвуют в вычислении статуса скоркарты `active/archive`.
- Только связи использования участвуют в правилах перевыпуска версий `Pilot` и `Deployment`.
- Источник в MVP ссылается на `simulation.id`, а использование допустимо только для `PilotVersion` и `DeploymentVersion`, то есть у этих связей разная адресация и разные инварианты.
- Единая полиморфная таблица потребовала бы условной логики по полям `relation_kind` и `entity_type`, ухудшила бы читаемость модели и ослабила бы контроль целостности на уровне схемы.

Вывод:
- отдельная `scorecard_source` нужна и оправдана;
- отдельные таблицы использования для `PilotVersion` и `DeploymentVersion` в MVP предпочтительнее одной общей полиморфной таблицы.

### Ограничения и индексы

Ограничения:
- CHECK `criticality IN ('low','high')` на `scorecard_version`
- CHECK `status IN ('active','archive')`
- UNIQUE `(scorecard_id, version)` на `scorecard_version`

Индексы:
- INDEX `scorecard(template_version_id)`
- INDEX `scorecard(status)`
- INDEX `scorecard(created_at)`
- INDEX `scorecard_version(scorecard_id, version DESC)`
- INDEX `scorecard_template(product_id)`
- INDEX `scorecard_template(name)`
- INDEX `scorecard_template_version(template_id, version DESC)`
- INDEX `scorecard_source(scorecard_version_id)`
- INDEX `scorecard_source(source_type, source_id)`
- INDEX `scorecard_version_pilot_version(scorecard_version_id)`
- INDEX `scorecard_version_pilot_version(pilot_version_id)`
- INDEX `scorecard_version_deployment_version(scorecard_version_id)`
- INDEX `scorecard_version_deployment_version(deployment_version_id)`

## Реализация серверной части

### Технические требования
- Использовать транзакцию на создание/обновление агрегата.
- Для деталки применять `select_related`/`prefetch_related` для шаблона, автора, источников, связей с версиями симуляций, пилотов и внедрений.
- Хранить `configuration` и `financial_effects` в JSONB.
- Подготовить слой сервисов для вычисления текущей версии и статуса.
- Изменение связей использования скоркарты с доменными элементами реализуется в сервисах `Pilot` / `Deployment`, но должно использовать единые таблицы связей версий скоркарты.
- Для `source_refs` в MVP поддерживать только источник типа `simulation`.

### История версий
- Для MVP допускается выдача только текущей версии в деталке.
- Внутренняя модель должна позволять последующее добавление отдельного API истории версий без миграции концепции.

## Интеграция и вычисление статуса

### Единая функция вычисления статуса

Необходимо реализовать доменную вспомогательную функцию вида:

```text
compute_scorecard_status(scorecard_id) -> status
```

Возможные результаты:
- `active`
- `archive`

### События пересчёта

Пересчёт статуса должен выполняться при событиях:
- создание новой связи со скоркартой из `Pilot` или `Deployment`;
- удаление связи со скоркартой из `Pilot` или `Deployment`;
- изменение состава связей в сервисах доменных элементов.

### Правило выбора статуса
- Если у скоркарты есть хотя бы одна связь с версией `Pilot` или `Deployment`, её статус равен `active`.
- Если у скоркарты не осталось ни одной связи с версиями `Pilot` или `Deployment`, её статус равен `archive`.
- Источники из `scorecard_source` не участвуют в вычислении статуса.

### Правила перевыпуска связанных сущностей
- Создание первой версии скоркарты создаёт новую версию доменного элемента из `parent_context`, потому что именно в ней появляется новая ссылка на скоркарту.
- Создание новой версии скоркарты создаёт новые версии всех доменных элементов, связанных с предыдущей актуальной версией скоркарты.
- Добавление связи существующей скоркарты к доменному элементу создаёт новую версию только этого доменного элемента и новую запись в одной из таблиц `scorecard_version_*_version`.
- Удаление связи существующей скоркарты с доменным элементом создаёт новую версию только этого доменного элемента и удаляет или закрывает соответствующую связь на новой версии.
- Если после удаления связи новая версия доменного элемента не содержит ни одной скоркарты, backend в той же транзакции создаёт новую скоркарту на основе актуальной версии шаблона продукта этого доменного элемента, формирует для неё `ScorecardVersion v1` со значениями по умолчанию и привязывает её к новой версии доменного элемента.
- Если у затронутого доменного элемента был активный цикл согласования, он прерывается и перезапускается по его собственным правилам жизненного цикла.

## Ошибки и валидация

### HTTP-коды
- `400` — некорректный JSON или формат параметров
- `403` — недостаточно прав
- `404` — скоркарта, шаблон или источник не найден
- `409` — конфликт доменных ограничений
- `422` — нарушение бизнес-валидации

### Бизнес-валидация
- Недопустимое значение `criticality` блокируется.
- Пустой `configuration` разрешён только если это валидный JSON-объект по схеме шаблона.
- `template_version`, `parent_context` и все `source_refs` должны принадлежать одному продукту.
- `parent_context.entity_type` в MVP допускает только `pilot` или `deployment`.
- Если переданы `source_refs`, каждый `source_type` должен быть равен `simulation`.
- Нельзя обновлять скоркарту в статусе `archive`.
- Нельзя создавать не-ручную скоркарту без `source_refs`.
- При создании скоркарты `parent_context` обязателен всегда, даже для ручного сценария.

### Нефункциональные требования
- Все маршруты API покрываются модульными и интеграционными тестами.
- Логика вычисления статуса тестируется отдельно.
- Ошибки должны возвращаться в унифицированном формате API проекта.

## Миграция данных

### Цель миграции

Перенести существующие финансовые метрики, прикреплённые к пилотам, в новую доменную модель скоркарт.

### Область миграции

Миграция применяется ко всем существующим записям `Pilot`, у которых:
- есть заполненные финансовые метрики (поля финансовых эффектов не равны NULL или 0);
- ИЛИ требуется создать скоркарту с дефолтными значениями для соблюдения бизнес-правила "Пилот должен быть связан минимум с одной скоркартой".

### Алгоритм миграции

Для каждого пилота выполняется следующая последовательность действий:

#### Шаг 1. Определение шаблона скоркарты

1. Получить `product_id` пилота
2. Найти актуальную версию шаблона скоркарты для этого продукта:
   - Запрос к `scorecard_template_version` с фильтром по `product_id`
   - Выбрать версию с `valid_from <= NOW()` и (`valid_to IS NULL` OR `valid_to >= NOW()`)
   - Если найдено несколько версий, выбрать с максимальным `valid_from`
3. Если шаблон не найден:
   - Логировать ошибку
   - Пропустить пилот
   - Продолжить миграцию остальных пилотов

#### Шаг 2. Подготовка данных скоркарты

1. **Название**: `"Скоркарта пилота {pilot.display_id}"`
2. **Индикаторы (configuration)**:
   - Взять `default_values.configuration` из найденного шаблона
   - Все значения остаются дефолтными (false)
3. **Финансовые эффекты (financial_effects)**:
   - Если у пилота есть заполненные финансовые метрики:
     - Скопировать значения из соответствующих полей пилота
     - Маппинг полей определяется структурой legacy-модели пилота
   - Если у пилота нет финансовых метрик:
     - Взять `default_values.financial_effects` из шаблона (все нули)
4. **Критичность (criticality)**:
   - Рассчитать по алгоритму из `criticality_thresholds` шаблона
   - Применить правило: если хотя бы один эффект превышает порог → `high`, иначе → `low`
5. **Метаданные**:
   - `is_manual = true` (скоркарта создана вручную, не из симуляции)
   - `description = "Автоматически создано при миграции финансовых метрик пилота"`
   - `created_by` = системный пользователь миграции
   - `created_at` = время выполнения миграции

#### Шаг 3. Создание скоркарты

1. Создать запись в таблице `scorecard`:
   - `id` = новый UUID
   - `template_version_id` = ID найденного шаблона
   - `name` = подготовленное название
   - `status` = `active` (так как сразу будет связь с пилотом)
   - `description` = подготовленное описание
   - `is_manual` = true
   - `created_at`, `created_by`, `updated_at`, `updated_by` = системные значения

2. Создать запись в таблице `scorecard_version`:
   - `id` = новый UUID
   - `scorecard_id` = ID созданной скоркарты
   - `version` = 1
   - `config` = подготовленные индикаторы (JSONB)
   - `financial_effects` = подготовленные финансовые эффекты (JSONB)
   - `criticality` = рассчитанная критичность
   - `created_at` = время миграции
   - `created_by` = системный пользователь

3. **НЕ создавать** записи в `scorecard_source`, так как `is_manual = true`

#### Шаг 4. Создание новой версии пилота

Согласно БП-1, создание связи со скоркартой требует создания новой версии пилота:

1. Получить текущую актуальную версию пилота (`pilot_version` с максимальным `version`)
2. Создать новую запись `pilot_version`:
   - `version` = текущая версия + 1
   - Скопировать все поля из текущей версии
   - `created_at` = время миграции
   - `created_by` = системный пользователь

#### Шаг 5. Создание связи

Создать запись в таблице `scorecard_version_pilot_version`:
- `scorecard_version_id` = ID созданной версии скоркарты (v1)
- `pilot_version_id` = ID новой версии пилота
- `linked_at` = время миграции
- `linked_by` = системный пользователь

### Обработка ошибок

- Миграция выполняется в транзакции для каждого пилота отдельно
- Если миграция одного пилота завершилась ошибкой:
  - Откатить транзакцию для этого пилота
  - Логировать ошибку с деталями (pilot_id, причина)
  - Продолжить миграцию остальных пилотов
- После завершения миграции вывести отчёт:
  - Количество успешно мигрированных пилотов
  - Количество пропущенных пилотов (нет шаблона)
  - Количество ошибок с детализацией

### Идемпотентность

Миграция должна быть идемпотентной:
- Перед созданием скоркарты проверить, не существует ли уже скоркарта с `description` содержащим "Автоматически создано при миграции" и связанная с данным пилотом
- Если такая скоркарта найдена, пропустить создание и перейти к следующему пилоту
- Альтернативный подход: добавить флаг `migrated` в таблицу `pilot` для отслеживания статуса миграции

### Откат миграции

Для отката миграции необходимо:
1. Удалить все записи `scorecard_version_pilot_version`, созданные миграцией (по `linked_by` = системный пользователь миграции)
2. Удалить все версии пилотов, созданные миграцией (по `created_by` и `created_at`)
3. Удалить все `scorecard_version`, созданные миграцией
4. Удалить все `scorecard`, созданные миграцией (по `description` содержащему "Автоматически создано при миграции")

### Маппинг полей финансовых метрик

Соответствие между legacy-полями пилота и полями скоркарты определяется следующим образом:

| Legacy-поле пилота | Поле скоркарты | Примечание |
|---|---|---|
| `ar_delta_pp` | `financial_effects.ar_delta_pp` | Изменение AR в п.п. |
| `issuance_volume_pct` | `financial_effects.issuance_volume_pct` | Изменение объёма выдач в % (дробь) |
| `rc_pct` | `financial_effects.rc_pct` | Изменение RC в % (дробь) |
| `npv_pct` | `financial_effects.npv_pct` | Изменение NPV в % (дробь) |
| `npl30_pct` | `financial_effects.npl30_pct` | Изменение NPL30 в % (дробь) |
| `f_rorwa_pct` | `financial_effects.f_rorwa_pct` | Изменение F-RoRWA в % (дробь) |

**Важно**: Если в legacy-модели значения хранятся в процентах (например, 5 для 5%), а в скоркарте ожидаются дроби (0.05 для 5%), необходимо выполнить преобразование: `value_fraction = value_percent / 100`.

### Валидация после миграции

После завершения миграции выполнить проверки:
1. Все пилоты имеют хотя бы одну связанную скоркарту
2. Все созданные скоркарты имеют статус `active`
3. Все созданные скоркарты имеют `version = 1`
4. Количество созданных связей `scorecard_version_pilot_version` равно количеству успешно мигрированных пилотов
5. Для каждого мигрированного пилота создана новая версия

### Порядок выполнения миграции

1. Создать резервную копию базы данных
2. Выполнить миграцию в тестовой среде
3. Проверить результаты миграции
4. Выполнить миграцию в продуктивной среде в окне обслуживания
5. Выполнить валидацию после миграции
6. Сохранить отчёт о миграции
