## Общий контур feature

- Назначение feature: встроить в интерфейс симуляций окно агента для диалога и формирования БТ по данным симуляции.
- Что уже есть в baseline/current: список симуляций, деталка симуляции, вкладки деталки, форма редактирования, данные симуляции и существующая ролевая модель.
- Какая дельта добавляется этой feature: глобальная точка входа в окно агента, правая неблокирующая панель диалога, восстановление сессии по `session_id`, действие `Сформировать БТ`, публикация БТ, сохранение `btUrl`, аудит сценария.
- Какие slice входят в контрольный документ: `agent-entrypoint`, `dialog-session`, `bt-publication`.

## Порядок slice для контроля

1. `01 agent-entrypoint — Точка входа и контекст окна агента`
2. `02 dialog-session — Сессия диалога и работа окна агента`
3. `03 bt-publication — Действия по БТ, черновик запроса и публикация`

## STORY-SIMULATION-BT-AGENT-001 — Точка входа и контекст окна агента

Slice card: `slices/agent-entrypoint/slice.md`  
Детализация FE: `slices/agent-entrypoint/requirements/frontend.md`  
Детализация BE: `slices/agent-entrypoint/requirements/backend.md`  
Прототип: `slices/agent-entrypoint/delivery-prototype/prototype.html`  
Planning story: `planning/stories/STORY-SIMULATION-BT-AGENT-001.md`

**Бизнес-требования**

- Цель: дать пользователю единую точку входа в окно агента на страницах симуляций.
- Бизнес-правила: окно агента доступно со списка, деталки, вкладок деталки и формы редактирования; вызов открытия окна использует единый REST только с `session_id`; действие `Сформировать БТ` доступно только для конкретной успешно завершённой симуляции с опцией вывода в ПРОМ.
- Ограничения: открытие окна не должно блокировать основной интерфейс; при открытии окна клиент не передаёт историю, `contextPrompt`, ФИО пользователя и дату-время старта диалога.

**Пользовательские требования к АС КОДА**

- Пользователь может открыть окно агента с любой страницы раздела симуляций.
- Пользователь остаётся на текущей странице, а окно открывается без смены маршрута.
- На странице подходящей симуляции пользователь видит в окне агента доступное действие `Сформировать БТ`.

**Критерии приемки**

1. Окно агента открывается со списка симуляций, с деталки, с вкладок деталки и с формы редактирования.
2. При открытии или восстановлении окна в серверный вызов передаётся только `session_id`.
3. Для страниц без подходящей симуляции окно агента остаётся доступным, но действие `Сформировать БТ` недоступно.

**USE CASES**

- **осн. сценарий 1** пользователь открывает окно агента со списка симуляций.
- **альт. сценарий 1.1** пользователь открывает окно агента с деталки симуляции или с её вкладки.
- **альт. сценарий 1.2** пользователь открывает окно агента из формы редактирования симуляции.
- **осн. сценарий 2** фронт использует активный `session_id` и восстанавливает состояние окна.

| Окно агента | Отображается состояние сессии и действия по БТ в зависимости от контекста страницы |

Требования на фронт:

- добавить единую точку входа в окно агента на ключевых страницах раздела симуляций;
- открыть окно как неблокирующую панель без сброса состояния страницы;
- при открытии и восстановлении отправлять только `session_id`;
- сохранять `session_id` и дату-время старта диалога в рамках пользовательской сессии АС КОДА.

- обеспечить выдачу признаков доступности и данных для черновика запроса через существующие страницы симуляций;
- не принимать дополнительные клиентские данные в вызове открытия окна.

Связанный детальный BE pack: `slices/agent-entrypoint/requirements/backend.md`

| База данных | `baseline/current/domain/aggregates/simulation.md` |
| Методы | `slices/agent-entrypoint/requirements/backend.md` |

**Изменения в ролевой модели**

- Используется существующая ролевая модель доступа к страницам симуляций.

## STORY-SIMULATION-BT-AGENT-002 — Сессия диалога и работа окна агента

Slice card: `slices/dialog-session/slice.md`  
Детализация FE: `slices/dialog-session/requirements/frontend.md`  
Детализация BE: `slices/dialog-session/requirements/backend.md`  
Прототип: `slices/dialog-session/delivery-prototype/prototype.html`  
Planning story: `planning/stories/STORY-SIMULATION-BT-AGENT-002.md`

**Бизнес-требования**

- Цель: обеспечить устойчивый диалог с агентом внутри АС КОДА без блокировки остального интерфейса.
- Бизнес-правила: история и контекст диалога принадлежат агенту; открытие окна и отправка сообщения разделены; `session_id` и дата-время старта диалога хранятся в пользовательской сессии АС КОДА и сбрасываются вместе с ней; при отправке сообщения передаются `user_prompt`, ФИО пользователя из СУДИР и дата-время старта диалога; статус агента показывается отдельным chip в шапке окна, а при ошибке пользователь видит причину и рекомендацию перезапустить сессию.
- Ограничения: клиентская история не является источником правды; Markdown должен корректно отображаться в сообщениях; текст пользовательского сообщения не валидируется и не интерпретируется со стороны АС КОДА.

**Пользовательские требования к АС КОДА**

- Диалог открывается в неблокирующей правой панели.
- Пользователь может продолжать работу со страницей симуляции вне панели агента.
- Пользователь может копировать текст из интерфейса АС КОДА в окно агента и из окна агента обратно в интерфейс АС КОДА.
- После ошибки пользователь видит понятное сообщение о проблеме агента и рекомендацию перезапустить сессию.

**Критерии приемки**

1. Окно агента работает как правая неблокирующая панель и не мешает навигации по интерфейсу.
2. История и состояние диалога восстанавливаются по `session_id`, который живёт в рамках пользовательской сессии АС КОДА.
3. При отправке текста в агент передаются ФИО пользователя из СУДИР и дата-время старта диалога.
4. Сообщения агента отображаются с поддержкой Markdown, а ошибка позволяет перезапустить сессию.

**USE CASES**

- **осн. сценарий 1** пользователь открывает окно, получает историю и продолжает переписку.
- **альт. сценарий 1.1** пользователь переходит между страницами симуляций, не закрывая окно.
- **альт. сценарий 1.2** во время ошибки пользователь перезапускает сессию и начинает новый диалог.
- **осн. сценарий 2** пользователь отправляет сообщение и получает обновлённую историю от агента.

### Функциональные требования

#### Реализация для FRONTEND

**Описание UI**

| Основной экран симуляций | Остаётся доступным вне панели агента |

Требования на фронт:

- показывать в шапке окна chip статуса агента со значением `Онлайн` или `Ошибка`, а по клику раскрывать детали статуса и рекомендации;
- размещать область чата сразу под шапкой без дополнительных информационных карточек между шапкой и перепиской;
- поддержать локальную блокировку только элементов управления окна на время отправки;

- обеспечить копирование текста в окно агента и из окна агента;

## STORY-SIMULATION-BT-AGENT-003 — Действия по БТ, черновик запроса и публикация

Slice card: `slices/bt-publication/slice.md`  
Детализация FE: `slices/bt-publication/requirements/frontend.md`  
Детализация BE: `slices/bt-publication/requirements/backend.md`  
Прототип: `slices/bt-publication/delivery-prototype/prototype.html`  
Planning story: `planning/stories/STORY-SIMULATION-BT-AGENT-003.md`

**Бизнес-требования**

- Цель: дать пользователю управляемый сценарий формирования БТ через окно агента и сохранить итоговую ссылку на созданный БТ.
- Бизнес-правила: кнопка `Сформировать БТ` доступна только для конкретной успешно завершённой симуляции с опцией вывода в ПРОМ; клик по кнопке только вставляет черновик запроса; пользователь может произвольно редактировать текст или не отправлять его; под полем ввода расположены только основные действия `Сформировать БТ` и `Отправить`; успешная публикация сохраняет `btUrl`.
- Ограничения: черновик запроса формируется на фронте по данным страницы; автоматическая отправка текста не допускается; сервер получает только тот текст, который пользователь отправил вручную.

**Пользовательские требования к АС КОДА**

- В подходящем контексте пользователь видит в окне агента кнопку `Сформировать БТ`.
- По клику пользователь получает черновик запроса в поле ввода и может его свободно менять.
- При наведении на кнопку пользователь видит подсказку с условиями доступности или с пояснением следующего шага.
- После успешной публикации пользователь видит `btUrl`, а ссылка на БТ появляется в данных симуляции.

**Критерии приемки**

1. Кнопка `Сформировать БТ` доступна только в подходящем контексте симуляции.
2. По клику на кнопку в поле ввода вставляется черновик запроса, который не отправляется автоматически.
3. После успешной публикации `btUrl` сохраняется и отображается в интерфейсе симуляции.
4. Неуспешная публикация не стирает ранее сохранённый `btLink`.

**USE CASES**

- **осн. сценарий 1** пользователь открывает окно из подходящей симуляции и вставляет черновик запроса.
- **альт. сценарий 1.1** окно открыто из неподходящей страницы, и действие `Сформировать БТ` недоступно.
- **альт. сценарий 1.2** пользователь редактирует черновик запроса или отказывается от отправки.
- **осн. сценарий 2** после успешной публикации пользователь получает ссылку на созданный БТ.

### Функциональные требования

#### Реализация для FRONTEND

**Описание UI**

| Страница симуляции | Отображается сохранённая ссылка на БТ после успешной публикации |

Требования на фронт:

- показать кнопку `Сформировать БТ` только в подходящем контексте;
- по клику вставлять в поле ввода черновик запроса по шаблону;
- выводить у `Сформировать БТ` tooltip с пояснением о доступности или о следующем шаге после вставки черновика;
- не отправлять черновик автоматически;
- после успеха отобразить `btUrl` и обновить ссылку на БТ в данных симуляции.

Связанный детальный FE pack: `slices/bt-publication/requirements/frontend.md`

- сохранить `btUrl` у симуляции;
- аудировать успешную и неуспешную публикацию.

Связанный детальный BE pack: `slices/bt-publication/requirements/backend.md`

| Методы | `slices/bt-publication/requirements/backend.md` |

**Изменения в ролевой модели**

- Дополнительные роли не вводятся.

**BUGS (Если есть)**
