# FEATURE-SIMULATION-BT-AGENT — Формирование БТ из симуляции через AI-агента RAIN

Статус: **в проработке**  
Квартал: `2026-Q2`  
Дата обновления: `2026-04-24`

## Цель
Встроить в существующую детальную страницу завершённой симуляции сценарий формирования бизнес-требования через AI-агента RAIN с публикацией результата в Confluence и сохранением ссылки на созданный БТ в АС КОДА.

## Контекст
Feature создана по результатам intake из внешнего источника `/home/reutov/Documents/AI/simulations_AI_agent`. По уточнению пользователя существующие симуляции и деталка симуляции уже реализованы в системе, поэтому новая feature ограничивается только AI-agent дельтой поверх существующей страницы-хоста и не включает переописание базового simulation UI.

## Ideal scope
- единый AI-сценарий подготовки БТ из симуляции;
- поддержка нескольких продуктовых контуров, а не только `Кредитные карты`;
- нормализованный контракт между АС КОДА, агентом RAIN и Confluence;
- восстановление сессии, retry, аудит и операционные ограничения без расхождений между источниками;
- каноническое baseline-описание simulation detail как страницы-хоста для этого сценария.

## MVP scope
- кнопка `Сформировать БТ` на существующей странице завершённой симуляции;
- проверка наличия существующего БТ и сбор `contextPrompt` из данных симуляции;
- диалог с агентом через `POST /dialog/init` и `POST /dialog/{sessionId}/continue`;
- UI-состояния ожидания, retry, restart, последовательная обработка сообщений и ограниченное восстановление истории;
- превью БТ, явное подтверждение публикации, получение `btUrl`, сохранение ссылки и аудит;
- первая итерация только для продукта `Кредитные карты`.

## Что исключено из MVP
- реализация или переписывание существующей деталки симуляции;
- расширение сценария на другие продукты;
- переустройство simulation domain model beyond уже известных baseline инвариантов;
- расширенные simulation-related инициативы из legacy-плана: snapshots, статусная модель, trace/artifacts actualization;
- автоматическая нормализация русских названий риск-параметров и обратная запись в ФП Риск-параметры.

## Риски и зависимости
- feature зависит от уже существующей деталки симуляции как страницы-хоста и от доступности данных, из которых собирается `contextPrompt`;
- в источниках расходится TTL UI-сессии: `30 минут` против `1 час`;
- нужно подтвердить, где находится канонический источник для поля `Ссылка на БТ` и как оно связано с `Simulation`;
- доступ завязан на существующую роль `experiment_editor_CC`, но каноническая фиксация этого правила в baseline ещё не выполнена;
- межсервисные требования по HTTPS/mTLS/OTT и аудит нужно сохранить в planning-скоупе, не смешивая их с уже существующим UI симуляции.
