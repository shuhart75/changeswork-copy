## Правило использования

- Этот файл вместе с `prototype.html` описывает общий root prototype всей feature.
- Root prototype показывается как пользовательский интерфейс для конечного пользователя.
- В root prototype не добавляются комментарии для фронтендера, handoff-пояснения и технические заметки внутри HTML.
- Пока статус выше не переключён на `да`, slice delivery prototypes трогать нельзя.

## Цель

- Показать единый пользовательский сценарий feature поверх существующего simulation host UI: вход в RAIN с разных страниц симуляций, подготовка черновика запроса, продолжение диалога и публикация БТ с сохранением ссылки обратно в симуляцию.

## Какие пользовательские сценарии покрыты

- открытие окна агента со списка симуляций;
- переход в завершённую деталку симуляции и на вкладку итогов;
- видимость действия `Сформировать БТ` только в подходящем контексте;
- вставка редактируемого черновика запроса;
- ручная отправка сообщения агенту;
- публикация БТ и мгновенное появление `btUrl` в итогах симуляции;
- недоступность BT-сценария для незавершённой симуляции;
- перезапуск сессии агента и базовый показ ошибки.

## Как показываем ideal target state

- Пользователь работает в реальном shell симуляций, открывает правую неблокирующую панель RAIN из завершённой симуляции, получает готовый к публикации результат и сразу видит сохранённую ссылку на БТ в итогах.

## Как показываем MVP cut

- MVP ограничен пространством `Кредитные карты`, одним окном агента, одним `session_id` и сценарием ручного подтверждения публикации без автозапуска и без мультипродуктового расширения.

## Навигация, fake data и важные состояния

- В прототипе есть список симуляций, завершённая симуляция `SIM-CC-148`, незавершённая `SIM-CC-147`, вкладки `Основная информация` и `Итоги симуляции`, а также правая панель RAIN.
- Fake data синхронизированы со скриншотами current system и с уже подготовленными requirement packs: ID симуляций, статусы, этапы, риск-параметры, business goal и `btUrl`.
- Окно агента не закрывается при смене страницы внутри раздела и сохраняет контекст текущего маршрута.

## Что пока не фиксируем на slice-уровне

- точные тексты системных сообщений агента и их окончательную Markdown-структуру;
- конечный UX замены уже введённого текста новым черновиком `Сформировать БТ`;
- детальную механику error/retry для каждого backend-сценария;
- возможные post-MVP состояния закрепления, потокового ответа и мультипродуктового контекста.

## Открытые вопросы

- Нужна ли дополнительная визуальная индикация уже существующего `btUrl` на списке симуляций, а не только на вкладке итогов.
- Нужно ли в общем prototype отдельно показать форму редактирования как самостоятельную страницу, если текущего переключения через деталку достаточно для согласования scope.
