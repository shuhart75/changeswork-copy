## Контекст и бизнес-смысл

### Цель

Использовать уже реализованный simulation detail API как источник всех нужных данных для локальной генерации черновика запроса и при этом оставить на стороне сервера ответственность только за финальную публикацию БТ, возврат `btUrl` и аудит.

### Источник правды

Источником правды для доступности действия по БТ и исходных данных страницы является уже реализованный `GET /api/v1/simulation/{number}`. Источником правды для опубликованного результата остаются Confluence и `btUrl`, возвращённый в текущий ответ агента; автоматическое сохранение этой ссылки в `Simulation` в MVP не выполняется.

### Затронутые bounded contexts / aggregates

- `Simulation`
- `Research and Execution`
- `Identity and Access`

### Термины и определения

- `Симуляция с доступным действием по БТ` — конкретная успешно завершённая симуляция, для которой доступна опция вывода в ПРОМ.
- `Данные для черновика запроса` — поля уже реализованной деталки симуляции и профиль пользователя, которые фронт использует локально.

## Бизнес-правила и системные ограничения

### BR-1. Действие по БТ доступно только для подходящей симуляции
- страница должна быть привязана к конкретной симуляции;
- симуляция должна быть успешно завершённой;
- для симуляции должна быть доступна опция вывода в ПРОМ;
- список симуляций и другие экраны без такого контекста не становятся подходящими сами по себе.

### BR-2. Черновик запроса не создаёт побочных эффектов на стороне сервера
- клик по кнопке `Сформировать БТ` не должен сам вызывать Confluence и не должен менять состояние сессии публикации;
- черновик запроса собирается на фронте по данным уже реализованного `GET /api/v1/simulation/{number}`;
- отдельный BT-specific preparatory endpoint или новый backend-метод для сборки черновика не вводится;
- агент получает только тот текст, который пользователь реально отправил после редактирования.

### BR-3. После успешной публикации сервер возвращает `btUrl` без автосохранения в симуляцию
- успешная публикация возвращает пользователю валидный `btUrl`;
- неуспешная публикация не должна возвращать фиктивную ссылку;
- автоматическая запись `btUrl` в модель или данные `Simulation` в MVP не выполняется.

## Границы MVP

### Входит в MVP

- использование уже реализованного simulation detail API для признаков доступности и данных черновика;
- публикация в Confluence;
- возврат `btUrl` в окно агента для ручного копирования;
- аудит бизнес-событий по БТ.

### Не входит в MVP

- сборка черновика запроса на стороне сервера при открытии окна;
- отдельная BT-specific ручка или специальное расширение существующей деталки симуляции, если текущий реализованный контракт уже достаточен;
- автоматическая отправка черновика запроса;
- автоматическая запись `btUrl` в данные симуляции;
- история всех публикаций как отдельный пользовательский модуль.

### Отложено после MVP

- несколько шаблонов черновика запроса;
- оценка качества отредактированного текста на стороне сервера;
- более богатая документная модель БТ.

## Пользовательские и системные сценарии

### Сценарий BE-1. Страница конкретной симуляции становится подходящей для действия по БТ
1. Уже реализованный `GET /api/v1/simulation/{number}` возвращает поля страницы симуляции, достаточные для вычисления доступности BT-сценария.
2. Фронт вычисляет признаки доступности.
3. В окне агента появляется действие по БТ.

### Сценарий BE-2. Публикация после ручной отправки черновика запроса
1. Пользователь вручную отправляет агенту черновик запроса или его отредактированную версию.
2. После последующих шагов агент создаёт страницу в Confluence.
3. Backend возвращает `btUrl` пользователю без автоматического сохранения ссылки у симуляции.

## Функциональные требования

### BE-FR-1. BT-сценарий использует признаки доступности из existing detail API

**Описание:**  
Существующая деталка симуляции должна давать фронту возможность определить, что текущая страница относится к подходящей симуляции, без новой BT-specific ручки.

**Правила и ограничения:**
- должен быть доступен статус симуляции;
- должен быть доступен признак или эквивалент уже реализованных полей, позволяющий определить доступность вывода в ПРОМ;
- должна быть однозначная привязка к конкретной симуляции;
- для списка без конкретной симуляции признаки доступности отсутствуют.

**Зависимости:**
- существующие API списка, деталки и редактирования симуляций;
- slice `agent-entrypoint`.

### BE-FR-2. Existing detail API уже отдаёт данные для черновика запроса

**Описание:**  
Уже реализованный `GET /api/v1/simulation/{number}` должен отдавать фронту данные симуляции, достаточные для локальной подстановки в черновик запроса.

**Правила и ограничения:**
- новые BT-specific поля и отдельный preparatory endpoint не добавляются, если текущий контракт уже содержит нужные данные;
- для релевантных записей деталки должны быть доступны риск-параметры и значения, которые используются во фронтовом шаблоне;
- текстовые поля страницы используются во фронтовом шаблоне только если они уже присутствуют в реализованной деталке;
- черновик запроса не формируется на сервере при открытии окна;
- имя пользователя может приходить из отдельного контекста аутентификации и профиля, а не из API симуляций.

**Зависимости:**
- существующие данные страницы симуляции;
- шаблон черновика запроса на фронтенде.

### BE-FR-3. Публикация и возврат `btUrl`

**Описание:**  
После реальной отправки текста пользователем и успешной работы агента сервер должен вернуть `btUrl` в ответ окна агента.

**Правила и ограничения:**
- сервер не отличает черновик запроса от любого другого пользовательского текста до тех пор, пока пользователь его не отправил;
- после успеха `btUrl` возвращается в ответ пользователю;
- backend не выполняет автоматическую запись `btUrl` в данные симуляции в рамках MVP;
- при ошибке публикации `btUrl` не должен подменяться фиктивным значением;
- аудит обязателен для успешной и неуспешной публикации.

**Зависимости:**
- публикация на стороне агента;
- интеграция с Confluence.

## Модель данных

### Основные сущности и поля

| Сущность / таблица | Поле | Тип | Обязательность | Описание |
|---|---|---|---|---|
| `Simulation` view | `status` | enum/string | условно обязательно | признак завершённой симуляции для действия по БТ |
| `Simulation` detail response | `availableActions` / эквивалент | array | условно обязательно | уже реализованный набор доступных действий страницы |
| `Simulation` detail response | `riskParameters` / `riskParamsChange` | object/array | условно обязательно | данные для черновика запроса |
| `Simulation` detail response | `releaseDescription` / `releaseEffects` / эквивалент | string/null | опционально | текстовые поля страницы для черновика |
| `Simulation` detail response | `btLink` | string/null | опционально | текущее поле ссылки на БТ на странице, если оно уже существует |

### Инварианты и ограничения

- действие по БТ не активируется без подходящей симуляции;
- отсутствие части текстовых полей страницы не запрещает вставку черновика запроса;
- автоматическая запись `btUrl` в `Simulation` не выполняется в рамках MVP.

### Индексы / уникальности / FK

- отдельные новые индексы для данных черновика запроса в MVP не требуются;
- отдельная модель хранения `btUrl` для автосохранения в симуляцию в MVP не требуется.

## API-контракт

### Эндпоинты

| Метод и маршрут | Назначение | Кто вызывает | Примечание |
|---|---|---|---|
| `GET /api/v1/simulation/{number}` / existing detail API | вернуть признаки доступности и данные для черновика запроса | фронтенд интерфейса симуляций | уже реализованный источник данных для действия по БТ; новый backend-метод не требуется |
| `POST /dialog/{session_id}/message` | получить от пользователя уже отредактированный текст, ФИО из СУДИР и дату-время старта диалога | окно агента на фронтенде | черновик уходит только после ручной отправки |

### OpenAPI fragment

```yaml
openapi: 3.0.3
info:
  title: API доступности БТ и публикации
  version: 1.0.0
paths:
  /api/v1/simulation/{number}:
    get:
      summary: Существующая деталка симуляции с данными для действия по БТ
      responses:
        "200":
          description: Деталка симуляции
          content:
            application/json:
              schema:
                type: object
                properties:
                  status:
                    type: string
                  availableActions:
                    type: array
                    items:
                      type: string
                  releaseDescription:
                    type: string
                    nullable: true
                  releaseEffects:
                    type: string
                    nullable: true
                  btLink:
                    type: string
                    nullable: true
                  riskParameters:
                    type: object
                    properties:
                      riskParameters:
                        type: array
                        items:
                          type: object
                          properties:
                            name:
                              type: string
                            userLowerValue: {}
                            userHigherValue: {}
  /dialog/{session_id}/message:
    post:
      summary: Отправить пользовательский текст для формирования БТ
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              required: [user_prompt, user_full_name, dialog_started_at]
              properties:
                user_prompt:
                  type: string
                user_full_name:
                  type: string
                dialog_started_at:
                  type: string
                  format: date-time
components: {}
```

## Интеграции, вычисления и фоновые процессы

- Внешние системы: Confluence, AI-агент RAIN;
- Асинхронные процессы: отдельно для формирования черновика запроса не требуются;
- Вычисление статусов / derived fields: доступность BT-сценария может вычисляться на фронте по уже реализованному ответу деталки симуляции;
- Идемпотентность / ретраи: повторная ручная отправка текста считается новым пользовательским действием.

## Ошибки и валидация

### Валидационные правила

- действие по БТ недоступно без подходящей симуляции;
- данные existing detail API должны оставаться достаточными для формирования черновика запроса без отдельной BT-specific ручки;
- отсутствие части текстовых полей страницы не запрещает ручное редактирование черновика пользователем;
- `dialog_started_at` должен соответствовать моменту первого открытия окна, а не моменту конкретной отправки;
- `btUrl`, если возвращён, должен быть валидным URI.

### Ошибки API

| Код/сценарий | Условие | Ответ |
|---|---|---|
| `400 bt_action_not_available` | попытка выполнить действие по БТ без подходящего контекста симуляции | ошибка бизнес-правила |
| `502 confluence_unavailable` | Confluence недоступен | ошибка публикации без `btUrl` |

## Миграция и обратная совместимость

- Нужны ли миграции данных: нет, потому что автоматическое хранение `btUrl` в рамках feature не добавляется;
- Нужен ли backfill: нет;
- Есть ли риски для текущего baseline: да, запуск БТ выполняется через черновик запроса, сформированный по данным страницы;
- Что должно попасть в release finalization: фиксация existing detail API как источника данных для BT-сценария, контракт возврата `btUrl` и явное отсутствие автосохранения в `Simulation` в MVP.

## Observability и аудит

- Логи: проверка доступности, результат публикации, `btUrl`;
- Метрики: количество вставок черновика запроса, публикаций, ошибок публикации;
- Audit trail: успешная и неуспешная публикация БТ без отдельного события автосохранения ссылки у симуляции.
