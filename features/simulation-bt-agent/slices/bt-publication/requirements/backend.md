## Контекст и бизнес-смысл

### Цель

Дать фронту все нужные данные для локальной генерации черновика запроса и при этом оставить на стороне сервера ответственность за финальную публикацию БТ, сохранение `btUrl` и аудит.

### Источник правды

Источником правды для доступности действия по БТ являются данные текущей страницы симуляции. Источником правды для опубликованного результата остаются Confluence и сохранённый `btUrl` у симуляции.

### Затронутые bounded contexts / aggregates

- `Simulation`
- `Research and Execution`
- `Identity and Access`

### Термины и определения

- `Симуляция с доступным действием по БТ` — конкретная успешно завершённая симуляция, для которой доступна опция вывода в ПРОМ.
- `Данные для черновика запроса` — значения `USER_NAME`, `RP_NAME`, `RP_TOBE_VAL`, `BUSINESS_GOAL`, которые фронт использует локально.

## Бизнес-правила и системные ограничения

### BR-1. Действие по БТ доступно только для подходящей симуляции
- страница должна быть привязана к конкретной симуляции;
- симуляция должна быть успешно завершённой;
- для симуляции должна быть доступна опция вывода в ПРОМ;
- список симуляций и другие экраны без такого контекста не становятся подходящими сами по себе.

### BR-2. Черновик запроса не создаёт побочных эффектов на стороне сервера
- клик по кнопке `Сформировать БТ` не должен сам вызывать Confluence и не должен менять состояние сессии публикации;
- черновик запроса собирается на фронте по данным текущей страницы;
- агент получает только тот текст, который пользователь реально отправил после редактирования.

### BR-3. Последний успешный `btUrl` остаётся текущей ссылкой у симуляции
- успешная публикация обновляет текущее значение ссылки;
- неуспешная публикация не стирает существующее значение;
- повторная успешная публикация может заменить текущую ссылку на новую.

## Границы MVP

### Входит в MVP

- признаки доступности для действия по БТ;
- контракт данных для черновика запроса;
- публикация в Confluence;
- сохранение `btUrl` в АС КОДА;
- аудит бизнес-событий по БТ.

### Не входит в MVP

- сборка черновика запроса на стороне сервера при открытии окна;
- автоматическая отправка черновика запроса;
- история всех публикаций как отдельный пользовательский модуль.

### Отложено после MVP

- несколько шаблонов черновика запроса;
- оценка качества отредактированного текста на стороне сервера;
- более богатая документная модель БТ.

## Пользовательские и системные сценарии

### Сценарий BE-1. Страница конкретной симуляции становится подходящей для действия по БТ
1. Существующие данные страницы симуляции возвращают статус симуляции и признак доступности вывода в ПРОМ.
2. Фронт вычисляет признаки доступности.
3. В окне агента появляется действие по БТ.

### Сценарий BE-2. Публикация после ручной отправки черновика запроса
1. Пользователь вручную отправляет агенту черновик запроса или его отредактированную версию.
2. После последующих шагов агент создаёт страницу в Confluence.
3. Backend сохраняет `btUrl` и возвращает результат пользователю.

## Функциональные требования

### BE-FR-1. Backend должен отдавать признаки доступности действия по БТ

**Описание:**  
Существующие страницы симуляций должны давать фронту возможность определить, что текущая страница относится к подходящей симуляции.

**Правила и ограничения:**
- должен быть доступен статус симуляции;
- должен быть доступен признак или эквивалент доступности вывода в ПРОМ;
- должна быть однозначная привязка к конкретной симуляции;
- для списка без конкретной симуляции признаки доступности отсутствуют.

**Зависимости:**
- существующие API списка, деталки и редактирования симуляций;
- slice `agent-entrypoint`.

### BE-FR-2. Backend должен отдавать данные для черновика запроса

**Описание:**  
Сервер должен отдавать фронту данные симуляции, достаточные для локальной подстановки в черновик запроса.

**Правила и ограничения:**
- для каждой релевантной записи риск-параметра должен быть доступен `name` и целевое значение `TO BE`;
- `BUSINESS_GOAL` передаётся только если он существует в доступном интерфейсе;
- черновик запроса не формируется на сервере при открытии окна;
- имя пользователя может приходить из отдельного контекста аутентификации и профиля, а не из API симуляций.

**Зависимости:**
- существующие данные страницы симуляции;
- шаблон черновика запроса на фронтенде.

### BE-FR-3. Публикация и сохранение `btUrl`

**Описание:**  
После реальной отправки текста пользователем и успешной работы агента сервер должен сохранить и вернуть `btUrl`.

**Правила и ограничения:**
- сервер не отличает черновик запроса от любого другого пользовательского текста до тех пор, пока пользователь его не отправил;
- после успеха `btUrl` сохраняется у симуляции;
- при ошибке публикации `btUrl` не должен подменяться фиктивным значением;
- аудит обязателен для успешной и неуспешной публикации.

**Зависимости:**
- публикация на стороне агента;
- интеграция с Confluence.

## Модель данных

### Основные сущности и поля

| Сущность / таблица | Поле | Тип | Обязательность | Описание |
|---|---|---|---|---|
| `Simulation` view | `status` | enum/string | условно обязательно | признак завершённой симуляции для действия по БТ |
| `Simulation` view | `can_promote_to_prod` / эквивалент | boolean/derived | условно обязательно | признак доступности вывода в ПРОМ |
| `Simulation` view | `risk_parameters[*].name` | string | условно обязательно | данные для черновика запроса |
| `Simulation` view | `risk_parameters[*].to_be_value` | scalar | условно обязательно | данные для черновика запроса |
| `Simulation` view | `business_goal` / эквивалент | string/null | опционально | данные для черновика запроса |
| `simulation_result` / эквивалент | `bt_link` | string/uri | опционально | текущая ссылка на БТ |

### Инварианты и ограничения

- действие по БТ не активируется без подходящей симуляции;
- отсутствие `business_goal` не запрещает вставку черновика запроса;
- `bt_link` обновляется только после успешной публикации.

### Индексы / уникальности / FK

- отдельные новые индексы для данных черновика запроса в MVP не требуются;
- модель хранения `bt_link` должна быть подтверждена к release finalization.

## API-контракт

### Эндпоинты

| Метод и маршрут | Назначение | Кто вызывает | Примечание |
|---|---|---|---|
| `GET /api/v1/simulation/{number}` / existing detail API | вернуть признаки доступности и данные для черновика запроса | фронтенд интерфейса симуляций | источник данных для действия по БТ |
| `POST /dialog/{session_id}/message` | получить от пользователя уже отредактированный текст, ФИО из СУДИР и дату-время старта диалога | окно агента на фронтенде | черновик уходит только после ручной отправки |
| `GET /api/v1/simulation/{number}` / запрос повторного чтения данных | вернуть обновлённый `btLink` | фронтенд интерфейса симуляций | после успешной публикации |

### OpenAPI fragment

```yaml
openapi: 3.0.3
info:
  title: API доступности БТ и публикации
  version: 1.0.0
paths:
  /api/v1/simulation/{number}:
    get:
      summary: Существующая деталка симуляции с данными для действия по БТ
      responses:
        "200":
          description: Деталка симуляции
          content:
            application/json:
              schema:
                type: object
                properties:
                  status:
                    type: string
                  canPromoteToProd:
                    type: boolean
                  businessGoal:
                    type: string
                    nullable: true
                  btLink:
                    type: string
                    nullable: true
                    format: uri
                  riskParameters:
                    type: array
                    items:
                      type: object
                      properties:
                        name:
                          type: string
                        toBeValue: {}
  /dialog/{session_id}/message:
    post:
      summary: Отправить пользовательский текст для формирования БТ
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              required: [user_prompt, user_full_name, dialog_started_at]
              properties:
                user_prompt:
                  type: string
                user_full_name:
                  type: string
                dialog_started_at:
                  type: string
                  format: date-time
components: {}
```

## Интеграции, вычисления и фоновые процессы

- Внешние системы: Confluence, AI-агент RAIN;
- Асинхронные процессы: отдельно для формирования черновика запроса не требуются;
- Вычисление статусов / derived fields: `can_promote_to_prod` может быть вычисляемым признаком существующего сервиса симуляций;
- Идемпотентность / ретраи: повторная ручная отправка текста считается новым пользовательским действием.

## Ошибки и валидация

### Валидационные правила

- действие по БТ недоступно без подходящей симуляции;
- `riskParameters[*].toBeValue` должны быть доступны для формирования черновика запроса;
- пустой `businessGoal` допустим;
- `dialog_started_at` должен соответствовать моменту первого открытия окна, а не моменту конкретной отправки;
- `btUrl`, если возвращён, должен быть валидным URI.

### Ошибки API

| Код/сценарий | Условие | Ответ |
|---|---|---|
| `400 bt_action_not_available` | попытка выполнить действие по БТ без подходящего контекста симуляции | ошибка бизнес-правила |
| `502 confluence_unavailable` | Confluence недоступен | ошибка публикации без `btUrl` |
| `500 bt_link_persist_failed` | `btUrl` получен, но не сохранён у симуляции | ошибка частичного успеха |

## Миграция и обратная совместимость

- Нужны ли миграции данных: возможно, если `bt_link` ещё не хранится канонически;
- Нужен ли backfill: нет;
- Есть ли риски для текущего baseline: да, запуск БТ выполняется через черновик запроса, сформированный по данным страницы;
- Что должно попасть в release finalization: признаки доступности, данные для черновика запроса, модель хранения `btLink`.

## Observability и аудит

- Логи: проверка доступности, результат публикации, `btUrl`, статус сохранения;
- Метрики: количество вставок черновика запроса, публикаций, ошибок сохранения ссылки;
- Audit trail: успешная и неуспешная публикация БТ, обновление ссылки у симуляции.

## Критерии приемки

### BE-AC-1. Признаки доступности и данные для черновика запроса
- [ ] Backend существующих страниц симуляций отдаёт признаки доступности для действия по БТ
- [ ] Backend существующих страниц симуляций отдаёт данные для формирования черновика запроса
- [ ] Для страниц без доступного сценария окно агента остаётся доступным без действия по БТ

### BE-AC-2. Результат публикации
- [ ] Успешный `btUrl` сохраняется как текущая ссылка у симуляции
- [ ] Неуспешная публикация не стирает существующий `btLink`
- [ ] При ручной отправке в агент уходят ФИО пользователя из СУДИР и дата-время старта диалога
- [ ] Аудит фиксирует успешную и неуспешную публикацию

## Открытые вопросы и допущения

- В требованиях используется условное поле `canPromoteToProd`; в реализации оно может быть выражено через существующие `available_actions` или другой эквивалентный признак.
- Черновик запроса может ссылаться на `businessGoal`, даже если это поле вычисляется не прямо из агрегата симуляции.
