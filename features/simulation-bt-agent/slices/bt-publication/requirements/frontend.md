## Контекст и бизнес-смысл

### Цель

Позволить пользователю быстро инициировать БТ-сценарий из подходящей страницы симуляции, оставить человеку полный контроль над отправляемым текстом и показать результат агента без автосохранения ссылки в симуляцию.

### Пользователи и роли

- Пользователь интерфейса симуляций, имеющий доступ к завершённой симуляции и действию вывода в ПРОМ.

### Бизнес-контекст

OpenAPI RAIN поддерживает optional `simulation_id` и `risk_params`, а результат успешного run отдаёт через `GET /chat/runs/{run_id}/result`: текст ответа находится в `message.content`, а ссылки и другие результаты публикации приходят в `artifacts[]`. Поэтому frontend показывает текст ответа и, если backend передал artifact типа `bt_page`, отображает его URL как структурированную ссылку для копирования.

### Термины и определения

- `Inline action по БТ` — подсказка и компактная кнопка над composer divider: `Сформировать БТ для текущей симуляции`.
- `Черновик запроса` — текст, который frontend вставляет в composer по данным текущей симуляции.
- `BT artifact` — элемент `artifacts[]` результата RAIN run с `type=bt_page`, `url` и optional `title`.

## Границы MVP

### Входит в MVP

- условный показ inline action по БТ в подходящем контексте;
- вставка редактируемого черновика запроса в composer;
- ручная отправка текста через общий async run;
- передача связанного контекста симуляции в backend для server-to-server вызова RAIN;
- отображение ответа агента и URL из `artifacts[]`;
- ручное копирование URL пользователем;
- отсутствие автоматического обновления `btLink`/`Ссылка на БТ` в данных симуляции.

### Не входит в MVP

- автоматическая отправка черновика;
- автоматическое сохранение ссылки на БТ в симуляцию;
- отдельное поле `btUrl` в frontend API;
- клиентский разбор JSON/технических структур от агента;
- автоматический retry публикации БТ при timeout.

### Отложено после MVP

- автоматическое сохранение `bt_page.url` в данные симуляции;
- идемпотентная публикация БТ;
- история всех созданных БТ по одной симуляции;
- автосохранение выбранной ссылки в симуляцию.

## Пользовательские сценарии

### Сценарий FE-1. Inline action доступен в подходящем контексте
1. Пользователь находится на странице конкретной успешно завершённой симуляции.
2. Agent status `ready`, active run отсутствует.
3. В окне агента над composer divider отображается inline action `Сформировать БТ для текущей симуляции`.

### Сценарий FE-2. Вставка и редактирование черновика
1. Пользователь нажимает inline action.
2. Frontend собирает человекочитаемый черновик по данным страницы и профиля пользователя.
3. Черновик вставляется в composer.
4. Пользователь свободно редактирует текст или не отправляет его.

### Сценарий FE-3. Отправка БТ-контекста
1. Пользователь отправляет черновик или изменённый текст.
2. Frontend передаёт backend текст и идентификатор текущей симуляции/режим БТ.
3. Дальше работает общий async run и polling из slice `dialog-session`.

### Сценарий FE-4. Отображение результата
1. После terminal status `succeeded` frontend обновляет историю.
2. Ответ RAIN из `message.content` отображается как Markdown/текст.
3. Если backend передал artifact `bt_page`, frontend показывает `artifact.url` как ссылку и даёт скопировать.
4. Данные симуляции не обновляются автоматически.

## UI-состав и навигация

### Экран/состояние 1. Inline action по БТ

- Назначение: предложить быстрый старт БТ-сценария.
- Откуда открывается: в уже открытом окне агента, если текущая страница подходит.
- Куда ведёт: к вставке черновика в composer.
- Что видно пользователю: подсказка `Сформировать БТ для текущей симуляции` и компактная кнопка/иконка.
- Какие действия доступны: вставить черновик, затем редактировать и отправить вручную.

### Экран/состояние 2. Ответ агента со ссылкой

- Назначение: показать результат публикации или дальнейшие вопросы агента.
- Откуда открывается: после успешного run.
- Куда ведёт: к ручному копированию ссылки или продолжению диалога.
- Что видно пользователю: текст ответа, Markdown, URL как кликабельная ссылка.
- Какие действия доступны: копировать текст/URL, продолжить диалог, вручную перенести ссылку в симуляцию.

## Функциональные требования

### FE-FR-1. Inline action по БТ показывается только в подходящем контексте

**Описание:**
Action по БТ должен отображаться только если текущая страница относится к конкретной завершённой симуляции, для которой доступно действие вывода в ПРОМ, агент готов и нет active run.

**Детали:**
- action не отображается на списке симуляций без конкретной симуляции;
- action не отображается для незавершённой симуляции;
- action блокируется при `agent_status != ready`;
- action блокируется при active run;
- tooltip объясняет доступность или причину недоступности.

**Зависимости:**
- existing simulation detail API;
- status chip;
- active run state.

### FE-FR-2. Черновик запроса формируется как человекочитаемый текст

**Описание:**
Нажатие inline action должно только вставить редактируемый черновик в composer.

**Детали:**
- черновик содержит ФИО, дату старта диалога, ID симуляции и риск-параметры AS IS/TO BE, если они доступны;
- черновик не отображается как JSON;
- пользователь может изменить любой текст перед отправкой;
- вставка черновика не создаёт run и не вызывает RAIN;
- длина вставленного черновика проверяется теми же лимитами, что и обычный prompt.

**Зависимости:**
- данные текущей симуляции;
- профиль пользователя;
- лимит prompt из `dialog-session`.

### FE-FR-3. Отправка БТ-контекста использует общий async run

**Описание:**
После ручной отправки БТ-текста frontend должен использовать общий endpoint отправки сообщения и polling.

**Детали:**
- frontend передаёт backend `message`;
- если текст связан с БТ, frontend передаёт `simulation_id` и `mode=bt_creation`;
- frontend не передаёт OTT и не вызывает RAIN;
- composer блокируется до terminal status;
- автоматический retry пользователю не предлагается как скрытое действие, если run мог создать БТ.

**Зависимости:**
- `dialog-session` async run API;
- backend проверка симуляции и сбор `risk_params`.

### FE-FR-4. Ответ RAIN отображается как `message.content`, URL БТ берётся из `artifacts[]`

**Описание:**
Frontend должен отображать текст ответа агента из истории и использовать структурированные `artifacts[]`, если backend передал результат RAIN run после `GET /chat/runs/{run_id}/result`.

**Детали:**
- весь ответ агента показывается в истории сообщений;
- URL в тексте могут подсвечиваться как обычные ссылки, но не считаются контрактным результатом публикации БТ;
- если artifact `type=bt_page` содержит `url`, UI показывает действие `Копировать ссылку`;
- если artifact отсутствует, UI не должен показывать фиктивную ссылку;
- автоматическое сохранение ссылки в симуляцию не выполняется.

**Зависимости:**
- backend history;
- backend result/artifacts projection;
- Markdown/linkify renderer.

## Интеграция с Backend API

| Метод и маршрут | Где используется | Что отправляем/читаем | Условия вызова | Примечание |
|---|---|---|---|---|
| `GET /api/v1/simulation/{number}` / existing detail API | чтение данных текущей симуляции для доступности action и шаблона | читаем статус, доступные действия, риск-параметры | при открытой подходящей странице симуляции | уже реализованный источник данных |
| `POST /dialog/message` | ручная отправка черновика или изменённого текста | отправляем `session_id`, `message`, `simulation_id`, `mode=bt_creation` | после ручной отправки | возвращает `session_id`, `can_send_message`, `dialog_status`; не полный ответ RAIN |
| `GET /dialog/status?session_id=...&known_sync_cursor=...` | ожидание результата | читаем `can_send_message`, `dialog_status`, optional `history_changed`, ошибку | пока run не terminal | общий polling по session_id, без `run_id` во frontend |
| `GET /dialog/messages?session_id=...` | отображение ответа | читаем страницу сообщений агента и `sync_cursor` | после terminal status или открытия истории | текст ответа приходит как `content`; структурированные artifacts backend может отобразить рядом с сообщением |

## Валидация на frontend

### Правила

- action по БТ доступен только при подходящем контексте, `agent_status=ready` и отсутствии active run;
- черновик запроса не уходит в агент автоматически;
- пользователь может редактировать черновик как угодно;
- пользователь может не отправлять вставленный текст;
- prompt не должен превышать configured limit;
- успешная ссылка показывается только если backend передал artifact `bt_page.url` или явный URL присутствует в тексте сообщения;
- автоматическое обновление ссылки на БТ в данных симуляции не выполняется.

### Сообщения об ошибках

| Ситуация | Сообщение | Где показываем |
|---|---|---|
| BT-сценарий недоступен | `Сформировать БТ можно только для завершённой симуляции, доступной к выводу в ПРОМ.` | tooltip/action |
| Агент не готов | `Агент сейчас не готов принимать сообщения.` | action/composer |
| Не удалось построить черновик | `Не удалось подготовить черновик запроса для БТ.` | composer |
| Черновик слишком длинный | `Черновик слишком длинный. Сократите текст перед отправкой.` | composer |
| Ошибка публикации или ответа агента | `Не удалось получить результат от агента. Попробуйте позже.` | история/статус run |

## Нефункциональные требования к UI

- accessibility / keyboard / tab order: inline action, tooltip и редактируемый черновик доступны с клавиатуры;
- responsive behavior: action не должен занимать место как отдельная большая кнопка и не должен ломать composer;
- performance expectations: вставка черновика локальная, без обращения к RAIN;
- audit / trace / telemetry if needed: можно логировать вставку черновика и ручную отправку БТ-сценария.

- Отдельное поле `btUrl` больше не требуется: согласованный RAIN contract возвращает результат публикации через `artifacts[]`.
- Нужно подтвердить финальный шаблон черновика и лимит prompt с учётом реального количества риск-параметров.
