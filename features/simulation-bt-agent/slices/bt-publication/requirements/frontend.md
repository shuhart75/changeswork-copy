## Контекст и бизнес-смысл

### Цель

Позволить пользователю быстро инициировать БТ-сценарий из подходящей страницы симуляции, но при этом оставить человеку полный контроль над тем, какой текст реально уйдёт агенту.

### Пользователи и роли

- Пользователь, работающий с конкретной завершённой симуляцией, может воспользоваться действием по БТ только если для этой симуляции доступна опция вывода в ПРОМ.

### Бизнес-контекст

Окно агента доступно на всех страницах симуляций, но кнопка `Сформировать БТ` появляется не всегда. Она доступна только внутри окна и только при наличии подходящего контекста текущей страницы. Нажатие на кнопку не запускает агент автоматически, а лишь вставляет в поле ввода черновик запроса.

### Термины и определения

- `Действие по БТ` — кнопка `Сформировать БТ` внутри окна агента.
- `Черновик запроса` — текст, который фронт вставляет в поле ввода на основании данных текущей симуляции.
- `Редактируемый черновик` — вставленный текст, который пользователь может произвольно менять или удалить.

## Границы MVP

### Входит в MVP

- условный показ кнопки `Сформировать БТ` внутри окна агента;
- tooltip-подсказка у `Сформировать БТ`, различающая недоступный и доступный сценарии;
- вставка черновика запроса по данным текущей симуляции;
- свободное редактирование вставленного текста пользователем;
- ручная отправка или отказ от отправки;
- показ `btUrl` и обновление ссылки на текущей симуляции после успешной публикации.

### Не входит в MVP

- автоматическая отправка текста после клика по кнопке;
- жёсткая форма вместо свободного текста;
- запрет пользователю переписывать вставленный текст.

### Отложено после MVP

- несколько шаблонов черновика запроса;
- сравнение между исходным черновиком и финально отправленным текстом;
- более сложные пошаговые сценарии формирования БТ.

## Пользовательские сценарии

### Сценарий FE-1. Кнопка БТ доступна в подходящем контексте
1. Пользователь открывает окно агента со страницы конкретной успешно завершённой симуляции.
2. Для этой симуляции доступна опция вывода в ПРОМ.
3. Внутри окна появляется кнопка `Сформировать БТ`, а при наведении пользователь получает поясняющую подсказку.

### Сценарий FE-2. Вставка и редактирование черновика запроса
1. Пользователь нажимает `Сформировать БТ`.
2. Фронт вставляет в поле ввода черновик запроса по шаблону.
3. Пользователь свободно редактирует этот текст, отправляет его агенту или вообще не отправляет.

### Сценарий FE-3. Успешная публикация
1. После дальнейшего взаимодействия агент возвращает `btUrl`.
2. Окно показывает ссылку.
3. Страница симуляции обновляет поле `Ссылка на БТ`.

## UI-состав и навигация

### Экран/состояние 1. Действие по БТ внутри окна

- Назначение: предложить пользователю быстрый старт БТ-сценария.
- Откуда открывается: в уже открытом окне агента.
- Куда ведёт: вставляет черновик запроса в поле ввода окна.
- Что видно пользователю: кнопка `Сформировать БТ` рядом с кнопкой `Отправить`; у кнопки есть tooltip с пояснением статуса.
- Какие действия доступны: вставить черновик запроса и понять по tooltip, почему действие доступно или недоступно.

### Экран/состояние 2. Редактируемый черновик запроса

- Назначение: дать пользователю редактируемый текст до отправки агенту.
- Откуда открывается: после клика `Сформировать БТ`.
- Куда ведёт: к ручной отправке текста в агент или к отказу от этого действия.
- Что видно пользователю: заполненное поле ввода с черновиком запроса; под полем остаются только кнопки `Сформировать БТ` и `Отправить`.
- Какие действия доступны: редактировать, отправить, не отправлять.

### Экран/состояние 3. Успешный результат публикации

- Назначение: показать, что БТ создан и доступен по ссылке.
- Откуда открывается: после успешного ответа агента с `btUrl`.
- Куда ведёт: переход по ссылке или закрытие окна.
- Что видно пользователю: `btUrl` и обновлённое состояние текущей симуляции.
- Какие действия доступны: открыть ссылку, продолжить работу, закрыть окно.

## Функциональные требования

### FE-FR-1. Доступность кнопки `Сформировать БТ`

**Описание:**  
Кнопка `Сформировать БТ` должна отображаться внутри окна агента только при подходящем контексте текущей страницы.

**Детали:**
- кнопка доступна, если окно открыто со страницы конкретной симуляции;
- эта симуляция должна быть успешно завершённой;
- для неё должна быть доступна опция вывода в ПРОМ;
- если окно открыто со списка, из общего контекста или с неподходящей симуляции, кнопка недоступна;
- логика доступности должна работать одинаково на любой вкладке деталки этой симуляции и в других её состояниях, если они относятся к той же симуляции.
- если кнопка недоступна, tooltip объясняет, что сценарий доступен только на завершённых деталках симуляций с возможностью вывода в ПРОМ.

**Зависимости:**
- slice `agent-entrypoint`;
- существующий контекст страницы симуляции.

### FE-FR-2. Клик по кнопке вставляет черновик запроса и не отправляет его автоматически

**Описание:**  
Нажатие `Сформировать БТ` только подготавливает текст запроса, но не запускает отправку агенту.

**Детали:**
- в поле ввода вставляется текстовый черновик;
- вставка не должна автоматически инициировать запрос в сеть;
- если в поле ввода уже есть текст, способ вставки должен быть явным: либо заменить поле целиком, либо запросить подтверждение на замену;
- после вставки поле остаётся полностью редактируемым.
- если кнопка доступна, tooltip объясняет, что по нажатию будет вставлен черновик запроса, после чего пользователь сможет его отредактировать и отправить вручную.

**Зависимости:**
- поле ввода окна агента;
- шаблон черновика запроса.

### FE-FR-3. Формат черновика запроса

**Описание:**  
Фронт должен вставлять черновик запроса по фиксированному шаблону на основании текущей симуляции и имени пользователя.

**Детали:**
- базовый шаблон:
  `Меня зовут {USER_NAME}. Поменяй значение риск-параметра {RP_NAME1} на {RP_TOBE_VAL1} и {RP_NAME2} на {RP_TOBE_VAL2}.`
  `Бизнес-цель -- {BUSINESS_GOAL}`
- если риск-параметров больше двух, фронт перечисляет все пары в той же фразе;
- если `BUSINESS_GOAL` отсутствует в контексте текущей страницы, фронт вставляет пустой хвост `Бизнес-цель -- ` и пользователь может дописать его вручную;
- имя пользователя подставляется из текущего профиля пользователя;
- черновик запроса всегда остаётся обычным редактируемым текстом, а не блоком только для чтения.

**Зависимости:**
- источник данных из slice `agent-entrypoint/backend.md`;
- текущий контекст профиля пользователя.

### FE-FR-4. Отражение успешного `btUrl`

**Описание:**  
После успешной публикации фронт должен отразить `btUrl` и обновить ссылку у текущей симуляции.

**Детали:**
- `btUrl` показывается как кликабельная ссылка в окне;
- страница текущей симуляции обновляет поле `Ссылка на БТ`;
- если пользователь ранее редактировал черновик запроса, интерфейс не обязан показывать сравнение версий, только результат;
- ошибка публикации не должна приводить к ложному обновлению поля `Ссылка на БТ`.

**Зависимости:**
- backend публикации;
- существующее обновление страницы симуляции.

## Валидация на frontend

### Правила

- кнопка `Сформировать БТ` доступна только при подходящем контексте;
- tooltip у `Сформировать БТ` должен объяснять либо причину недоступности, либо следующий шаг после вставки черновика;
- черновик запроса не уходит в агент автоматически;
- пользователь может редактировать черновик запроса как угодно;
- пользователь может вообще не отправлять вставленный текст;
- при ручной отправке вместе с текстом уходят ФИО пользователя из СУДИР и дата-время старта диалога;
- успешный `btUrl` отражается только после фактического ответа агента.

### Сообщения об ошибках

| Ситуация | Сообщение | Где показываем |
|---|---|---|
| Для текущей страницы BT-сценарий недоступен | `Сформировать БТ можно только для завершённой симуляции, доступной к выводу в ПРОМ.` | внутри окна рядом с кнопкой действия по БТ |
| Не удалось построить черновик запроса | `Не удалось подготовить черновик запроса для БТ.` | внутри окна |
| Ошибка публикации | `Не удалось создать БТ. Попробуйте повторить позже.` | внутри окна |

## Нефункциональные требования к UI

- accessibility / keyboard / tab order: кнопка `Сформировать БТ`, её tooltip и редактируемый черновик доступны с клавиатуры;
- responsive behavior: черновик запроса и результаты публикации должны оставаться читаемыми в компактном окне;
- performance expectations: вставка черновика запроса выполняется мгновенно и локально;
- audit / trace / telemetry if needed: можно логировать факт вставки черновика запроса и факт ручной отправки пользователем.

## Критерии приемки

### FE-AC-1. Доступность действия по БТ
- [ ] Кнопка `Сформировать БТ` доступна только для конкретной завершённой симуляции с опцией вывода в ПРОМ
- [ ] Для всех остальных страниц симуляций окно агента открывается, но кнопка недоступна
- [ ] Логика доступности одинаково работает на любой вкладке деталки этой симуляции
- [ ] Tooltip у кнопки объясняет причину недоступности на неподходящих страницах

### FE-AC-2. Черновик запроса и отправка
- [ ] По клику на `Сформировать БТ` в поле ввода вставляется черновик запроса
- [ ] Пользователь может произвольно отредактировать вставленный текст
- [ ] Черновик запроса не отправляется агенту автоматически
- [ ] Пользователь может отправить или не отправить этот текст
- [ ] В доступном состоянии tooltip у кнопки поясняет, что после вставки черновик можно отредактировать и отправить вручную

### FE-AC-3. Результат публикации
- [ ] При успешной публикации `btUrl` отображается как ссылка
- [ ] Поле `Ссылка на БТ` обновляется у текущей симуляции
- [ ] При ошибке публикации ссылка не обновляется ложноположительно

## Открытые вопросы и допущения

- Если в поле ввода уже есть пользовательский текст, способ замены черновика запроса должен быть отдельно подтверждён UX-решением.
- Формат перечисления более чем двух риск-параметров зафиксирован семантически, а не как строгое пунктуационное правило.
