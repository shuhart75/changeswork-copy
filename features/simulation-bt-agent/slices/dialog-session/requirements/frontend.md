## Контекст и бизнес-смысл

### Цель

Сделать окно агента постоянным помощником в рамках интерфейса симуляций: пользователь может держать его открытым, продолжать работать в основном интерфейсе и возвращаться к уже существующей истории по тому же `session_id`.

### Пользователи и роли

- Пользователь интерфейса симуляций взаимодействует с окном агента в рамках своих прав доступа к этим страницам.

### Бизнес-контекст

Окно агента не является модальным шагом процесса. Оно открывается как правая боковая панель, не блокирует остальной интерфейс и получает контекст разговора от агента по `session_id`, а не из клиентского `contextPrompt`.

### Термины и определения

- `Восстановление сессии` — восстановление состояния окна по `session_id`.
- `Локальная блокировка окна` — временная блокировка только поля ввода и элементов управления окна во время запроса.
- `Неблокирующая работа основного интерфейса` — возможность продолжать работу в интерфейсе симуляций при открытом окне.
- `Дата и время старта диалога` — значение, которое фиксируется в момент первого открытия окна и далее остаётся неизменным для этой сессии.

## Границы MVP

### Входит в MVP

- правая боковая панель агента;
- неблокирующее поведение окна;
- восстановление истории и состояния по `session_id`;
- отправка пользовательских сообщений после восстановления сессии;
- локальная блокировка только элементов управления окна на время запроса;
- повторное открытие окна на других страницах симуляций с тем же `session_id`;
- отображение Markdown в сообщениях чата;
- возможность копировать текст из интерфейса АС КОДА в окно агента и из окна агента обратно в интерфейс АС КОДА;
- понятное сообщение об ошибке агента с рекомендацией перезапустить сессию.

### Не входит в MVP

- модальный режим, блокирующий остальной интерфейс;
- клиентское хранение истории как источника правды;
- параллельные независимые ветки одного диалога.

### Отложено после MVP

- расширенные состояния закрепления и сворачивания окна;
- перенос сессии между устройствами;
- потоковая выдача частичных ответов.

## Пользовательские сценарии

### Сценарий FE-1. Открытие и восстановление окна
1. Пользователь открывает окно агента на любой странице симуляций.
2. Фронт вызывает REST восстановления сессии с `session_id`.
3. Агент возвращает историю и текущее состояние разговора.

### Сценарий FE-2. Пользователь продолжает работу в основном интерфейсе
1. Окно агента остаётся открытым справа.
2. Пользователь переключается между списком, деталью и редактированием симуляций.
3. Окно не мешает навигации и остаётся доступным.

### Сценарий FE-3. Отправка сообщения
1. Пользователь вводит или редактирует сообщение в окне.
2. Фронт отправляет сообщение агенту.
3. Вместе с текстом фронт передаёт ФИО пользователя из СУДИР и дату-время старта диалога.
4. На время запроса блокируются только элементы управления окна.
5. После ответа история обновляется, а основной интерфейс остаётся доступным.

### Сценарий FE-4. Ошибка и перезапуск сессии
1. Агент или интеграционный слой возвращает ошибку.
2. В окне показывается понятное сообщение об ошибке.
3. Пользователь может воспользоваться рекомендацией перезапустить сессию и начать новый диалог.

## UI-состав и навигация

### Экран/состояние 1. Правая боковая панель агента

- Назначение: основная поверхность работы с агентом.
- Откуда открывается: из точки входа на любой странице симуляций.
- Куда ведёт: остаётся поверх текущей страницы без смены маршрута.
- Что видно пользователю: в шапке окна заголовок и chip статуса агента; сразу под шапкой расположена область чата; ниже находятся поле ввода и основные действия.
- Какие действия доступны: писать агенту, копировать текст, скрывать или закрывать окно, раскрывать детали статуса агента и параллельно работать с основным интерфейсом.

### Экран/состояние 2. Локальная загрузка и ошибка внутри окна

- Назначение: показать, что окно сейчас синхронизируется, ожидает ответ или столкнулось с ошибкой.
- Откуда открывается: после восстановления сессии или отправки сообщения.
- Куда ведёт: обратно в обычное состояние окна или в новую сессию после перезапуска.
- Что видно пользователю: индикатор загрузки, текст ошибки и рекомендации по восстановлению работы агента.
- Какие действия доступны: ждать, повторить действие после восстановления, ознакомиться с рекомендацией и продолжать пользоваться основным интерфейсом.

## Функциональные требования

### FE-FR-1. Восстановление сессии выполняется по `session_id`

**Описание:**  
После открытия или повторного открытия окна фронт должен восстановить состояние окна через единый REST, передавая только `session_id`.

**Детали:**
- фронт не отправляет историю и клиентский контекст при открытии окна;
- если `session_id` уже существует, фронт запрашивает восстановление;
- если `session_id` новый, фронт создаёт пустую или стартовую сессию через тот же REST;
- история, пришедшая от агента, считается канонической;
- `session_id` хранится в рамках пользовательской сессии АС КОДА и сбрасывается вместе с ней.

**Зависимости:**
- backend API восстановления сессии;
- slice `agent-entrypoint`.

### FE-FR-2. Окно не блокирует основной интерфейс

**Описание:**  
Пока окно агента открыто, пользователь должен иметь доступ к интерфейсу симуляций.

**Детали:**
- окно отображается как правая боковая панель;
- окно не должно создавать модальную подложку, запрещающую взаимодействие с основной страницей;
- локальная загрузка внутри окна не должна блокировать клики и навигацию за его пределами;
- если окно открыто во время редактирования симуляции, пользователь может продолжать редактирование;
- закрытие окна не должно сбрасывать страницу симуляций.

**Зависимости:**
- общая оболочка интерфейса симуляций.

### FE-FR-3. Отправка пользовательского текста отделена от открытия окна

**Описание:**  
Открытие окна и восстановление истории не должны смешиваться с отправкой пользовательского текста.

**Детали:**
- АС КОДА передаёт агенту только текст пользователя и служебные поля вызова, но не валидирует и не интерпретирует содержимое текста;
- текст пользователя отправляется отдельным действием после того, как окно уже открыто;
- вместе с текстом всегда отправляются ФИО пользователя из СУДИР и дата-время старта диалога;
- дата и время старта диалога берутся из момента первого открытия окна и не меняются до сброса пользовательской сессии АС КОДА;
- фронтенд может временно блокировать поле ввода и кнопку отправки внутри окна на время запроса;
- поле ввода отображается как область высотой в 5 строк текста и при переполнении получает внутреннюю вертикальную прокрутку;
- история, возвращаемая после отправки, обновляет окно без перерисовки основной страницы.

**Зависимости:**
- backend API отправки сообщения;
- slice `bt-publication`.

### FE-FR-4. Сообщения чата отображаются с поддержкой Markdown

**Описание:**  
Окно агента должно корректно отображать сообщения агента с разметкой Markdown.

**Детали:**
- поддерживаются обычный текст, списки, заголовки, кодовые блоки и ссылки;
- отображение Markdown не должно ломать вёрстку окна;
- пользователь должен иметь возможность выделять и копировать текст из сообщений без ограничений.

**Зависимости:**
- формат ответа агента.

### FE-FR-5. Статус агента показывается в шапке окна, а ошибка раскрывает рекомендации

**Описание:**  
Окно должно показывать текущее состояние агента отдельным chip в шапке, а при проблеме на стороне агента или интеграции раскрывать понятное сообщение и рекомендацию по восстановлению работы.

**Детали:**
- в штатном состоянии chip показывает `Онлайн`;
- при проблеме chip переключается в состояние `Ошибка`;
- по клику на chip пользователь видит текст ошибки и рекомендацию перезапустить сессию;
- при перезапуске создаётся новая сессия и новый `session_id`;
- при перезапуске история предыдущей сессии не считается активной.

**Зависимости:**
- backend правила перезапуска сессии.

## Интеграция с Backend API

| Метод и маршрут | Где используется | Что отправляем/читаем | Условия вызова | Примечание |
|---|---|---|---|---|

## Валидация на frontend

### Правила

- при открытии и восстановлении нельзя отправлять backend ничего кроме `session_id`;
- при отправке текста нельзя считать клиентскую историю источником правды;
- при отправке текста ФИО берётся из текущего профиля СУДИР, а дата-время старта диалога берутся из момента первого открытия окна;
- локальная блокировка применяется только к элементам окна;
- основной интерфейс остаётся интерактивным;
- текст пользовательского сообщения не валидируется и не интерпретируется со стороны АС КОДА.

### Сообщения об ошибках

| Ситуация | Сообщение | Где показываем |
|---|---|---|
| Не удалось восстановить окно | `Не удалось восстановить диалог агента.` | внутри окна |
| Не удалось отправить сообщение | `Не удалось отправить сообщение агенту.` | внутри окна |
| Ошибка на стороне агента | `Во время работы агента произошла ошибка. Рекомендуется перезапустить сессию.` | внутри окна в деталях status chip |

## Нефункциональные требования к UI

- accessibility / keyboard / tab order: окно доступно с клавиатуры, но не перехватывает навигацию по всей странице;
- responsive behavior: окно должно оставаться удобным при параллельной работе со списком, деталью и формой редактирования;
- поле ввода должно сохранять фиксированную высоту в 5 строк и внутренний скролл без увеличения всей панели;
- performance expectations: восстановление окна не должно перезагружать текущую страницу;
- audit / trace / telemetry if needed: можно логировать открытие, восстановление, отправку сообщения и перезапуск сессии.

## Критерии приемки

### FE-AC-1. Восстановление сессии
- [ ] При открытии окна вызывается единый REST с одним `session_id`
- [ ] История и состояние восстанавливаются из ответа агента
- [ ] Клиентский `contextPrompt` и история не отправляются в REST открытия окна

### FE-AC-2. Неблокирующее поведение
- [ ] Основной интерфейс симуляций остаётся доступным при открытом окне
- [ ] Окно отображается как правая боковая панель
- [ ] На время запроса блокируются только элементы управления окна

### FE-AC-3. Служебные поля отправки
- [ ] При отправке текста агенту фронт передаёт ФИО пользователя из СУДИР
- [ ] При отправке текста агенту фронт передаёт дату-время старта диалога, зафиксированные при первом открытии окна
- [ ] При повторных сообщениях в рамках той же пользовательской сессии дата-время старта диалога не меняются

### FE-AC-4. Отображение и ошибки
- [ ] Сообщения агента отображаются с поддержкой Markdown
- [ ] Пользователь может копировать текст в окно агента и из окна агента
- [ ] В шапке окна отображается chip статуса агента
- [ ] При ошибке chip переключается в состояние `Ошибка` и по клику раскрывает текст проблемы с рекомендацией перезапустить сессию
- [ ] Поле ввода отображается как зона на 5 строк с внутренней вертикальной прокруткой при переполнении

## Открытые вопросы и допущения

- В этих требованиях зафиксирован рабочий маршрут `POST /dialog/{session_id}/message` для отправки текста после открытия окна; итоговый маршрут может быть скорректирован, если сохранится разделение между открытием или восстановлением и отправкой сообщения.
- Если пользователь открывает окно на нескольких страницах подряд, считается, что речь идёт об одном и том же окне агента, пока жив текущий `session_id`.
