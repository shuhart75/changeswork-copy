## Контекст и бизнес-смысл

### Цель

Сделать окно агента независимым от конкретной страницы и от одноразового набора клиентских данных, при этом сохранив для фронта доступ к данным страницы, необходимым для действий по БТ внутри окна.

### Источник правды

Источником правды для истории и состояния диалога является агент, который восстанавливает их по `session_id`. Источником правды для доступности действия по БТ остаются данные текущей страницы симуляции.

### Затронутые bounded contexts / aggregates

- `Research and Execution`
- `Simulation`
- `Identity and Access`

### Термины и определения

- `REST открытия и восстановления сессии` — единый REST вызова окна агента.
- `Признаки доступности` — признаки, по которым фронт решает, показывать ли в окне кнопку `Сформировать БТ`.
- `Данные для черновика запроса` — данные симуляции, из которых фронт локально собирает текстовый черновик запроса агенту.

## Бизнес-правила и системные ограничения

### BR-1. История и контекст диалога хранятся на стороне агента
- при вызове окна фронт не передаёт историю сообщений;
- при вызове окна фронт не передаёт `contextPrompt`;
- при вызове окна фронт не передаёт ФИО пользователя и дату-время старта диалога;
- агент должен восстановить историю и состояние по одному `session_id`.

### BR-2. Доступность действия по БТ определяется текущей страницей симуляций
- действие по БТ доступно только если текущая страница относится к конкретной симуляции;
- эта симуляция должна быть успешно завершённой;
- для неё должна быть доступна опция вывода в ПРОМ;
- на других страницах окно агента открывается, но действие по БТ остаётся недоступным.

## Границы MVP

### Входит в MVP

- единый backend-метод открытия и восстановления окна по `session_id`;
- возврат состояния окна и истории из агента;
- сохранение и выдача данных текущих страниц симуляций, нужных фронту для черновика запроса.

### Не входит в MVP

- передача структурированного контекста симуляции в API агента при каждом открытии;
- отдельный подготовительный метод для генерации текста запроса на БТ;
- блокировка навигации по интерфейсу симуляций.

### Отложено после MVP

- унификация контекста страницы для других доменных разделов;
- сборка черновика запроса на стороне сервера вместо фронта.

## Пользовательские и системные сценарии

### Сценарий BE-1. Восстановление окна по `session_id`
1. Фронт вызывает единый REST открытия или восстановления сессии.
2. Backend получает только `session_id`.
3. Агент возвращает сохранённую историю и текущее состояние окна.

### Сценарий BE-2. Страница конкретной симуляции отдаёт данные для черновика запроса на БТ
1. Пользователь открывает деталку или редактирование конкретной симуляции.
2. Существующий backend-контракт страницы возвращает статус, признак доступности вывода в ПРОМ, риск-параметры и поле бизнес-цели.
3. Фронт локально решает, показывать ли кнопку `Сформировать БТ`.

## Функциональные требования

### BE-FR-1. Единый REST вызова окна принимает только `session_id`

**Описание:**  
Backend должен иметь единый REST открытия или восстановления окна агента, в который передаётся только `session_id`.

**Правила и ограничения:**
- request body содержит только `session_id`;
- backend не принимает `contextPrompt`, `riskParam`, `businessEffect`, `btLink` или историю сообщений;
- backend не принимает в этом вызове ФИО пользователя и дату-время старта диалога;
- если `session_id` известен агенту, возвращается восстановленное состояние;
- если `session_id` новый, создаётся пустая или стартовая сессия по правилам агента.

**Зависимости:**
- slice `dialog-session`;
- хранилище сессий на стороне агента.

### BE-FR-2. Страницы симуляций должны отдавать признаки доступности

**Описание:**  
Существующие страницы симуляций должны возвращать фронту данные, достаточные для вычисления доступности кнопки `Сформировать БТ`.

**Правила и ограничения:**
- должен быть доступен идентификатор конкретной симуляции;
- должен быть доступен её текущий статус;
- должен быть доступен признак или эквивалентный набор данных, позволяющий понять, что для симуляции доступна опция вывода в ПРОМ;
- для страниц без конкретной симуляции такой набор признаков может отсутствовать.

**Зависимости:**
- существующие контракты списка, деталки и редактирования в интерфейсе симуляций;
- slice `bt-publication`.

### BE-FR-3. Страницы симуляций должны отдавать данные для черновика запроса

**Описание:**  
Backend существующих страниц симуляций должен отдавать фронту данные, из которых фронт сможет локально собрать черновик запроса для кнопки `Сформировать БТ`.

**Правила и ограничения:**
- должны быть доступны риск-параметры и их целевые значения `TO BE`;
- должно быть доступно бизнес-поле, которое используется как источник `BUSINESS_GOAL`, если оно есть в текущем интерфейсе;
- имя пользователя может приходить не из API симуляций, а из текущего профиля пользователя;
- черновик запроса не строится на стороне агента в момент открытия окна.

**Зависимости:**
- контракты текущей страницы;
- текущий контекст аутентификации и профиля пользователя.

## Модель данных

### Основные сущности и поля

| Сущность / таблица | Поле | Тип | Обязательность | Описание |
|---|---|---|---|---|
| `agent_window_session` / хранилище сессий | `session_id` | UUID/string | обязательно | идентификатор окна агента |
| `Simulation` view | `id` | UUID/string | условно обязательно | нужен для страниц конкретной симуляции |
| `Simulation` view | `status` | enum/string | условно обязательно | используется для определения доступности действия по БТ |
| `Simulation` view | `can_promote_to_prod` / эквивалент | boolean/derived | условно обязательно | признак доступности вывода в ПРОМ |
| `Simulation` view | `risk_parameters[*].name` | string | условно обязательно | часть черновика запроса |
| `Simulation` view | `risk_parameters[*].to_be_value` | scalar | условно обязательно | часть черновика запроса |
| `Simulation` view | `business_goal` / эквивалент | string/null | опционально | источник `BUSINESS_GOAL` для черновика запроса |

### Инварианты и ограничения

- `session_id` уникален в рамках сессии окна агента;
- отсутствие конкретной симуляции на странице означает, что действие по БТ недоступно;
- отсутствие `business_goal` не запрещает показать окно агента.

### Индексы / уникальности / FK

- уникальный индекс по `session_id`;
- новые внешние ключи для признаков доступности в MVP не требуются.

## API-контракт

### Эндпоинты

| Метод и маршрут | Назначение | Кто вызывает | Примечание |
|---|---|---|---|
| `POST /dialog/session` | открыть или восстановить окно агента | frontend agent window | request содержит только `session_id` |
| `GET /api/v1/simulations` / existing list API | отдать контекст экрана для списка | simulation UI frontend | используется без привязки к конкретному действию по БТ |
| `GET /api/v1/simulation/{number}` / existing detail API | отдать данные конкретной симуляции | simulation UI frontend | источник признаков доступности и данных для черновика запроса |

### OpenAPI fragment

```yaml
openapi: 3.0.3
info:
  title: API открытия и восстановления сессии окна агента
  version: 1.0.0
paths:
  /dialog/session:
    post:
      summary: Открыть или восстановить сессию окна агента
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              required: [session_id]
              properties:
                session_id:
                  type: string
                  format: uuid
      responses:
        "200":
          description: Состояние окна агента восстановлено
  /api/v1/simulation/{number}:
    get:
      summary: Существующая деталка симуляции с признаками доступности действия по БТ
      responses:
        "200":
          description: Данные страницы симуляции
          content:
            application/json:
              schema:
                type: object
                properties:
                  id:
                    type: string
                  status:
                    type: string
                  canPromoteToProd:
                    type: boolean
                  businessGoal:
                    type: string
                    nullable: true
                  riskParameters:
                    type: array
                    items:
                      type: object
                      properties:
                        name:
                          type: string
                        toBeValue: {}
components: {}
```

## Интеграции, вычисления и фоновые процессы

- Внешние системы: backend хранения сессий агента;
- Асинхронные процессы: не требуются отдельно в этом slice;
- Вычисление статусов / derived fields: `can_promote_to_prod` может быть вычисляемым полем существующего сервиса симуляций;
- Идемпотентность / ретраи: повторный вызов `POST /dialog/session` с тем же `session_id` должен восстанавливать одну и ту же сессию.

## Ошибки и валидация

### Валидационные правила

- `session_id` обязателен;
- при вызове окна не принимаются дополнительные поля;
- `can_promote_to_prod` должен быть вычислим или явно возвращаем для страниц конкретной симуляции;
- `riskParameters[*].toBeValue` может отсутствовать только если действие по БТ для данной страницы недоступно.

### Ошибки API

| Код/сценарий | Условие | Ответ |
|---|---|---|
| `400 invalid_session_open_payload` | запрос содержит поля помимо `session_id` | ошибка валидации |
| `404 simulation_not_found` | страница конкретной симуляции запрашивает несуществующую симуляцию | стандартная ошибка существующего API симуляций |

## Критерии приемки

### BE-AC-1. Контракт открытия сессии
- [ ] Единый REST вызова окна принимает только `session_id`
- [ ] История и состояние окна восстанавливаются на стороне агента
- [ ] При открытии окна фронт не передаёт `contextPrompt`, историю сообщений, ФИО пользователя и дату-время старта диалога

### BE-AC-2. Признаки доступности
- [ ] Для страниц конкретной симуляции backend отдаёт статус и признак доступности вывода в ПРОМ
- [ ] Для страниц конкретной симуляции backend отдаёт данные для черновика запроса
- [ ] Для страниц без конкретной симуляции действие по БТ может оставаться недоступным без ошибки backend

## Открытые вопросы и допущения

- Предполагается, что единый REST вызова окна не отвечает за отправку пользовательских сообщений, а только за открытие и восстановление.
- Точное техническое имя признака `can_promote_to_prod` может отличаться от указанного в требованиях, если сервис симуляций отдаёт равнозначный признак.
