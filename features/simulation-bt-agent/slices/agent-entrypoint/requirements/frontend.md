## Контекст и бизнес-смысл

### Цель

Дать пользователю постоянную и единообразную точку входа в окно агента на всех страницах интерфейса симуляций: в списке, на деталке, на вкладках деталки, в форме редактирования и на других связанных экранах.

### Пользователи и роли

- Пользователь, имеющий доступ к соответствующей странице симуляций, может открыть окно агента.
- Доступность действия `Сформировать БТ` внутри окна определяется не самим фактом открытия окна, а контекстом текущей страницы и состоянием конкретной симуляции.

### Бизнес-контекст

Окно агента не привязано только к завершённой деталке симуляции. Оно доступно на страницах интерфейса симуляций, а действие по БТ включается внутри окна только тогда, когда текущая страница относится к конкретной завершённой симуляции, доступной к выводу в ПРОМ.

### Термины и определения

- `Окно агента` — неблокирующее окно-панель, доступное поверх интерфейса симуляций.
- `Контекст страницы` — локальные данные о том, с какой страницы открыто окно и связана ли она с конкретной симуляцией.
- `Признаки доступности` — набор признаков, позволяющий показать или скрыть в окне кнопку `Сформировать БТ`.

## Границы MVP

### Входит в MVP

- открытие окна агента со списка симуляций;
- открытие окна агента с деталки симуляции и с любой её вкладки;
- открытие окна агента с формы редактирования симуляции;
- повторное открытие того же окна по активному `session_id`;
- передача в backend только `session_id` при открытии или восстановлении окна.

### Не входит в MVP

- отдельный маршрут для окна агента;
- блокировка основного интерфейса на время работы агента;
- отправка структурированного `contextPrompt` на backend при открытии окна.

### Отложено после MVP

- открытие окна агента вне раздела симуляций;
- поддержка нескольких независимых окон агента одновременно;
- передача одной и той же сессии между разными пользователями.

## Пользовательские сценарии

### Сценарий FE-1. Открытие окна со списка симуляций
1. Пользователь находится на списковой форме симуляций.
2. Пользователь открывает окно агента.
3. Окно появляется поверх списка и не блокирует работу со страницей.

### Сценарий FE-2. Открытие окна со страницы конкретной симуляции
1. Пользователь находится на деталке конкретной симуляции, на любой вкладке или в форме её редактирования.
2. Пользователь открывает окно агента.
3. Окно привязывается к контексту текущей страницы этой симуляции.

### Сценарий FE-3. Повторное открытие уже активного окна
1. Пользователь закрывает или сворачивает окно агента.
2. Пользователь повторно открывает окно из другой страницы раздела симуляций.
3. Фронт использует тот же активный `session_id` и запрашивает восстановление состояния окна.

## UI-состав и навигация

### Экран/состояние 1. Глобальная точка входа в окно агента

- Назначение: открыть окно агента из любого экрана симуляций.
- Откуда открывается: список, деталка, вкладки деталки, редактирование и другие страницы симуляций.
- Куда ведёт: в окно агента без смены маршрута.
- Что видно пользователю: доступная точка входа в окно агента.
- Какие действия доступны: открыть окно, повторно открыть уже существующую сессию.

### Экран/состояние 2. Неблокирующее окно агента

- Назначение: работать с агентом параллельно с основным интерфейсом.
- Откуда открывается: из точки входа.
- Куда ведёт: остаётся поверх текущего состояния страницы.
- Что видно пользователю: история диалога, поле ввода, действия по БТ в зависимости от контекста страницы.
- Какие действия доступны: отправлять сообщения, сворачивать окно, закрывать окно, продолжать работать в основном интерфейсе.

## Функциональные требования

### FE-FR-1. Окно агента открывается с любой страницы симуляций

**Описание:**  
Фронт должен предоставлять способ открыть окно агента на всех ключевых страницах интерфейса симуляций, а не только на завершённой деталке.

**Детали:**
- точка входа доступна на списковой форме симуляций;
- точка входа доступна на деталке симуляции и на любой её вкладке;
- точка входа доступна на форме редактирования симуляции;
- открытие окна не меняет текущий маршрут и не сбрасывает состояние страницы.

**Зависимости:**
- существующие экраны симуляций;
- единая модель пользовательского взаимодействия для точки входа.

### FE-FR-2. При открытии и восстановлении окна отправляется только `session_id`

**Описание:**  
Фронт не должен отправлять в агент клиентский контекст симуляции, историю и дополнительные данные запроса при открытии окна.

**Детали:**
- при первом открытии фронт создаёт или получает `session_id`;
- при первом открытии фронт фиксирует дату и время старта диалога;
- при повторном открытии фронт использует уже активный `session_id`;
- в REST вызов открытия или восстановления передаётся только `session_id`;
- дата и время старта диалога сохраняются в рамках пользовательской сессии АС КОДА и затем переиспользуются при отправке текста агенту;
- никакой `contextPrompt`, список риск-параметров, история сообщений или дополнительные поля на этом шаге в backend не передаются.

**Зависимости:**
- backend API открытия и восстановления сессии;
- slice `dialog-session`.

### FE-FR-3. Окно агента не блокирует основной интерфейс

**Описание:**  
Окно агента должно работать как неблокирующий слой поверх интерфейса симуляций.

**Детали:**
- пользователь может переключаться по страницам симуляций, не закрывая окно;
- пользователь может взаимодействовать с основным интерфейсом, пока окно открыто;
- блокировка во время запроса может применяться только к локальным элементам окна агента, но не к остальной странице;
- окно должно сохранять своё состояние при навигации внутри интерфейса симуляций, пока актуален `session_id`.

**Зависимости:**
- slice `dialog-session`;
- архитектура общей оболочки интерфейса симуляций.

## Валидация на frontend

### Правила

- окно агента должно открываться с любой страницы симуляций;
- при открытии и восстановлении нельзя передавать backend что-либо кроме `session_id`;
- дата и время старта диалога фиксируются один раз в момент первого открытия окна и не пересчитываются при последующих сообщениях;
- состояние открытого окна не должно блокировать основную страницу;
- действия по БТ зависят от контекста текущей страницы и описаны в slice `bt-publication`.

### Сообщения об ошибках

| Ситуация | Сообщение | Где показываем |
|---|---|---|
| Не удалось открыть окно агента | `Не удалось открыть окно агента. Попробуйте ещё раз.` | внутри окна или рядом с точкой входа |
| Не удалось восстановить активную сессию | `Не удалось восстановить текущую сессию агента.` | внутри окна |

## Нефункциональные требования к UI

- accessibility / keyboard / tab order: точка входа и окно агента доступны с клавиатуры, а фокус не блокирует основной интерфейс;
- responsive behavior: окно корректно работает на компьютере и не перекрывает критичные элементы интерфейса симуляций;
- performance expectations: открытие окна не приводит к перезагрузке текущей страницы;
- audit / trace / telemetry if needed: можно логировать событие открытия окна и восстановления `session_id`.

## Открытые вопросы и допущения

- Под `вызовом диалога` в этих требованиях зафиксирован вызов открытия или восстановления окна, а не дальнейшая отправка пользовательских сообщений.
- Предполагается, что `session_id` живёт дольше одного маршрута и хранится на стороне клиента ровно как идентификатор активного окна, а не как источник истории.
- Предполагается, что дата и время старта диалога живут в той же пользовательской сессии АС КОДА, что и `session_id`, и сбрасываются вместе с ней.
