## Контекст и бизнес-смысл

### Цель

Дать пользователю постоянную и единообразную точку входа в окно агента на страницах интерфейса симуляций и сразу показать, готов ли агент принимать сообщения.

### Пользователи и роли

- Пользователь, имеющий доступ к соответствующей странице симуляций, может открыть окно агента.
- Доступность действия по БТ внутри окна определяется контекстом текущей страницы и состоянием конкретной симуляции.

### Бизнес-контекст

Окно агента доступно не только на завершённой деталке. Оно открывается как правый неблокирующий помощник, а readiness/liveness агента влияет на возможность ввода и отправки сообщений.

### Термины и определения

- `Окно агента` — неблокирующая правая панель поверх интерфейса симуляций.
- `UI-сессия` — состояние окна агента в АС КОДА, связанное с backend-managed `session_id`.
- `session_id` — opaque identifier UI-сессии, который создаёт backend АС КОДА и возвращает frontend для продолжения диалога.
- `start_datetime` — дата и время первого открытия окна, фиксируется backend при создании UI-сессии и передаётся в RAIN при первом run.
- `agent status` — нормализованное frontend-состояние по backend status API: `ready`, `not_ready`, `unavailable`, `unknown`.

## Границы MVP

### Входит в MVP

- открытие окна агента со списка симуляций;
- открытие окна агента с деталки симуляции и с любой её вкладки;
- открытие окна агента с формы редактирования симуляции;
- получение и переиспользование `session_id` в рамках пользовательской СУДИР-сессии;
- фиксация `start_datetime` на backend при первом открытии окна;
- status chip агента в шапке окна;
- блокировка composer/actions при неготовом агенте.

### Не входит в MVP

- прямое подключение frontend к RAIN;
- работа frontend с OTT/mTLS;
- вызов RAIN при простом открытии окна;
- отдельный маршрут для окна агента;
- блокировка основного интерфейса на время работы агента.

### Отложено после MVP

- открытие окна агента вне раздела симуляций;
- несколько независимых окон агента одновременно;
- перенос UI-сессии между устройствами;
- подробная диагностика readiness в UI.

## Пользовательские сценарии

### Сценарий FE-1. Открытие окна со списка симуляций
1. Пользователь находится на списковой форме симуляций.
2. Пользователь открывает окно агента.
3. Frontend вызывает backend без `session_id`, если нужно создать новую сессию, или передаёт ранее полученный `session_id`, если нужно продолжить диалог.
4. Окно появляется поверх списка и не блокирует работу со страницей.
5. Frontend запрашивает статус агента через backend.

### Сценарий FE-2. Открытие окна со страницы конкретной симуляции
1. Пользователь находится на деталке конкретной симуляции, на любой вкладке или в форме её редактирования.
2. Пользователь открывает окно агента.
3. Окно привязывается к текущему контексту страницы, но RAIN ещё не вызывается.

### Сценарий FE-3. Агент не готов
1. Backend status API возвращает `not_ready` или `unavailable`.
2. Status chip показывает ошибочное/неготовое состояние.
3. Composer и inline actions заблокированы.
4. Пользователь может продолжать работать с основным интерфейсом и дождаться обновления статуса.

## UI-состав и навигация

### Экран/состояние 1. Глобальная точка входа в окно агента

- Назначение: открыть окно агента из любого экрана симуляций.
- Откуда открывается: список, деталка, вкладки деталки, редактирование и другие страницы симуляций.
- Куда ведёт: в окно агента без смены маршрута.
- Что видно пользователю: доступная точка входа в окно агента.
- Какие действия доступны: открыть окно, повторно открыть уже существующую UI-сессию.

### Экран/состояние 2. Шапка окна со status chip

- Назначение: показать готовность агента к работе.
- Откуда открывается: сразу после открытия окна.
- Куда ведёт: к обычному вводу, если агент готов, или к blocked composer, если не готов.
- Что видно пользователю: `Онлайн`, `Не готов`, `Недоступен` или `Проверка`.
- Какие действия доступны: обновить статус, раскрыть подробность ошибки, перезапустить UI-сессию.

## Функциональные требования

### FE-FR-1. Окно агента открывается с любой страницы симуляций

**Описание:**
Frontend должен предоставлять способ открыть окно агента на всех ключевых страницах интерфейса симуляций.

**Детали:**
- точка входа доступна на списковой форме симуляций;
- точка входа доступна на деталке симуляции и на любой её вкладке;
- точка входа доступна на форме редактирования симуляции;
- открытие окна не меняет текущий маршрут и не сбрасывает состояние страницы.

**Зависимости:**
- существующие экраны симуляций;
- общая оболочка интерфейса.

### FE-FR-2. Frontend продолжает backend-managed UI-сессию

**Описание:**
Frontend получает `session_id` от backend АС КОДА при первом открытии окна и переиспользует его до перезапуска, сброса окна или окончания пользовательской СУДИР-сессии.

**Детали:**
- при первом открытии окна frontend вызывает `POST /dialog/session` без `session_id`;
- backend возвращает `session_id`, который frontend хранит как opaque token;
- при повторном открытии или polling frontend передаёт `session_id` в параметрах запроса, а не в path;
- если frontend вызывает `POST /dialog/session` без `session_id`, это означает сброс текущей UI-сессии и создание новой;
- frontend не генерирует `session_id` и не фиксирует `start_datetime`;
- при открытии окна RAIN не вызывается;
- frontend вызывает backend АС КОДА для открытия/восстановления UI-сессии и загрузки истории;
- при перезапуске UI-сессии backend создаёт новые `session_id` и `start_datetime`.

**Зависимости:**
- backend UI session endpoint;
- slice `dialog-session`.

### FE-FR-3. Status chip использует backend status API

**Описание:**
Frontend должен отображать готовность агента на основании нормализованного ответа backend АС КОДА, а не прямых вызовов RAIN.

**Детали:**
- status API вызывается при открытии окна;
- status API может переопрашиваться периодически, рекомендуемо раз в `15-30` секунд, или по действию пользователя;
- `ready` включает composer при отсутствии active run;
- `not_ready`, `unavailable` и `unknown` блокируют composer и agent actions;
- tooltip/details status chip объясняет причину, если backend её вернул.

**Зависимости:**
- `GET /dialog/agent/status`;
- RAIN `/health/liveness` и `/health/readiness` на backend.

### FE-FR-4. Окно агента не блокирует основной интерфейс

**Описание:**
Окно агента должно работать как неблокирующий слой поверх интерфейса симуляций.

**Детали:**
- пользователь может переключаться по страницам симуляций, не закрывая окно;
- пользователь может взаимодействовать с основным интерфейсом, пока окно открыто;
- блокировка при active run или неготовом агенте применяется только к composer/actions окна;
- окно сохраняет состояние при навигации внутри интерфейса симуляций, пока актуален `session_id`.

**Зависимости:**
- slice `dialog-session`;
- архитектура UI shell.

## Валидация на frontend

### Правила

- окно агента должно открываться с любой страницы симуляций;
- frontend не генерирует `session_id`, а хранит значение, полученное от backend;
- отсутствие `session_id` в dialog API означает создание новой UI-сессии/сброс текущего окна;
- при открытии окна нельзя отправлять `message`, историю, риск-параметры или `contextPrompt` в RAIN;
- status chip должен переводить composer/actions в blocked state при `status != ready`;
- основная страница остаётся интерактивной.

### Сообщения об ошибках

| Ситуация | Сообщение | Где показываем |
|---|---|---|
| Не удалось открыть окно агента | `Не удалось открыть окно агента. Попробуйте ещё раз.` | внутри окна или рядом с точкой входа |
| Агент не готов | `Агент сейчас не готов принимать сообщения.` | status chip / composer |
| Агент недоступен | `Не удалось проверить доступность агента.` | status chip / composer |
| Не удалось восстановить UI-сессию | `Не удалось восстановить текущую сессию агента.` | внутри окна |

## Нефункциональные требования к UI

- accessibility / keyboard / tab order: точка входа, chip и окно агента доступны с клавиатуры;
- responsive behavior: окно корректно работает на компьютере и не перекрывает критичные элементы интерфейса симуляций;
- performance expectations: открытие окна не вызывает RAIN `/chat` и не ждёт долгий агентский ответ;
- audit / trace / telemetry if needed: можно логировать открытие окна, проверку статуса и восстановление UI-сессии.

- [ ] Status chip отображает нормализованный статус агента
- [ ] При `status != ready` composer и actions заблокированы

## Открытые вопросы и допущения

- Частота автоматического обновления status chip должна быть подтверждена с эксплуатацией; рекомендуемый диапазон `15-30` секунд.
- Если backend вернёт stale status, frontend должен показать время последней проверки.
