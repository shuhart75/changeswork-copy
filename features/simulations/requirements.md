## Общий контур feature

- Назначение feature: зафиксировать existing baseline coverage по симуляциям в одном контрольном документе.
- Что уже есть в baseline/current: список, деталка, форма, lifecycle и связанные сущности/артефакты.
- Какая дельта добавляется этой feature: удобный feature-level документ контроля без необходимости читать slices по отдельности.
- Какие slice входят в контрольный документ: `list`, `detail`, `form`, `lifecycle`, `artifacts-related`.

## Порядок slice для контроля

1. `01 list — Список симуляций`

4. `04 lifecycle — Жизненный цикл симуляции`
5. `05 artifacts-related — Артефакты и связанные сущности`

**Бизнес-требования**

- Цель: дать пользователю обзорный список симуляций с навигацией к деталке.
- Бизнес-правила: список поддерживает фильтры, поиск, сортировку и пагинацию.

## SIMULATIONS-LIFECYCLE — Жизненный цикл симуляции

Slice card: `slices/lifecycle/slice.md`
Детализация FE: `slices/lifecycle/requirements/frontend.md`
Детализация BE: `slices/lifecycle/requirements/backend.md`
Прототип: `slices/lifecycle/delivery-prototype/prototype.html`
Planning story: `не применяется, existing coverage`

**Бизнес-требования**

- Цель: зафиксировать независимый lifecycle симуляции.
- Бизнес-правила: симуляция стартует из `draft`, выполняющаяся не редактируется, завершённая immutable.
- Ограничения: lifecycle не смешивается с approval/ratification.

**Пользовательские требования к АС КОДА**

- Пользователь видит доступные действия по lifecycle и ограничения по статусу.

**Критерии приемки**

1. Разрешённые и запрещённые переходы статусов определены явно.
2. Lifecycle работает независимо от approval flow.
3. Ограничения редактирования зависят от статуса симуляции.

**USE CASES**

- **осн. сценарий 1** пользователь запускает симуляцию из `draft`.
- **альт. сценарий 1.1** система запрещает недопустимый переход.
- **альт. сценарий 1.2** completed simulation не допускает редактирование.
- **осн. сценарий 2** пользователь наблюдает статус и доступные действия.
