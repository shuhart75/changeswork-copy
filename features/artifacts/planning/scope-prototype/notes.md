# Заметки к scope prototype

## Цель
Согласовать reusable-контур артефактов как shared механизма для `Pilot`, `Simulation` и `Deployment`, не превращая scope в файловое хранилище.

## Какой идеальный сценарий показываем
- на host screen есть единый блок артефактов;
- пользователь с правами editor добавляет, редактирует и удаляет внешние ссылки;
- viewer открывает ссылки и видит метаданные без права изменения;
- поведение одинаково для `Pilot`, `Simulation` и `Deployment`.

## Какой MVP cut показываем отдельно
- только внешние URL, без upload и preview;
- базовые поля `title`, `url`, `description/type` по зафиксированной модели;
- наследование прав от родительской сущности;
- единый CRUD/API-контур и одинаковый UI-блок на host screens.

## Какие planning stories обсуждаем через этот прототип
- `imported-shared-scope`
- `STORY-ARTIFACTS-001` — контрольный feature-level story для согласования требований

## Mock data
- у пилота 2 артефакта: ссылка на BRD и ссылка на Confluence-страницу;
- у симуляции 1 артефакт: ссылка на результат БТ;
- у внедрения 1 артефакт: ссылка на релизный документ;
- одна невалидная ссылка для показа серверной/клиентской валидации.

## Открытые вопросы
- нужна ли в MVP фиксированная типизация артефактов или достаточно свободного текстового признака;
- обязателен ли порядок сортировки вручную или хватает reverse-chronological списка;
- какой минимальный объём audit-событий нужен для операций add/edit/delete.
