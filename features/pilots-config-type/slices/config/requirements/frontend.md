## Контекст и бизнес-смысл

### Цель

Сделать контур пилота явным в UI, чтобы пользователь не создавал offline/combined пилоты как неотличимые от online.

### Пользователи и роли

Пользователи и права наследуются из существующего workspace пилотов.

### Термины и определения

- `online` — Онлайн.
- `offline` — Офлайн.
- `online+offline` — Онлайн+Офлайн.

## Границы MVP

### Входит в MVP

- обязательный select в создании/редактировании;
- default `online`;
- отображение значения в деталке;
- клиентская валидация enum.

### Не входит в MVP

- отдельный экран управления типами процесса;
- новая фильтрация списка пилотов, если она не требуется текущим workspace;
- изменение прав доступа.

### Отложено после MVP

- актуализация delivery prototype в prototype mode.

## Пользовательские сценарии

### Сценарий FE-1. Создание пилота с типом процесса
1. Пользователь открывает форму создания пилота.
2. Система показывает поле `Тип процесса` со значением `Онлайн`.
3. Пользователь оставляет default или выбирает другое допустимое значение.
4. Frontend отправляет wire-value `processType`.

### Сценарий FE-2. Просмотр типа процесса
1. Пользователь открывает деталку пилота.
2. Frontend читает `processType` из ответа backend.
3. Значение отображается как `Онлайн`, `Офлайн` или `Онлайн+Офлайн`.

## UI-состав и навигация

### Экран/состояние 1. Форма создания/редактирования пилота

- Назначение: задать или изменить тип процесса вместе с остальными параметрами пилота.
- Откуда открывается: workspace пилотов.
- Куда ведёт: сохранение пилота.
- Что видно пользователю: select `Тип процесса`.
- Какие действия доступны: выбор одного из трёх значений.

### Экран/состояние 2. Детальная страница пилота

- Назначение: показать сохранённый тип процесса.
- Откуда открывается: список или связанный переход.
- Куда ведёт: при наличии прав — в форму редактирования.
- Что видно пользователю: read-only поле типа процесса.
- Какие действия доступны: редактирование только через существующий edit flow.

## Функциональные требования

### FE-FR-1. Select типа процесса

**Описание:** В форме пилота должен быть select `Тип процесса`.

**Детали:**
- значения: `online`, `offline`, `online+offline`;
- подписи: `Онлайн`, `Офлайн`, `Онлайн+Офлайн`;
- default при создании: `online`;
- поле обязательное.

**Зависимости:**
- backend принимает и возвращает `processType`.

### FE-FR-2. Валидация и блокировки

**Описание:** Frontend должен валидировать enum и наследовать существующие правила редактирования пилота.

**Детали:**
- если pilot не редактируем по статусу/роли, `processType` также не редактируем;
- неизвестное значение из API отображается как ошибка данных;
- форма не отправляется с пустым или неизвестным значением.

**Зависимости:**
- существующая логика формы пилота.

| Элемент UI | Источник данных | Пустое состояние | Loading | Error | Примечание |
|---|---|---|---|---|---|
| Select `Тип процесса` | `processType` формы/деталки | `online` при создании | disabled/skeleton вместе с формой | текст ошибки у поля | wire-value не переводится |
| Read-only тип процесса | `processType` detail response | `Онлайн`, если backend прислал пусто только для legacy записи до миграции | skeleton вместе с деталкой | ошибка данных | после миграции пустого значения быть не должно |

## Интеграция с Backend API

| Метод и маршрут | Где используется | Что отправляем/читаем | Условия вызова | Примечание |
|---|---|---|---|---|
| `POST /api/v1/experiment` | создание пилота/эксперимента | `processType` | submit формы | в текущем UI может быть фасадный route пилотов |
| `PUT /api/v1/experiment/{id}` | редактирование | `processType` | пилот редактируем | права наследуются |
| `POST /api/v1/experiments*`, `GET /api/v1/experiment/{number}` | список/деталка | `processType` | загрузка данных | точные routes см. BE pack |

## Валидация на frontend

### Правила

- `processType` обязателен.
- Допустимые значения: `online`, `offline`, `online+offline`.

### Сообщения об ошибках

| Ситуация | Сообщение | Где показываем |
|---|---|---|
| значение пустое | `Выберите тип процесса` | под select |
| значение неизвестно | `Неизвестный тип процесса` | под select или в detail error state |

## Нефункциональные требования к UI

- accessibility / keyboard / tab order: select участвует в обычном tab order формы.
- responsive behavior: поле занимает стандартную ширину формы.
- performance expectations: без отдельных запросов за справочником, значения статичны.
- audit / trace / telemetry if needed: отдельная frontend telemetry не требуется.

## Критерии приемки

### FE-AC-1. Создание
- [ ] При открытии формы создания выбран `Онлайн`.
- [ ] Пользователь может выбрать `Офлайн` или `Онлайн+Офлайн`.
- [ ] В request отправляется wire-value `processType`.

### FE-AC-2. Деталка и редактирование
- [ ] Деталка показывает сохранённое значение.
- [ ] Редактирование доступно только вместе с остальными редактируемыми полями пилота.
- [ ] Unknown/empty value не сохраняется молча.
