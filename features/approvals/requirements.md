## STORY-APPROVALS-001 — Core approval/ratification процесс

Slice card: `slices/core-process/slice.md`
Детализация FE: `slices/core-process/requirements/frontend.md`
Детализация BE: `slices/core-process/requirements/backend.md`
Прототип: `slices/core-process/delivery-prototype/prototype.html`
Planning story: `planning/stories/STORY-APPROVALS-001.md`

**Бизнес-требования**

- Цель: обеспечить единый процесс approval и ratification для `PilotVersion` и `DeploymentVersion`.
- Бизнес-правила: этапы идут последовательно; решения принимаются назначенными участниками; reject/cancel/recall возвращают процесс в согласованное состояние.
- Ограничения: граница между core process и package flows не должна размываться.

**Пользовательские требования к АС КОДА**

- Система должна уметь запускать, продолжать и завершать процесс согласования для целевых доменных сущностей.
- Автор и участники процесса должны получать согласованное представление статусов и результатов.

**Критерии приемки**

1. Approval и ratification переходы работают по зафиксированным правилам.
2. Решения и комментарии сохраняются и доступны для просмотра.
3. Интеграция с lifecycle `Pilot` и `Deployment` не расходится с доменными правилами.

**USE CASES**

- **осн. сценарий 1** автор отправляет версию в процесс согласования.
- **альт. сценарий 1.1** участник отклоняет элемент с причиной.
- **альт. сценарий 1.2** инициатор отзывает элемент из процесса.
- **осн. сценарий 2** после approval элемент переходит в ratification.

## STORY-APPROVALS-002 — Страница Согласования

Slice card: `slices/page/slice.md`
Детализация FE: `slices/page/requirements/frontend.md`
Детализация BE: `slices/page/requirements/backend.md`
Прототип: `slices/page/delivery-prototype/prototype.html`
Planning story: `planning/stories/STORY-APPROVALS-002.md`

**Бизнес-требования**

- Цель: дать пользователям рабочее место для просмотра и обработки назначений на согласование и утверждение.
- Бизнес-правила: страница должна поддерживать individual и package контекст без потери границы ответственности.
- Ограничения: расширенные пользовательские фильтры вне текущего MVP не входят.

**Пользовательские требования к АС КОДА**

- Пользователь видит свои назначения и может принять решение по ним.
- Для package-context пользователь видит связанную детализацию и групповые сценарии, если они входят в текущий scope.

**Критерии приемки**

1. Страница отображает релевантные назначения и их статусы.
2. Пользователь может открыть детализацию и принять допустимое решение.
3. Индивидуальные и пакетные сценарии отображаются без противоречий.

**USE CASES**

- **осн. сценарий 1** пользователь открывает список назначений и выбирает элемент.
- **альт. сценарий 1.1** пользователь фильтрует или меняет контекст просмотра.
- **альт. сценарий 1.2** пакетный сценарий открывает привязанный package context.
- **осн. сценарий 2** пользователь подтверждает или отклоняет назначение.

- поддержать individual и package views по зафиксированным сценариям.

Требования на бэк:

- поддержать package-related представления, если они входят в текущий scope.

Связанный детальный BE pack: `slices/page/requirements/backend.md`
