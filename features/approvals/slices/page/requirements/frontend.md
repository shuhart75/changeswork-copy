# Страница "Согласования" (Frontend)

Статус: **исключено из MVP**  
Область: MVP  
Дата обновления: 2026-05-25  
Decision ID: `DEC-2026-05-25-APPROVALS-SBERDOCS-001`

## Решение

Отдельная frontend-страница `Согласования` не реализуется. Согласующие и утверждающие работают в SberDocs, а АС КОДА показывает только read-only статус и on-demand историю на host screen сущности.

## Superseded scope

Следующий прежний scope отменён для MVP:

- вкладки `На согласовании` и `На утверждении`;
- список назначений пользователя внутри АС КОДА;
- детальные карточки `ApprovalChain`;
- карточки пакетов;
- индивидуальные actions согласующего `approve`, `reject`, `ratify`; кнопка `Отозвать` в АС КОДА не входит в MVP, отзыв выполняется в SberDocs автором/соавтором;
- массовые actions;
- deep link на деталку согласования внутри страницы `Согласования`.

## Новый frontend scope

Актуальный frontend scope находится в `../core-process/requirements/frontend.md` и включает:

- подготовку `new`-версии брифа и участников отправки;
- preview брифа;
- отправку `new`-версии в SberDocs через backend;
- read-only отображение SberDocs `systemNumber`, ссылки, статуса и отдельную загрузку approval sheet на host screen.

## Критерии приемки отмены

- [ ] В навигации MVP нет отдельного пункта `Согласования` для принятия решений.
- [ ] В АС КОДА нет кнопок принятия решения за согласующего/подписанта.
- [ ] Пользователь видит ссылку `Открыть в SberDocs`, если нужно выполнить действие.
- [ ] Старый delivery prototype страницы считается obsolete до отдельного решения о возобновлении.

## Обработка legacy-ссылок

Если в старых материалах или прототипах есть ссылка на страницу `Согласования`, она должна быть заменена одним из вариантов:

| Сценарий | Новое поведение |
|---|---|
| посмотреть статус/историю | открыть host screen сущности с блоком SberDocs status |
| принять решение | открыть документ в SberDocs |
| посмотреть пакет | сценарий отсутствует в MVP |
