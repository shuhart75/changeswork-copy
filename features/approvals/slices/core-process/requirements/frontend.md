## Контекст и бизнес-смысл

### Цель

Пользователь должен в АС КОДА подготовить бриф, подписанта и получателей, отправить документ в SberDocs и затем видеть текущий статус, историю и ссылку на документ без реализации собственного рабочего места согласующего.

### Пользователи и роли

- методолог, который редактирует бриф и отправляет документ в SberDocs;
- ПРМ, который передаётся в SberDocs как соавтор `additionalAuthorList[]`;
- подписант, которому backend отправляет письмо автоматически, когда задача подписи становится активной;
- согласующие не выбираются и не хранятся в АС КОДА; пользователь настраивает маршрут уже в SberDocs.

### Бизнес-контекст

Frontend не владеет approval actions и не управляет отправкой письма. Он формирует данные для отправки, показывает preview HTML, получает итоговый статус и read-only отображает состояние, пришедшее из backend.

## Границы MVP

### Входит в MVP

- блок/вкладка `Согласование через SberDocs` на host screen;
- локальное редактирование подписанта и получателей в `new`-версии;
- локальное создание и редактирование JSON документа в `new`-версии;
- read-only блок скоркарты во всех состояниях;
- поле `Доп. эффекты` под блоком скоркарты до отправки;
- предпросмотр HTML-письма;
- подтверждение отправки в SberDocs;
- отображение `systemNumber`, ссылки на документ, mapped status, оригинального `documentState/stateName` и даты последней синхронизации;
- read-only статус документа из SberDocs, on-demand загрузка поимённой истории и скачивание актуального DOCX;
- блокировку действий доменного элемента, пока с ним связан `ApprovalChain`;
- отображение ошибок создания/синхронизации.

### Не входит в MVP

- отдельная страница `Согласования`;
- вкладки `На согласовании` / `На утверждении`;
- UI-кнопки `approve`, `reject`, `ratify`, `sign` в АС КОДА;
- кнопка ручной отправки письма;
- отдельный UI-статус отправки письма;
- массовые действия;
- карточки пакетов и страница `Пакеты`;
- локальная frontend-агрегация статусов участников сверх данных SberDocs.

## Пользовательские сценарии

### Сценарий FE-1. Подготовка `new`-версии
1. Пользователь открывает host screen доменной сущности.
2. Если утверждающий не назначен, UI скрывает кнопку `Создать документ` и показывает скоркарту read-only.
3. Если утверждающий назначен и документ ещё не создан, UI показывает кнопку `Создать документ`.
4. Пользователь заполняет поля документа и `Доп. эффекты`, затем сохраняет `new`-версию без запуска согласования.

### Сценарий FE-2. Предпросмотр письма и отправка
1. Пользователь открывает `Предпросмотр письма`.
2. UI отправляет текущий JSON документа и `Доп. эффекты` на backend.
3. UI показывает HTML письма, сформированный backend из JSON, скоркарты и `Доп. эффекты`.
4. Пользователь подтверждает отправку.
5. UI вызывает backend submit endpoint и блокирует повторную отправку до ответа.
6. После успешного ответа UI сразу показывает `systemNumber`, ссылку SberDocs и новый статус.

### Сценарий FE-3. Мониторинг после отправки
1. После успешной отправки UI скрывает редактирование брифа, подписанта и получателей.
2. UI показывает `systemNumber`, ссылку на SberDocs, mapped status, оригинальный `documentState/stateName` и `lastSyncedAt`.
3. UI не показывает ни кнопку ручной отправки письма, ни отдельный статус отправки письма.
4. История согласования загружается отдельным вызовом, когда пользователь открывает соответствующий раздел.

### Сценарий FE-4. Ошибка SberDocs
1. Backend возвращает validation error, `sberdocs_create_error`, transport error или признак деградации SberDocs из фонового мониторинга.
2. UI показывает понятное сообщение и технический код/описание из SberDocs.
3. Если `documentId` не создан, сохранённая `new`-версия остаётся редактируемой, и пользователь может исправить поля и отправить повторно теми же endpoint-ами.
4. При сбое фонового health-check backend дополнительно уведомляет поддержку АС[СТ, РСП, КОДА, СРО].
5. Если backend вернул `requiresManualResolution = true` после аномального success-ответа SberDocs, UI не показывает повторный submit и сообщает, что нужна ручная проверка, так как в SberDocs мог уже появиться документ.

### Сценарий FE-5. Отклонение в SberDocs
1. SberDocs вернул raw `REJECTED`.
2. UI отображает mapped status `on_approval`, raw `REJECTED`, ссылку `Открыть в SberDocs` и загружает комментарии из approval sheet отдельным методом.
3. UI не открывает локальное редактирование автоматически: правки и повторная отправка выполняются в SberDocs.

### Сценарий FE-6. Отзыв или редактирование в SberDocs
1. В интерфейсе АС КОДА кнопка `Отозвать` в MVP не показывается.
2. Если нужно отозвать или отредактировать уже созданный документ, пользователь открывает документ по ссылке `Открыть в SberDocs`.
3. В SberDocs действие выполняет методолог-автор, ПРМ-соавтор или заместитель автора.
4. Если SberDocs возвращает `ON_DELETING` или `DELETED`, UI показывает local status `cancelled`.

### Сценарий FE-7. Просмотр истории
1. Пользователь открывает раздел истории/листа согласования.
2. UI вызывает отдельный backend метод `approval-sheet`.
3. UI отображает поимённые решения и комментарии из SberDocs без ожидания, что они хранятся в основном snapshot.

### Сценарий FE-8. Скачать актуальный документ
1. После отправки или согласования пользователь нажимает `Скачать документ`.
2. UI вызывает backend метод получения актуального файла из SberDocs.
3. UI скачивает DOCX, который вернул SberDocs, а не локально сгенерированный preview.

## UI-состав и навигация

### Экран/состояние 1. New document and participants

- Назначение: подготовить всё, что уйдёт в SberDocs.
- Откуда открывается: host screen `Pilot` / `Deployment` / релевантная сущность.
- Что видно пользователю: поля документа, поле `Краткое содержание`, read-only подписант, получатели, read-only scorecard block, поле `Доп. эффекты`, preview email button, save new version, send to SberDocs.
- Какие действия доступны: `Создать документ`, `Сохранить`, `Сохранить и отправить на утверждение`, `Предпросмотр письма`, `Отмена`.

### Экран/состояние 2. SberDocs status snapshot

- Назначение: показать состояние после отправки.
- Что видно пользователю: локальный статус, `systemNumber`, URL, исходный `documentState`, `stateName`, `lastSyncedAt`, раздел загрузки истории и кнопка скачивания актуального DOCX.
- Какие действия доступны: `Открыть в SberDocs`, `Обновить статус`, `История`, `Скачать документ`. Кнопок `Отозвать` и `Отправить уведомление` в MVP нет.

## Функциональные требования

### FE-FR-1. New-версия брифа и участников отправки

**Описание:**
Frontend должен редактировать бриф, подписанта и получателей в статусе `new` только до успешной отправки документа в SberDocs.

**Детали:**
- UI не показывает редактор этапов согласования и не сохраняет согласующих в локальной версии;
- UI должен явно донести, что согласующие и маршрут согласования настраиваются в SberDocs;
- поле `Краткое содержание` уходит в SberDocs `summary`;
- поля `Адресат`, `Наименование документа`, `Цель` обязательны при сохранении/отправке;
- в `Периметр применения` должно быть заполнено хотя бы одно поле;
- поле `Доп. эффекты` опционально, сохраняется в JSON, включается в письмо и не включается в DOCX;
- скоркарта отображается read-only во всех состояниях, включается в письмо и не включается в DOCX;
- preview письма доступен до отправки и не блокируется валидацией обязательных полей;
- после успешной отправки поля становятся read-only.

### FE-FR-2. Отправка в SberDocs

**Описание:**
UI не отправляет запрос в SberDocs напрямую, а вызывает backend АС КОДА.

**Детали:**
- submit button блокируется во время запроса;
- повторный submit запрещён backend, если версия уже отправлена;
- успешный ответ submit возвращает `documentId`, `systemNumber`, `documentUrl`, новый `status` и `lastSyncedAt`;
- UI не получает `jobId`, raw SberDocs response и технические snapshot-поля;
- после успеха UI закрывает диалог отправки и переводит экран в read-only режим.

### FE-FR-3. Read-only status и on-demand history

**Описание:**
UI отображает status snapshot из backend, а историю участников получает отдельным методом при открытии раздела истории.

**Детали:**
- показывать mapped status и оригинальные поля SberDocs из status snapshot;
- не показывать отдельный статус письма и не предполагать, что пользователь управляет рассылкой;
- не вычислять решения локально;
- не показывать action buttons согласующего;
- unmapped SberDocs status показывать как `Требует проверки` с оригинальным кодом;
- для unmapped статуса можно показывать пояснение, что backend уведомил поддержку, если такой признак пришёл в status snapshot;
- историю участников не ожидать в основном GET: для неё вызывать `GET /approval-sheet`.

### FE-FR-4. Предпросмотр письма

**Описание:**
Frontend должен отображать HTML, который backend сформировал по тем же правилам, что и реальное письмо подписанту.

**Детали:**
- UI не преобразует JSON в HTML самостоятельно;
- preview использует те же бизнес-данные, что и реальное письмо: JSON документа, скоркарту и `Доп. эффекты`;
- поля, которые появляются только после создания документа в SberDocs, не должны участвовать в шаблоне письма, чтобы preview и реальное письмо совпадали по составу;
- preview не запускает отправку документа и не инициирует email.

### FE-FR-5. Отказ от отдельной страницы

**Описание:**
Навигация АС КОДА не должна вести пользователя на отдельную страницу `Согласования` для принятия решений.

**Детали:**
- ссылки из host screen ведут либо на SberDocs URL, либо остаются внутри host screen;
- вкладки `На согласовании`/`На утверждении` не создаются;
- package cards не отображаются.

### FE-FR-6. Отзыв через SberDocs

**Описание:**
UI АС КОДА не предоставляет действие `Отозвать` в MVP. Отзыв или редактирование уже созданного документа выполняется в интерфейсе SberDocs.

**Детали:**
- на host screen остаётся ссылка `Открыть в SberDocs`;
- UI может показывать пояснение, что автор документа в SberDocs — методолог-отправитель, а соавтор — ПРМ;
- после отзыва/редактирования в SberDocs UI получает новый статус только через sync/status refresh;
- `ON_DELETING`/`DELETED` отображаются как `cancelled`.

### FE-FR-7. Блокировка действий доменного элемента

**Описание:**
Пока у доменного элемента есть связанный `ApprovalChain`, UI должен запретить действия с доменным элементом.

**Детали:**
- блокируются все action-кнопки host entity, кроме исключений, которые будут описаны отдельными требованиями;
- рядом со статусом согласования UI показывает пояснение: действие выполняется в SberDocs, если `documentId` уже получен;
- primary action для пользователя после создания SberDocs-документа — `Открыть в SberDocs`;
- действия локального approval block (`История`, `Скачать документ`, `Обновить статус`, подготовка `new` до отправки или после ошибки создания без `documentId`) не считаются действиями доменного элемента.

### FE-FR-8. Скачивание актуального документа

**Описание:**
UI должен скачивать документ из SberDocs через backend, потому что основной DOCX мог измениться в SberDocs после отправки.

**Детали:**
- действие доступно после появления `documentId`, а основной happy path — после `approved`;
- UI вызывает `GET /api/v1/approval-chains/{targetType}/{targetId}/document-file`;
- UI не пытается скачать локальный preview вместо документа SberDocs;
- при ошибке показывает, что актуальный документ временно недоступен, не меняя статус согласования.

## Интеграция с Backend API

| Метод и маршрут | Где используется | Что отправляем/читаем | Условия вызова | Примечание |
|---|---|---|---|---|
| `GET /api/v1/approval-chains/{targetType}/{targetId}` | host screen | active version + status snapshot | при открытии вкладки/блока | внутренний API АС КОДА |
| `PUT /api/v1/approval-chains/{targetType}/{targetId}/new-version` | редактор `new`-версии | document JSON, `additionalEffects`, signer, recipients, change reason | до отправки в SberDocs | сохраняет `new`-версию |
| `POST /api/v1/approval-chains/{targetType}/{targetId}/notification-preview` | предпросмотр письма | текущий document JSON + `additionalEffects` | до отправки, валидация не блокирует | backend добавляет скоркарту и возвращает HTML письма |
| `POST /api/v1/approval-chains/{targetType}/{targetId}/submit-to-sberdocs` | send action | подтверждённая версия | если версия ещё не отправлена | в ответе приходят `documentId`, `systemNumber`, `documentUrl`, `status` |
| `POST /api/v1/approval-chains/{targetType}/{targetId}/sync-sberdocs` | manual refresh | nothing или force flag | после отправки | backend обновляет status snapshot без загрузки истории |
| `GET /api/v1/approval-chains/{targetType}/{targetId}/approval-sheet` | раздел истории | nothing | когда пользователь открывает историю | backend читает approval sheet из SberDocs и не сохраняет локально |
| `GET /api/v1/approval-chains/{targetType}/{targetId}/document-file` | скачать документ | nothing | после появления `documentId`, особенно после согласования | backend получает актуальный основной DOCX из SberDocs |

## Валидация на frontend

### Правила

- нельзя отправить документ без подписанта и получателя;
- нельзя отправить бриф без обязательных полей `Адресат`, `Наименование документа`, `Цель`;
- нельзя сохранить/отправить документ, если в `Периметр применения` не заполнено ни одно поле;
- нельзя повторно отправить уже созданный документ;
- нельзя редактировать `new`-версию после успешной отправки;
- preview письма открывается даже при незаполненных обязательных полях.

### Сообщения об ошибках

| Ситуация | Сообщение | Где показываем |
|---|---|---|
| SberDocs validation error | `SberDocs не принял документ: <descriptionMessage>` | banner + field-level details when available |
| фоновый мониторинг SberDocs сообщает о деградации | `Мониторинг SberDocs показывает проблему: <message>. Поддержка уведомлена.` | status block |
| SberDocs недоступен во время business-вызова | `Не удалось выполнить запрос к SberDocs: <message>` | submit dialog + status block |
| sync unavailable | `Не удалось обновить статус SberDocs. Показаны последние сохранённые данные.` | status block |
| unmapped status | `Получен неизвестный статус SberDocs: <code>. Поддержка уведомлена.` | status block |
| actual document unavailable | `Не удалось получить актуальный документ из SberDocs. Попробуйте позже или откройте документ в SberDocs.` | download action |

## Нефункциональные требования

- UI должен явно показывать источник статуса: `SberDocs`.
- UI должен сохранять введённую `new`-версию при ошибке создания документа.
- UI должен быть устойчив к временному отсутствию `approvalSheetId`; после его появления история загружается отдельным методом при открытии раздела истории.
- UI не должен создавать ощущение, что решение можно принять в АС КОДА.
- UI не должен обещать, что HTML-письмо совпадает с финальным DOCX: письмо и DOCX имеют разный состав, а финальный DOCX скачивается из SberDocs.
- UI не должен показывать элементы ручной отправки письма или его отдельный delivery-status.

## Чеклист тестирования

- [ ] `new`-версия сохраняется без отправки в SberDocs.
- [ ] Preview письма не меняет статус и открывается даже при незаполненных обязательных полях.
- [ ] Submit блокирует повторный submit.
- [ ] Submit требует подписанта, получателя, `Адресат`, `Наименование документа`, `Цель` и хотя бы одно поле `Периметр применения`.
- [ ] Успешный submit сразу показывает `documentId`, `systemNumber`, ссылку и новый status.
- [ ] Если submit или другой business-вызов к SberDocs завершился ошибкой, UI показывает её без предположения, что запрос был заранее остановлен pre-check-ом.
- [ ] `APPROVED`, `TRANSITION_TO_REGISTRATION`, `ON_REGISTRATION`, `REGISTRATION_REJECTED`, `REGISTERED`, `TRANSITION_TO_EXECUTION`, `ON_EXECUTION`, `EXECUTED` отображаются как согласовано.
- [ ] Raw `REJECTED` отображается как `on_approval` с предложением открыть SberDocs.
- [ ] Raw `ON_DELETING` / `DELETED` отображают локальную цепочку как `cancelled`.
