# Command Cheatsheet

Короткая шпаргалка по естественным командам для LLM. Это не shell-команды и не slash-команды. Пиши их обычным сообщением в чат, а после первой строки добавляй контекст: пути, feature, slice, task ids, даты, expected output.

## Базовые переключатели режима

1. `новая фича`
2. `занимаемся планированием`
3. `делаем требования`
4. `делаем презентационный прототип`
5. `делаем прототип для разработки`
6. `обновляем прогресс`
7. `финализируем релиз`

## Готовые фразы для ежедневной работы

### Аналитик: планирование

- `новая фича, в папке лежит внешний пакет материалов, сначала сделай intake и не создавай feature раньше времени`
- `новая фича, нужно отделить existing baseline coverage от реальной новой дельты`
- `занимаемся планированием, нужно разложить доработку на planning stories и дать HLE в человеко-днях по AN/FE/BE/QA`
- `занимаемся планированием, собери quarter-plan и commander-plan по текущему scope`
- `разложи фичу на planning stories, отдельно покажи ideal scope и MVP`
- `подготовь HLE, я хочу якорные оценки по stories в разрезе AN/FE/BE/QA для обсуждения с командой`
- `спланируй фичу, зафиксируй допущения, риски и связь источники -> дельта -> planning stories -> срезы`
- `сравни план и факт по фиче, покажи где уже выходим за commander plan`

### Аналитик: требования

- `делаем требования, в папке лежат скрины и change request, нужно сначала собрать общий документ requirements.md`
- `делаем требования в новом формате, обнови root requirements и потом разложи их по коротким срезам с чеклистами для тестировщика`
- `делаем требования в старом формате, сохрани подробную Confluence-style структуру`
- `разложи требования на срезы, проверь полноту срезов под капотом и спроси только если есть противоречия или непроверяемые правила`
- `подготовь детальные требования по срезам, сначала обнови root requirements при найденных пробелах, затем собери slice cards и FE/BE packs`
- `актуализируй требования по feature, проверь влияние на соседние фичи и baseline`
- `проверь хвосты требований по feature, дочисти старые endpoint names, field names и статусы после последней правки`
- `зафиксируй доменное решение и обнови domain-impact`
- `проверь консистентность требований по feature и выпиши что нужно дотянуть`

### Аналитик: прототипы

- `делаем презентационный прототип, сначала найди существующие прототипы и помоги выбрать базу`
- `делаем презентационный прототип, за основу возьми этот экран/скриншот/старый макет`
- `делаем презентационный прототип, нужен один общий кликабельный прототип на всю feature, без комментариев для фронта`
- `актуализируй общий прототип фичи, slice-макеты пока не делай`
- `общий прототип согласован, теперь разложи его по slice prototypes для фронтенда`
- `делаем прототип для разработки, если root prototype не подтверждён, ничего в slices не меняй и скажи что сначала надо окнуть общий прототип`
- `создай прототип среза для фронта, автоматически сверь его с требованиями среза и общим прототипом`

### Прогресс и задачи

- `обновляем прогресс, задача RSCON-2445 завершена вчера`
- `обновляем прогресс, задачу RSCON-2451 взял второй фронтендер позавчера`
- `добавь реальные задачи вместо story STORY-DEPLOYMENTS-003`
- `обнови реальный прогресс и пересобери actual-progress gantt вместе с Confluence-копией`
- `добавь milestone релиза на 2026-04-30`
- `собери puml без инклюдов для actual-progress, чтобы вставить в Confluence`

### Разработчик

- `возьми срез в разработку, собери краткий handoff по требованиям, прототипу и проверкам`
- `разбери срез по коду, найди затронутые модули, API, данные, интеграции, тесты и риски`
- `предложи план реализации, каждая задача должна ссылаться на требование или критерий проверки`
- `начни реализацию, выполняй только согласованный план и остановись при противоречии с требованиями`
- `продолжи реализацию, восстановись с последнего чекпойнта и покажи что уже сделано и что осталось`
- `проверь реализацию среза, сверяй с требованиями, прототипом и обязательными проверками`
- `подготовь к ревью, дай краткую сводку изменений, проверок и рисков`

### Тестировщик

- `подготовь проверки по срезу, собери пользовательские, серверные, интеграционные и технические проверки`
- `собери негативные сценарии по срезу, отдельно пометь предположения`
- `сверь проверки с требованиями, покажи матрицу покрытие требование -> проверки`
- `проверь прототип по срезу, сравни поведение макета с требованиями и проверками`
- `проверь реализацию по срезу, сравни готовое поведение с требованиями, прототипом и test-plan`
- `зафиксируй найденные пробелы, верни вопросы в требования или прототип`

### Релиз и baseline

- `финализируем релиз, собери релизный пакет по вовлечённым фичам`
- `зафиксируй итоговые требования по релизу`
- `закрой consistency backlog по релизу или явно пометь отложенное`
- `промоуть в baseline релиз releases/2026-Q2/rscon-2438`
- `откати решение DEC-2026-04-23-SCORECARDS-001`

## Рекомендуемый шаблон сообщения

```text
делаем презентационный прототип
Feature: `packages`
Материалы: `context/source-materials/change-requests/packages-filtering`
За основу возьми текущий прототип `features/packages/prototype.html` и скрин страницы из current-system.
Нужен один общий кликабельный root prototype на всю feature, как его видит пользователь. Не делай slice-макеты и не добавляй комментарии для фронта внутрь HTML.
```

```text
общий прототип согласован
Feature: `packages`
Отметь в `features/packages/prototype-notes.md`, что общий prototype подтверждён и можно переходить к slice delivery prototypes.
После этого разложи подтверждённый root prototype на slice handoff-макеты для фронтендера.
```

## Короткие правила

- первая строка — команда;
- дальше — конкретный контекст;
- если меняется режим работы, начинай с базового переключателя режима;
- требования всегда сначала правятся в `features/<feature>/requirements.md`, а потом режутся на slices;
- для новых требований можно явно указать формат: `новый лёгкий` или `старый подробный`; если формат не указан, существующая фича сохраняет свой формат, новая фича по умолчанию получает новый лёгкий формат;
- после точечной правки требований сначала делай быструю feature-local дочистку хвостов по старым названиям, endpoint'ам, полям, статусам и option labels, а не полный аудит всего репозитория;
- planning estimates всегда хранятся с разрезом `AN / FE / BE / QA`, а не только общей story-оценкой;
- при работе с прототипами сначала выбирается визуальная база и домучивается общий prototype в корне feature;
- root prototype — это пользовательский кликабельный макет всей feature, без комментариев для фронтенда внутри HTML;
- slice delivery prototypes делаются только после явного подтверждения пользователя и отметки `да` в `prototype-notes.md`;
- если root prototype не подтверждён, ничего в `slices/*/delivery-prototype/*` менять нельзя;
- если изменение влияет на домен или соседние фичи, в том же сообщении проси обновить `domain-impact.md` и `consistency-backlog.md`.
- standalone PlantUML для Confluence собирай отдельным export-файлом, не переписывая include-based gantt source.
- состав команды и допустимые resource lanes смотри в `.workflow/team.md`;
- при actual-progress не начатые задачи раскладывай без перегруза ресурсов выше 100%; если executor не задан или `TBD_*`, ресурс выбирается автоматически по роли/префиксу задачи.
- context summaries, `.research/`, handoff, implementation-plan и test-plan — вспомогательные артефакты; принятые решения нужно переносить в требования, планирование, прототипы, задачи, релиз или baseline.
- пользователь не обязан просить `собери контекст` или `исследуй срез`: обвязка делает это автоматически под предметные команды.

## Полезные синонимы

- Вместо `делаем требования`: `пишем требования`, `соберем требования`, `давай сделаем требования`
- Вместо `разложи требования на срезы`: `разложи по срезам`, `нарежь требования на срезы`, `выдели срезы`
- Вместо `подготовь детальные требования по срезам`: `детализируй срезы`, `подготовь требования по каждому срезу`, `подготовь FE/BE требования`
- Вместо `занимаемся планированием`: `давай планировать`, `переходим в планирование`, `пора планировать`
- Вместо `делаем презентационный прототип`: `нужен демо-макет`, `собери кликабельный макет для заказчика`, `давай общий прототип`
- Вместо `общий прототип согласован`: `прототип согласован`, `можно резать на слайсы`, `разложи общий прототип по слайсам`
- Вместо `создай прототип среза для фронта`: `собери прототип среза для фронтенда`, `подготовь handoff-прототип среза`, `сделай макет среза для фронта`
- Вместо `делаем прототип для разработки`: `собери handoff-макет`, `нужен макет для фронтендера`
- Вместо `обновляем прогресс`: `обновим статус задач`, `зафиксируем факт`, `обнови фактический прогресс`
- Вместо `финализируем релиз`: `готовим релизный пакет`, `закрываем релизный цикл`
- Вместо `новая фича`: `разбери новую фичу`, `сделай feature intake`, `это новая фича`
- Вместо `возьми срез в разработку`: `подготовь срез для разработки`, `начинаем разработку среза`
- Вместо `начни реализацию`: `приступай к реализации`, `выполни первую задачу реализации`, `стартуй реализацию среза`
- Вместо `продолжи реализацию`: `продолжи по плану`, `возобнови реализацию`, `продолжи с последнего чекпойнта`
- Вместо `подготовь к ревью`: `собери summary к ревью`, `подготовь review notes`, `подготовь PR summary`
- Вместо `подготовь проверки по срезу`: `собери тест-дизайн`, `собери QA-проверки по срезу`
- Вместо `проверь прототип по срезу`: `сверь прототип со срезом`, `проверь макет по требованиям среза`
- Вместо `проверь реализацию по срезу`: `проверь готовый срез`, `сверь реализацию с требованиями среза`
- Вместо `зафиксируй найденные пробелы`: `запиши пробелы`, `верни пробелы в требования`, `оформи найденные вопросы`
