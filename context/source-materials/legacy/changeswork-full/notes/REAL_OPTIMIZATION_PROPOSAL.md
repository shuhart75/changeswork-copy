## Решение 2: Полезные skills для разработки

### Skill 1: **spec-sync** (критично)

**Назначение:** Синхронизация изменений между связанными документами

**Когда использовать:**
- Изменили domain_model.md → автоматически обновить data_model.puml, state_machine.puml, GUIDE.md
- Переименовали сущность → обновить везде
- Добавили бизнес-правило → проверить, что оно отражено в state machine

**Что делает:**
```bash
# Пример использования
/spec-sync --source domain_model.md --targets data_model.puml,state_machine.puml,GUIDE.md

# Или автоматически при сохранении
/spec-sync --auto
```

**Экономия времени:** 30-40% на синхронизации документов

---

### Skill 2: **version-manager** (важно)

**Назначение:** Управление версиями спецификации

**Когда использовать:**
- Создать новую версию спецификации (v3.2)
- Сравнить версии (что изменилось между v3.0 и v3.1)
- Откатиться к предыдущей версии

**Что делает:**
```bash
# Создать новую версию
/version-manager create v3.2 --from current

# Сравнить версии
/version-manager diff v3.0 v3.1

# Откатиться
/version-manager rollback v3.0
```

**Экономия времени:** 20-30% на управлении версиями

---

### Skill 3: **prototype-validator** (критично сейчас)

**Назначение:** Проверка и исправление HTML прототипов

**Когда использовать:**
- Прототип не открывается в браузере
- Нужно проверить валидность HTML/JS
- Нужно обновить CDN ссылки

**Что делает:**
```bash
# Проверить прототип
/prototype-validator check prototypes/current.html

# Исправить проблемы
/prototype-validator fix prototypes/current.html

# Обновить CDN
/prototype-validator update-cdn prototypes/current.html
```

**Экономия времени:** Мгновенное решение проблем с прототипом

---

### Skill 4: **task-generator** (полезно)

**Назначение:** Генерация задач разработки из спецификации

**Когда использовать:**
- Добавили новую сущность в domain model → сгенерировать задачи (backend, frontend, tests)
- Изменили state machine → сгенерировать задачи на обновление lifecycle
- Нужен план разработки для новой фичи

**Что делает:**
```bash
# Сгенерировать задачи для новой сущности
/task-generator entity Initiative --output planning/mvp/tasks/

# Сгенерировать задачи для изменения
/task-generator change "Добавить rollback для Deployment" --estimate
```

**Экономия времени:** 40-50% на создании задач

---

### Skill 5: **diagram-updater** (полезно)

**Назначение:** Автоматическое обновление PlantUML диаграмм

**Когда использовать:**
- Добавили таблицу в data model → обновить PlantUML
- Изменили state machine → обновить диаграмму
- Нужно визуализировать изменения

**Что делает:**
```bash
# Обновить data model диаграмму
/diagram-updater data-model --add-table deployment_history

# Обновить state machine
/diagram-updater state-machine Initiative --add-transition "active->archived"

# Сгенерировать PNG из PlantUML
/diagram-updater render spec/data_model.puml
```

**Экономия времени:** 30-40% на работе с диаграммами

## Решение 3: Когда НЕ нужны агенты

**Агенты НЕ нужны для:**
- Синхронизации документации (skill достаточно)
- Управления версиями (skill достаточно)
- Валидации прототипов (skill достаточно)

**Агенты НУЖНЫ для:**
- Реальной разработки кода (когда начнёте писать Django/React)
- Сложных рефакторингов
- Миграции данных
- Интеграционного тестирования

**Вывод:** Сейчас нужны skills, не агенты. Агенты понадобятся позже, когда начнётся кодирование.

## Оценка эффекта

### Без оптимизации (текущее состояние):
- Поиск нужного файла: 2-5 минут
- Синхронизация документов: 30-60 минут
- Управление версиями: 20-40 минут
- Проверка прототипа: 10-20 минут
- **Итого overhead:** 1-2 часа в день

### С оптимизацией:
- Поиск нужного файла: 10-30 секунд (структура)
- Синхронизация документов: 5-10 минут (spec-sync skill)
- Управление версиями: 2-5 минут (version-manager skill)
- Проверка прототипа: 1-2 минуты (prototype-validator skill)
- **Итого overhead:** 10-20 минут в день

**Экономия:** 40-100 минут в день = 7-17 часов в неделю
