### 2. Разделение ролей Approver и Ratifier
**Приоритет:** Критичный
**Статус:** Требует внимания

Важно не забыть про четкое разделение ролей в процессе согласования:

- [ ] **Approver** - роль согласующего на этапах Approval
  - Может согласовывать только на этапах "Согласование"
  - Не может утверждать на этапе Ratification
  - Видит только те Pilot/Change, где назначен согласующим

- [ ] **Ratifier** - роль утверждающего на этапе Ratification
  - Может утверждать только на этапе "Утверждение"
  - Не может согласовывать на этапах Approval
  - Видит только те Pilot/Change/Package, где назначен утверждающим

- [ ] Реализация в Backend (mvp_tasks_approval_process_backend.md)
  - Проверка прав доступа при назначении согласующих
  - Валидация: для Approval этапов только Approver
  - Валидация: для Ratification этапа только Ratifier
  - Нельзя выбрать самого себя при отправке на согласование

- [ ] Реализация в Frontend
  - UI для выбора согласующих с фильтрацией по ролям
  - Отображение только релевантных согласований для каждой роли
  - Раздельные списки для Approver и Ratifier

## Дополнительные заметки

### Множественные роли
Пользователь может иметь несколько ролей одновременно:
- ПРМ и методолог могут быть назначены Approver или Ratifier
- При наличии нескольких ролей пользователь видит объединенный список задач

### Процесс согласования
Двухэтапный процесс:
1. **Approval** (Согласование) - риск-менеджеры с ролью Approver
2. **Ratification** (Утверждение) - руководство с ролью Ratifier
