# Презентация системы управления внедрениями риск-стратегий

**Версия:** 3.1
**Дата:** 2026-03-16
**Для:** Руководство

---

## 1. Краткое резюме

### Что это за система?

**Система управления внедрениями риск-стратегий** — это корпоративное решение для управления полным жизненным циклом изменений кредитных риск-стратегий: от исследования и тестирования до продуктового внедрения.

### Ключевые возможности

- **Управление инициативами** — группировка всей работы по стратегическим изменениям
- **Трёхфазный процесс** — Симуляции → Пилоты → Внедрения
- **Двухэтапное согласование** — Approval (согласование) → Ratification (утверждение)
- **Управление скоркартами** — версионирование и отслеживание источников
- **Визуализация происхождения** — прозрачность цепочки решений
- **Ролевая модель** — разграничение прав доступа по ролям

### Бизнес-ценность

✅ **Прозрачность** — полная история всех изменений и решений
✅ **Контроль** — двухэтапное согласование критичных изменений
✅ **Трассируемость** — отслеживание происхождения от симуляции до внедрения
✅ **Эффективность** — автоматизация процессов согласования
✅ **Безопасность** — ролевая модель с разграничением прав

## 2. Архитектура системы

### Инициатива-центричная архитектура

Система построена вокруг концепции **Инициативы** — верхнеуровневой группировки всей работы по стратегическому изменению.

```
Инициатива (Initiative)
├── Симуляции (0..*)      — Анализ исторических данных
├── Пилоты (0..*)         — Контролируемые эксперименты
└── Внедрения (0..*)      — Продуктовые внедрения
```

### Трёхфазный процесс внедрения

**Фаза 1: Симуляция (Simulation)**
- Анализ исторических данных
- Прогнозирование финансовых эффектов
- Генерация рекомендаций для скоркарт

**Фаза 2: Пилот (Pilot)**
- Контролируемый эксперимент с реальными клиентами
- Сбор метрик и валидация прогнозов
- Требует согласования перед активацией

**Фаза 3: Внедрение (Deployment)**
- Продуктовое внедрение стратегии
- Требует двухэтапного согласования
- Отслеживание конфигурации и результатов

### Визуализация происхождения (Lineage)

Система автоматически отслеживает цепочку происхождения решений:

```
Симуляции → Скоркарты → Пилоты → Скоркарты → Внедрения
```

**Допустимые пути:**
- Полный путь: Симуляция → Пилот → Внедрение
- Быстрый путь: Симуляция → Внедрение (пропуск пилота)
- Прямое внедрение: Внедрение (без предварительного анализа)
- Только исследование: Симуляция (без внедрения)

**Преимущества:**
- Полная прозрачность принятия решений
- Аудит всех изменений
- Понимание источников каждой скоркарты

## 3. Управление скоркартами

### Что такое скоркарта?

**Скоркарта (Scorecard)** — это конфигурация кредитного решения, определяющая правила оценки заёмщиков.

### Ключевые возможности

**Создание скоркарт:**
- Вручную методологом
- Автоматически из результатов симуляций
- Автоматически из результатов пилотов

**Версионирование:**
- Каждое изменение создаёт новую версию
- История всех изменений сохраняется
- Возможность сравнения версий

**Отслеживание источников:**
- Связь с симуляциями-источниками
- Связь с пилотами-источниками
- Полная трассируемость происхождения

**Связи:**
- Многие-ко-многим с пилотами (A/B/C тестирование)
- Многие-ко-многим с внедрениями (мульти-скоркартные стратегии)
- Связь с продуктами и подпродуктами

### Жизненный цикл скоркарты

```
draft → requires_approval → in_approval → in_ratification →
approved → active → archived
```

**Особенности:**
- Скоркарта проходит согласование вместе с пилотами/внедрениями
- Статус синхронизируется со связанными сущностями
- Нельзя удалить скоркарту, используемую в активных пилотах/внедрениях

## 4. Процесс согласования

### Двухэтапное согласование

Система реализует строгий процесс согласования критичных изменений:

**Этап 1: Approval (Согласование)**
- Выполняется риск-менеджерами с ролью Approver
- Переменное количество этапов (0..*)
- Этапы выполняются последовательно
- Внутри этапа согласующие работают параллельно
- Все должны одобрить для перехода на следующий этап
- Один отклонил = процесс завершён с отказом

**Этап 2: Ratification (Утверждение)**
- Выполняется руководством с ролью Ratifier
- Опциональный этап (0..1)
- Ровно один утверждающий принимает решение
- Финальное утверждение перед внедрением

### Гибкая настройка маршрутов

При отправке на согласование пользователь настраивает:
- Количество этапов Approval (0..*)
- Список согласующих для каждого этапа
- Наличие этапа Ratification (да/нет)
- Режим утверждения (автоматический или через очередь)

**Минимальное требование:** хотя бы один этап (Approval или Ratification)

### Пакетное утверждение (Package)

Для эффективного утверждения множества изменений система поддерживает **пакеты**:

**Что такое Package:**
- Группировка пилотов и внедрений для совместного утверждения
- Создаётся только при отправке на Ratification (нет черновиков)
- Может содержать элементы из разных инициатив
- Поддерживает частичное утверждение (каждый элемент решается отдельно)

**Преимущества:**
- Утверждение нескольких изменений за один раз
- Экономия времени руководства
- Гибкость: можно утвердить часть, отклонить часть
- Критичность пакета = максимальная критичность элементов

**Workflow:**
1. ПРМ/Методолог формирует пакет из элементов в очереди ожидания
2. Выбирает утверждающего (Ratifier)
3. Пакет отправляется на утверждение
4. Ratifier принимает решение по каждому элементу отдельно или по всем сразу
5. История решений сохраняется для каждого элемента

### Email-уведомления

Система автоматически отправляет уведомления при ключевых событиях:
- Отправка на согласование → уведомление согласующим
- Принятие решения → уведомление создателю
- Переход на следующий этап → уведомление следующим согласующим
- Комментарии согласующих → уведомление заинтересованным сторонам

## 5. Ролевая модель и права доступа

### Роли в системе

**ПРМ (Product Risk Manager)**
- Управляет инициативами своего продукта
- Создаёт и редактирует симуляции, пилоты, внедрения
- Отправляет на согласование
- Может быть назначен согласующим/утверждающим
- Привязан к конкретному продукту

**Методолог (Methodologist)**
- Управляет скоркартами и шаблонами
- Создаёт и редактирует скоркарты
- Управляет артефактами всех сущностей
- Может быть назначен согласующим/утверждающим
- Глобальная роль (все продукты)

**Approver (Согласующий)**
- Согласует изменения на этапах Approval
- Видит все сущности системы (только чтение)
- В разделе "Согласования" видит только свои назначения
- Глобальная роль

**Ratifier (Утверждающий)**
- Утверждает изменения на этапе Ratification
- Видит все сущности системы (только чтение)
- В разделе "Согласования" видит только свои назначения
- Глобальная роль

**Администратор (Admin)**
- Полный доступ ко всем функциям
- Управление пользователями и ролями
- Глобальная роль

### Множественные роли

Пользователь может иметь несколько ролей одновременно:
- ПРМ продукта A + Approver
- Методолог + Ratifier
- ПРМ продукта A + ПРМ продукта B

**Правила совмещения:**
✅ Разрешено: (ПРМ ИЛИ Методолог ИЛИ Админ) + Approver + Ratifier
❌ Запрещено: ПРМ + Методолог (конфликт интересов)

### Матрица прав доступа

| Действие | ПРМ (свой продукт) | Методолог | Approver | Ratifier | Админ |
|----------|-------------------|-----------|----------|----------|-------|
| Просмотр всех сущностей | ✓ | ✓ | ✓ (только чтение) | ✓ (только чтение) | ✓ |
| Создание инициативы | ✓ | ✗ | ✗ | ✗ | ✓ |
| Управление симуляциями/пилотами/внедрениями | ✓ | ✗ | ✗ | ✗ | ✓ |
| Создание/редактирование скоркарт | ✗ | ✓ | ✗ | ✗ | ✓ |
| Управление артефактами (своих сущностей) | ✓ | ✓ (всех) | ✗ | ✗ | ✓ |
| Отправка на согласование | ✓ | ✓ | ✗ | ✗ | ✓ |
| Формирование пакетов | ✓ | ✓ | ✗ | ✗ | ✓ |
| Просмотр раздела "Согласования" | ✓ (свои) | ✓ (свои) | ✓ (свои) | ✓ (свои) | ✓ (все) |
| Согласование (Approval) | ✓ (если назначен) | ✓ (если назначен) | ✓ | ✗ | ✓ |
| Утверждение (Ratification) | ✓ (если назначен) | ✓ (если назначен) | ✗ | ✓ | ✓ |
| Архивирование инициатив | ✓ | ✗ | ✗ | ✗ | ✓ |
| Управление пользователями и ролями | ✗ | ✗ | ✗ | ✗ | ✓ |

**Ключевые принципы:**
- Разделение обязанностей: создание ≠ согласование ≠ утверждение
- Прозрачность: Approver/Ratifier видят все сущности для контекста
- Фокус: в разделе "Согласования" видны только свои назначения
- Гибкость: ПРМ/Методолог могут быть назначены согласующими

## 6. Жизненные циклы сущностей

### Инициатива (Initiative)

**Жизненный цикл:** draft → active → deployed → archived

**Особенности:**
- Статус рассчитывается автоматически на основе вложенных сущностей
- **draft:** пустая ИЛИ все вложенные сущности в draft
- **active:** хотя бы одна вложенная сущность не в draft
- **deployed:** хотя бы одно внедрение в статусе deployed
- **archived:** переводится вручную

**Правила архивирования:**
- Нельзя архивировать, если есть deployed внедрения
- Нельзя архивировать, если есть неархивированные вложенные сущности
- НЕТ каскадного архивирования (вложенные сущности не архивируются автоматически)

### Симуляция (Simulation)

**Жизненный цикл:** draft → running → completed/failed → archived

**Особенности:**
- Анализ исторических данных
- Может быть источником для скоркарт
- Не требует согласования
- Нельзя изменить после завершения
- Архивируется вручную

### Пилот (Pilot)

**Жизненный цикл:** draft → requires_activation → active → completed → archived

**Дополнительные статусы:**
- **approval_rejected:** отклонён на этапе Approval (можно редактировать и переотправить)
- **ratification_rejected:** отклонён на этапе Ratification (можно редактировать и переотправить)

**Особенности:**
- Требует согласования перед активацией (статус requires_activation)
- Связан минимум с одной скоркартой
- Может быть источником для скоркарт
- Версионирование: любые изменения создают новую версию
- Архивируется вручную

### Внедрение (Deployment)

**Жизненный цикл:** draft → requires_approval → approved → deployed

**Дополнительные статусы:**
- **approval_rejected:** отклонён на этапе Approval (можно редактировать и переотправить)
- **ratification_rejected:** отклонён на этапе Ratification (можно редактировать и переотправить)

**Особенности:**
- Требует двухэтапного согласования (статус requires_approval)
- Связано минимум с одной скоркартой
- Только одно активное внедрение (deployed) на сущность
- Версионирование: любые изменения создают новую версию
- Нельзя удалить после развёртывания

## 7. Ключевые функциональные возможности

### Управление артефактами

**Что такое артефакты:**
- Ссылки на внешние документы (URL)
- НЕ загрузка файлов в систему
- Привязка к инициативам, симуляциям, пилотам, внедрениям

**Типы артефактов:**
- Business Case — бизнес-обоснование
- ROV (Return on Value) — расчёт эффективности
- Other — прочие документы

**Управление:**
- ПРМ управляет артефактами своих сущностей
- Методолог управляет артефактами всех сущностей
- Валидация формата URL
- Метаданные: название, URL, тип, автор, дата

### Отслеживание связей

**Связи через скоркарты:**
- Пилот ↔ Скоркарты (многие-ко-многим)
- Внедрение ↔ Скоркарты (многие-ко-многим)
- Скоркарта ↔ Источники (Симуляции/Пилоты)

**Наследование связей:**
- Внедрение наследует связи с симуляциями через скоркарты
- Внедрение наследует связи с пилотами через скоркарты
- Автоматическое создание записей в таблицах источников

**Визуализация Lineage:**
- Автоматическое построение графа происхождения
- Показывает путь: Симуляции → Пилоты → Внедрения
- Только чтение (не CRUD-сущность)
- API endpoints для получения данных визуализации

### Связи с продуктами

**Иерархия:**
- Product (Кредитный продукт)
- SubProduct (Вариант продукта)

**Правила:**
- Инициатива принадлежит одному Product
- Инициатива может быть связана с несколькими SubProduct
- Скоркарта может относиться к Product в целом или к конкретным SubProduct
- Если подпродукты не указаны, сущность относится ко всем подпродуктам Product
- Пилоты и внедрения наследуют связи с подпродуктами через скоркарты

## 8. Типичные сценарии использования

### Сценарий 1: Полный цикл внедрения с пилотом

**Шаг 1: Создание инициативы**
- ПРМ создаёт инициативу для нового продукта
- Указывает продукт и подпродукты
- Прикрепляет бизнес-обоснование (артефакт)

**Шаг 2: Симуляция**
- ПРМ создаёт симуляцию на исторических данных
- Запускает анализ (статус: running)
- Получает результаты и прогнозы (статус: completed)

**Шаг 3: Создание скоркарты**
- Методолог создаёт скоркарту из результатов симуляции
- Система автоматически отслеживает источник (simulation_id)
- Скоркарта получает статус draft

**Шаг 4: Пилот**
- ПРМ создаёт пилот, выбирает скоркарту
- Указывает размеры пилотной и контрольной групп
- Отправляет на согласование (статус: requires_activation)
- Настраивает маршрут: 2 этапа Approval + Ratification
- Выбирает согласующих для каждого этапа

**Шаг 5: Согласование пилота**
- Approver этапа 1 согласуют (все должны одобрить)
- Approver этапа 2 согласуют (все должны одобрить)
- Автоматический переход на Ratification
- Ratifier утверждает
- Пилот получает статус approved

**Шаг 6: Активация пилота**
- ПРМ активирует пилот (статус: active)
- Проводится эксперимент
- Собираются метрики
- Пилот завершается (статус: completed)

**Шаг 7: Внедрение**
- ПРМ создаёт внедрение, выбирает скоркарту из пилота
- Система автоматически связывает внедрение с пилотом через скоркарту
- Указывает критичность и плановую дату
- Отправляет на согласование (статус: requires_approval)

**Шаг 8: Согласование внедрения**
- Проходит этапы Approval
- Попадает в очередь ожидания утверждения (awaiting_ratification)
- ПРМ формирует Package из нескольких внедрений
- Ratifier утверждает пакет (может частично)
- Внедрение получает статус approved

**Шаг 9: Развёртывание**
- ПРМ развёртывает внедрение (статус: deployed)
- Инициатива автоматически получает статус deployed
- Отслеживаются результаты

**Результат:**
- Полная трассируемость: Симуляция → Скоркарта → Пилот → Скоркарта → Внедрение
- История всех согласований сохранена
- Визуализация Lineage доступна

### Сценарий 2: Быстрое внедрение без пилота

**Когда используется:**
- Низкорисковые изменения
- Срочные внедрения
- Изменения, уже протестированные на других продуктах

**Процесс:**
1. ПРМ создаёт инициативу
2. Создаёт симуляцию и получает результаты
3. Методолог создаёт скоркарту из симуляции
4. ПРМ создаёт внедрение напрямую (пропуск пилота)
5. Отправляет на согласование
6. После утверждения развёртывает

**Преимущества:**
- Экономия времени (нет фазы пилота)
- Сохранение трассируемости (Симуляция → Скоркарта → Внедрение)
- Полный контроль через согласование

### Сценарий 3: Пакетное утверждение

**Когда используется:**
- Множество внедрений готовы к утверждению
- Оптимизация времени руководства
- Связанные изменения

**Процесс:**
1. Несколько пилотов/внедрений прошли этапы Approval
2. Все попали в очередь ожидания утверждения (awaiting_ratification)
3. ПРМ/Методолог формирует Package из 5-10 элементов
4. Выбирает Ratifier
5. Отправляет пакет на утверждение
6. Ratifier просматривает все элементы
7. Принимает решение: по всем сразу или по каждому отдельно
8. Каждый элемент получает свой статус (ratified/ratification_rejected)

**Преимущества:**
- Эффективное использование времени руководства
- Гибкость: частичное утверждение
- Сохранение истории по каждому элементу
- Критичность пакета = максимальная критичность элементов

### Сценарий 4: Отклонение и повторная отправка

**Ситуация:** Внедрение отклонено на этапе Approval

**Процесс:**
1. Approver отклоняет с комментарием
2. Внедрение получает статус approval_rejected
3. ПРМ видит комментарий и причину отклонения
4. ПРМ редактирует внедрение (статус остаётся approval_rejected)
5. ПРМ повторно отправляет на согласование
6. Создаётся новый ApprovalInstance
7. Процесс согласования начинается заново

**Альтернатива:** Архивирование
- Если изменения невозможны или нецелесообразны
- ПРМ может архивировать внедрение в статусе approval_rejected

## 9. Бизнес-ценность и преимущества

### Прозрачность и контроль

**Полная история решений:**
- Каждое изменение отслеживается от идеи до внедрения
- История всех согласований сохраняется
- Комментарии и обоснования доступны для аудита
- Визуализация Lineage показывает происхождение каждого решения

**Контроль рисков:**
- Двухэтапное согласование критичных изменений
- Разделение ролей: создание ≠ согласование ≠ утверждение
- Невозможно обойти процесс согласования
- Автоматические уведомления предотвращают задержки

### Эффективность процессов

**Автоматизация:**
- Автоматическое отслеживание связей через скоркарты
- Автоматические email-уведомления
- Автоматический расчёт статусов инициатив
- Автоматическое построение визуализации Lineage

**Оптимизация времени:**
- Пакетное утверждение множества изменений
- Параллельное согласование внутри этапов
- Гибкая настройка маршрутов (не нужно проходить все этапы)
- Быстрый путь для низкорисковых изменений (пропуск пилота)

**Снижение ошибок:**
- Валидация обязательных полей
- Невозможно удалить используемые скоркарты
- Невозможно развернуть без утверждения
- Версионирование предотвращает потерю данных

### Трассируемость и аудит

**Для регуляторов:**
- Полная история всех изменений риск-стратегий
- Обоснование каждого решения
- Список всех согласующих и утверждающих
- Временные метки всех действий

**Для внутреннего аудита:**
- Соответствие процессов корпоративным политикам
- Разделение обязанностей
- Отслеживание источников скоркарт
- История версий

**Для бизнеса:**
- Понимание эффективности изменений
- Анализ времени прохождения согласований
- Выявление узких мест в процессах
- Метрики по продуктам и инициативам

### Гибкость и масштабируемость

**Гибкая настройка процессов:**
- Переменное количество этапов согласования
- Выбор согласующих при отправке (не предзаданные маршруты)
- Опциональное утверждение руководством
- Автоматическое или ручное утверждение

**Поддержка разных сценариев:**
- Полный цикл: Симуляция → Пилот → Внедрение
- Быстрый путь: Симуляция → Внедрение
- Прямое внедрение: без предварительного анализа
- A/B/C тестирование: множество скоркарт в одном пилоте

**Масштабируемость:**
- Поддержка множества продуктов
- Множество инициатив параллельно
- Множественные роли у пользователей
- Пакетное утверждение для больших объёмов

**Domain-Driven Design:**
- Ограниченные контексты (Bounded Contexts)
- Корневые агрегаты (Aggregate Roots)
- Единый язык (Ubiquitous Language)
- Бизнес-правила в домене

**Безопасность:**
- Ролевая модель с разграничением прав
- Middleware для проверки доступа

- Lazy loading для больших списков

**Качество кода:**
- Test coverage > 80% (Backend), > 70% (Frontend)
- E2E тесты для критичных сценариев
- Code review обязателен
- Автоматизированное тестирование в CI/CD

### Ограниченные контексты (Bounded Contexts)

Система разделена на 9 ограниченных контекстов:

1. **Идентификация и доступ** — пользователи, роли, права
2. **Каталог продуктов** — Product/SubProduct
3. **Шаблоны скоркарт** — версионированные шаблоны
4. **Скоркарты** — экземпляры скоркарт с источниками
5. **Инициативы** — группировка верхнего уровня
6. **Исследования и выполнение** — Simulation/Pilot/Deployment
7. **Lineage** — визуализация происхождения
8. **Пакеты** — группировка для утверждения
9. **Согласование** — процесс Approval/Ratification

**Преимущества:**
- Чёткое разделение ответственности
- Независимая разработка контекстов
- Упрощение тестирования
- Возможность масштабирования

## Приложение A: Глоссарий терминов

**Инициатива (Initiative)**
Верхнеуровневая группировка всей работы по стратегическому изменению. Содержит симуляции, пилоты и внедрения.

**Симуляция (Simulation)**
Анализ исторических данных для прогнозирования эффектов изменения стратегии.

**Пилот (Pilot)**
Контролируемый эксперимент с реальными клиентами для валидации прогнозов.

**Внедрение (Deployment)**
Продуктовое внедрение модификации стратегии в production.

**Скоркарта (Scorecard)**
Конфигурация кредитного решения, определяющая правила оценки заёмщиков.

**Lineage (Происхождение)**
Визуализация цепочки: Симуляции → Пилоты → Внедрения через связи со скоркартами.

**Package (Пакет)**
Группировка пилотов и внедрений для совместного утверждения руководством.

**Approval (Согласование)**
Первый этап процесса согласования, выполняется риск-менеджерами с ролью Approver.

**Ratification (Утверждение)**
Второй этап процесса согласования, выполняется руководством с ролью Ratifier.

**ПРМ (Product Risk Manager)**
Роль пользователя, управляющего инициативами своего продукта.

**Методолог (Methodologist)**
Роль пользователя, управляющего скоркартами и методологическими документами.

**Approver (Согласующий)**
Роль пользователя, согласующего изменения на этапах Approval.

**Ratifier (Утверждающий)**
Роль пользователя, утверждающего изменения на этапе Ratification.

**Артефакт (Artifact)**
Ссылка на внешний документ (URL), прикреплённая к сущности системы.
