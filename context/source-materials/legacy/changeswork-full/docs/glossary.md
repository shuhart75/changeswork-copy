# Система управления внедрениями риск-стратегий
## Глоссарий терминов

---

## Основные сущности

### Инициатива (Initiative)
**Определение:** Верхнеуровневая группировка всей работы по стратегическому изменению кредитной риск-стратегии.

**Назначение:** Объединяет все связанные симуляции, пилоты, внедрения и скоркарты в единый контекст.

**Пример:** "Инициатива по внедрению новой скоринговой модели для Premium сегмента кредитных карт"

**Жизненный цикл:**
- **draft** — черновик, работа только началась
- **active** — активная работа, есть запущенные симуляции/пилоты
- **deployed** — хотя бы одно внедрение развернуто в продакшен
- **archived** — работа завершена и архивирована

**Ключевые характеристики:**
- Принадлежит одному продукту
- Может быть связано с несколькими подпродуктами
- Статус рассчитывается автоматически на основе вложенных сущностей
- Содержит описание, автора, даты создания и обновления

---

### Симуляция (Simulation)
**Определение:** Анализ влияния изменений риск-стратегии на исторических данных.

**Назначение:** Прогнозирование финансовых эффектов до запуска реальных экспериментов.

**Пример:** "Симуляция изменения весов факторов на данных 2025 года"

**Жизненный цикл:**
- **draft** — настройка параметров
- **running** — симуляция выполняется
- **completed** — симуляция завершена успешно
- **failed** — симуляция завершена с ошибкой

**Ключевые характеристики:**
- Период исторических данных (от и до)
- Режим симуляции (стандартный, ускоренный, детальный)
- Результаты: AR(абс), Объём выдач, NPV, RC
- Может быть источником для создания скоркарт

**Связи:**
- Принадлежит одной Инициативе
- Может быть связана с несколькими Пилотами (через скоркарты)
- Может быть связана с несколькими Внедрениями (через скоркарты)

---

### Пилот (Pilot)
**Определение:** Контролируемый эксперимент с реальными клиентами для валидации изменений риск-стратегии.

**Назначение:** Проверка гипотез на реальных данных перед полным внедрением.

**Пример:** "A/B тест новой скоринговой модели на 10% трафика"

**Жизненный цикл:**
- **draft** — настройка параметров
- **requires_activation** — готов к запуску, требует согласования
- **in_approval** — на этапе согласования
- **in_ratification** — на этапе утверждения
- **approved** — утвержден, готов к активации
- **active** — пилот запущен и работает
- **requires_correction** — требует корректировки параметров
- **completed** — пилот успешно завершен
- **rejected** — отклонен на этапе согласования

**Ключевые характеристики:**
- Связан с одной или несколькими скоркартами (для A/B/C тестов)
- Параметры: доля трафика, длительность, целевые метрики
- Результаты: фактические метрики по итогам пилота
- Может требовать корректировки в процессе

**Связи:**
- Принадлежит одной Инициативе
- Связан с несколькими Симуляциями (опционально)
- Связан с одной или несколькими Скоркартами (обязательно)
- Может быть связан с несколькими Внедрениями (через скоркарты)

---

### Внедрение (Deployment)
**Определение:** Продуктовое внедрение модификации риск-стратегии в промышленную эксплуатацию.

**Назначение:** Финальный этап — развертывание скоркарты в продакшен.

**Пример:** "Внедрение новой скоринговой модели Premium v3.0 на 100% трафика"

**Жизненный цикл:**
- **draft** — подготовка к внедрению
- **requires_approval** — готово к согласованию
- **in_approval** — на этапе согласования
- **in_ratification** — на этапе утверждения
- **approved** — утверждено, готово к внедрению
- **deployed** — внедрено в продакшен
- **rolled_back** — откачено из-за проблем

**Ключевые характеристики:**
- Связано с одной или несколькими скоркартами
- Плановая дата внедрения
- Фактическая дата внедрения
- Дата отката (если применимо)
- Только одно активное внедрение на изменение (без отката)

**Связи:**
- Принадлежит одной Инициативе
- Связано с несколькими Симуляциями (через скоркарты)
- Связано с несколькими Пилотами (через скоркарты)
- Связано с одной или несколькими Скоркартами (обязательно)

---

### Скоркарта (Scorecard)
**Определение:** Скоринговая карта кредитного решения с конфигурацией, версионированием и отслеживанием источников.

**Назначение:** Конкретная конфигурация модели принятия решений, которая используется в пилотах и внедрениях.

**Пример:** "Скоркарта Premium v2.3 — улучшенная модель с новыми факторами"

**Статусы:**
- **draft** — черновик, в разработке
- **in_approval** — на согласовании
- **approved** — утверждена, готова к использованию
- **active** — используется в пилоте или продакшене
- **archived** — архивирована, больше не используется

**Ключевые характеристики:**
- Версионирование (v1, v2, v3...)
- Критичность (low, medium, high, critical)
- Источник создания (симуляция, пилот, ручное создание)
- Финансовые эффекты (прогнозные или фактические)
- Связь с продуктами и подпродуктами

**Связи:**
- Может быть связана с несколькими Пилотами (многие-ко-многим)
- Может быть связана с несколькими Внедрениями (многие-ко-многим)
- Имеет источники (Симуляции, Пилоты, или ручное создание)

---

### Цепочка (Chain)
**Определение:** Концепция, связывающая этапы жизненного цикла одного стратегического изменения.

**Назначение:** Представление полного пути от симуляций до продуктового внедрения.

**Структура:**
```
Симуляции (0..*) → Пилоты (0..*) → Изменение (1)
```

**Допустимые варианты:**
1. Только симуляции (исследование без внедрения)
2. Симуляции + Изменение (пропуск пилота)
3. Симуляции + Пилот (эксперимент без внедрения)
4. Симуляции + Пилот + Изменение (полный цикл)
5. Пилот + Изменение (пропуск симуляций)
6. Только Изменение (прямое внедрение)

**Ключевые характеристики:**
- Цепочка должна иметь ровно ОДНО Изменение
- Цепочка может иметь НЕСКОЛЬКО Пилотов
- Цепочка может иметь НЕСКОЛЬКО Симуляций
- Все сущности должны принадлежать одному Внедрению
- Приоритет цепочки определяет порядок обработки

**Связи:**
- Принадлежит одному Внедрению
- Связь Пилотов с Изменением реализуется через общие Скоркарты

---

### Пакет (Package)
**Определение:** Группа цепочек с одинаковым маршрутом согласования для координированного утверждения.

**Назначение:** Пакетное согласование связанных изменений, которые должны быть внедрены синхронно.

**Пример:** "Комплексное изменение стратегии Q1 2026" (включает изменения для 3 продуктов)

**Ключевые характеристики:**
- Все цепочки в пакете имеют одинаковый маршрут согласования
- Критичность пакета = максимальная критичность среди всех цепочек
- Может быть отправлен на согласование как единое целое
- Согласующие видят все изменения в контексте

**Связи:**
- Содержит несколько Цепочек
- Цепочки могут принадлежать разным Внедрениям

---

## Процессы

### Согласование (Approval)
**Определение:** Первый этап двухэтапного процесса согласования изменений.

**Участники:** Риск-менеджеры, аналитики, эксперты предметной области.

**Назначение:** Проверка корректности анализа, методологии, результатов.

**Возможные решения:**
- **Одобрить** — переход на этап утверждения
- **Отклонить** — возврат на доработку с указанием причин
- **Запросить доработки** — требуются изменения перед повторным согласованием

**Применяется к:**
- Пилотам (когда status = requires_activation или requires_correction)
- Изменениям (когда status = requires_approval)

---

### Утверждение (Ratification)
**Определение:** Второй этап двухэтапного процесса согласования изменений.

**Участники:** Руководство (CRO, Head of Risk, другие топ-менеджеры).

**Назначение:** Финальное решение о внедрении на основе бизнес-рисков и стратегии.

**Возможные решения:**
- **Утвердить** — разрешение на активацию/внедрение
- **Отклонить** — запрет на внедрение с указанием причин

**Применяется к:**
- Пилотам (после успешного согласования)
- Изменениям (после успешного согласования)

---

### Маршрутизация (Routing)
**Определение:** Автоматическое определение согласующих и утверждающих на основе параметров изменения.

**Факторы маршрутизации:**
- Продукт и подпродукт
- Критичность изменения
- Тип изменения (пилот, продуктовое внедрение)
- Организационная структура

**Результат:** Список согласующих и утверждающих с порядком прохождения.

---

### Откат (Rollback)
**Определение:** Возврат к предыдущей версии скоркарты при обнаружении проблем после внедрения.

**Триггеры:**
- Метрики отклоняются от прогноза
- Обнаружены технические проблемы
- Регуляторные требования
- Решение руководства

**Процесс:**
1. Инициация отката с указанием причины
2. Автоматическое возвращение предыдущей версии
3. Фиксация даты и причины отката
4. Уведомление всех заинтересованных
5. Анализ причин для будущих улучшений

---

## Метрики и эффекты

### AR(абс) — Absolute Approval Rate
**Определение:** Абсолютное изменение уровня одобрения заявок.

**Единица измерения:** Рубли или процентные пункты.

**Пример:** "+2.3M" означает увеличение дохода на 2.3 миллиона рублей.

**Использование:** Ключевая метрика для оценки финансового эффекта изменений.

---

### Объём выдач (Volume)
**Определение:** Изменение объема выданных кредитов.

**Единица измерения:** Проценты или абсолютные значения.

**Пример:** "+15%" означает увеличение объема выдач на 15%.

**Использование:** Оценка влияния на масштаб бизнеса.

---

### NPV — Net Present Value
**Определение:** Чистая приведенная стоимость будущих денежных потоков.

**Единица измерения:** Рубли.

**Пример:** "+5.2M" означает увеличение NPV на 5.2 миллиона рублей.

**Использование:** Оценка долгосрочной финансовой ценности изменения.

---

### RC — Risk Cost
**Определение:** Стоимость риска (ожидаемые потери от дефолтов).

**Единица измерения:** Проценты или рубли.

**Пример:** "-0.3%" означает снижение стоимости риска на 0.3 процентных пункта.

**Использование:** Оценка влияния на уровень риска портфеля.

---

### Approval Rate
**Определение:** Доля одобренных заявок от общего числа заявок.

**Единица измерения:** Проценты.

**Пример:** "68%" означает, что 68% заявок одобрены.

**Использование:** Ключевая операционная метрика кредитного процесса.

---

### Default Rate
**Определение:** Доля заемщиков, допустивших дефолт.

**Единица измерения:** Проценты.

**Пример:** "3.2%" означает, что 3.2% заемщиков в дефолте.

**Использование:** Ключевая метрика кредитного риска.

---

## Атрибуты и характеристики

### Критичность (Criticality)
**Определение:** Оценка важности и влияния изменения на бизнес.

**Уровни:**
- **Low** — низкая критичность, минимальное влияние
- **Medium** — средняя критичность, умеренное влияние
- **High** — высокая критичность, значительное влияние
- **Critical** — критичная, максимальное влияние на бизнес

**Факторы оценки:**
- Объем затрагиваемых клиентов
- Финансовое влияние
- Риски при неудаче
- Регуляторные требования

**Использование:** Определяет приоритет и маршрут согласования.

---

### Приоритет (Priority)
**Определение:** Порядок обработки изменений при наличии нескольких параллельных работ.

**Уровни:**
- **Low** — низкий приоритет, можно отложить
- **Medium** — средний приоритет, стандартная обработка
- **High** — высокий приоритет, требует ускоренной обработки

**Использование:** Управление очередью работ и распределение ресурсов.

---

### Источник (Source)
**Определение:** Происхождение скоркарты.

**Типы источников:**
- **Simulation** — создана на основе результатов симуляции
- **Pilot** — создана на основе результатов пилота
- **Manual** — создана вручную (регуляторные требования, срочные фиксы)

**Использование:** Трейсабилити и понимание обоснования изменений.

---

### Версия (Version)
**Определение:** Номер версии скоркарты в рамках одного шаблона.

**Формат:** Целое число (1, 2, 3...)

**Правила:**
- Каждое изменение скоркарты создает новую версию
- Версии не удаляются, только архивируются
- В продакшене может быть только одна активная версия

**Использование:** Отслеживание эволюции скоркарты, возможность отката.

## Роли пользователей

### Риск-менеджер (Risk Manager)
**Ответственность:**
- Создание и управление внедрениями
- Принятие решений о запуске пилотов
- Управление процессом согласования
- Мониторинг результатов

**Права:**
- Создание/редактирование внедрений, пилотов, изменений
- Отправка на согласование
- Просмотр всех данных по своим продуктам

---

### Аналитик данных (Data Analyst)
**Ответственность:**
- Настройка и запуск симуляций
- Анализ результатов
- Создание скоркарт на основе симуляций
- Прогнозирование эффектов

**Права:**
- Создание/редактирование симуляций
- Создание скоркарт
- Просмотр результатов симуляций

---

### Продуктовый менеджер (Product Manager)
**Ответственность:**
- Управление пилотами
- Мониторинг A/B тестов
- Принятие решений о масштабировании
- Координация с разработкой

**Права:**
- Создание/редактирование пилотов
- Просмотр результатов пилотов
- Активация/остановка пилотов

---

### Согласующий (Approver)
**Ответственность:**
- Проверка корректности анализа
- Оценка методологии
- Согласование или отклонение изменений
- Запрос доработок при необходимости

**Права:**
- Просмотр изменений на согласовании
- Одобрение/отклонение
- Добавление комментариев

---

### Утверждающий (Ratifier)
**Ответственность:**
- Финальное решение о внедрении
- Оценка бизнес-рисков
- Соответствие стратегии компании
- Утверждение или отклонение

**Права:**
- Просмотр изменений на утверждении
- Утверждение/отклонение
- Просмотр полной истории

---

### Администратор (Administrator)
**Ответственность:**
- Управление пользователями и правами
- Настройка маршрутов согласования
- Управление справочниками
- Техническая поддержка

**Права:**
- Полный доступ ко всем функциям
- Управление конфигурацией системы
- Просмотр аудит-логов

---

## Технические термины

### Трейсабилити (Traceability)
**Определение:** Возможность отследить полную историю изменения от идеи до продакшена.

**Включает:**
- Связи между сущностями
- Историю решений
- Обоснования изменений
- Результаты на каждом этапе

---

### Аудит-лог (Audit Log)
**Определение:** Журнал всех действий пользователей в системе.

**Содержит:**
- Кто выполнил действие
- Что было сделано
- Когда это произошло
- Какие данные были изменены

---

### Маппинг (Mapping)
**Определение:** Связывание сущностей между собой.

**Примеры:**
- Маппинг скоркарт на пилоты
- Маппинг симуляций на изменения
- Маппинг продуктов на подпродукты

---

### Webhook
**Определение:** Механизм автоматического уведомления системы о событиях во внешних системах.

**Использование:**
- Уведомление о завершении симуляции
- Обновление статуса пилота
- Подтверждение деплоя скоркарты

---

## Сокращения

- **AR** — Approval Rate (уровень одобрения)
- **NPV** — Net Present Value (чистая приведенная стоимость)
- **RC** — Risk Cost (стоимость риска)
- **CRO** — Chief Risk Officer (директор по рискам)
- **A/B/C тест** — Эксперимент с несколькими вариантами
- **MVP** — Minimum Viable Product (минимально жизнеспособный продукт)
- **ROI** — Return on Investment (возврат инвестиций)
- **SLA** — Service Level Agreement (соглашение об уровне обслуживания)
- **API** — Application Programming Interface (программный интерфейс)
- **UI** — User Interface (пользовательский интерфейс)
- **FTE** — Full-Time Equivalent (эквивалент полной занятости)
