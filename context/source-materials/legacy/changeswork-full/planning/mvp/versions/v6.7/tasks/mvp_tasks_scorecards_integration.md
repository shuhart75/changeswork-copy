# Задачи: Интеграция скоркарт (hook-места для статусов и "использования")

**Источник правды:** [`spec/domain_model.md`](/home/reutov/Documents/AI/changesWork/spec/domain_model.md) → "Жизненный цикл скоркарты"

## Обзор

**Компонент:** Интеграция `Scorecard`/`ScorecardVersion` с процессами Pilot/Deployment + Approval/Ratification  
**Статус:** Новая разработка (MVP v6.7)

### Зачем нужна задача

В MVP скоркарта "проходит согласование" **не самостоятельно**, а вместе с Pilot/Deployment, к которым она прикреплена. Поэтому нужно:
- определить правила вычисления/синхронизации статуса скоркарты
- добавить hook-места (event handlers), которые будут пересчитывать/обновлять статус при изменениях ЖЦ PilotVersion/DeploymentVersion и при событиях ApprovalInstance

## BE-SC_INT1: Backend интеграции скоркарт (hook-места)

**Тип:** Story  
**Приоритет:** Высокий  
**Оценка:** 2 дня (как в Gantt v6)

### Summary
Добавить минимально достаточную реализацию для синхронизации статуса скоркарты с наиболее продвинутым этапом среди связанных PilotVersion/DeploymentVersion (как в spec), и сделать это доступным для списков/деталок скоркарт.

### Scope (MVP, минимально)

1. **Единая функция вычисления статуса скоркарты**

- Реализовать доменный helper (условно):
  - `compute_scorecard_status(scorecard_id) -> status`
  - статус вычисляется как "наиболее продвинутый" среди всех связанных сущностей (PilotVersion/DeploymentVersion), которые используют скоркарту.
- Зафиксировать порядок статусов (как в spec):
  - `draft` → `requires_approval` → `in_approval` → `in_ratification` → `approved` → `active` → `archived`
  - rejected-статусы: `approval_rejected` / `ratification_rejected` (если отражаем на скоркарте)

2. **Hook-места пересчёта**

- Добавить вызовы пересчёта статуса скоркарты в точках:
  - `PilotVersion/DeploymentVersion` submit_for_approval
  - завершение ApprovalInstance (approved/rejected/cancelled)
  - публикация новой версии пилота (`active`) / deploy внедрения (`deployed`) для перевода скоркарты в `active`
  - rollback / завершение пилота (возможный пересчет к `archived`, если скоркарта больше не используется)

3. **Доступность статуса в API скоркарт**

- Убедиться, что вычисленный/синхронизированный статус доступен:
  - в `GET /api/scorecards/` (список)
  - в `GET /api/scorecards/{id}/` (деталка)

> Техническое допущение для MVP (чтобы уложиться в 2 дня): допускается вычислять статус "на лету" (read-time) при выдаче списка/деталки, без сложных фоновых джоб и без оптимизации на большие объёмы.

### Acceptance Criteria

- [ ] Зафиксирован порядок статусов скоркарты (как в spec) и реализована функция вычисления статуса
- [ ] Есть hook-места, которые вызываются из `BE-INT1`/ЖЦ сущностей и обеспечивают консистентность
- [ ] Статус присутствует в API списка и деталки скоркарт
- [ ] Минимальные тесты на вычисление статуса по комбинациям "используется в пилоте/внедрении"

### Dependencies

- `BE_INT1` (интеграция ЖЦ с ApprovalInstance)
- `BE_PL1`, `BE_CL1` (жизненные циклы версий Pilot/Deployment)
- `BE_SC1`/`BE_SCD1` (API скоркарт, куда нужно добавить статус)
