## 🎨 Архитектура MVP

### Основные сущности

```
Deployment (Внедрение)
├── Simulation (Симуляция) [0..*]
├── Pilot (Пилот) [0..*]
├── Change (Изменение) [0..*]
└── Artifact (Артефакт) [0..*] (URL ссылки)

Scorecard (Скоркарта)
├── связь с Pilot [многие-ко-многим]
└── связь с Change [многие-ко-многим]

Lineage (Визуализация происхождения)
├── Данные читаются из таблиц источников
├── Связи через Scorecards
└── Граф: Simulation → Pilot → Change

Package (Пакет)
├── Pilot/Change [1..*]
├── Создается ТОЛЬКО при отправке на утверждение
└── ApprovalInstance [0..1]

ApprovalInstance (Экземпляр согласования)
├── Target: Pilot/Change/Package (полиморфная связь)
├── ApprovalStage [1..*] (динамически создаются при отправке)
│   ├── Approval stages (0..*)
│   └── Ratification stage (0..1)
└── ApprovalDecision [0..*]
```

### Процесс согласования

```
1. ПРМ создает Pilot/Change
2. ПРМ отправляет Pilot/Change на согласование
   - При отправке выбирает согласующих для каждого этапа:
     - Approval stages (0..*): для каждого этапа указывает список согласующих
     - Ratification stage (0..1): опциональный
       - если auto_ratification = true: указывает 1 утверждающего
       - если auto_ratification = false: утверждающий выбирается при запуске ratification (индивидуально или через Package)
     - Флаг auto_ratification (если есть Ratification stage)
3. Создается ApprovalInstance с выбранными этапами
4. Этапы Approval (если есть):
   - Последовательное выполнение между этапами
   - Параллельная работа внутри этапа
   - Все должны согласовать для перехода к следующему этапу
   - Если хотя бы один отклонил → target переходит в `approval_rejected` (не в draft)
5. Переход на Ratification (если есть):
   - Если auto_ratification = true → автоматически на Ratification
   - Если auto_ratification = false → в `awaiting_ratification` (очередь), далее по команде пользователя: Ratification индивидуально или через Package
6. Этап Ratification:
   - Для индивидуального Pilot/Change: 1 утверждающий
   - Для Package: решение может быть принято по всем элементам сразу или по каждому элементу отдельно (статус хранится по item)
7. При успешном прохождении: Pilot/Change переходит в статус "approved/ratified"
8. ПРМ может активировать Pilot или внедрить Change
```

### Ролевая модель

| Роль | Привязка к продукту | Основные права |
|------|---------------------|----------------|
| **ПРМ** | Да (product_id NOT NULL) | Создание и управление Deployment/Simulation/Pilot/Change своего продукта, отправка на согласование |
| **Методолог** | Нет (product_id IS NULL) | Создание и управление Scorecard для всех продуктов, управление артефактами всех сущностей |
| **Approver** | Нет (product_id IS NULL) | Согласование Pilot/Change/Package на этапе Approval |
| **Ratifier** | Нет (product_id IS NULL) | Утверждение Pilot/Change/Package на этапе Ratification |
| **Админ** | Нет (product_id IS NULL) | Полный доступ ко всем сущностям, управление ролями |

## ⚠️ Риски и митигация

### Высокие риски

1. **Сложность процесса согласования**
   - Митигация: Выделить лучшего разработчика, начать раньше, заложить буфер

2. **Интеграция Frontend и Backend**
   - Митигация: Backend работает на 1-2 дня впереди, ежедневные sync встречи, mock данные

3. **Недооценка трудозатрат**
   - Митигация: Буфер 23-32%, приоритизация критичных задач

4. **Изменение требований**
   - Митигация: Зафиксировать скоуп, change request процесс, демо после каждого спринта
