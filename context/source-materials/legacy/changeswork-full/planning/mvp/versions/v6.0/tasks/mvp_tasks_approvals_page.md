# Задачи для страницы "Согласования" (Approvals View)

## Обзор страницы

**Страница:** Согласования (Approvals View)
**Путь в макете:** Боковое меню → "Согласования"
**Статус:** Новая разработка

### Что показывает страница (из макета):

**2 вкладки:**
1. **На согласовании** - Pilot и Change со статусом 'in_approval' (этап согласования)
2. **На утверждении** - Pilot и Change со статусом 'in_ratification' (этап утверждения)

**Функционал:**
- Группировка по Package (если элементы в пакете) + отдельные элементы
- Checkbox для выбора элементов (batch операции)
- Кнопки batch действий:
  - "Согласовать выбранные (N)" / "Утвердить выбранные (N)"
  - "Отклонить выбранные (N)"
- Кнопки для отдельных элементов:
  - "Согласовать" / "Утвердить"
  - "Отклонить"
- Кнопки для пакетов:
  - "Согласовать пакет" / "Утвердить пакет"
  - "Отклонить пакет"
- Диалог детального просмотра элемента
- Диалог комментария для действий (approve/reject/ratify)
- Отображение EntityScorecardBlock для каждого элемента
- Отображение информации о внедрении
- Клик на элемент → детальный просмотр

**Права доступа:**
- Согласующий видит только элементы, где он назначен согласующим
- ПРМ/Методолог, назначенные согласующими, видят свои назначения
- Админ видит все элементы

**Особенности:**
- Только назначенные согласующие могут принимать решения
- Комментарий обязателен при отклонении
- Batch операции для множественного согласования/отклонения
- Группировка по Package для координированного согласования

## BE-APR1: Backend для страницы "Согласования"

**Тип:** Story
**Приоритет:** Критический
**Оценка:** 6 дней

### Summary
Разработка Backend для страницы Согласования с batch операциями и фильтрацией по назначениям

### Description

Разработать Backend для страницы Согласования:

1. **API Endpoint: GET /api/approvals/my/**

   **Параметры запроса:**
   - `stage` (optional) - фильтр по этапу (approval/ratification)
   - `page` (optional, default=1) - номер страницы
   - `page_size` (optional, default=20) - размер страницы

   **Формат ответа:**
   ```json
   {
     "count": 42,
     "results": {
       "packages": [
         {
           "package": {
             "id": "uuid",
             "display_id": "PKG-001",
             "name": "Пакет изменений Premium Q1 2026",
             "deployment": {
               "id": "uuid",
               "display_id": "DEP-001",
               "name": "Оптимизация риск-стратегии Premium карт"
             }
           },
           "items": [
             {
               "id": "uuid",
               "type": "pilot",
               "display_id": "PLT-001",
               "name": "Пилот Premium Q1 2026",
               "status": "in_approval",
               "stage": "approval",
               "scorecards": [...],
               "created_at": "2026-03-01T10:00:00Z"
             }
           ]
         }
       ],
       "standalone": [
         {
           "id": "uuid",
           "type": "change",
           "display_id": "CHG-001",
           "name": "Внедрение новой скоркарты Premium",
           "status": "in_ratification",
           "stage": "ratification",
           "scorecards": [...],
           "deployment": {...},
           "created_at": "2026-03-01T10:00:00Z"
         }
       ]
     }
   }
   ```

2. **API Endpoint: POST /api/approvals/approve/**

   **Параметры запроса:**
   ```json
   {
     "items": [
       {"type": "pilot", "id": "uuid"},
       {"type": "change", "id": "uuid"}
     ],
     "comment": "Согласовано без замечаний"
   }
   ```

   **Бизнес-логика:**
   - Валидация: текущий пользователь назначен согласующим для всех элементов
   - Обновление ApprovalDecision для текущего пользователя → 'approved'
   - Проверка: все ли согласующие этапа приняли решение
   - Если все согласовали → переход к следующему этапу или завершение
   - Отправка уведомлений


3. **API Endpoint: POST /api/approvals/reject/**

   **Параметры запроса:**
   ```json
   {
     "items": [
       {"type": "pilot", "id": "uuid"},
       {"type": "change", "id": "uuid"}
     ],
     "comment": "Требуется доработка обоснования"
   }
   ```

   **Бизнес-логика:**
   - Валидация: текущий пользователь назначен согласующим для всех элементов
   - Валидация: комментарий обязателен
   - Обновление ApprovalDecision для текущего пользователя → 'rejected'
   - Обновление ApprovalWorkflow → 'rejected'
   - Возврат элементов в предыдущий статус
   - Отправка уведомлений автору с причиной

4. **API Endpoint: POST /api/approvals/ratify/**

   **Параметры запроса:**
   ```json
   {
     "items": [
       {"type": "pilot", "id": "uuid"},
       {"type": "change", "id": "uuid"}
     ],
     "comment": "Утверждено"
   }
   ```

   **Бизнес-логика:**
   - Валидация: текущий пользователь назначен утверждающим (ratifier) для всех элементов
   - Обновление ApprovalDecision для текущего пользователя → 'approved'
   - Завершение процесса согласования
   - Обновление статусов элементов (Pilot → active, Change → approved)
   - Отправка уведомлений

5. **Бизнес-логика:**

   **БП-1: Фильтрация по назначениям**
   - Показывать только элементы, где текущий пользователь назначен согласующим
   - Для этапа approval: проверка ApprovalDecision.approver_id = current_user
   - Для этапа ratification: проверка ratifier_id = current_user

   **БП-2: Группировка по Package**
   - Элементы в Package группируются вместе
   - Отдельные элементы (без Package) показываются отдельно
   - Возможность согласовать/отклонить весь Package целиком

   **БП-3: Batch операции**
   - Можно согласовать/отклонить несколько элементов одновременно
   - Валидация: пользователь назначен согласующим для всех выбранных элементов
   - Комментарий применяется ко всем элементам

6. **Права доступа:**
   - Просмотреть может: только назначенный согласующий/утверждающий
   - Согласовать/отклонить может: только назначенный согласующий/утверждающий
   - HTTP 403 если пользователь не назначен

9. **Loading и Error состояния:**
   - Skeleton loader при загрузке
   - Error message при ошибке API
   - Empty state если нет данных

10. **API интеграция:**
