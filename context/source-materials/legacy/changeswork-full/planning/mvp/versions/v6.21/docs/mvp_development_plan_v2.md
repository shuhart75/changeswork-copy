# План разработки MVP - Двухфазная стратегия

## Обзор

**Подход:** Двухфазная разработка без разбиения на спринты
**Команда:** Аналитики, Backend разработчики, Frontend разработчики, QA

---

## Фаза 1: Процесс согласования и минимальная интеграция

### Цель
Реализовать процесс согласования/утверждения и подключить к нему существующие симуляции и пилоты. Добавить артефакты и скоркарты к существующим сущностям без адаптации UI.

### Задачи

#### Backend - Процесс согласования (21 день)
```
BE-AP1 (8д) - Процесс согласования
  ↓
BE-INT1 (3д) - Интеграция жизненных циклов
  ↓
BE-PL1 (3д) - ЖЦ пилота
  ↓
BE-NOT1 (5д) - Уведомления

Параллельно:
BE-SL1 (2д) - ЖЦ симуляции
```

**BE-AP1 (8 дней) - Процесс согласования:**
- Двухэтапный процесс (Approval → Ratification)
- Универсальный для Pilot, Change, Package
- Маршруты согласования
- Роли Approver/Ratifier

**BE-INT1 (3 дня) - Интеграция ЖЦ:**
- Pilot: requires_activation → active
- Change: requires_approval → approved
- Автоматические переходы при согласовании/отклонении

**BE-PL1 (3 дня) - ЖЦ пилота:**
- Состояния: draft → requires_activation → active → completed
- Интеграция с процессом согласования

**BE-NOT1 (5 дней) - Уведомления:**
- Email уведомления для всех событий
- Шаблоны с брендингом
- История уведомлений

**BE-SL1 (2 дня) - ЖЦ симуляции:**
- Состояния: draft → running → completed/failed
- Независимый от согласования

#### Backend - Артефакты и скоркарты (5 дней)
```
BE-ART1 (3д) - Подключение артефактов к существующим Simulation/Pilot
BE-SC-INT1 (2д) - Подключение скоркарт к существующим Pilot
```

**BE-ART1 (3 дня) - Артефакты:**
- API для управления артефактами (ссылки на внешние документы)
- Привязка к Simulation/Pilot
- Права доступа (ПРМ для своих, методолог для всех)

**BE-SC-INT1 (2 дня) - Скоркарты:**
- Связь многие-ко-многим Pilot ↔ Scorecard
- Наследование связей с подпродуктами

#### Аналитика - Процесс согласования и страницы (4 дня)
```
AN-AP1 (2д) - Процесс согласования
AN-APR1 (2д) - Страница Согласования (зависит от AN-AP1)
AN-PKG1 (2д) - Страница Пакеты (параллельно)
```

**AN-AP1 (2 дня) - Процесс согласования:**
- Анализ двухэтапного процесса (Approval → Ratification)
- Диалог настройки маршрута согласования
- Выбор согласующих (Approver) и утверждающих (Ratifier)

**AN-APR1 (2 дня) - Страница Согласования:**
- Анализ требований для страницы Согласования
- 2 вкладки (На согласовании, На утверждении)
- Batch операции

**AN-PKG1 (2 дня) - Страница Пакеты:**
- Анализ требований для страницы Пакеты
- Создание пакетов, группировка Pilot/Change
- Отправка на утверждение

#### Frontend - Страницы Согласования и Пакеты (22 дня)
```
BE-APR1 (6д) → FE-APR1 (7д)
BE-PKG1 (7д) → FE-PKG1 (8д)
```

**BE-APR1 (6 дней) - Backend страницы Согласования:**
- API для страницы Согласования
- Фильтрация по назначениям
- Batch согласование/отклонение

**FE-APR1 (7 дней) - Frontend страницы Согласования:**
- Страница Согласования
- 2 вкладки
- Группировка по Package
- Batch операции

**BE-PKG1 (7 дней) - Backend страницы Пакеты:**
- API для страницы Пакеты с batch операциями
- Группировка Pilot/Change для совместного утверждения
- Автоматический расчет критичности Package

**FE-PKG1 (8 дней) - Frontend страницы Пакеты:**
- Страница Пакеты (4 вкладки, batch операции, группировка)
- Диалог выбора маршрута утверждения
- Предпросмотр email брифа

#### QA (3 дня)
```
QA-PHASE1 (3д) - Интеграционное тестирование процесса согласования
```

### Результаты Фазы 1
- ✅ Процесс согласования/утверждения работает
- ✅ Существующие симуляции и пилоты подключены к процессу
- ✅ Артефакты доступны для Simulation/Pilot
- ✅ Скоркарты связаны с Pilot
- ✅ Страница Согласования работает
- ✅ Страница Пакеты работает
- ✅ Уведомления отправляются

### Критический путь Фазы 1
```
Поток 1: AN-AP1 (2д) → BE-AP1 (8д) → BE-INT1 (3д) → BE-PL1 (3д) → BE-NOT1 (5д) = 21 день
Поток 2: AN-PKG1 (2д) → BE-PKG1 (7д) → FE-PKG1 (8д) = 17 дней
Поток 3: AN-APR1 (2д, зависит от AN-AP1) → BE-APR1 (6д) → FE-APR1 (7д) = 17 дней

QA-PHASE1 (3д) зависит от BE-NOT1, FE-PKG1, FE-APR1

Критический путь: BE-AP1 (8д) → BE-INT1 (3д) → BE-PL1 (3д) → BE-NOT1 (5д) → QA-PHASE1 (3д) = 22 дня
```

## Фаза 2: Внедрения, изменения и полная адаптация

### Цель

**BE-D1 (3 дня):**
- API для списка внедрений с фильтрацией и пагинацией

**BE-DD1 (3 дня):**

**BE-SCD1 (4 дня):**

**BE-SCF1 (4 дня):**
- API для создания скоркарты

#### Frontend - Внедрения (18 дней)

FE-D1 (3д) → FE-DD1 (10д)

FE-C1 (3д) → FE-CHD1 (6д) → FE-CHF1 (3д)

## Риски и митигация

### Высокие риски

**1. Процесс согласования (BE-AP1: 8 дней)**
- Риск: Критический компонент, блокирует всю Фазу 1
- Митигация: Начать первым, выделить лучшего backend разработчика

**2. Деталка внедрения (FE-DD1: 10 дней)**
- Риск: Самая сложная страница, 4 вкладки, 3 inline формы
- Митигация: Выделить опытного frontend разработчика

**3. Страница Пакеты (BE-PKG1 + FE-PKG1: 15 дней)**
- Риск: Сложная логика группировки и batch операций
- Митигация: Code review, тщательное тестирование

**4. Форма скоркарты (BE-SCF1 + FE-SCF1: 9 дней)**
- Риск: JSON editor, сложная валидация
- Митигация: Использовать готовую библиотеку для JSON editor

### Средние риски

**5. Интеграция артефактов и скоркарт**
- Риск: Нужно не сломать существующие симуляции и пилоты
- Митигация: Тщательное тестирование, постепенное внедрение

**6. Адаптация дизайна**
- Риск: Может потребовать больше времени, чем оценено
- Митигация: Заложен буфер 20%

## Рекомендации

### Приоритеты

1. **Фаза 1 - критический приоритет** - без процесса согласования система не работает
2. **Параллелизация в Фазе 2** - максимально распараллелить работу команды
3. **Code review для критических компонентов** - процесс согласования, пакеты, деталка внедрения
4. **Регулярное тестирование** - не откладывать QA на конец

### Буферы

- Заложить **20% буфер** на непредвиденные задачи
- 66 дней (базовая оценка) + 20% = **79 дней** (реалистичная оценка)
- С учетом буфера и выходных: **~3.5 месяца**

### Контрольные точки

- **День 22:** Демо процесса согласования (Фаза 1)
- **День 40:** Демо внедрений и изменений
- **День 66:** Финальное демо MVP

## Отличия от предыдущего плана

### Что изменилось

1. **Убрали разбиение на спринты** - единый поток в рамках каждой фазы
2. **Фаза 1 фокусируется на согласовании** - минимальная интеграция с существующими сущностями
3. **Не адаптируем UI в Фазе 1** - только подключаем артефакты и скоркарты к существующим страницам
4. **Не трогаем меню в Фазе 1** - обновление навигации перенесено в Фазу 2
5. **Фаза 2 включает всё остальное** - внедрения, изменения, полная адаптация дизайна

### Преимущества нового подхода

- ✅ Быстрее получаем работающий процесс согласования
- ✅ Можно начать использовать систему после Фазы 1
- ✅ Меньше рисков - фокус на одной большой задаче в Фазе 1
- ✅ Проще планировать - две четкие фазы вместо шести
