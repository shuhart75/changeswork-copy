## Риски и митигация

### Высокие риски

#### Риск 1: Сложность процесса согласования
**Вероятность:** Средняя (снижена благодаря увеличенному сроку)
**Влияние:** Критическое
**Описание:** Двухэтапное согласование с маршрутами - самая сложная часть системы.

**Митигация:**
- Выделить лучшего Backend разработчика на BE-12
- Начать разработку в Sprint 3 (после готовности всех API)
- Буфер 46-66 дней позволяет справиться с задержками
- Провести code review с архитектором
- Аналитик готовит детальные сценарии в AN-5

#### Риск 2: Интеграция Frontend и Backend
**Вероятность:** Низкая (снижена благодаря увеличенному сроку)
**Влияние:** Среднее
**Описание:** Проблемы при интеграции могут задержать релиз.

**Митигация:**
- Backend работает на 1-2 дня впереди Frontend
- Ежедневные sync встречи между FE и BE (15 мин)
- Использовать mock данные на Frontend до готовности API
- Sprint 4 полностью выделен на интеграцию
- Буфер 45-53 дней покрывает задержки
- 3 Backend разработчика обеспечивают быструю поддержку

#### Риск 3: Недооценка трудозатрат
**Вероятность:** Низкая (снижена благодаря увеличенному сроку)
**Влияние:** Среднее
**Описание:** Задачи могут занять больше времени, чем оценено.

**Митигация:**
- Буфер 56-83% заложен в план
- Приоритизация: критичные задачи в первую очередь
- 4-месячный срок вместо 2-месячного снижает риски

#### Риск 4: Изменение требований
**Вероятность:** Средняя
**Влияние:** Среднее
**Описание:** Заказчик может изменить требования в процессе разработки.

**Митигация:**
- Скоуп зафиксирован и утвержден
- Change request процесс: новые требования → версия 2.0
- Демо после каждого спринта для раннего feedback
- Аналитики работают на опережение, готовя документацию

### Средние риски

#### Риск 5: Болезни/отпуска
**Вероятность:** Средняя
**Влияние:** Низкое (снижено благодаря большой команде)

**Митигация:**
- Буфер 56-83% покрывает 3-4 недели простоя
- 3 Backend разработчика обеспечивают взаимозаменяемость
- Парное программирование для knowledge sharing
- Документация кода

#### Риск 6: Технические проблемы
**Вероятность:** Низкая
**Влияние:** Низкое

**Митигация:**
- Использовать проверенные библиотеки
- Настроить окружение в Sprint 1
- Выделить время на баг-фиксы в Sprint 4
- 2 тестировщика обеспечивают раннее выявление проблем

## Рекомендации по организации работы

### Daily Standup (15 мин)
- Что сделано вчера
- Что планируется сегодня
- Есть ли блокеры

### Sprint Planning (2 часа в начале спринта)
- Обзор задач спринта
- Уточнение требований
- Распределение задач

### Sprint Review (1 час в конце спринта)
- Демо реализованного функционала
- Feedback от заказчика
- Корректировка плана при необходимости

### Sprint Retrospective (1 час в конце спринта)
- Что прошло хорошо
- Что можно улучшить
- Action items на следующий спринт

### Sync встречи FE-BE (30 мин, 2 раза в неделю)
- Обсуждение API контрактов
- Решение проблем интеграции
- Планирование совместной работы

## Заключение

MVP реализуем за 4 месяца с командой 2 Аналитика + 3 Backend + 2 Frontend + 2 Тестировщика при условии:
- ✅ Четкое следование плану
- ✅ Ежедневная коммуникация
- ✅ Приоритизация критичных задач
- ✅ Использование большого буфера времени (56-83%) для непредвиденных задач
- ✅ Фиксация скоупа (новые требования → версия 2.0)
- ✅ Параллельная работа команды (3 Backend разработчика)
- ✅ Качественное тестирование на всех этапах

**Вероятность успеха:** 95%

**Ключевые факторы успеха:**
1. Большой буфер времени (56-83%) для каждой роли
2. Расширенная команда (9 человек вместо 4)
3. Аналитики обеспечивают готовность требований
4. 3 Backend разработчика позволяют параллельную работу
5. 2 Тестировщика обеспечивают качество с первого спринта
