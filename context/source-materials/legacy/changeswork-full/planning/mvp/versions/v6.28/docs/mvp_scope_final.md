# MVP Scope - ФИНАЛЬНАЯ УТВЕРЖДЕННАЯ ВЕРСИЯ

**Дата утверждения:** 2026-03-04
**Срок реализации:** 3 месяца (12 недель, 60 рабочих дней)
**Команда:** 2 Backend + 2 Frontend разработчика

## ✅ Что ВКЛЮЧЕНО в MVP

### Основные сущности

#### 1. Initiative (Инициатива)
**Жизненный цикл:** draft → active → deployed → **archived**
- Группировка всей работы по внедрению
- Связь с Product
- Артефакты (документы)
- **Архивирование включено** ✅

#### 2. Simulation (Симуляция)
**Жизненный цикл:** draft → running → completed/failed → **archived**
- Анализ исторических данных
- Связь с Initiative (обязательная)
- Может быть источником для Scorecard
- **Архивирование включено** ✅

#### 3. Pilot (Пилот)
**Жизненный цикл:** draft → requires_activation → active → requires_correction/completed → **archived**
- Контролируемый эксперимент
- Связь с Initiative (обязательная)
- Связь со Scorecard (многие-ко-многим, минимум 1)
- Может быть источником для Scorecard
- **Архивирование включено** ✅

#### 4. Deployment (Внедрение)
**Жизненный цикл:** draft → requires_approval → approved → deployed
- Продуктовое внедрение
- Связь с Initiative (обязательная)
- Связь со Scorecard (многие-ко-многим, минимум 1)
- Критичность (low/medium/high/critical)
- Плановая дата внедрения
- Артефакты (документы)
- **Rollback НЕ включен** ❌

#### 5. Scorecard (Скоркарта)
**Упрощенная версия без версионирования**
- Название, описание, версия (строка), конфигурация (JSON)
- Связь с Product
- Связь с Pilot (многие-ко-многим)
- Связь с Deployment (многие-ко-многим)
- Отслеживание источника (simulation_id, pilot_id)
- **Audit log:** created_at, updated_at, created_by, updated_by
- **Версионирование (отдельная таблица) НЕ включено** ❌

#### 6. ScorecardTemplate (Шаблон скоркарты)
**Новая сущность**
- Шаблоны скоркарт по продуктам
- Название, описание, product_id
- Конфигурация по умолчанию (JSON)
- При создании Scorecard можно выбрать шаблон

#### 7. Lineage (Визуализация происхождения)
**Визуализация связей "создано из"**
- НЕ является CRUD-сущностью (только визуализация)
- Данные читаются из таблиц источников (simulation_source, pilot_source, change_source)
- Связи отслеживаются через Scorecards
- Визуализация показывает граф: Simulation → Pilot → Change
- API: только GET endpoints для получения данных визуализации

**Допустимые пути:**
- Simulation(ы) → Pilot(ы) → Change(я)
- Simulation(ы) → Change(я)
- Pilot(ы) → Change(я)
- Change(я) (прямое внедрение)
- Pilot(ы) без Change (только эксперимент)
- Simulation(ы) без Pilot/Deployment (только исследование)

#### 8. Package (Пакет)
**Группировка для утверждения**
- Создаётся ТОЛЬКО при отправке на утверждение (НЕТ draft состояния)
- Создание и отправка происходят одновременно
- Группирует Pilot и/или Change (не Chain)
- Пользователь выбирает Ratifier при создании
- Package НЕ имеет статусов - статусы принадлежат PackageItem
- Критичность = MAX(priority) среди всех Pilot/Deployment (вычисляется на лету)

#### 9. ApprovalInstance (Экземпляр согласования)
**Динамический процесс согласования**
- Связь с Pilot, Change или Package (полиморфная: target_type + target_id)
- Маршрут создаётся при отправке (не предзадан)
- Флаг auto_ratification: boolean
- Этапы согласования (ApprovalStage):
  - Approval stages (0..*): переменное количество
  - Ratification stage (0..1): опциональный
  - Минимум 1 этап в сумме
- Текущий этап (current_stage_order)
- Статус: pending → approved/rejected
- История решений (ApprovalDecision)

**Логика auto_ratification:**
- Если `true`: после Approval → сразу на Ratification (индивидуально)
- Если `false`: после Approval → в `awaiting_ratification` (очередь), далее по команде пользователя: Ratification индивидуально или через Package

**Для Package:**
- Связь с Package (target_type = 'package')
- Поддерживает частичное согласование через таблицу package_item

#### 11. ApprovalDecision (Решение согласующего)
**Решение на этапе согласования**
- Связь с ApprovalInstance
- Связь с ApprovalStage
- Пользователь (user_id)
   - Решение: approved/rejected
   - Комментарий (опционален)
- Дата решения

#### 12. Artifact (Артефакт)
**Ссылки на внешние документы**
- Для Deployment, Simulation, Pilot и Change
- Хранятся как URL ссылки (НЕ загрузка файлов)
- Типы: business_case, rov, other
- Валидация формата URL
- Метаданные: name, url, type, uploaded_by, uploaded_at

#### 13. UserRole (Роль пользователя)
**Ролевая модель**
- **Множественные роли у пользователя** ✅
- Роли:
  - **ПРМ (prm)** - привязан к product_id (NOT NULL)
  - **Методолог (methodologist)** - глобальная роль (product_id IS NULL)
  - **Approver (approver)** - глобальная роль (product_id IS NULL), согласует на этапе Approval
  - **Ratifier (ratifier)** - глобальная роль (product_id IS NULL), утверждает на этапе Ratification
  - **Админ (admin)** - глобальная роль (product_id IS NULL)

**Примеры множественных ролей:**
- ПРМ продукта A + Approver
- Методолог + Ratifier
- ПРМ продукта A + ПРМ продукта B

#### 14. Email уведомления
**Интеграция с существующим email сервисом**
- Email шаблоны для событий:
  - Package отправлен на согласование → уведомление согласующим
  - Решение принято → уведомление создателю
  - Переход на следующий этап → уведомление согласующим
  - Комментарии согласующих
- Все пользователи получают уведомления (настройки в v2.0)

## ❌ Что НЕ ВКЛЮЧЕНО в MVP

### Исключено полностью:

1. **Версионирование Scorecard**
   - Отдельная таблица `scorecard_version`
   - История версий с возможностью отката
   - Сравнение версий
   - **Вместо этого:** Audit log (created_at, updated_at, updated_by)

2. **Rollback Deployment**
   - Функциональность отката внедрения
   - Статус `rolled_back`
   - **Вместо этого:** Только статусы draft → requires_approval → approved → deployed

3. **SubProduct**
   - Иерархия Product → SubProduct
   - Связи Deployment/Scorecard с SubProduct
   - **Вместо этого:** Работаем только с Product

4. **Продвинутая аналитика**
   - Дашборды с графиками
   - Отчеты и экспорт
   - Статистика по внедрениям
   - **Вместо этого:** Базовый Dashboard с метриками

5. **Email настройки пользователя**
   - Настройки "получать/не получать уведомления"
   - **Вместо этого:** Все получают уведомления по умолчанию

## 📋 Процесс согласования (детально)

### Workflow:

#### 1. **ПРМ создает Pilot или Change**
   - **Для Pilot:**
     - Заполняет: название, описание, deployment_id
     - Выбирает Scorecard (многие-ко-многим, минимум 1)
     - Указывает размеры групп (pilot_group_size, control_group_size)
     - Статус: draft

   - **Для Change:**
     - Заполняет: название, описание, deployment_id
     - Выбирает Scorecard (многие-ко-многим, минимум 1)
     - Указывает критичность (low/medium/high/critical)
     - Указывает плановую дату внедрения
     - Прикрепляет артефакты (опционально)
     - Статус: draft

#### 2. **ПРМ просматривает Lineage (визуализация происхождения)**
   - Визуализация автоматически строится на основе связей через Scorecards
   - Показывает граф: Simulation → Pilot → Change
   - Данные читаются из таблиц источников (simulation_source, pilot_source, change_source)
   - **Lineage НЕ является CRUD-сущностью** (только визуализация)

#### 3. **ПРМ отправляет Pilot/Deployment на согласование**
   - Pilot: статус requires_activation или requires_correction
   - Change: статус requires_approval
   - При отправке пользователь выбирает:
     - Этапы Approval (0..*): для каждого этапа указывает список согласующих (минимум 1)
     - Этап Ratification (0..1): опциональный
       - если `auto_ratification=true`: указывает ровно 1 утверждающего (Ratifier)
       - если `auto_ratification=false`: утверждающий выбирается при запуске ratification (индивидуально или через Package)
     - Флаг auto_ratification (если есть Ratification stage)
     - Минимум 1 этап в сумме
   - Создается ApprovalInstance с выбранными этапами и назначениями
   - **Email уведомление** согласующим первого этапа Approval

#### 4. **Этапы Approval (Согласование)**
   - Pilot/Deployment проходит индивидуально
   - Количество этапов: 0..* (переменное)
   - Этапы выполняются последовательно (stage_order: 1, 2, 3...)
   - Внутри этапа согласующие работают **параллельно**
   - Каждый принимает решение: approve или reject (комментарий опционален)
   - **Email уведомление** при каждом решении
   - Если **хотя бы один отклонил** → ApprovalInstance.status = rejected, target → `approval_rejected`, процесс завершен
   - Если **все в этапе одобрили** → переход к следующему этапу Approval
   - Если **все этапы Approval пройдены** → переход к Ratification (если есть) или завершение

#### 5. **Переход на Ratification (зависит от auto_ratification)**

   **Вариант A: auto_ratification = true (индивидуальное утверждение)**
   - Pilot/Deployment автоматически переходит на этап Ratification
   - **Email уведомление** утверждающему
   - Утверждение проходит индивидуально

   **Вариант B: auto_ratification = false (очередь ожидания утверждения)**
   - Pilot/Deployment попадает в `awaiting_ratification`
   - Далее по команде пользователя:
     - запустить Ratification индивидуально (без Package) или
     - создать Package из выбранных элементов `awaiting_ratification` и отправить на Ratification

#### 6. **Этап Ratification (Утверждение)**
   - Для индивидуального Pilot/Deployment: 1 утверждающий принимает решение
   - Для Package: решение может быть принято по всем item сразу или по каждому item отдельно (итоговый статус хранится по item)
   - Каждый принимает решение: ratify или reject
   - **Email уведомление** при каждом решении
   - Если отклонено → target → `ratification_rejected`
   - Если утверждено → ratified/approved
   - **Email уведомление** создателю

#### 7. **При успешном прохождении**
   - Pilot: может быть активирован (status = active)
   - Deployment: может быть развёрнут (status = deployed)

#### 8. **При отклонении**
   - Reject не возвращает в `draft`
   - Target получает статус `approval_rejected` (reject на Approval) или `ratification_rejected` (reject на Ratification)
   - В этих статусах доступны редактирование, повторная отправка на согласование/утверждение и архивирование
## �� Бизнес-правила (критичные)

### БП-1: Управление Deployment
- Статус рассчитывается автоматически:
  - draft: пустое ИЛИ все вложенные в draft
  - active: хотя бы одна вложенная сущность не в draft
  - deployed: хотя бы одно Deployment в deployed
- Архивирование: вручную, нельзя если есть deployed Deployments
- При архивировании: все вложенные сущности архивируются

### БП-2: Управление Simulation
- Обязательная связь с Initiative
- Может быть источником для Scorecard
- Архивирование: вручную

### БП-3: Управление Pilot
- Обязательная связь с Deployment
- Обязательная связь со Scorecard (минимум 1)
- Может быть источником для Scorecard
- Архивирование: вручную

### БП-4: Управление Change
- Обязательная связь с Deployment
- Обязательная связь со Scorecard (минимум 1)
- Только одно активное внедрение (deployed_at NOT NULL, rolled_back_at IS NULL)
- Нельзя редактировать после отправки на согласование

### БП-5: Управление Scorecard
- Может быть создана вручную или из Simulation/Pilot
- Нельзя удалить, если используется в Pilot или Change
- Audit log: все изменения логируются (updated_at, updated_by)

### БП-6: Lineage (визуализация происхождения)
- НЕ является CRUD-сущностью (только визуализация)
- Данные читаются из таблиц источников (simulation_source, pilot_source, change_source)
- Связи отслеживаются через Scorecards
- Backend автоматически создаёт записи в таблицах источников при создании Pilot/Deployment с Scorecards
- API: только GET endpoints для визуализации

### БП-7: Управление Package
- Создаётся ТОЛЬКО при отправке на утверждение (НЕТ draft состояния)
- Создание и отправка происходят одновременно
- Группирует Pilot и/или Change (не Chain)
- Пользователь выбирает Ratifier при создании
- Package НЕ имеет статусов - статусы принадлежат PackageItem
- Критичность = MAX(priority) среди всех Pilot/Deployment (вычисляется на лету)
- Поддерживает частичное согласование (каждый элемент решается индивидуально через PackageItem)

### БП-8: Процесс согласования
- Pilot и Deployment проходят согласование индивидуально
- Двухэтапное согласование: Approval → Ratification
- **Этап Approval:** Approver согласуют изменения
  - Все Approver этапа должны одобрить для перехода на следующий этап
  - Один отклонил = rejected, процесс завершён
- **Этап Ratification:** Ratifier утверждает изменения
  - Ровно 1 Ratifier принимает решение
- **Package:** поддерживает частичное согласование (каждый элемент решается индивидуально через PackageItem)
- История всех решений сохраняется
