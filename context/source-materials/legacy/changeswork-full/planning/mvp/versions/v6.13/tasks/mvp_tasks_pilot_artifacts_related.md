# Задачи: Пилоты (добавить блоки "Артефакты" и "Связанные сущности" на страницы Пилота)

## Обзор

**Компонент:** Пилоты (UI + API)  
**Страницы:** Пилот (создание/редактирование/деталка)  
**Статус:** Новая разработка (MVP v6.8)

### Что делаем в MVP

Добавляем на страницы Пилота блоки:
- **Артефакты**: список, добавление/удаление ссылок (URL) на документы (с правами, как в spec)
- **Связанные сущности**: связи с Симуляциями и Внедрениями через Скоркарты (read-only)

> Примечание (важно): Симуляции не дорабатываем и не включаем в approval/ratification. Здесь они нужны только как источники скоркарт в блоке "Связанные сущности".

### Поведение на разных страницах

- **Создание пилота**: блоки присутствуют в UI (как в макете), но в disabled/empty состоянии с текстом "Доступно после создания пилота" (т.к. нет `pilot_id` для загрузки артефактов/получения связей).
- **Редактирование пилота (версия draft/rejected)**: блоки активны.
- **Детальная страница пилота**: блоки активны.

## AN-PA1: Аналитика (Артефакты + Связанные сущности на Пилоте)

**Тип:** Story  
**Приоритет:** Высокий  
**Оценка:** 3 дня

### Summary
Зафиксировать требования к UI/UX и API для блоков "Артефакты" и "Связанные сущности" на страницах Пилота.

### Scope

- Согласовать формат артефакта как ссылки: поля `name`, `type`, `url`; валидация URL (как в spec).
- Права доступа:
  - кто видит артефакты
  - кто может добавлять/удалять
- Состав "Связанных сущностей":
  - Симуляции: источники скоркарт
  - Внедрения: через общие скоркарты
  - формат "через скоркарты: SC-001, SC-002"
- Поведение на create/edit/detail (см. выше).
- Отображение истории/комментариев не входит: это отдельно в approval/process.

### Deliverables
- UI/UX спецификация блоков для Пилота (markdown)
- API контракт для endpoints артефактов и related

## BE-PA1: Backend (Артефакты + Связанные сущности на Пилоте)

**Тип:** Story  
**Приоритет:** Высокий  
**Оценка:** 3 дня

### Summary
Реализовать endpoints для артефактов пилота и получения связанных сущностей.

### API

- Артефакты пилота реализуются через общий механизм `BE-ART1` (URL-ссылки), поэтому в рамках `BE-PA1`:
  - используем `/api/pilots/{id}/artifacts/` как nested alias (если он нужен для UI) или `/api/artifacts/?entity_type=pilot&entity_id=...`
  - создание/удаление артефактов проксируется в `BE-ART1`
- `GET /api/pilots/{id}/related/`

### Notes

- В ответе `related` вернуть:
  - `simulations[]` (read-only)
  - `deployments[]` (read-only)
  - для каждого элемента: id/display_id/name/status + список скоркарт, через которые связь построена
- Связи считаем по скоркартам:
  - `Pilot` -> `Scorecard` (M2M)
  - `Simulation` -> `Scorecard` (источник скоркарты)
  - `Deployment`/`DeploymentVersion` -> `Scorecard` (M2M)
- Права: ПРМ видит все продукты, но модифицировать (add/delete artifacts links) может только для своего продукта; админ может всё.

### Acceptance Criteria

- [ ] Endpoints артефактов работают (list/create/delete) с правами (артефакты как URL)
- [ ] Endpoints related возвращает симуляции и внедрения через скоркарты
- [ ] Формат ответа стабилен и задокументирован
- [ ] Тесты для прав и базовых сценариев

## FE-PA1: Frontend (Артефакты + Связанные сущности на Пилоте)

**Тип:** Story  
**Приоритет:** Высокий  
**Оценка:** 3 дня

### Summary
Добавить блоки "Артефакты" и "Связанные сущности" на страницы пилота: create/edit/detail.

### UI

- **ArtifactsBlock**
  - list
  - add link (name + type + url)
  - delete (при наличии прав)
- **RelatedEntitiesBlock**
  - секция "Симуляции (N)"
  - секция "Внедрения (N)"
  - отображение "через скоркарты"

### Поведение

- Create: disabled/empty state "Доступно после создания пилота"
- Edit/Detail: загрузка данных, loading/skeleton, error states

### Acceptance Criteria

- [ ] Блоки отображаются на create/edit/detail как описано
- [ ] Добавление/удаление артефактов (URL) работает (при наличии прав)
- [ ] Related сущности отображаются и кликабельны (переход в деталку)
- [ ] Empty/error/loading состояния реализованы
