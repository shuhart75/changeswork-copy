# Модель домена v3.0 - Архитектура на основе Внедрений

**Версия:** 3.0
**Дата:** 2026-02-24
**Архитектура:** Внедрение-центричная с явным управлением Скоркартами

---

## Краткое резюме

Версия 3.0 представляет фундаментальную реструктуризацию модели домена:

- **Внедрение** становится агрегатом верхнего уровня, группирующим всю связанную работу
- Сущность **Stage** декомпозирована на три отдельных агрегата: **Симуляция**, **Пилот**, **Изменение**
- **Скоркарта** становится полноценным агрегатом со связями многие-ко-многим
- **Lineage** представляет визуализацию происхождения (Симуляции → Пилоты → Изменения) через таблицы источников
- Явное отслеживание источников Скоркарт (Симуляции и Пилоты)
- **Package** создаётся только при отправке на утверждение (Ratification), без статусов
- Роли **Approver** и **Ratifier** разделены для этапов Approval и Ratification

## Ограниченные контексты

### 1. Контекст идентификации и доступа

**Назначение:** Управление пользователями, ролями и правами доступа.

**Сущности:**
- `User` - Пользователь системы
- `UserRole` - Роль пользователя с привязкой к продукту (опционально)
- `ApprovalAssignment` - Назначение пользователя на этап согласования

**Роли:**
- `prm` (Product Risk Manager) - ПРМ, управляет внедрениями своего продукта
- `methodologist` - Методолог, управляет скоркартами и документами
- `approver` - Согласующий на этапах Approval
- `ratifier` - Утверждающий на этапе Ratification
- `admin` - Администратор, полный доступ

**Ответственность:**
- Управление пользователями и их ролями
- Разграничение доступа к сущностям
- Назначение согласующих на этапы согласования
- Поддержка множественных ролей для одного пользователя

**Бизнес-правила:**
- Пользователь может иметь несколько ролей
- Роль ПРМ привязана к конкретному продукту (product_id NOT NULL)
- Роли методолог, approver, ratifier, админ глобальные (product_id IS NULL)
- ПРМ и методолог могут быть назначены approver или ratifier
- Approver видит только те согласования на этапах Approval, где он назначен
- Ratifier видит только те согласования на этапе Ratification, где он назначен
- Правила совмещения ролей: ✅ (ПРМ ИЛИ Методолог ИЛИ Админ) + Approver + Ratifier; ❌ ПРМ + Методолог

---

### 2. Контекст каталога продуктов

**Назначение:** Управление иерархией и классификацией продуктов.

**Сущности:**
- `Product` - Кредитный продукт (например, "Кредитная карта")
- `SubProduct` - Вариант продукта (например, "Карта с кэшбэком")

**Ответственность:**
- Управление каталогом продуктов
- Связь продукт-стратегия

---

### 3. Контекст шаблонов скоркарт

**Назначение:** Управление версионированными шаблонами для создания скоркарт.

**Сущности:**
- `ScorecardTemplate` - Определение шаблона для продукта
- `ScorecardTemplateVersion` - Версионированный шаблон с конфигурацией по умолчанию

**Ответственность:**
- Версионирование шаблонов во времени
- Значения полей и структура по умолчанию
- Управление шаблонами для конкретных продуктов

**Бизнес-правила:**
- Шаблоны версионируются по продуктам
- Каждая версия имеет период действия (valid_from, valid_to)
- default_config хранится как JSONB

---

### 4. Контекст скоркарт

**Назначение:** Управление фактическими скоркартами, используемыми в пилотах и изменениях.

**Сущности:**
- `Scorecard` - Экземпляр скоркарты, созданный из шаблона
- `ScorecardVersion` - Версионированная конфигурация скоркарты
- `ScorecardSource` - Отслеживает источники (Симуляции/Пилоты) для скоркарты

**Ответственность:**
- Создание скоркарт (вручную или из источников)
- Версионирование скоркарт при изменении
- Трассируемость источников
- Управление финансовыми эффектами
- Управление жизненным циклом согласования/утверждения

**Бизнес-правила:**
- Скоркарта создается из ScorecardTemplateVersion
- Может быть создана вручную (is_manual = true) или из результатов Симуляции/Пилота
- Каждое изменение создает новую ScorecardVersion
- financial_effects хранятся в JSONB внутри ScorecardVersion
- Связь многие-ко-многим с Пилотами и Изменениями
- Источники отслеживаются в таблице scorecard_source
- Может быть связана с несколькими SubProduct в рамках Product (многие-ко-многим)
- Если подпродукты не указаны, Скоркарта относится ко всем подпродуктам Product
- Любая сущность (Пилот, Изменение), связанная со Скоркартой, наследует связи с подпродуктами этой Скоркарты
- Скоркарта проходит полный цикл согласования/утверждения вместе с Пилотами и Изменениями
- Жизненный цикл: draft → requires_approval → in_approval → in_ratification → approved → active → archived

---

### 5. Контекст внедрений

**Назначение:** Группировка верхнего уровня всей работы, связанной со стратегическим изменением.

**Корневой агрегат:**
- `Deployment`

**Ответственность:**
- Группировка связанных Симуляций, Пилотов и Изменений
- Предоставление контекста для всех активностей
- Отслеживание общего статуса внедрения

**Бизнес-правила:**
- Внедрение принадлежит одному Product
- Может быть связано с несколькими SubProduct в рамках Product (многие-ко-многим)
- Если подпродукты не указаны, Внедрение относится ко всем подпродуктам Product
- Может содержать множество Симуляций, Пилотов и Изменений
- Жизненный цикл: draft → active → deployed → archived
- Статус рассчитывается автоматически на основе вложенных сущностей

---

### 6. Контекст исследований и выполнения

**Назначение:** Управление тремя фазами развертывания стратегии.

**Корневые агрегаты:**
- `Simulation` - Анализ исторических данных
- `Pilot` - Реальный эксперимент
- `Change` - Продуктовое внедрение

#### Симуляция

**Ответственность:**
- Запуск анализа на исторических данных
- Генерация прогнозов финансовых эффектов
- Служит источником для Скоркарт

**Бизнес-правила:**
- Принадлежит одному Внедрению
- Может быть источником для множества Скоркарт
- Множество Симуляций могут быть связаны через Lineage (визуализация происхождения)
- Жизненный цикл: draft → running → completed/failed
- Результаты хранятся как JSONB

#### Пилот

**Ответственность:**
- Выполнение контролируемого эксперимента
- Сбор реальных метрик
- Валидация прогнозов скоркарты
- Служит источником для Скоркарт

**Бизнес-правила:**
- Принадлежит одному Внедрению
- Может быть связан максимум с ОДНИМ Изменением через общие Скоркарты (0..1)
- Связан с одной или несколькими Скоркартами (многие-ко-многим)
- Может быть источником для Скоркарт
- Если создан из Скоркарт, источником которых были Симуляции, наследует связи с Симуляциями
- Жизненный цикл: draft → requires_activation → active → requires_correction/completed
- Может быть отправлен на согласование в статусе "requires_activation" или "requires_correction"

#### Изменение

**Ответственность:**
- Развертывание стратегии в продакшн
- Отслеживание конфигурации развертывания
- Управление откатом

**Бизнес-правила:**
- Принадлежит одному Внедрению
- Может иметь несколько Пилотов через общие Скоркарты (0..*)
- Связано с одной или несколькими Скоркартами (многие-ко-многим)
- Если создано из Скоркарт, источником которых были Симуляции → наследует связи с Симуляциями
- Если создано из Скоркарт, источником которых были Пилоты → наследует связи с Пилотами
- Жизненный цикл: draft → requires_approval → approved → deployed → rolled_back
- Может быть отправлено на согласование в статусе "requires_approval"
- Только одно активное развертывание на Изменение (rollback_at = null)

### 7. Контекст отслеживания происхождения (Lineage)

**Назначение:** Отслеживание линии происхождения сущностей через связи "создано из".

**Сущности:**
- `SimulationSource` - Источник Симуляции (из какого Пилота создана)
- `PilotSource` - Источники Пилота (из каких Симуляций создан)
- `ChangeSource` - Источники Изменения (из каких Пилотов/Симуляций создано)

**Ответственность:**
- Автоматическое установление связей "создано из" при создании сущностей
- Вычисление lineage на лету для визуализации
- Трассируемость происхождения сущностей

**Бизнес-правила:**
- Lineage НЕ является отдельной сущностью - это визуализация существующих связей
- Lineage устанавливается автоматически при создании Pilot/Change с выбором Scorecards
- Backend проверяет источники Scorecards и создаёт записи в *_source таблицах
- Одна Симуляция может быть создана из одного Пилота (one-to-one)
- Один Пилот может быть создан из нескольких Симуляций (many-to-many)
- Одно Изменение может быть создано из нескольких Пилотов и/или Симуляций (many-to-many)
- Lineage вычисляется рекурсивно для отображения всей цепочки происхождения

---

### 8. Контекст пакетов

**Назначение:** Группировка согласованных Pilot/Change для координированного утверждения (Ratification) руководством.

**Корневой агрегат:**
- `Package`

**Сущности:**
- `PackageItem` - Элемент пакета (Pilot или Change) с индивидуальным статусом

**Ответственность:**
- Группировка Pilot/Change для совместного утверждения
- Координация процесса Ratification
- Хранение истории утверждения для аудита

**Бизнес-правила:**
- Package НЕ имеет статусов - статусы принадлежат items (PackageItem.status: pending/ratified/ratification_rejected)
- Создаётся ТОЛЬКО при отправке на утверждение (нет draft состояния)
- При создании Package сразу отправляется на Ratification
- Может содержать Pilot и/или Change из ЛЮБЫХ Deployments
- Все Pilot/Change должны быть в статусе approved + awaiting_ratification
- Поддерживает частичное утверждение (каждый item решается индивидуально Ratifier)
- Priority вычисляется на лету: MAX(item.priority)
- Package хранится навсегда для истории утверждения (не удаляется)

---

### 9. Контекст согласования

**Назначение:** Многоэтапный процесс согласования и утверждения для Pilot и Change без предзаданных маршрутов.

**Корневой агрегат:**
- `ApprovalInstance` - Экземпляр процесса согласования для конкретного Pilot/Change/Package

**Сущности:**
- `ApprovalStage` - Этапы согласования (Approval stages) и утверждения (Ratification stage)
- `ApprovalDecision` - Индивидуальные решения согласующих
- `ApprovalAssignment` - Назначение согласующих на этапы

**Ответственность:**
- Управление многоэтапным процессом согласования
- Отслеживание решений согласующих на каждом этапе
- Поддержка автоматического и ручного утверждения
- Поддержка частичного утверждения Package

**Бизнес-правила:**
- Маршрут создаётся при отправке на согласование (не предзадан)
- Пользователь выбирает согласующих для каждого этапа при отправке
- ApprovalInstance имеет флаг `auto_ratification: boolean`
- **Этапы согласования (Approval stages):**
  - Количество: 0..* (переменное)
  - Внутри этапа: согласующие работают параллельно
  - Между этапами: последовательное выполнение
  - Минимум 1 согласующий на этап
  - Все согласующие этапа должны одобрить для перехода на следующий этап
  - Любое отклонение провалит весь процесс
- **Этап утверждения (Ratification stage):**
  - Количество: 0..1 (опциональный)
  - Ровно 1 утверждающий
- **Общие правила:**
  - Минимум 1 этап (любой: Approval или Ratification)
  - Один человек не может быть в нескольких этапах
- **После всех Approval этапов:**
  - Если есть Ratification stage И `auto_ratification = true` → автоматический переход на Ratification
  - Если есть Ratification stage И `auto_ratification = false` → попадает в "Ожидают утверждения"
  - Если нет Ratification stage → процесс завершён (approved)
- **Package для согласования:**
  - Создаётся вручную из Pilot/Change в состоянии awaiting_ratification
  - Может содержать Pilot/Change из ЛЮБЫХ Deployments
  - При отправке пользователь выбирает согласующих для каждого этапа:
    - Этапы Approval (0..*): для каждого этапа указывает список согласующих (минимум 1)
    - Этап Ratification (0..1): указывает ровно 1 утверждающего
    - Минимум 1 этап в сумме
  - Согласующий может принять решение по каждому item индивидуально (частичное согласование)
  - Можно удалить отдельный item из Package (даже после отправки)

## Корневые агрегаты

### Внедрение

**Идентификатор:** UUID
**Жизненный цикл:** draft → active → deployed → archived

**Содержит:**
- Симуляции (0..*)
- Пилоты (0..*)
- Изменения (0..*)
- Артефакты (0..*)

**Инварианты:**
- Должно принадлежать одному Product
- Может быть связано с несколькими SubProduct (многие-ко-многим через deployment_sub_product)
- Если подпродукты не указаны, относится ко всем подпродуктам Product
- Имя должно быть уникальным в рамках Product
- Статус рассчитывается автоматически:
  - draft: пустое ИЛИ все вложенные сущности в draft
  - active: хотя бы одна вложенная сущность не в draft
  - deployed: хотя бы одно Изменение в deployed
  - archived: переведено вручную (НЕТ каскадного архивирования)
- Нельзя архивировать Внедрение, если есть deployed Изменения или неархивированные вложенные сущности

---

### Симуляция

**Идентификатор:** UUID
**Жизненный цикл:** draft → running → completed/failed

**Содержит:**
- Результаты (JSONB)

**Инварианты:**
- Должна принадлежать одному Внедрению
- Период данных должен быть валидным (from < to)
- Не может быть изменена после завершения

**Может быть источником для:**
- Скоркарт (через scorecard_source)

---

### Пилот

**Идентификатор:** UUID
**Жизненный цикл:** draft → requires_activation → active → requires_correction/completed → archived

**Содержит:**
- Метрики (JSONB)

**Инварианты:**
- Должен принадлежать одному Внедрению
- Должен быть связан минимум с одной Скоркартой
- pilot_group_size + control_group_size > 0
- Не может быть удален, если находится в процессе согласования

**Связи:**
- Многие-ко-многим со Скоркартами (через pilot_scorecard)
- Связь с ApprovalInstance (полиморфная, создаётся при отправке на согласование)

**Процесс согласования:**
- Pilot проходит согласование индивидуально (не в составе Lineage)
- При отправке (submit) пользователь указывает:
  - Этапы согласования (0..* Approval stages)
  - Этап утверждения (0..1 Ratification stage, optional)
  - Флаг auto_ratification (если есть Ratification stage)
- Статусы requires_activation и requires_correction требуют согласования
- После всех Approval этапов: auto_ratification определяет дальнейший путь
- Можно отозвать (recall) с согласования, история сохраняется

**Может быть источником для:**
- Скоркарт (через scorecard_source)

---

### Изменение

**Идентификатор:** UUID
**Жизненный цикл:** draft → requires_approval → approved → deployed → rolled_back

**Содержит:**
- Конфигурация развертывания (JSONB)
- План отката (Text)

**Инварианты:**
- Должно принадлежать одному Внедрению
- Должно быть связано минимум с одной Скоркартой
- Только одно активное развертывание (rollback_at = null)
- Не может быть удалено (hard delete), если развернуто

**Связи:**
- Многие-ко-многим со Скоркартами (через change_scorecard)
- Связь с ApprovalInstance (полиморфная, создаётся при отправке на согласование)

**Процесс согласования:**
- Change проходит согласование индивидуально (не в составе Lineage)
- При отправке (submit) пользователь указывает:
  - Этапы согласования (0..* Approval stages)
  - Этап утверждения (0..1 Ratification stage, optional)
  - Флаг auto_ratification (если есть Ratification stage)
- Статус requires_approval требует согласования
- После всех Approval этапов: auto_ratification определяет дальнейший путь
- Можно отозвать (recall) с согласования, история сохраняется

---

### Скоркарта

**Идентификатор:** UUID
**Жизненный цикл:** draft → requires_approval → in_approval → in_ratification → approved → active → archived

**Содержит:**
- Версии (1..*)
- Источники (0..*)

**Инварианты:**
- Должна быть создана из ScorecardTemplateVersion
- Каждое изменение создает новую версию
- Номера версий - последовательные целые числа
- Если не ручная, должна иметь минимум один источник
- Может быть связана с несколькими SubProduct (многие-ко-многим через scorecard_sub_product)
- Если подпродукты не указаны, относится ко всем подпродуктам Product
- Скоркарта проходит согласование/утверждение вместе с Пилотами и Изменениями, к которым она прикреплена
- Статус Скоркарты синхронизируется со статусами связанных Пилотов/Изменений

**Связи:**
- Многие-ко-многим с Пилотами (через pilot_scorecard)
- Многие-ко-многим с Изменениями (через change_scorecard)
- Многие-ко-многим с источниками (через scorecard_source)
- Многие-ко-многим с SubProduct (через scorecard_sub_product)
- Пилоты и Изменения, связанные со Скоркартой, наследуют связи с подпродуктами

---

### Lineage (визуализация происхождения)

**НЕ является CRUD-сущностью** - это визуализация связей "создано из"

**Отслеживание через таблицы источников:**
- `simulation_source` - связь Симуляция → источник (Pilot)
- `pilot_source` - связь Пилот → источник (Simulation)
- `change_source` - связь Изменение → источник (Pilot/Simulation)

**Правила:**
- Все сущности в Lineage должны быть из одного Deployment
- Backend автоматически создаёт записи в таблицах источников при создании Pilot/Change с указанием Scorecards
- Frontend визуализирует Lineage, читая данные из таблиц источников

**Назначение:**
- Визуализация пути: Симуляции → Пилоты → Изменения
- Отслеживание линии происхождения
- НЕ является объектом согласования (Pilot и Change проходят согласование индивидуально)

### Пакет

**Идентификатор:** UUID
**Жизненный цикл:** НЕТ - Package не имеет статусов, статусы принадлежат PackageItem

**Содержит:**
- PackageItem (1..*) - каждый item ссылается на Pilot или Change
- События (0..*)

**Инварианты:**
- Создаётся ТОЛЬКО при отправке на утверждение (Ratification)
- НЕТ draft состояния - создание и отправка происходят одновременно
- Может содержать Pilot и/или Change из ЛЮБЫХ Deployments
- При создании пользователь выбирает утверждающего (Ratifier)
- Критичность = MAX(priority) среди всех Pilot/Change (вычисляется на лету)
- Package хранится навсегда для истории утверждения (аудит)

**Назначение:**
- Группировка Pilot/Change для совместного утверждения (Ratification)
- Экономия времени утверждающих (утверждение за один приём)
- История утверждения для аудита

**Частичное утверждение:**
- Утверждающий может принять решение по каждому item индивидуально
- Каждый Pilot/Change в Package имеет свой статус в таблице package_item: pending/ratified/ratification_rejected
- Package может быть частично утверждён (часть items одобрена, часть отклонена)

## Основные концепции

### Источники скоркарт

Скоркарты могут быть созданы из нескольких источников:

1. **Ручное создание**
   - Пользователь создает скоркарту напрямую
   - is_manual = true
   - Нет записей в scorecard_source

2. **Из Симуляции**
   - Симуляция завершается с результатами
   - Пользователь создает Скоркарту из результатов
   - scorecard_source: (scorecard_version_id, 'simulation', simulation_id)

3. **Из Пилота**
   - Пилот завершается с метриками
   - Пользователь создает Скоркарту из метрик
   - scorecard_source: (scorecard_version_id, 'pilot', pilot_id)

4. **Комбинированные источники**
   - Скоркарта может иметь множество источников
   - Пример: создана из Симуляции, затем обновлена результатами Пилота
   - Множество записей в scorecard_source

### Lineage внедрения

Lineage представляет визуализацию линии происхождения и трассировку:

```
Симуляции (0..*) → Пилоты (0..*) → Изменения (0..*)
```

**Допустимые пути Lineage:**
1. Только симуляции (фаза исследования)
2. Симуляции + Изменение(я) (пропуск пилота)
3. Симуляции + Пилот(ы) (фаза эксперимента)
4. Симуляции + Пилот(ы) + Изменение(я) (полное внедрение)
5. Пилот(ы) + Изменение(я) (пропуск симуляций)
6. Только Изменение(я) (прямое внедрение)
7. Пилот(ы) без Изменений (только эксперимент)

**Правила Lineage:**
- Множество Симуляций разрешено (итерации)
- Множество Пилотов разрешено (A/B/C тестирование, итерации)
- Множество Изменений разрешено (0..*)
- Все сущности должны принадлежать одному Внедрению
- Lineage НЕ является объектом согласования (только визуализация)
- Pilot и Change проходят согласование индивидуально

### Связи скоркарт

```
Скоркарта (многие) ←→ (многие) Пилот
Скоркарта (многие) ←→ (многие) Изменение
```

**Примеры:**

1. **Одна Скоркарта, Множество Пилотов (A/B/C тестирование)**
   - Скоркарта "Модель риска v2.3"
   - Пилот A: 10% трафика
   - Пилот B: 15% трафика
   - Пилот C: 20% трафика

2. **Множество Скоркарт, Один Пилот**
   - Пилот "Эксперимент Q1 2026"
   - Скоркарта "Application Score"
   - Скоркарта "Behavioral Score"
   - Скоркарта "Income Verification"

3. **Множество Скоркарт, Одно Изменение**
   - Изменение "Новая риск-стратегия"
   - Скоркарта "Application Score v3"
   - Скоркарта "Behavioral Score v2"

### Связь Симуляция-Пилот-Изменение

Когда Пилот создается из Скоркарт, источником которых были Симуляции:

1. Пользователь запускает Симуляцию S1
2. Пользователь создает Скоркарту SC1 из результатов S1
   - scorecard_source: (SC1.version_id, 'simulation', S1.id)
3. Пользователь создает Пилот P1 и связывает SC1
   - pilot_scorecard: (P1.id, SC1.id)
4. Пилот P1 теперь неявно связан с Симуляцией S1 через Скоркарту SC1

Запрос для поиска Симуляций для Пилота:
```sql
SELECT DISTINCT s.*
FROM simulation s
JOIN scorecard_source ss ON ss.source_id = s.id AND ss.source_type = 'simulation'
JOIN scorecard_version sv ON sv.id = ss.scorecard_version_id
JOIN scorecard sc ON sc.id = sv.scorecard_id
JOIN pilot_scorecard ps ON ps.scorecard_id = sc.id
WHERE ps.pilot_id = :pilot_id
```

## Бизнес-правила

### БП-1: Управление внедрением

- Статус Внедрения рассчитывается автоматически:
  - draft: Внедрение пустое ИЛИ все вложенные сущности в статусе draft
  - active: хотя бы одна вложенная сущность перешла в следующий статус (running, requires_activation, requires_approval, active, approved и т.д.)
  - deployed: хотя бы одно Изменение в статусе deployed
  - archived: переведено вручную пользователем
- **НЕТ каскадного архивирования** - вложенные сущности НЕ архивируются автоматически
- Нельзя архивировать Внедрение, если:
  - Хотя бы одно Изменение в статусе deployed
  - Есть неархивированные вложенные сущности (Симуляции, Пилоты, Изменения)
- Архивированное Внедрение не может быть изменено
- Можно создавать Симуляции, Пилоты и Изменения во Внедрении с любым статусом (кроме archived)

### БП-2: Выполнение симуляции

- Симуляция может быть запущена только из статуса 'draft'
- Выполняющаяся Симуляция не может быть изменена
- Результаты завершенной Симуляции неизменяемы
- Провалившаяся Симуляция может быть отредактирована и перезапущена

### БП-3: Создание и управление скоркартой

- Скоркарта должна быть создана из валидной ScorecardTemplateVersion
- Ручная Скоркарта: is_manual = true, источники не требуются
- Скоркарта из источника: is_manual = false, требуется минимум один источник
- Скоркарта может иметь множество источников (Симуляции и/или Пилоты)
- Каждое редактирование создает новую ScorecardVersion с инкрементированным номером версии
- Скоркарта проходит согласование/утверждение вместе с Пилотами и Изменениями, к которым она прикреплена
- Статус Скоркарты синхронизируется со статусами связанных Пилотов/Изменений:
  - Когда Пилот/Изменение переходит в requires_approval → Скоркарта переходит в requires_approval
  - Когда Пилот/Изменение переходит в in_approval → Скоркарта переходит в in_approval
  - Когда Пилот/Изменение переходит в in_ratification → Скоркарта переходит в in_ratification
  - Когда Пилот/Изменение одобрено (approved) → Скоркарта переходит в approved
  - Когда Пилот активирован (active) или Изменение развернуто (deployed) → Скоркарта переходит в active
  - Когда Пилот/Изменение отклонено (rejected) → Скоркарта переходит в rejected
- Если Скоркарта прикреплена к нескольким Пилотам/Изменениям на разных этапах, её статус определяется наиболее продвинутым этапом
- Скоркарта в статусе active или archived не может быть изменена

### БП-4: Управление пилотом

- Пилот должен быть связан минимум с одной Скоркартой
- Пилот проходит согласование индивидуально (не в составе Lineage)
- При отправке на согласование пользователь выбирает согласующих для каждого этапа
- Пилот может быть отправлен на согласование когда status = 'requires_activation' или 'requires_correction'
- После этапов Approval: флаг auto_ratification определяет дальнейший путь
  - Если auto_ratification = true → сразу на Ratification индивидуально
  - Если auto_ratification = false → попадает в awaiting_ratification, ожидает формирования Package
- Можно отозвать Пилот из согласования в любой момент (история сохраняется)
- Активный Пилот не может быть удален (hard delete)
- Пилот может быть источником для новых Скоркарт после завершения

### БП-5: Управление изменением

- Изменение должно быть связано минимум с одной Скоркартой
- Изменение проходит согласование индивидуально (не в составе Lineage)
- При отправке на согласование пользователь выбирает согласующих для каждого этапа
- Изменение может быть отправлено на согласование когда status = 'requires_approval'
- После этапов Approval: флаг auto_ratification определяет дальнейший путь
  - Если auto_ratification = true → сразу на Ratification индивидуально
  - Если auto_ratification = false → попадает в awaiting_ratification, ожидает формирования Package
- Можно отозвать Изменение из согласования в любой момент (история сохраняется)
- Развернутое Изменение не может быть удалено (hard delete)
- Только одно активное развертывание на Изменение (rollback_at = null)
- После отката должно быть создано новое Изменение

### БП-6: Lineage (визуализация происхождения)

- Lineage НЕ является CRUD-сущностью - это визуализация связей "создано из"
- Отслеживание происхождения через таблицы источников:
  - simulation_source: связь Симуляция → источник (Pilot)
  - pilot_source: связь Пилот → источник (Simulation)
  - change_source: связь Изменение → источник (Pilot/Simulation)
- Lineage может включать множество Изменений (0..*)
- Lineage может включать множество Пилотов (0..*)
- Lineage может включать множество Симуляций (0..*)
- Все сущности (Симуляции, Пилоты, Изменения) должны принадлежать одному Внедрению
- Lineage НЕ является объектом согласования
- Lineage используется только для визуализации и трассировки линии происхождения
- Backend автоматически создаёт записи в таблицах источников при создании Pilot/Change с указанием Scorecards

### БП-7: Согласование Pilot и Change

- Pilot и Change проходят согласование индивидуально (не в составе Lineage)
- Pilot может быть отправлен на согласование когда status = 'requires_activation' или 'requires_correction'
- Change может быть отправлен на согласование когда status = 'requires_approval'
- При отправке (submit) пользователь указывает:
  - Этапы согласования (0..* Approval stages)
  - Для каждого этапа: список согласующих (минимум 1) - **только пользователи с ролью Approver**
  - Этап утверждения (0..1 Ratification stage, optional)
  - Для утверждения: 1 утверждающий - **только пользователь с ролью Ratifier**
  - Флаг auto_ratification (если есть Ratification stage)
- Минимум 1 этап (любой: Approval или Ratification)
- Один человек не может быть в нескольких этапах
- **Нельзя выбрать самого себя**
- Можно отозвать (recall) с согласования, история сохраняется (ApprovalInstance.status = 'cancelled')

### БП-8: Управление пакетом

- Package создаётся ТОЛЬКО при отправке на утверждение (Ratification)
- НЕТ draft состояния - создание и отправка происходят одновременно
- Package может содержать Pilot и/или Change из ЛЮБЫХ Deployments
- При создании Package пользователь выбирает:
  - Этап Ratification: указывает ровно 1 утверждающего - **только пользователь с ролью Ratifier**
- Package не может быть создан, если список Pilot/Change пуст
- Критичность Package = MAX(priority) среди всех Pilot/Change (вычисляется на лету)
- Утверждающий может принять решение по каждому item индивидуально (частичное утверждение)
- Каждый Pilot/Change в Package имеет свой статус в таблице package_item: pending/ratified/ratification_rejected
- Package хранится навсегда для истории утверждения (аудит)

### БП-9: Процесс согласования

**Отправка на согласование:**
- Пользователь выбирает согласующих при отправке (маршрут не предзадан)
- Создаётся ApprovalInstance с этапами и назначениями
- Email уведомления согласующим первого этапа

**Этапы согласования (Approval stages):**
- Количество: 0..* (переменное)
- Внутри этапа: согласующие работают параллельно
- Между этапами: последовательное выполнение
- Все согласующие этапа должны одобрить для перехода на следующий этап
- Любое отклонение провалит весь процесс (статус = rejected)
- После прохождения всех Approval этапов: статус = "approved"

**Переход на Ratification:**
- Если есть Ratification stage И auto_ratification = true:
  - Автоматически создаётся новый ApprovalInstance для Ratification
  - Email уведомление утверждающему
  - Pilot/Change проходит утверждение индивидуально
- Если есть Ratification stage И auto_ratification = false:
  - Pilot/Change попадает в состояние awaiting_ratification (флаг в таблице)
  - Пользователь вручную создаёт Package для совместного утверждения
  - При создании Package выбирает утверждающего (Ratifier)
  - Package создаётся и отправляется на утверждение одновременно (нет draft состояния)
- Если нет Ratification stage:
  - Процесс завершён (approved)

**Этап утверждения (Ratification):**
- Для индивидуального Pilot/Change: 1 утверждающий принимает решение
- Для Package: утверждающий может принять решение по каждому item индивидуально
- После утверждения: статус = "ratified"
- После отклонения: статус = "ratification_rejected"

**Отзыв с согласования:**
- Можно отозвать Pilot/Change/Package с любого этапа
- ApprovalInstance.status = 'cancelled'
- История полностью сохраняется
- Сущность возвращается в предыдущий статус

### БП-10: Управление доступом

**1. Scope доступа ПРМ:**
- ПРМ видит все сущности всех продуктов (только просмотр)
- ПРМ управляет только Deployment'ами своего Product (создание, редактирование, архивирование)
- При создании Deployment автоматически проверяется product_id из UserRole
- ПРМ может управлять Simulation/Pilot/Change только в рамках своих Deployment'ов
- ПРМ не может создавать/редактировать Scorecard напрямую (только методолог)
- ПРМ может отправлять свои Pilot/Change на согласование
- ПРМ может формировать Package из своих Pilot/Change

**2. Scope доступа Методолога:**
- Методолог видит все сущности всех продуктов
- Методолог управляет только Scorecard и артефактами (документами)
- Методолог не может создавать Deployment/Simulation/Pilot/Change
- Методолог может отправлять Pilot/Change на согласование (если связаны со Scorecard, которые он создал)
- Методолог может формировать Package

**3. Назначение согласующих:**
- При отправке Pilot/Change/Package на согласование пользователь выбирает согласующих для каждого этапа:
  - Для этапов **Approval**: только пользователи с ролью **Approver**
  - Для этапа **Ratification**: только пользователи с ролью **Ratifier**
- Нельзя выбрать самого себя при отправке на согласование
- Approver видит только раздел "Согласования" и только те Pilot/Change/Package, где он назначен согласующим на этапах Approval
- Ratifier видит только раздел "Согласования" и только те Pilot/Change/Package, где он назначен утверждающим на этапе Ratification
- ПРМ/Методолог, назначенные Approver/Ratifier, видят свои назначения + свою обычную работу

**4. Множественные роли:**
- User может иметь несколько UserRole (например, prm для Product A + approver)
- При входе в раздел "Согласования" показываются все назначения пользователя
- Админ видит все согласования всех пользователей

**5. Матрица прав доступа:**

| Действие | ПРМ (свой продукт) | Методолог | Approver | Ratifier | Админ |
|----------|-------------------|-----------|----------|----------|-------|
| Просмотр всех сущностей | ✓ | ✓ | ✗ | ✗ | ✓ |
| Создание Deployment | ✓ | ✗ | ✗ | ✗ | ✓ |
| Управление Simulation/Pilot/Change | ✓ | ✗ | ✗ | ✗ | ✓ |
| Создание/редактирование Scorecard | ✗ | ✓ | ✗ | ✗ | ✓ |
| Управление артефактами (своих сущностей) | ✓ | ✓ (всех) | ✗ | ✗ | ✓ |
| Отправка на согласование | ✓ | ✓ | ✗ | ✗ | ✓ |
| Формирование Package | ✓ | ✓ | ✗ | ✗ | ✓ |
| Просмотр раздела "Согласования" | ✓ (свои) | ✓ (свои) | ✓ (свои) | ✓ (свои) | ✓ (все) |
| Согласование (Approval stages) | ✓ (если назначен) | ✓ (если назначен) | ✓ | ✗ | ✓ |
| Утверждение (Ratification stage) | ✓ (если назначен) | ✓ (если назначен) | ✗ | ✓ | ✓ |
| Архивирование Deployment | ✓ | ✗ | ✗ | ✗ | ✓ |
| Rollback Change | ✓ | ✗ | ✗ | ✗ | ✓ |

### БП-10: Управление артефактами

- **Артефакты - это ссылки на внешние документы** (URL), НЕ загружаемые файлы
- Артефакты могут быть привязаны к: Deployment, Simulation, Pilot, Change
- entity_type + entity_id идентифицируют родительскую сущность
- Типы артефактов: business_case, rov, other
- Скоркарты НЕ являются артефактами (отдельная сущность)
- Валидация: проверка формата URL

**Права на управление артефактами:**
- **ПРМ** может управлять артефактами, прикрепленными к своим Deployment/Simulation/Pilot/Change
- **Методолог** может управлять артефактами, прикрепленными к любым сущностям (глобальный доступ)
- **Администратор** может управлять всеми артефактами
- Артефакты наследуют права доступа родительской сущности

## Управление жизненным циклом

### Жизненный цикл внедрения

```
draft → active → deployed
  ↓       ↓
archived (только если нет deployed изменений)
```

**Автоматический расчет статуса:**
- **draft**: Внедрение пустое ИЛИ все вложенные сущности в draft
- **active**: Хотя бы одна вложенная сущность перешла в следующий статус (running, requires_activation, requires_approval, active, approved и т.д.)
- **deployed**: Хотя бы одно Изменение в статусе deployed

**Ручной переход:**
- **любой статус → archived**: Пользователь архивирует Внедрение
  - Все вложенные сущности также переводятся в archived
  - **Ограничение**: Нельзя архивировать, если есть Изменения в deployed

**Восстановление:**
- archived → draft: Пользователь восстанавливает Внедрение (все вложенные сущности восстанавливаются)

### Жизненный цикл симуляции

```
draft → running → completed
                → failed
```

**Переходы:**
- draft → running: Пользователь запускает Симуляцию
- running → completed: Симуляция успешно завершена
- running → failed: Симуляция провалилась
- running → draft: Пользователь отменяет Симуляцию
- failed → draft: Пользователь редактирует и перезапускает

### Жизненный цикл пилота

```
draft → requires_activation → active → completed
                                     → requires_correction → active
```

**Переходы:**
- draft → requires_activation: Пользователь отправляет на активацию
- requires_activation → active: Одобрено и активировано
- requires_activation → draft: Пользователь редактирует
- active → completed: Пилот успешно завершен
- active → requires_correction: Обнаружены проблемы, требуется корректировка
- requires_correction → active: Одобрено и реактивировано
- requires_correction → draft: Пользователь редактирует

### Жизненный цикл изменения

```
draft → requires_approval → approved → deployed → rolled_back
```

**Переходы:**
- draft → requires_approval: Пользователь отправляет на одобрение
- requires_approval → approved: Одобрено всеми согласующими
- requires_approval → draft: Пользователь редактирует
- approved → deployed: Пользователь разворачивает в продакшн
- deployed → rolled_back: Пользователь откатывает развертывание

### Жизненный цикл скоркарты

```
draft → requires_approval → in_approval → in_ratification → approved → active → archived
                                                          → rejected → draft
```

**Переходы:**
- draft → requires_approval: Скоркарта прикреплена к Пилоту/Изменению, которое отправлено на согласование
- requires_approval → in_approval: Процесс согласования начат
- in_approval → in_ratification: Этап согласования пройден, переход к утверждению
- in_approval → rejected: Отклонено на этапе согласования
- in_ratification → approved: Утверждено руководством
- in_ratification → rejected: Отклонено на этапе утверждения
- approved → active: Связанный Пилот активирован или Изменение развернуто
- active → archived: Скоркарта больше не используется
- rejected → draft: Пользователь редактирует и переотправляет

**Важно:**
- Скоркарта проходит согласование/утверждение вместе с Пилотами и Изменениями, к которым она прикреплена
- Статус Скоркарты синхронизируется со статусами связанных Пилотов/Изменений
- Одна Скоркарта может быть прикреплена к нескольким Пилотам/Изменениям, находящимся на разных этапах согласования
- В таком случае статус Скоркарты определяется наиболее продвинутым этапом среди всех связанных сущностей

### Жизненный цикл пакета

**Package НЕ имеет жизненного цикла и статусов** - это просто группировочная сущность для истории утверждения.

**Статусы принадлежат PackageItem:**

```
pending → ratified
        → ratification_rejected
```

**Переходы PackageItem:**
- pending: Создан при отправке Package на утверждение
- pending → ratified: Утверждающий одобрил
- pending → ratification_rejected: Утверждающий отклонил

**Важно:**
- Package создаётся ТОЛЬКО при отправке на утверждение (нет draft состояния)
- Package хранится навсегда для истории утверждения (аудит)
- Каждый Pilot/Change в Package имеет свой индивидуальный статус (PackageItem.status)
- Package поддерживает частичное утверждение (каждый Pilot/Change решается индивидуально)

## Процесс согласования

### Динамическое согласование

Система использует **динамическое создание маршрутов согласования** при отправке Pilot/Change на согласование.

**Ключевые принципы:**
- Нет предзаданных маршрутов согласования (ApprovalRoute)
- Пользователь выбирает согласующих при отправке на согласование
- Переменное количество этапов Approval (0..*)
- Опциональный этап Ratification (0..1)
- Минимум 1 этап в сумме (Approval + Ratification)
- Один человек не может быть в нескольких этапах одного ApprovalInstance

**Этапы Approval (Согласование)**
- Переменное количество этапов (0..*)
- Каждый этап: минимум 1 согласующий
- Последовательное выполнение между этапами
- Параллельная работа согласующих внутри этапа
- Все согласующие этапа должны одобрить для перехода к следующему этапу
- Любое отклонение провалит процесс

**Этап Ratification (Утверждение)**
- Опциональный этап (0..1)
- Ровно 1 утверждающий
- Финальное решение руководства
- Может быть автоматическим (индивидуально) или ручным (через Package)

### Объекты согласования

- **Pilot**: Индивидуальное согласование пилота (статусы: requires_activation, requires_correction)
- **Change**: Индивидуальное согласование изменения (статус: requires_approval)
- **Package**: Группировка Pilot/Change для совместного утверждения
  - Может содержать Pilot/Change из разных Deployments
  - Поддерживает частичное согласование (каждый элемент решается индивидуально)
  - Создаётся вручную пользователем при отправке на согласование
- **Скоркарта**: Проходит согласование вместе с Pilot/Change, к которым она прикреплена

### Workflow согласования

**Шаг 1: Отправка на согласование**
- ПРМ отправляет Pilot (requires_activation/requires_correction) или Change (requires_approval)
- При отправке пользователь выбирает:
  - Этапы Approval (0..*): для каждого этапа указывает список согласующих (минимум 1)
    - **Можно выбрать только пользователей с ролью Approver**
  - Этап Ratification (0..1): указывает ровно 1 утверждающего
    - **Можно выбрать только пользователей с ролью Ratifier**
  - Минимум 1 этап в сумме
  - Один человек не может быть в нескольких этапах
  - **Нельзя выбрать самого себя**
- Создаётся ApprovalInstance с target_type/target_id (Pilot/Change)
- Создаются ApprovalStage для каждого этапа
- Создаются ApprovalAssignment для каждого согласующего
- Email уведомления согласующим первого этапа

**Шаг 2: Этапы Approval (если есть)**
- Этапы выполняются последовательно (stage_order: 1, 2, 3...)
- Внутри этапа согласующие работают параллельно
- Каждый согласующий: approve (с комментарием) или reject (комментарий обязателен)
- Если хотя бы один reject → ApprovalInstance.status = rejected, процесс завершён
- Если все в этапе approve → переход к следующему этапу Approval
- Если все этапы Approval пройдены → переход к Ratification (если есть) или завершение

**Шаг 3: Этап Ratification (если есть)**
- Проверяется флаг ApprovalInstance.auto_ratification

**Вариант A: auto_ratification = true (индивидуальное утверждение)**
- Pilot/Change автоматически переходит на этап Ratification
- Email уведомление утверждающему
- Утверждающий принимает решение: ratify или reject
- Если ratify → Pilot/Change переходит в approved/ratified
- Если reject → Pilot/Change переходит в rejected

**Вариант B: auto_ratification = false (утверждение через Package)**
- Pilot/Change попадает в состояние "awaiting_ratification" (флаг в таблице)
- ПРМ/Методолог вручную создаёт Package:
  - Выбирает Pilot/Change из любых Deployments в состоянии awaiting_ratification
  - При создании Package выбирает согласующих для этапов Approval (0..*) и Ratification (0..1)
  - Создаётся ApprovalInstance с target_type = 'package'
- Отправляет Package на согласование
- Package проходит те же этапы Approval (если есть), затем Ratification

**Шаг 4: Частичное согласование Package**
- При согласовании Package согласующий видит список всех Pilot/Change в Package
- Согласующий может принять решение по каждому элементу индивидуально:
  - approve/ratify конкретный Pilot/Change
  - reject конкретный Pilot/Change
- Решения сохраняются в таблице package_item (status: pending/ratified/ratification_rejected)
- Package считается обработанным, когда все согласующие приняли решения по всем элементам

**Шаг 5: Отзыв из согласования (Recall)**
- ПРМ может отозвать Pilot/Change/Package из согласования в любой момент
- История согласования сохраняется (ApprovalInstance.status = 'cancelled')
- Pilot/Change возвращается в предыдущий статус (draft или текущий рабочий статус)
- Можно переотправить на согласование с новыми согласующими

**Шаг 6: Удаление элементов из Package**
- ПРМ может удалить отдельный Pilot/Change из Package даже после отправки на согласование
- Удалённый элемент возвращается в awaiting_ratification
- Package продолжает согласование с оставшимися элементами

**Шаг 7: После утверждения**
- Pilot: может быть активирован (status = active)
- Change: может быть развёрнут (status = deployed)

### Правила согласования

- Pilot может быть отправлен на согласование в статусах: requires_activation, requires_correction
- Change может быть отправлен на согласование в статусе: requires_approval
- Lineage НЕ является объектом согласования (только визуализация связей)
- Package создаётся ТОЛЬКО при отправке на утверждение (Ratification), НЕТ draft состояния
- Package может содержать Pilot/Change из разных Deployments
- При отправке на согласование пользователь выбирает согласующих для каждого этапа
- Минимум 1 этап согласования (Approval или Ratification)
- Один человек не может быть в нескольких этапах одного ApprovalInstance
- Package поддерживает частичное согласование (каждый элемент решается индивидуально)
- Можно отозвать Pilot/Change/Package из согласования в любой момент (история сохраняется)
- Можно удалить отдельный Pilot/Change из Package даже после отправки на согласование
- Скоркарты, прикрепленные к Pilot/Change, автоматически переходят в соответствующие статусы
- Статус Скоркарты синхронизируется со статусами всех связанных Pilot/Change

## Единый язык

### Основные термины

- **Внедрение** - Группировка верхнего уровня всей работы, связанной со стратегическим изменением
- **Симуляция** - Анализ исторических данных для прогноза эффектов
- **Пилот** - Контролируемый эксперимент с реальными клиентами
- **Изменение** - Продуктовое внедрение модификации стратегии
- **Lineage** - Визуализация происхождения, связывающая Симуляции → Пилоты → Изменения через таблицы источников
- **Скоркарта** - Скоркарта кредитного решения с конфигурацией и финансовыми эффектами
- **Шаблон скоркарты** - Версионированный шаблон для создания скоркарт
- **Пакет** - Группа Pilot/Change для совместного утверждения (Ratification), создаётся только при отправке на утверждение

### Термины ролевой модели

- **Пользователь** - Участник системы с одной или несколькими ролями
- **ПРМ** (Product Risk Manager) - Управляет внедрениями своего продукта, может отправлять на согласование и формировать пакеты
- **Методолог** - Управляет скоркартами и документами, может отправлять на согласование и формировать пакеты
- **Approver** - Согласующий на этапах Approval, видит только раздел "Согласования" и свои назначения
- **Ratifier** - Утверждающий на этапе Ratification, видит только раздел "Согласования" и свои назначения
- **Администратор** - Полный доступ ко всем функциям системы
- **Назначение на согласование** - Привязка пользователя к этапу согласования
- **Множественные роли** - Пользователь может иметь несколько ролей одновременно (например, ПРМ + Approver + Ratifier)
- **Scope доступа** - Область видимости и управления для роли (продукт для ПРМ, глобальная для методолога/Approver/Ratifier)

### Термины скоркарт

- **Шаблон скоркарты** - Определение шаблона для продукта
- **Версия шаблона скоркарты** - Версионированный шаблон с конфигурацией по умолчанию
- **Скоркарта** - Экземпляр, созданный из шаблона
- **Версия скоркарты** - Версионированная конфигурация скоркарты
- **Источник скоркарты** - Происхождение данных скоркарты (Симуляция или Пилот)
- **Ручная скоркарта** - Скоркарта, созданная напрямую пользователем
- **Скоркарта из источника** - Скоркарта, созданная из результатов Симуляции или Пилота
- **Согласование скоркарты** - Скоркарта проходит согласование вместе с Пилотами/Изменениями, к которым она прикреплена
- **Синхронизация статусов** - Статус Скоркарты автоматически синхронизируется со статусами связанных Пилотов/Изменений

### Термины жизненного цикла

- **Черновик** - Начальное состояние, может быть отредактировано
- **Активна** - В процессе, выполняется
- **Требует активации** - Пилот готов к согласованию для запуска
- **Требует корректировки** - Пилот нуждается в корректировке
- **Требует одобрения** - Изменение готово к согласованию для развертывания
- **Одобрено** - Одобрено всеми согласующими, готово к развертыванию
- **Внедрено** - В продакшне
- **Завершено** - Успешно завершено
- **Откачено** - Возвращено к предыдущему состоянию
- **Архивировано** - Удалено из активного использования

### Термины согласования

- **Процесс согласования** - Многоэтапный процесс с динамическим выбором согласующих
- **Динамический маршрут** - Маршрут согласования создаётся при отправке, не предзадан
- **Этап Approval** - Этап согласования риск-менеджерами (0..* этапов)
- **Этап Ratification** - Этап утверждения руководством (0..1 этап)
- **Согласующий** - Лицо, принимающее решение на этапе Approval
- **Утверждающий** - Лицо, принимающее финальное решение на этапе Ratification
- **Решение по согласованию** - Индивидуальный выбор (approve/reject/ratify)
- **Частичное согласование** - Возможность принять решение по каждому элементу Package индивидуально
- **Отзыв из согласования** - Возврат Pilot/Change/Package из процесса согласования с сохранением истории
- **Автоматическое утверждение** - Pilot/Change переходит на Ratification индивидуально (auto_ratification = true)
- **Ручное утверждение** - Pilot/Change ожидает формирования Package для совместного утверждения (auto_ratification = false)
- **Approval** - Этап согласования риск-менеджерами (может быть несколько этапов)
- **Ratification** - Этап утверждения руководством (опциональный, один этап)

### Термины связей

- **Lineage** - Визуализация происхождения: Симуляции → Пилоты → Изменения (через таблицы источников)
- **Связь** - Ассоциация между сущностями (например, Пилот связан со Скоркартой)
- **Источник** - Происхождение данных (Симуляция или Пилот как источник для Скоркарты)
- **Объект согласования** - Сущность, проходящая процесс согласования (Pilot, Change или Package)

## Ключевые изменения от v2.1

### Архитектурные изменения

1. **Введение Внедрения**
   - Новый агрегат верхнего уровня
   - Группирует всю связанную работу
   - Предоставляет контекст для Симуляций, Пилотов, Изменений

2. **Декомпозиция Stage**
   - v2.1: Единая полиморфная сущность Stage
   - v3.0: Три отдельных агрегата (Симуляция, Пилот, Изменение)
   - Более четкое разделение ответственности
   - Упрощенная модель данных (без полиморфизма)

3. **Скоркарта как полноценный агрегат**
   - v2.1: Скоркарты были артефактами
   - v3.0: Скоркарты - отдельные сущности с версионированием
   - Связи многие-ко-многим с Пилотами и Изменениями
   - Явное отслеживание источников

4. **Lineage как визуализация происхождения**
   - v2.1: Внедрение было первичным агрегатом
   - v3.0: Lineage представляет визуализацию цепочки "создано из" (Симуляции → Пилоты → Изменения)
   - Lineage НЕ является CRUD-сущностью, только визуализация
   - Отслеживание происхождения через таблицы источников (simulation_source, pilot_source, change_source)

5. **Шаблоны скоркарт**
   - Новый контекст для управления шаблонами
   - Версионированные шаблоны с конфигурациями по умолчанию
   - Разделение шаблона и экземпляра

### Изменения связей

1. **Связи скоркарт**
   - v2.1: Скоркарта ссылалась из Stage
   - v3.0: Многие-ко-многим с Пилотами и Изменениями
   - Позволяет A/B тестирование и мульти-скоркартные стратегии

2. **Связь Пилот-Изменение**
   - v2.1: Оба были типами Stage
   - v3.0: Связь Пилот (многие) ←→ Изменение (многие) через Скоркарты
   - Пилот может быть связан с несколькими Изменениями через общие Скоркарты
   - Изменение наследует связи через источники Скоркарт

3. **Независимость Симуляции**
   - v2.1: Симуляция была типом Stage
   - v3.0: Симуляция - независимый агрегат
   - Может существовать без Пилота или Изменения
   - Множество Симуляций на Цепочку

### Изменения статусов

1. **Lineage вместо статусов**
   - v2.1: draft → simulation → approval → pilot → prod → archive
   - v3.0: Lineage - визуализация происхождения через таблицы источников (simulation_source, pilot_source, change_source)
   - Статусы перенесены в отдельные сущности (Simulation, Pilot, Change)
   - Lineage НЕ является CRUD-сущностью

2. **Новые статусы**
   - Пилот: requires_activation, requires_correction
   - Изменение: requires_approval
   - Внедрение: draft, active, completed, archived

### Изменения согласования

1. **Время согласования**
   - v2.1: Сущности могли быть отправлены на согласование на любом этапе
   - v3.0: Нельзя отправить Lineage на согласование (только визуализация)
   - Pilot и Change проходят согласование индивидуально
   - Должны быть выполнены специфические условия (пилот требует активации, изменение требует одобрения)

2. **Ограничения пакета**
   - v2.1: Сущности могли группироваться свободно
   - v3.0: Package создаётся ТОЛЬКО при отправке на утверждение (Ratification), может содержать любые Pilot/Change из любых Deployments
   - Package НЕ имеет статусов - статусы принадлежат PackageItem

### Изменения модели данных

1. **Удаленные таблицы**
   - stage (заменена на simulation, pilot, change)
   - stage_type (больше не нужна)
   - criticality_assessment (перенесена в deployment priority)

2. **Новые таблицы**
   - initiative
   - scorecard_template
   - scorecard_template_version
   - scorecard
   - scorecard_version
   - scorecard_source
   - pilot_scorecard
   - change_scorecard
   - deployment_simulation

3. **Измененные таблицы**
   - deployment: теперь ссылается на change_id (UK), pilot_id, initiative_id
   - package: структура без изменений
   - artifact: теперь полиморфная (entity_type + entity_id)

## Путь миграции от v2.1 к v3.0

### Шаги миграции данных

1. **Создание Внедрений**
   - Сгруппировать существующие сущности по product
   - Создать Внедрение для каждой группы
   - Создать записи в deployment_sub_product для связи с подпродуктами
   - Установить статус на основе статусов вложенных сущностей

2. **Декомпозиция Stages**
   - stage.stage_type = 'simulation' → таблица simulation
   - stage.stage_type = 'experiment' → таблица pilot
   - stage.stage_type = 'implementation' → таблица change
   - Сохранить все полиморфные поля

3. **Создание шаблонов скоркарт**
   - Извлечь уникальные конфигурации скоркарт
   - Создать шаблоны по продуктам
   - Создать версии шаблонов с конфигурациями по умолчанию

4. **Конвертация артефактов в скоркарты**
   - artifact.doc_type = 'scorecard' → таблица scorecard
   - Создать scorecard_version для каждого артефакта
   - Связать с пилотами и изменениями

5. **Создание записей Lineage (таблицы источников)**
   - Для каждой связи v2.1 между сущностями:
   - Создать записи в simulation_source, pilot_source, change_source
   - Отслеживать происхождение через Скоркарты
   - Lineage НЕ создаётся как отдельная сущность - только записи в таблицах источников

6. **Обновление Пакетов**
   - Удалить статусы Package (статусы переносятся в PackageItem)
   - Создать PackageItem для каждого Pilot/Change в Package
   - Package теперь создаётся только при отправке на утверждение

### Миграция кода

1. **Обновление доменных моделей**
   - Заменить Stage на Simulation/Pilot/Change
   - Добавить агрегат Deployment
   - Добавить агрегат Scorecard

2. **Обновление репозиториев**
   - Создать DeploymentRepository
   - Разделить StageRepository на SimulationRepository, PilotRepository, ChangeRepository
   - Создать ScorecardRepository

3. **Обновление сервисов**
   - Создать LineageService для визуализации происхождения (чтение из таблиц источников)
   - Создать ScorecardService для управления скоркартами
   - Обновить ApprovalService для новых правил согласования (разделение Approver/Ratifier)

4. **Обновление UI**
   - Добавить экраны управления Внедрениями
   - Разделить экраны Simulation/Pilot/Change
   - Добавить экраны управления Скоркартами
   - Добавить компонент LineageVisualization для отображения визуализации происхождения
