# CLAUDE.md

Этот файл содержит руководство для Claude Code (claude.ai/code) при работе с кодом в этом репозитории.

## Обзор проекта

Это модель домена и спецификация архитектуры для **Системы управления внедрениями риск-стратегий** (v3.1). Система управляет жизненным циклом изменений кредитных риск-стратегий через симуляции, пилоты и продуктовые внедрения.

## Архитектура

**Архитектура на основе Инициатив**, где Инициатива является агрегатом верхнего уровня, группирующим всю связанную работу:

- **Инициатива (Initiative)** - Группировка стратегических изменений (draft → active → deployed → archived)
  - Статус рассчитывается автоматически на основе вложенных сущностей
  - draft: пустое ИЛИ все вложенные в draft
  - active: хотя бы одна вложенная сущность не в draft
  - deployed: хотя бы одно Внедрение в deployed
  - archived: переведено вручную (НЕТ каскадного архивирования)
- **Симуляция (Simulation)** - Фаза анализа исторических данных (draft → running → completed/failed)
- **Пилот (Pilot)** - Фаза реального эксперимента (черновик → требует активации → активен → требует корректировки/завершен)
- **Внедрение (Deployment)** - Фаза продуктового внедрения (черновик → требует одобрения → одобрено → внедрено → откачено)
- **Lineage** - Визуализация происхождения: Симуляции (0..*) → Пилоты (0..*) → Внедрения (0..*), отслеживается через таблицы источников
- **Скоркарта (Scorecard)** - Скоркарта кредитного решения с версионированием и отслеживанием источников
- **Пакет (Package)** - Группирует Pilot/Deployment для совместного утверждения (Ratification), создаётся только при отправке

## Ролевая модель

**Роли:**
- **ПРМ (prm)** - Product Risk Manager, управляет инициативами своего продукта
- **Методолог (methodologist)** - управляет скоркартами и документами
- **Approver (approver)** - согласующий на этапах Approval
- **Ratifier (ratifier)** - утверждающий на этапе Ratification
- **Администратор (admin)** - полный доступ

**Матрица прав доступа:**

| Действие | ПРМ (свой продукт) | Методолог | Approver | Ratifier | Админ |
|----------|-------------------|-----------|----------|----------|-------|
| Просмотр всех сущностей | ✓ | ✓ | ✗ | ✗ | ✓ |
| Создание Initiative | ✓ | ✗ | ✗ | ✗ | ✓ |
| Управление Simulation/Pilot/Deployment | ✓ | ✗ | ✗ | ✗ | ✓ |
| Создание/редактирование Scorecard | ✗ | ✓ | ✗ | ✗ | ✓ |
| Управление артефактами (своих сущностей) | ✓ | ✓ (всех) | ✗ | ✗ | ✓ |
| Отправка на согласование | ✓ | ✓ | ✗ | ✗ | ✓ |
| Формирование Package | ✓ | ✓ | ✗ | ✗ | ✓ |
| Просмотр раздела "Согласования" | ✓ (свои) | ✓ (свои) | ✓ (свои) | ✓ (свои) | ✓ (все) |
| Согласование (Approval stages) | ✓ (если назначен) | ✓ (если назначен) | ✓ | ✗ | ✓ |
| Утверждение (Ratification stage) | ✓ (если назначен) | ✓ (если назначен) | ✗ | ✓ | ✓ |
| Архивирование Initiative | ✓ | ✗ | ✗ | ✗ | ✓ |
| Rollback Deployment | ✓ | ✗ | ✗ | ✗ | ✓ |

**Управление артефактами:**
- **Артефакты - это ссылки на внешние документы** (URL), НЕ загружаемые файлы
- Артефакты могут быть прикреплены к Initiative, Simulation, Pilot, Deployment
- ПРМ управляет артефактами своих сущностей (своего продукта)
- Методолог управляет артефактами всех сущностей (глобальный доступ)
- Артефакты наследуют права доступа родительской сущности

**Множественные роли:**
- Пользователь может иметь несколько ролей одновременно
- ПРМ и методолог могут быть назначены Approver или Ratifier
- Роль ПРМ привязана к конкретному продукту (product_id NOT NULL)
- Роли методолог, approver, ratifier, админ глобальные (product_id IS NULL)

**Назначение на согласование:**
- При отправке Pilot/Deployment на согласование пользователь выбирает согласующих для каждого этапа:
  - Для этапов **Approval**: только пользователи с ролью **Approver**
  - Для этапа **Ratification**: только пользователи с ролью **Ratifier**
- При создании Package для утверждения пользователь выбирает утверждающего (Ratifier)
- Нельзя выбрать самого себя при отправке на согласование
- Approver видит только те Pilot/Deployment, где он назначен согласующим на этапах Approval
- Ratifier видит только те Pilot/Deployment/Package, где он назначен утверждающим на этапе Ratification
- ПРМ/Методолог, назначенные Approver/Ratifier, видят свои назначения + свою обычную работу

## Ключевые концепции домена

### Lineage инициативы
```
Симуляции (множество) → Пилоты (опционально, множество) → Внедрения (множество)
```

Допустимые пути Lineage:
1. Только симуляции (исследование)
2. Симуляции + Внедрение (пропуск пилота)
3. Симуляции + Пилот (эксперимент)
4. Симуляции + Пилот + Внедрение (полное внедрение)
5. Пилот + Внедрение (пропуск симуляций)
6. Только Внедрение (прямое внедрение)

**Важно:** Lineage НЕ является CRUD-сущностью - это визуализация происхождения через таблицы источников (simulation_source, pilot_source, deployment_source).

### Связи Скоркарт
- Многие-ко-многим с Пилотами (позволяет A/B/C тестирование)
- Многие-ко-многим с Внедрениями (позволяет мульти-скоркартные стратегии)
- Многие-ко-многим с Подпродуктами (если не указаны, относится ко всем подпродуктам Product)
- Скоркарты могут создаваться вручную или из результатов Симуляций/Пилотов
- Отслеживание источников через таблицу `scorecard_source`
- Пилоты и Внедрения, связанные со Скоркартой, наследуют связи с подпродуктами

### Наследование связей через Скоркарты
- **Внедрение наследует связи через источники Скоркарт**:
  - Если Внедрение создано из Скоркарт, источником которых были Симуляции → наследует связи с Симуляциями
  - Если Внедрение создано из Скоркарт, источником которых были Пилоты → наследует связи с Пилотами
- **Пилот может быть связан с несколькими Внедрениями** через общие Скоркарты (многие-ко-многим)

### Связи с подпродуктами
- Инициатива принадлежит одному Product и может быть связана с несколькими SubProduct (многие-ко-многим)
- Скоркарта может относиться к Product в целом или к одному/нескольким SubProduct (многие-ко-многим)
- Если подпродукты не указаны, сущность относится ко всем подпродуктам Product
- Пилоты и Внедрения наследуют связи с подпродуктами через связанные Скоркарты

### Процесс согласования
- Двухэтапное согласование: Approval (согласование) → Ratification (утверждение)
- Нельзя отправить Lineage на согласование (только визуализация)
- Pilot и Deployment проходят согласование индивидуально
- Согласование Пилота: когда status = 'requires_activation' или 'requires_correction'
- Согласование Внедрения: когда status = 'requires_approval'
- Package создаётся ТОЛЬКО при отправке на утверждение (Ratification), НЕТ draft состояния

## Технические паттерны и best practices

### Критичный CSS-паттерн: width: '100%' для корневых flex-контейнеров

**ВАЖНО:** Все корневые Box компоненты в представлениях и формах ДОЛЖНЫ иметь `width: '100%'`.

**Проблема:**
- Flex-элементы по умолчанию имеют `min-width: auto`, что означает "не сжимайся меньше размера контента"
- Если контент внутри вкладок имеет разную ширину, flex-контейнер будет "распираться" и менять ширину
- Это приводит к "скачущей" ширине при переключении между вкладками или изменении контента

**Решение:**
```jsx
// ПРАВИЛЬНО - корневой Box с width: '100%'
function MyView() {
    return (
        <Box sx={{ width: '100%' }}>
            {/* контент */}
        </Box>
    );
}

// НЕПРАВИЛЬНО - без width
function MyView() {
    return (
        <Box>
            {/* контент */}
        </Box>
    );
}
```

**Где применять:**
- Корневой Box в App компоненте (с `display: 'flex'`)
- Все компоненты представлений (InitiativesView, SimulationsView, PilotsView, DeploymentsView, ScorecardsView, PackagesView)
- Все детальные представления (InitiativeDetailView, SimulationDetailView, PilotDetailView, DeploymentDetailView, ScorecardDetailView)
- Все формы (CreateInitiativeForm и т.д.)

**Дополнительно:**
- Для корневого flex-контейнера также можно добавить `min-width: 0` (хотя `width: '100%'` обычно достаточно)
- Комментарий в коде (строка ~1701) содержит подробное объяснение

## Бизнес-правила (критичные)

**БП-1: Управление инициативой**
- Статус Инициативы рассчитывается автоматически:
  - draft: пустое ИЛИ все вложенные в draft
  - active: хотя бы одна вложенная сущность не в draft
  - deployed: хотя бы одно Внедрение в deployed
- **НЕТ каскадного архивирования** - вложенные сущности НЕ архивируются автоматически
- Нельзя архивировать Инициативу, если есть deployed Внедрения или неархивированные вложенные сущности

**БП-6: Lineage (визуализация происхождения)**
- Lineage НЕ является CRUD-сущностью - это визуализация связей "создано из"
- Отслеживание происхождения через таблицы источников (simulation_source, pilot_source, deployment_source)
- Lineage может включать НЕСКОЛЬКО Внедрений (0..*)
- Lineage может включать НЕСКОЛЬКО Пилотов (0..*)
- Lineage может включать НЕСКОЛЬКО Симуляций (0..*)
- Все сущности должны принадлежать одной Инициативе
- Связь Пилотов с Внедрениями реализуется через общие Скоркарты (многие-ко-многим)
- Backend автоматически создаёт записи в таблицах источников при создании Pilot/Deployment с указанием Scorecards

**БП-4: Управление Пилотом**
- Пилот может быть связан с несколькими Внедрениями (многие-ко-многим через Скоркарты)
- Пилот должен быть связан минимум с одной Скоркартой

**БП-5: Управление Внедрением**
- Внедрение должно быть связано минимум с одной Скоркартой
- Только одно активное внедрение на Внедрение (rollback_at = null)

**БП-8: Управление Пакетом**
- Package создаётся ТОЛЬКО при отправке на утверждение (Ratification)
- НЕТ draft состояния - создание и отправка происходят одновременно
- Package может содержать Pilot и/или Deployment из ЛЮБЫХ Initiatives
- Критичность Package = MAX(priority) среди всех Pilot/Deployment (вычисляется на лету)
- Package НЕ имеет статусов - статусы принадлежат PackageItem

## Ограниченные контексты (Bounded Contexts)

1. **Идентификация и доступ** - Управление пользователями, ролями и правами доступа
2. **Каталог продуктов** - Иерархия Product/SubProduct
3. **Шаблоны скоркарт** - Версионированные шаблоны с конфигурациями по умолчанию (без UI, всегда используется default)
4. **Скоркарты** - Фактические экземпляры скоркарт с версионированием и отслеживанием источников
5. **Инициативы** - Группировка верхнего уровня
6. **Исследования и выполнение** - Агрегаты Simulation, Pilot, Deployment
7. **Lineage** - Визуализация происхождения через таблицы источников (simulation_source, pilot_source, deployment_source)
8. **Пакеты** - Группировка Pilot/Deployment для совместного утверждения (Ratification)
9. **Согласование** - Универсальный многоэтапный процесс согласования с разделением ролей Approver/Ratifier

## Единый язык (Ubiquitous Language)

- **Инициатива** (Initiative) - Верхнеуровневая группировка всей работы по стратегическому изменению
- **Симуляция** (Simulation) - Анализ исторических данных для прогноза эффектов
- **Пилот** (Pilot) - Контролируемый эксперимент с реальными клиентами
- **Внедрение** (Deployment) - Продуктовое внедрение модификации стратегии
- **Lineage** - Визуализация происхождения, связывающая Симуляции → Пилоты → Внедрения через таблицы источников
- **Скоркарта** (Scorecard) - Скоркарта кредитного решения с конфигурацией и финансовыми эффектами
- **Пакет** (Package) - Группа Pilot/Deployment для совместного утверждения (Ratification), создаётся только при отправке
- **Approval** - Этап согласования (риск-менеджеры с ролью Approver)
- **Ratification** - Этап утверждения (руководство с ролью Ratifier)
- **Approver** - Роль согласующего на этапах Approval
- **Ratifier** - Роль утверждающего на этапе Ratification
