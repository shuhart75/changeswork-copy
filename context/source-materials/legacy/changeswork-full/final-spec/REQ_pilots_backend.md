## Бизнес-требования

### Цель и границы задачи

#### Цель
Реализовать **Backend API для управления Пилотами** — контролируемыми экспериментами с реальными клиентами для тестирования новых риск-стратегий, включая CRUD операции, управление жизненным циклом, связи со скоркартами и интеграцию с процессом согласования/утверждения.

### Бизнес-правила

#### БП-1: Создание и управление Пилотами
- Пилот создаётся в статусе `draft`
- Пилот должен быть связан минимум с одной Скоркартой
- Пилот принадлежит одной Инициативе
- Пилот может быть связан с несколькими Подпродуктами (через Скоркарты)
- ПРМ может создавать Пилоты только для своего продукта
- Методолог может создавать Пилоты только для своего продукта

#### БП-2: Жизненный цикл Пилота
Статусы Пилота:
- `draft` — черновик, редактируется
- `requires_activation` — отправлен на согласование для активации
- `in_approval` — на этапе согласования (Approval)
- `approved` — согласован, ожидает утверждения
- `in_ratification` — на этапе утверждения (Ratification)
- `ratified` — утверждён, готов к активации
- `active` — активен, проводится эксперимент
- `requires_correction` — требует корректировки (после анализа результатов)
- `completed` — завершён

Переходы статусов:
- `draft` → `requires_activation` (отправка на согласование)
- `requires_activation` → `in_approval` (начало согласования)
- `in_approval` → `approved` (все этапы Approval пройдены)
- `in_approval` → `draft` (отклонено на Approval или отозвано)
- `approved` → `in_ratification` (отправка на утверждение)
- `in_ratification` → `ratified` (утверждено)
- `in_ratification` → `draft` (отклонено на Ratification или отозвано)
- `ratified` → `active` (активация пилота)
- `active` → `requires_correction` (требует корректировки)
- `active` → `completed` (успешное завершение)
- `requires_correction` → `requires_activation` (отправка скорректированного пилота)
- `requires_correction` → `draft` (возврат к редактированию)

#### БП-3: Связи Пилота
- **Со Скоркартами**: многие-ко-многим, минимум 1 скоркарта обязательна
- **С Инициативой**: многие-к-одному, обязательная связь
- **С Подпродуктами**: наследуется через Скоркарты
- **С Симуляциями**: многие-ко-многим через общие Скоркарты (отслеживание происхождения)
- **С Внедрениями**: многие-ко-многим через общие Скоркарты (для последующего внедрения)

#### БП-4: Версионирование
- Каждый Пилот имеет версию (PilotVersion)
- При создании создаётся версия v1
- При корректировке может создаваться новая версия
- Только одна версия может быть активной одновременно

#### БП-5: Финансовые эффекты
- Пилот наследует финансовые эффекты от связанных Скоркарт
- Финансовые эффекты суммируются по всем Скоркартам
- Отображаются: NPV, IRR, Payback Period, ожидаемый эффект по годам
- Во время активного пилота собираются фактические метрики

#### БП-6: Интеграция с процессом согласования
- Пилот отправляется на согласование через ApprovalInstance
- Может проходить несколько этапов Approval
- Обязательный этап Ratification перед активацией
- Может быть отозван (recalled) на любом этапе согласования
- После корректировки может быть повторно отправлен на согласование

### Ограничения

**Не входит в MVP:**
- Автоматический сбор метрик из внешних систем
- A/B тестирование на уровне отдельных клиентов
- Расширенная аналитика по результатам пилота
- Автоматическое завершение пилота по критериям
- Интеграция с системами мониторинга рисков

## Пользовательские требования к АС КОДА

### Определения

- **Пилот (Pilot)** — контролируемый эксперимент с реальными клиентами для тестирования новой риск-стратегии.
- **Версия Пилота (PilotVersion)** — конкретная версия Пилота с уникальным набором параметров и статусом.
- **Активный Пилот** — версия Пилота в статусе `active` (только одна версия может быть активной).
- **Корректировка** — изменение параметров активного пилота на основе промежуточных результатов.
- **Lineage (Происхождение)** — связь Пилота с Симуляциями и Внедрениями через общие Скоркарты.

### USE CASES

#### Основной сценарий 1: Создание Пилота
1. ПРМ/Методолог открывает форму создания Пилота
2. Выбирает Инициативу (своего продукта)
3. Заполняет основные поля:
   - Название
   - Описание
   - Критичность
   - Периметр тестирования
   - Планируемая длительность
4. Выбирает минимум одну Скоркарту
5. Система автоматически определяет Подпродукты через выбранные Скоркарты
6. Нажимает "Создать"
7. Система создаёт Пилот в статусе `draft` с версией v1

**Альтернативный сценарий 1.1: Создание из Симуляции**
1. ПРМ/Методолог открывает детали завершённой Симуляции
2. Нажимает "Создать Пилот"
3. Система предзаполняет форму данными из Симуляции (название, описание, Скоркарты)
4. Пользователь корректирует данные при необходимости
5. Создаёт Пилот
6. Система автоматически создаёт связь через общие Скоркарты (lineage)

#### Основной сценарий 2: Редактирование Пилота
1. ПРМ/Методолог открывает детали Пилота в статусе `draft`
2. Редактирует поля (название, описание, критичность, периметр)
3. Изменяет связанные Скоркарты (добавляет/удаляет)
4. Сохраняет изменения
5. Система обновляет Пилот

**Альтернативный сценарий 2.1: Попытка редактировать не-draft Пилот**
1. Пользователь пытается редактировать Пилот не в статусе `draft`
2. Система блокирует редактирование
3. Отображается сообщение "Редактирование возможно только в статусе Черновик"

#### Основной сценарий 3: Отправка на согласование
1. ПРМ/Методолог открывает детали Пилота в статусе `draft`
2. Нажимает "Отправить на согласование"
3. Система открывает диалог настройки маршрута согласования
4. Пользователь выбирает:
   - Этапы Approval (0 или более)
   - Для каждого этапа: список Approvers
   - Требуется ли Ratification (обязательно для активации)
   - Ratifier (если Ratification требуется)
5. Подтверждает отправку
6. Система создаёт ApprovalInstance
7. Пилот переходит в статус `requires_activation`

**Альтернативный сценарий 3.1: Попытка отправить без Скоркарт**
1. Пользователь пытается отправить Пилот без связанных Скоркарт
2. Система блокирует отправку
3. Отображается сообщение "Пилот должен быть связан минимум с одной Скоркартой"

#### Основной сценарий 4: Активация Пилота
1. ПРМ/Методолог открывает детали Пилота в статусе `ratified`
2. Нажимает "Активировать Пилот"
3. Система запрашивает подтверждение
4. Пользователь подтверждает
5. Система переводит Пилот в статус `active`
6. Если есть другая активная версия этого Пилота, она автоматически завершается

**Альтернативный сценарий 4.1: Корректировка активного Пилота**
1. ПРМ/Методолог открывает детали Пилота в статусе `active`
2. Анализирует промежуточные результаты
3. Нажимает "Требует корректировки"
4. Система запрашивает причину корректировки
5. Пользователь вводит причину и подтверждает
6. Система переводит Пилот в статус `requires_correction`
7. Пользователь может отредактировать параметры и повторно отправить на согласование

---## Критерии приемки

### Критерий 1: CRUD операции с Пилотами
- ПРМ/Методолог может создать Пилот для своего продукта
- Не может создать Пилот для чужого продукта (403)
- Может редактировать Пилот в статусе `draft`
- Не может редактировать Пилот в других статусах (422)
- Может удалить Пилот в статусе `draft`
- Не может удалить Пилот в других статусах (422)

### Критерий 2: Связи со Скоркартами
- Пилот должен быть связан минимум с одной Скоркартой
- Попытка создать Пилот без Скоркарт блокируется (422)
- Попытка удалить последнюю Скоркарту блокируется (422)
- Подпродукты автоматически определяются через Скоркарты
- Финансовые эффекты корректно суммируются по всем Скоркартам

### Критерий 3: Жизненный цикл
- Пилот создаётся в статусе `draft`
- Переходы статусов соответствуют БП-2
- Недопустимые переходы блокируются (422)
- При активации другая активная версия автоматически завершается
- При корректировке создаётся запись с причиной

### Критерий 4: Интеграция с согласованием
- Пилот можно отправить на согласование из статуса `draft` или `requires_correction`
- Создаётся ApprovalInstance с выбранным маршрутом
- Статус Пилота синхронизируется со статусом ApprovalInstance
- Пилот можно отозвать на любом этапе согласования
- После отзыва Пилот возвращается в статус `draft`

### Критерий 5: Версионирование
- При создании создаётся версия v1
- При корректировке можно создать новую версию
- Только одна версия может быть в статусе `active`
- История версий сохраняется

### Критерий 6: Lineage (происхождение)
- При создании Пилота из Симуляции автоматически создаётся связь через Скоркарты
- Lineage корректно отображается в UI
- Можно проследить путь: Симуляции → Пилоты → Внедрения

## Функциональные требования

### ФТ-1: Управление Пилотами
Система должна предоставлять API для создания, чтения, обновления и удаления Пилотов с валидацией прав доступа по продукту.

### ФТ-2: Управление жизненным циклом
Система должна предоставлять API для управления статусами Пилота с валидацией допустимых переходов и автоматической синхронизацией с ApprovalInstance.

### ФТ-3: Управление связями
Система должна предоставлять API для управления связями Пилота со Скоркартами с валидацией минимального количества (минимум 1).

### ФТ-4: Версионирование
Система должна поддерживать версионирование Пилотов с автоматическим управлением активной версией.

### ФТ-5: Интеграция с согласованием
Система должна интегрироваться с Approval Service для отправки Пилотов на согласование и синхронизации статусов.

### ФТ-6: Отслеживание происхождения
Система должна автоматически создавать связи для отслеживания происхождения (lineage) через общие Скоркарты.

### ФТ-7: Финансовые эффекты
Система должна вычислять и возвращать агрегированные финансовые эффекты по всем связанным Скоркартам.

### ФТ-8: Сбор метрик
Система должна предоставлять API для записи и получения метрик активного пилота.

## Реализация для FRONTEND

### Описание UI

#### Экран: Список Пилотов

**Структура:**
- Таблица/список Пилотов с фильтрацией и сортировкой
- Фильтры: по продукту, статусу, критичности, инициативе
- Кнопка "Создать Пилот"
- Поиск по названию

**Колонки таблицы:**
- ID и название (кликабельное)
- Статус (цветной чип)
- Критичность (цветной чип)
- Продукт
- Инициатива
- Скоркарты (количество)
- Дата создания
- Действия (кнопки)

#### Экран: Детали Пилота

**Вкладки:**
1. **Основная информация**
   - Название, описание
   - Статус, критичность
   - Продукт, подпродукты
   - Инициатива
   - Периметр тестирования
   - Планируемая длительность
   - Даты (создания, активации, завершения)

2. **Скоркарты**
   - Список связанных Скоркарт
   - Кнопки добавления/удаления (если draft)
   - Финансовые эффекты по каждой Скоркарте

3. **Финансовые эффекты**
   - Агрегированные показатели (NPV, IRR, Payback)
   - График эффектов по годам
   - Разбивка по Скоркартам

4. **Метрики пилота**
   - Фактические метрики (если active)
   - Сравнение план/факт
   - Графики динамики

5. **История согласования**
   - Этапы согласования
   - Участники и их решения
   - Комментарии
   - Даты

6. **Lineage (Происхождение)**
   - Визуализация связей: Симуляции → Пилоты → Внедрения
   - Кликабельные элементы для перехода

7. **История версий**
   - Список всех версий
   - Статус каждой версии
   - Даты активации/завершения
   - Причины корректировок

**Действия (кнопки):**
- Редактировать (если draft)
- Удалить (если draft)
- Отправить на согласование (если draft или requires_correction)
- Отозвать (если in_approval/in_ratification)
- Активировать (если ratified)
- Требует корректировки (если active)
- Завершить (если active)

#### Экран: Форма создания/редактирования Пилота

**Поля:**
- Инициатива (выбор из списка своего продукта)
- Название (обязательное)
- Описание (обязательное)
- Критичность (выбор: LOW/MEDIUM/HIGH/CRITICAL)
- Периметр тестирования (текст)
- Планируемая длительность (число, дни)
- Скоркарты (множественный выбор, минимум 1)

**Валидация:**
- Все обязательные поля заполнены
- Выбрана минимум одна Скоркарта
- Инициатива принадлежит продукту пользователя

### Результат

После успешного выполнения действия:
- Создание: переход на детали созданного Пилота
- Редактирование: обновление данных, уведомление об успехе
- Удаление: возврат к списку, уведомление об успехе
- Отправка на согласование: обновление статуса, уведомление
- Активация: обновление статуса, уведомление
- Корректировка: обновление статуса, запрос причины, уведомление
- Завершение: обновление статуса, уведомление

## Реализация BACKEND

### Сервис ПКАП1200 АС КОДА

**Сервис:** Pilot Service (Сервис управления Пилотами)

**Ответственность:**
- Управление жизненным циклом Пилотов
- CRUD операции с Пилотами
- Управление связями со Скоркартами
- Версионирование Пилотов
- Интеграция с Approval Service
- Отслеживание происхождения (lineage)
- Вычисление финансовых эффектов
- Сбор и хранение метрик пилота

### База данных

**Используемые таблицы:**

1. `pilots` — основная информация о Пилотах
2. `pilot_versions` — версии Пилотов
3. `pilot_scorecards` — связь Пилотов со Скоркартами (многие-ко-многим)
4. `pilot_source` — отслеживание происхождения (lineage)
5. `pilot_metrics` — метрики активного пилота
6. `initiatives` — инициативы (связь с Пилотом)
7. `scorecards` — скоркарты (связь с Пилотом)
8. `approval_instances` — процесс согласования (связь с PilotVersion)

**Связи:**
- `pilots` → `pilot_versions` (1:N)
- `pilot_versions` → `pilot_scorecards` → `scorecards` (N:M)
- `pilot_versions` → `initiatives` (N:1)
- `pilot_versions` → `approval_instances` (1:1, через target_type='PILOT_VERSION')
- `pilot_source` → `simulation_versions` (отслеживание происхождения)
- `pilot_versions` → `pilot_metrics` (1:N)

### Структура данных

#### Таблица: pilots

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| id | UUID | да | PK |
| name | VARCHAR(200) | да | Название Пилота |
| description | TEXT | да | Описание |
| initiative_id | UUID | да | FK → initiatives.id |
| created_at | TIMESTAMP | да | Дата создания |
| created_by | UUID | да | Кто создал |
| updated_at | TIMESTAMP | да | Дата последнего изменения |
| updated_by | UUID | да | Кто изменил |

**Индексы:**
- INDEX `(initiative_id)` — для фильтрации по инициативе
- INDEX `(created_by, created_at)` — для списка Пилотов пользователя

---

#### Таблица: pilot_versions

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| id | UUID | да | PK |
| pilot_id | UUID | да | FK → pilots.id |
| version | INT | да | Номер версии (1, 2, 3...) |
| status | VARCHAR(32) | да | draft \| requires_activation \| in_approval \| approved \| in_ratification \| ratified \| active \| requires_correction \| completed |
| criticality | VARCHAR(32) | да | LOW \| MEDIUM \| HIGH \| CRITICAL |
| test_scope | TEXT | нет | Периметр тестирования |
| planned_duration_days | INT | нет | Планируемая длительность в днях |
| activated_at | TIMESTAMP | нет | Дата активации |
| completed_at | TIMESTAMP | нет | Дата завершения |
| correction_reason | TEXT | нет | Причина корректировки |
| correction_requested_at | TIMESTAMP | нет | Дата запроса корректировки |
| created_at | TIMESTAMP | да | Дата создания версии |
| created_by | UUID | да | Кто создал версию |

**Ограничения:**
- UNIQUE `(pilot_id, version)` — уникальность версии в рамках Пилота
- CHECK `status IN ('draft', 'requires_activation', 'in_approval', 'approved', 'in_ratification', 'ratified', 'active', 'requires_correction', 'completed')`
- CHECK `criticality IN ('LOW', 'MEDIUM', 'HIGH', 'CRITICAL')`
- CHECK: только одна версия может быть в статусе `active` для одного pilot_id

**Индексы:**
- INDEX `(pilot_id, version)` — для получения версий
- INDEX `(status)` — для фильтрации по статусу
- INDEX `(activated_at)` — для фильтрации по дате активации

---

#### Таблица: pilot_scorecards

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| id | UUID | да | PK |
| pilot_version_id | UUID | да | FK → pilot_versions.id |
| scorecard_id | UUID | да | FK → scorecards.id |
| added_at | TIMESTAMP | да | Когда добавлена связь |
| added_by | UUID | да | Кто добавил связь |

**Ограничения:**
- UNIQUE `(pilot_version_id, scorecard_id)` — одна Скоркарта не может быть добавлена дважды

**Индексы:**
- INDEX `(pilot_version_id)` — для получения Скоркарт Пилота
- INDEX `(scorecard_id)` — для получения Пилотов по Скоркарте

---

#### Таблица: pilot_source

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| id | UUID | да | PK |
| pilot_version_id | UUID | да | FK → pilot_versions.id |
| source_type | VARCHAR(32) | да | SIMULATION_VERSION |
| source_id | UUID | да | ID источника |
| created_at | TIMESTAMP | да | Когда создана связь |

**Ограничения:**
- CHECK `source_type IN ('SIMULATION_VERSION')`

**Индексы:**
- INDEX `(pilot_version_id)` — для получения источников Пилота
- INDEX `(source_type, source_id)` — для получения Пилотов по источнику

**Примечание:** Связи создаются автоматически при создании Пилота из Симуляции или при выборе Скоркарт, которые были созданы из Симуляций.

---

#### Таблица: pilot_metrics

| Поле | Тип | Обяз. | Описание |
|------|-----|-------|----------|
| id | UUID | да | PK |
| pilot_version_id | UUID | да | FK → pilot_versions.id |
| metric_name | VARCHAR(100) | да | Название метрики |
| metric_value | DECIMAL(15,4) | да | Значение метрики |
| metric_date | DATE | да | Дата метрики |
| recorded_at | TIMESTAMP | да | Когда записана |
| recorded_by | UUID | да | Кто записал |

**Ограничения:**
- UNIQUE `(pilot_version_id, metric_name, metric_date)` — одна метрика на дату

**Индексы:**
- INDEX `(pilot_version_id, metric_date)` — для получения метрик по датам
- INDEX `(pilot_version_id, metric_name)` — для получения метрик по типу

### Методы

#### Метод 1: Получить список Пилотов

**Endpoint:** `GET /api/pilots`

**Query параметры:**
- `productId` (optional): UUID — фильтр по продукту
- `status` (optional): string — фильтр по статусу
- `criticality` (optional): string — фильтр по критичности
- `initiativeId` (optional): UUID — фильтр по инициативе

**Ответ:**
```json
{
  "pilots": [
    {
      "id": "uuid",
      "name": "Пилот новой скоркарты Premium",
      "description": "...",
      "initiative": {
        "id": "uuid",
        "name": "Инициатива Premium Q1 2026",
        "product": { "id": "uuid", "name": "Premium" }
      },
      "currentVersion": {
        "id": "uuid",
        "version": 1,
        "status": "draft",
        "criticality": "HIGH",
        "scorecards": [
          { "id": "uuid", "name": "Скоркарта Premium v2.0" }
        ]
      },
      "createdAt": "2026-03-16T10:00:00Z",
      "createdBy": "uuid"
    }
  ]
}
```

**Логика:**
1. Получить текущего пользователя
2. Найти все Pilots с учётом фильтров
3. Для каждого Pilot загрузить:
   - Initiative с Product
   - Текущую версию (последнюю по version)
   - Скоркарты текущей версии
4. Применить права доступа (ПРМ/Методолог видят все, но могут фильтровать по своему продукту)
5. Вернуть список

**Валидация:**
- Пользователь аутентифицирован

**Ошибки:**
- `401` — не аутентифицирован

#### Метод 3: Создать Пилот

**Endpoint:** `POST /api/pilots`

**Body:**
```json
{
  "name": "Пилот новой скоркарты Premium",
  "description": "...",
  "initiativeId": "uuid",
  "criticality": "HIGH",
  "testScope": "Клиенты сегмента Premium",
  "plannedDurationDays": 90,
  "scorecardIds": ["uuid1", "uuid2"]
}
```

**Ответ:**
```json
{
  "pilot": {...},
  "version": {...}
}
```

**Логика:**
1. Получить текущего пользователя
2. Найти Initiative по initiativeId
3. Проверить права: пользователь может создавать Пилоты только для своего продукта
4. Проверить, что выбрана минимум одна Скоркарта
5. Создать Pilot
6. Создать PilotVersion (version=1, status='draft')
7. Создать связи со Скоркартами (pilot_scorecards)
8. Автоматически создать связи lineage (pilot_source) если Скоркарты имеют источники
9. Вернуть созданный Пилот

**Валидация:**
- Пользователь аутентифицирован
- Пользователь имеет роль ПРМ, Методолог или Админ
- Initiative принадлежит продукту пользователя (для ПРМ/Методолога)
- Выбрана минимум одна Скоркарта
- Все обязательные поля заполнены

**Ошибки:**
- `401` — не аутентифицирован
- `403` — нет прав (Initiative чужого продукта)
- `404` — Initiative или Scorecard не найдены
- `422` — ошибки валидации:
  - Не выбрана ни одна Скоркарта
  - Отсутствуют обязательные поля

```json
  "name": "Новое название",
  "description": "Новое описание",
  "criticality": "CRITICAL",

  "scorecardIds": ["uuid1", "uuid2", "uuid3"]
```

**Ответ:**

```json
}
```

**Логика:**
1. Получить текущего пользователя
2. Найти Pilot по id
3. Проверить, что текущая версия в статусе `draft`
4. Проверить права: пользователь — создатель или Админ
5. Обновить поля Pilot (name, description)
6. Обновить поля PilotVersion (criticality, testScope, plannedDurationDays)
7. Обновить связи со Скоркартами (если изменились)
8. Проверить, что осталась минимум одна Скоркарта
9. Обновить lineage (если изменились Скоркарты)
10. Вернуть обновлённый Пилот

**Валидация:**
- Пользователь аутентифицирован
- Текущая версия в статусе `draft`

- Если изменяются Скоркарты, минимум одна должна остаться

**Ошибки:**
- `401` — не аутентифицирован
- `403` — нет прав
- `404` — Pilot не найден
- `422` — ошибки валидации:
  - Текущая версия не в статусе `draft`
  - Попытка удалить все Скоркарты

- Текущая версия в статусе `draft` или `requires_correction`

**Ошибки:**
- `401` — не аутентифицирован
- `403` — нет прав
- `404` — Pilot не найден
- `422` — ошибки валидации

```json
}
```

**Логика:**
1. Получить текущего пользователя
2. Найти Pilot по id
3. Проверить, что текущая версия в статусе `active`
4. Создать запись в pilot_metrics
5. Вернуть созданную метрику

**Валидация:**
- Пользователь аутентифицирован
- Текущая версия в статусе `active`
- Пользователь имеет роль ПРМ, Методолог или Админ
- Все обязательные поля заполнены

**Ошибки:**
- `401` — не аутентифицирован
- `403` — нет прав
- `404` — Pilot не найден
- `422` — ошибки валидации:
  - Текущая версия не в статусе `active`
  - Метрика с таким именем и датой уже существует

```java
public PilotDTO createPilot(CreatePilotRequest request, User currentUser) {
    // 1. Проверить роль
    if (!currentUser.hasAnyRole("prm", "methodologist", "admin")) {
        throw new ForbiddenException("Недостаточно прав для создания Пилота");
    }
    
    // 2. Найти Initiative
    Initiative initiative = initiativeRepository.findById(request.getInitiativeId())
        .orElseThrow(() -> new NotFoundException("Initiative не найдена"));
    
    // 3. Проверить product scoping (для prm/methodologist)
    if (!currentUser.hasRole("admin")) {
        String productCode = initiative.getProduct().getCode();
        if (!currentUser.getPermittedSpaces().contains(productCode)) {
            throw new ForbiddenException("Пилот может быть создан только для вашего продукта");
        }
    }
    
    // 4. Создать Pilot
    // ...
}
```

### Интеграция с системой аудита

**Метод отправки:**
```java
public void sendAuditEvent(AuditEvent event) {
    // Отправка в Kafka topic "audit-events"
    kafkaTemplate.send("audit-events", event);
}
```

**Вызов после каждой операции:**
```java
public PilotDTO createPilot(CreatePilotRequest request, User currentUser) {
    // 1. Создать Pilot
    Pilot pilot = // ...

    // 2. Отправить аудит-событие
    AuditEvent event = AuditEvent.builder()
        .eventType("PILOT_CREATED")
        .timestamp(Instant.now())
        .userId(currentUser.getId())
        .userName(currentUser.getFullName())
        .entityType("PILOT")
        .entityId(pilot.getId())
        .entityName(pilot.getName())
        .details(Map.of(
            "initiativeId", pilot.getInitiative().getId(),
            "initiativeName", pilot.getInitiative().getName(),
            "productCode", pilot.getInitiative().getProduct().getCode(),
            "criticality", pilot.getCurrentVersion().getCriticality(),
            "scorecardIds", pilot.getCurrentVersion().getScorecards().stream()
                .map(Scorecard::getId)
                .collect(Collectors.toList())
        ))
        .build();

    auditService.sendAuditEvent(event);

    // 3. Вернуть результат
    return pilotMapper.toDTO(pilot);
}
```

### Требования к аудиту

1. **Все операции должны логироваться** — создание, обновление, удаление, изменение статуса
2. **Аудит должен быть синхронным** — событие отправляется сразу после успешного выполнения операции
3. **Аудит не должен блокировать операцию** — если отправка в Kafka не удалась, операция всё равно считается успешной (логируется ошибка)
4. **Аудит должен содержать достаточно информации** для восстановления контекста операции
5. **Аудит должен быть неизменяемым** — записи не могут быть изменены или удалены
