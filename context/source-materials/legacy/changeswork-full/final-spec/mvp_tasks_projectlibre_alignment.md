# MVP Tasks Alignment (source: ProjectLibre `Q2.pod`)

## Source Files

- `sources/projectlibre/Q2.pod` (ProjectLibre)
- `sources/projectlibre/Q2.json` (exported from `.pod` via MPXJ)

## Goal

Использовать `Q2.pod` как источник правды для **декомпозиции и оценок dev-задач** MVP.

Дополнения к `Q2.pod` в рамках этого репозитория:

- Добавлены **задачи аналитики**: по одной аналитической задаче на каждую dev-задачу.
- Добавлена **QA задача** (интеграционная).
- Уточнение скоупа по FE пилотов: в задачу "блок артефактов" включены UI-доработки под адаптированный ЖЦ (+3 дня к оценке, см. ниже).

## Team / Resources

- `A1`, `A2`: Analyst
- `B1`, `B2`, `B3`: Backend
- `F1`: Frontend (100%)
- `F2`: Frontend (50%)
- `Q1`: QA

## Task List (MVP v6.15)

Ниже приведены **все** dev-задачи из `Q2.pod` и добавленные к ним аналитика/QA. Для каждой dev-задачи указан скоуп, который нужно реализовать в рамках задачи.

### 1. Роли

- `AN_ROLES` (1d, A1): роли/права/матрица доступов для MVP.
- `BE_ROLES` (3d, B3): **BE новые роли**
  - Добавить/уточнить роли, используемые в MVP (PRM/approver/admin/etc).
  - Зафиксировать правила видимости и редактирования по продукту.
  - API/guards: единообразная проверка прав для артефактов, согласований, пакетов, скоркарт, trace.

### 2. Артефакты

- `AN_ART_BE` (1d, A2): модель артефактов, права, API контракты (URL only).
- `BE_ART` (3d, B2): **BE артефакты (ссылки)**
  - БД/модель артефакта (url, name, type, created_at/by, owner entity).
  - API: list/add/delete; валидация URL; права.
  - Поддержка сущностей в MVP: Pilot, Simulation (и Deployment, если применимо по текущей спецификации проекта).

- `AN_FE_PILOT_ART` (1d, A2): UX блока артефактов на пилоте (create/edit/detail) + состояния (empty/loading/error).
- `FE_PILOT_ART` (6d, F1): **FE пилоты: блок артефактов + UI под ЖЦ**
  - UI блока артефактов на пилоте: create/edit/detail.
  - Действия add/delete, состояния, права (read-only/disabled).
  - Дополнительно (добавлено сверх `Q2.pod`): UI-доработки пилота под адаптированный ЖЦ (submit/resubmit/recall, отображение статуса текущей версии и т.п.).

- `AN_FE_SIM_ART` (1d, A2): UX блока артефактов на симуляции (create/edit/detail).
- `FE_SIM_ART` (6d, F2 50%): **FE симуляции: блок артефактов**
  - UI блока артефактов на симуляции: create/edit/detail.
  - Состояния, права, валидация.

### 3. Согласования / утверждения

- `AN_AP_CORE` (2d, A1): правила процесса approval/ratification, статусы, recall/reject/resubmit, комментарии per-item.
- `BE_AP_CORE` (10d, B1): **BE процесс согласования и утверждения (core)**
  - БД/логика: ApprovalInstance/Stage/Assignment/Decision.
  - Per-item статусы, комментарии, история.
  - Recall: item, selected items, package.
  - Reject: `approval_rejected` / `ratification_rejected` (без возврата в draft).

- `AN_PILOT_LC_AP` (1d, A2): как approval/ratification встраивается в ЖЦ пилота и версионирование.
- `BE_PILOT_LC_AP` (5d, B2): **BE ЖЦ пилота: согласование/утверждение**
  - Адаптировать существующий ЖЦ пилота для поддержки approval/ratification (через версии).
  - Старая версия активна до момента согласования/утверждения новой версии; потом новая становится current, старая non-active.

- `AN_AP_PAGE_BE` (1d, A1): контракты/фильтры/действия страницы "Согласования" (BE).
- `BE_AP_PAGE` (10d, B1): **BE страница "Согласования": API/логика**
  - Endpoints под FE: списки, фильтры, карточка item, действия approve/reject/recall/resubmit, история/комментарии.

- `AN_AP_PAGE_FE` (1d, A2): UX страницы "Согласования" (FE).
- `FE_AP_PAGE` (10d, F1): **FE страница "Согласования"**
  - UI, фильтры, действия, просмотр истории/комментариев.

- `AN_PKG_BE` (1d, A1): контракты и UX массовых действий страницы "Пакеты" (BE).
- `BE_PKG_PAGE` (10d, B3): **BE страница "Пакеты": API/логика**
  - Пакет как удобная обертка, статусы per-item.
  - Решение по пакету: по всем, по каждому, только по выбранным (checked).
  - Минимальный размер пакета 2 элемента.

- `AN_PKG_FE` (1d, A2): UX страницы "Пакеты" (FE).
- `FE_PKG_PAGE` (10d, F2 50%): **FE страница "Пакеты"**
  - UI выбора items, approve/reject для выбранных, просмотр комментариев и истории.

### 4. Внедрения (added to MVP)

Эти задачи **не входят** в `Q2.pod`, но обязательны для MVP (страница внедрений в прототипе, БД/API и ЖЦ внедрения).

- `AN_DEP_BE` (1d, A1): модель/контракты/права для внедрений (BE).
- `BE_DEP_API` (5d, B3): **BE внедрений: БД+API**
  - БД: сущность Deployment (минимальный набор полей под список и форму).
  - API: list/get/create/update, фильтры по продукту, права (PRM видит все, изменяет только свой продукт).
  - Интеграция с блоками (если включены в MVP): артефакты, связанные сущности (trace).

- `AN_DEP_LC` (1d, A2): правила ЖЦ внедрения (статусы, переходы, ограничения редактирования).
- `BE_DEP_LC` (5d, B3): **BE ЖЦ внедрений**
  - Реализация переходов и правил на уровне БД/сервиса/API.

- `AN_FE_DEP_LIST` (1d, A2): UX списка внедрений (фильтры/бейджи статусов/доступы).
- `FE_DEP_LIST` (5d, F2 50%): **FE списка внедрений**
  - Страница списка, фильтры по продукту, отображение статусов, действия (если предусмотрены).
  - Зависимость: старт после готовности `BE_DEP_API`.

- `AN_FE_DEP_FORM` (1d, A1): UX формы внедрения (создание/редактирование) + edge-cases.
- `FE_DEP_FORM` (10d, F2 50%): **FE формы внедрения (создание/редактирование)**
  - Форма создания и редактирования, валидация, права.
  - Зависимость: старт после готовности `BE_DEP_LC`.

### 5. Скоркарты (без списка)

- `AN_SC_BE` (1d, A1): модель скоркарт/версий/шаблонов и правила валидации.
- `BE_SC_API` (10d, B2): **BE скоркарты (включая шаблоны): БД + API**
  - CRUD/версии/шаблоны скоркарт, необходимые для деталки/формы.
  - Hook-места/интеграция со статусами (если требуется отдельной задачей).

- `AN_SCD_BE` (1d, A2): деталка скоркарты (BE API).
- `BE_SCD` (5d, B3): **BE деталка скоркарты**
  - `GET /api/scorecards/{id}` + sources/usage (если в MVP).

- `AN_SCD_FE` (1d, A1): UX деталки скоркарты (без перехода из списка).
- `FE_SCD` (10d, F1): **FE деталка скоркарты**
  - UI, табы sources/usage, навигация к связанным сущностям.

- `AN_SCF_BE` (1d, A2): форма скоркарты (BE API).
- `BE_SCF` (5d, B1): **BE форма скоркарты (создание/редактирование)**
  - Endpoints create/update, загрузка конфигурации из шаблона.

- `AN_SCF_FE` (1d, A1): UX формы скоркарты (JSON editor, шаблоны).
- `FE_SCF` (10d, F1): **FE форма скоркарты (создание/редактирование)**
  - UI, выбор версии шаблона, загрузка default config, JSON editor.

### 6. Уведомления

- `AN_NOT` (1d, A2): перечень событий, шаблоны писем, получатели.
- `BE_NOT` (5d, B3): **BE уведомления**
  - Email события по approval/ratification, пакетам и т.п.
  - Шаблоны + отправка.

### 7. Trace

- `AN_TRACE_BE` (1d, A1): модель связей/запросов для trace, производительность.
- `BE_TRACE` (5d, B2): **BE Trace: БД + логика + API**
  - Хранение/получение связей между пилотами, симуляциями, внедрениями (через скоркарты и/или created_from).
  - API для блока "Связанные сущности" (read-only).

- `AN_TRACE_FE` (1d, A2): UX виджета "Связанные сущности".
- `FE_TRACE` (5d, F1): **FE Trace: блок "Связанные сущности"**
  - Виджет/блок связей на страницах пилота/симуляции/внедрения (в зависимости от того, где он есть в прототипе).
