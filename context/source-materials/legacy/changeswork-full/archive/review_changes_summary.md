## 🔴 КРИТИЧЕСКИЕ АРХИТЕКТУРНЫЕ ИЗМЕНЕНИЯ

### 1. Chain - НЕ сущность, а визуализация lineage

**Было:**
- Chain - отдельная сущность, создаваемая вручную
- Таблицы: `chain`, `chain_simulation`, `chain_pilot`, `chain_change`
- BE-10: API Chain (CRUD)
- FE-10: Раздел Chains с формами создания/редактирования

**Стало:**
- Chain - визуализация существующих связей "создано из" (lineage)
- Таблицы lineage tracking:
  - `simulation_source`: simulation_id, source_type ('pilot'), source_id
  - `pilot_source`: pilot_id, source_type ('simulation'), source_id (many-to-many)
  - `change_source`: change_id, source_type ('pilot'/'simulation'), source_id (many-to-many)
- BE-10: GET /api/{entity}/{id}/lineage/ (вычисление на лету)
- FE-10: Компонент LineageVisualization (встраивается в детальные страницы)
- НЕТ отдельного раздела "Цепочки" в меню

**Lineage устанавливается автоматически:**
- При создании Pilot/Change с выбором Scorecards
- Backend проверяет источники Scorecards и создаёт записи в lineage таблицах

**Изменения в задачах:**
- BE-1: Удалить таблицы chain*, добавить *_source таблицы
- BE-10: Полностью переписать (GET lineage вместо CRUD)
- BE-11: Убрать зависимость от BE-10
- FE-10: Полностью переписать (визуализация вместо CRUD)

### 2. Package - НЕ имеет статусов, только группировка

**Было:**
- Package со статусами: draft → pending_ratification → ratified/rejected
- Можно создать в draft, редактировать, потом отправить
- Поле priority в таблице package

**Стало:**
- Package БЕЗ статусов - просто группировочная сущность для истории
- Статусы принадлежат items: package_item.status (pending/ratified/ratification_rejected)
- Package создаётся ТОЛЬКО при отправке на утверждение (нет draft)
- Priority вычисляется на лету из items

**Таблица `package`:**
```sql
- id, name, description, created_at, created_by
- ❌ УДАЛИТЬ: status, priority
```

**Workflow:**
1. Выбор Pilot/Change (approved + awaiting_ratification)
2. Кнопка "Отправить на утверждение"
3. Диалог: название, описание, утверждающий
4. Package создаётся И сразу отправляется
5. При отмене: Package НЕ создаётся

**Хранение:**
- Package хранится навсегда для истории утверждения (аудит)
- Не удаляется после завершения

**Изменения в задачах:**
- BE-1: Удалить поля status, priority из package
- BE-11: Упростить API (нет draft, нет PATCH, нет DELETE)
- FE-11: Убрать PackagesView, CreatePackageForm, EditPackageForm

### 3. Артефакты - только ссылки, НЕ файлы

**Было:**
- Загрузка файлов (multipart/form-data)
- Хранение на диске/S3
- Валидация размера (10MB), mime_type (PDF/DOCX/XLSX)
- Скачивание файлов

**Стало:**
- Только ссылки на внешние документы
- Валидация URL формата
- entity_type расширен: deployment, simulation, pilot, change (было только deployment/change)

**Таблица `artifact`:**
```sql
- id, entity_type, entity_id, name, url, type, uploaded_by, uploaded_at
- ❌ УДАЛИТЬ: file_path, file_size, mime_type
- ❌ УДАЛИТЬ: constraint на file_size, mime_type
```

**API:**
- POST /api/artifacts/ - добавление ссылки (НЕ загрузка файла)
- GET /api/artifacts/ - список
- DELETE /api/artifacts/{id}/ - удаление ссылки
- ❌ УДАЛИТЬ: GET /api/artifacts/{id}/download/

**Изменения в задачах:**
- BE-3: Обновить таблицу artifact
- BE-13: Полностью переписать (ссылки вместо файлов)
- FE-13: Убрать FileUpload, добавить URL input

### 5. Роли: Approver и Ratifier - две отдельные роли

**Было:**
- Роль "Согласующий" для всех этапов

**Стало:**
- **Approver** - для этапов Approval
- **Ratifier** - для этапа Ratification
- Две отдельные роли, могут совмещаться

**Таблица user_role:**
```sql
role ENUM('prm', 'methodologist', 'approver', 'ratifier', 'admin')
```

**Правила совмещения:**
- ✅ (ПРМ ИЛИ Методолог ИЛИ Админ) + Approver + Ratifier
- ❌ ПРМ + Методолог

**Ограничения при отправке:**
- Для Approval: только пользователи с ролью Approver
- Для Ratification: только пользователи с ролью Ratifier
- Нельзя выбрать самого себя

**Изменения в задачах:**
- BE-3: Обновить enum роли
- BE-12: Уточнить валидацию выбора согласующих
- BE-14: Обновить middleware для множественных ролей
- FE-7, FE-9, FE-11, FE-12: Обновить UI выбора согласующих
