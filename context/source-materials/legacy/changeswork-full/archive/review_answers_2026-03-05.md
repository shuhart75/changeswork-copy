## BE-7: API Pilot (адаптация + Archive)

### Вопрос: Можно ли создать Pilot из нескольких Simulation?
**Ответ:** Можно. Pilot связан с несколькими Simulations **косвенно через Scorecards**:
1. Simulation → создаёт Scorecard (записывается в `scorecard_source`)
2. Pilot → связывается с несколькими Scorecards (таблица `pilot_scorecard`)
3. Таким образом Pilot связан с несколькими Simulations через их Scorecards

В API при создании Pilot:
- Пользователь выбирает несколько Scorecards (которые могут быть из разных Simulations)
- Pilot создаётся с этими Scorecards
- Линия происхождения сохраняется через `scorecard_source`

## BE-9: API Change (CRUD)

### Вопрос 1: Можно ли создать Change из нескольких Pilot?
**Ответ:** Можно. Работает в точности также, как с созданием одного пилота из нескольких симуляций.

Change связан с несколькими Pilots **косвенно через Scorecards**:
1. Pilot → связан с несколькими Scorecards (таблица `pilot_scorecard`)
2. Change → связывается с несколькими Scorecards (таблица `change_scorecard`)
3. Таким образом Change связан с несколькими Pilots через их общие Scorecards

В API при создании Change:
- Пользователь выбирает несколько Scorecards (которые могут быть из разных Pilots)
- Change создаётся с этими Scorecards
- Линия происхождения сохраняется через связи Scorecards

### Вопрос 2: Что происходит при deployed - нужно ли заполнять deployed_at автоматически?
**Ответ:** Change проходит согласование (Approval stages) → Change утверждается (Ratification stage) → статус становится 'approved' (или 'ratified') → Позже кто-то **вручную** переводит Change в статус 'deployed' → При переводе в 'deployed' автоматически заполняется `deployed_at` = текущая дата/время.

## BE-10: API Chain (CRUD)

### Вопрос 1: Нужна ли валидация на минимальное количество сущностей в Chain?
**Ответ:** Пустую цепочку создавать нельзя.

### КРИТИЧЕСКОЕ АРХИТЕКТУРНОЕ ИЗМЕНЕНИЕ: Chain

**Текущий дизайн в документации (НЕПРАВИЛЬНЫЙ):**
- Chain - это отдельная сущность, которую пользователь создаёт вручную
- Таблицы: `chain`, `chain_simulation`, `chain_pilot`, `chain_change`
- Связи через Scorecards (косвенные)

**ПРАВИЛЬНЫЙ дизайн:**

**1. Chain - это НЕ сущность, а UI визуализация существующих связей "создано из"**
- Таблицы `chain`, `chain_simulation`, `chain_pilot`, `chain_change` **УДАЛИТЬ**
- Chain вычисляется на лету при просмотре детальной страницы
- API endpoint: `GET /api/simulations/{id}/lineage/` или `GET /api/pilots/{id}/lineage/` или `GET /api/changes/{id}/lineage/`

**2. Новые таблицы для отслеживания происхождения (lineage tracking):**
- `simulation_source`: simulation_id, source_type ('pilot'), source_id
  - Одна симуляция может быть создана из одного пилота (one-to-one)
- `pilot_source`: pilot_id, source_type ('simulation'), source_id
  - Один пилот может быть создан из нескольких симуляций (many-to-many)
- `change_source`: change_id, source_type ('pilot'/'simulation'), source_id
  - Одно изменение может быть создано из нескольких пилотов и/или симуляций (many-to-many)

**3. Связи со Scorecards (остаются БЕЗ ИЗМЕНЕНИЙ, существуют параллельно):**
- `pilot_scorecard`: pilot_id, scorecard_id (many-to-many)
- `change_scorecard`: change_id, scorecard_id (many-to-many)
- `scorecard_source`: scorecard_id, source_type ('simulation'/'pilot'/'manual'), source_id

**Пример:**
- К одной скоркарте привязано: Пилот 1, Пилот 2, Симуляция 1
- Источник скоркарты: "создана вручную"
- Связи "создано из": Пилот 1 → Симуляция 1 → Пилот 2

Это две разные системы связей:
- **Scorecards** - показывают, какие скоркарты используются в пилотах/изменениях
- **Lineage** - показывают, на основе чего создана каждая сущность

**4. Package (упрощается):**
- `package`: id, name, description, status, priority, created_at, updated_at, created_by
- `package_item`: id, package_id, item_type ('pilot'/'change'), item_id, status, created_at
- Package группирует отдельные Pilot/Change (НЕ цепочки)

**5. Валидация:**
- `deployment_id` НЕ нужно в lineage таблицах
- Валидация "все из одного Deployment" проверяется на уровне API

**Что нужно изменить в документации:**
- BE-1: Удалить таблицы `chain`, `chain_simulation`, `chain_pilot`, `chain_change`
- BE-1: Добавить таблицы `simulation_source`, `pilot_source`, `change_source`
- BE-10: Полностью переписать - Chain не CRUD API, а GET lineage API
- BE-11: Убрать зависимость от BE-10 (Chain)

## BE-12: Бизнес-логика согласования

### Вопрос 1: Можно ли отозвать свое решение до завершения этапа?
**Ответ:** Нет, решение окончательное сразу после отправки. Изменить свое решение нельзя.

### Вопрос 2: Что происходит если согласующий удален из системы во время процесса?
**Ответ:** Важно состояние в момент согласования:
- **Если согласующий удален ДО того, как принял решение:**
  - Возвращаем Pilot/Change в предшествующий статус (до отправки на согласование)
  - ApprovalInstance → status = 'cancelled'
  - Добавляем комментарий с объяснением причины
  - Отмена происходит **автоматически**
- **Если согласующий удален ПОСЛЕ того, как принял решение:**
  - Его решение остается в силе
  - История сохраняется

### Вопрос 3: Есть ли timeout для принятия решения?
**Ответ:** Нет, timeout нет. Согласующий может принять решение в любое время.

### Вопрос 4: Можно ли изменить список согласующих после отправки?
**Ответ:** Нет, список согласующих нельзя изменить после отправки на согласование. Список фиксируется в момент отправки.

## BE-14: Middleware для проверки прав доступа

### Вопрос 1: Как обрабатывать случай, когда пользователь имеет несколько ролей?
**Ответ:**

**Ролевая модель (уточнение):**
- **Согласующий (Approver)** - отдельная роль для этапов Approval
- **Утверждающий (Ratifier)** - отдельная роль для этапа Ratification
- Это две разные роли, которые могут совмещаться

**Таблица user_role:**
```sql
role ENUM('prm', 'methodologist', 'approver', 'ratifier', 'admin')
```
Один пользователь может иметь несколько записей с разными ролями.

**Правила совмещения ролей:**
- ✅ Можно: (ПРМ ИЛИ Методолог ИЛИ Админ) + Approver + Ratifier
- ✅ Можно: ПРМ + Approver + Ratifier
- ❌ Нельзя: ПРМ + Методолог одновременно

**Права при совмещении:**
- Пользователь получает ВСЕ права своих ролей
- Пример: ПРМ + Approver = все права ПРМ + доступ к разделу "Согласования"

**Ограничения при отправке на согласование:**
- Для этапов **Approval**: можно выбрать только пользователей с ролью **Approver**
- Для этапа **Ratification**: можно выбрать только пользователей с ролью **Ratifier**
- Пользователь **НЕ может выбрать сам себя** при отправке на согласование

### FE-7: Адаптация Pilots + Archive

**Вопрос:** Как создается Pilot из нескольких Simulation?
**Ответ:** Реализовано в макете. В списке Simulations пользователь выбирает несколько симуляций и нажимает кнопку "Создать пилот из выбранных".

**Workflow:**
1. Открывается форма создания нового пилота (CreatePilotFormInline)
2. Автоматически создаются Scorecards по числу выбранных Simulations
3. Для каждой Scorecard:
   - Указано, на базе какой Simulation она создана (sourceSimulationId)
   - Часть полей предзаполнена по результатам Simulation
   - Поля, не определенные в Simulation, остаются дефолтными
4. Если пользователь вручную правит предзаполненные значения → источник Scorecard меняется на 'manual'
5. При сохранении создаются:
   - Новый Pilot
   - Новые Scorecards (если не выбраны существующие)
   - Записи в `pilot_scorecard` (связь Pilot с Scorecards)
   - Записи в `pilot_source` (lineage: Pilot создан из Simulations)
   - Записи в `scorecard_source` (источник каждой Scorecard)

### FE-9: Раздел Changes

**Вопрос:** Можно ли создать Change из нескольких Pilot - как это работает в UI?
**Ответ:** Да, можно. Работает аналогично созданию Pilot из Simulations.

**Два способа создания Change из Pilots:**

1. **Из списка Pilots:** "Создать изменение из выбранных"
   - Пользователь выбирает несколько Pilots в списке
   - Нажимает кнопку "Создать изменение из выбранных"
   - Открывается форма CreateChangeFormInline
   - Автоматически создаются Scorecards по числу выбранных Pilots
   - Для каждой Scorecard предзаполняются поля из Pilot
   - Если пользователь правит значения → источник меняется на 'manual'

2. **При ручном создании Change:**
   - Пользователь создает Change вручную (не из списка)
   - В форме CreateChangeForm выбирает существующие Scorecards
   - Если выбранные Scorecards принадлежат Pilots → автоматически создаются записи в `change_source`
   - Это тоже считается "Change создан из Pilots"

**Backend логика (автоматическое установление lineage):**
- При создании Change с выбранными Scorecards
- Backend проверяет: какие Pilots используют эти Scorecards (через `pilot_scorecard`)
- Автоматически создает записи в `change_source` для этих Pilots
- Lineage устанавливается автоматически

**Аналогично для Pilot из Simulations:**
- При ручном создании Pilot, если выбираются Scorecards из Simulations
- Backend автоматически создает записи в `pilot_source`

### FE-11: Раздел Packages

**После утверждения:**
