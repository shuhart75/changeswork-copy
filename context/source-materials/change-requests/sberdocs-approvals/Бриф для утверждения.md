## Требования Frontend

### 1. Термины и разграничение понятий

| Объект | Назначение | Получатель | Содержимое |
| --- | --- | --- | --- |
| **Документ** | Основной объект согласования, хранится в SberDocs | Утверждающий (подписант) | Структурированные данные формы без скоркарты и доп. эффектов |
| **Письмо-уведомление** | HTML-письмо, отправляемое утверждающему при переходе задачи в работу | Утверждающий (подписант) | Данные из заполненной формы + скоркарта с метриками + поле «Доп. эффекты» |

Frontend не управляет содержимым документа и не отвечает за формирование письма. Оба объекта формируются на backend.

### 3. Детальное описание поведения

#### 3.1 Кнопка «Создать документ»

* Отображается, если документ в текущей записи ещё не создан и утверждающий назначен.
* При нажатии добавляет форму создания на текущую страницу — без перехода на отдельный экран.
* Скрывается или блокируется, если документ уже создан.

#### 3.2 Кнопка «Редактировать документ»

* Отображается только если документ уже сохранён и ещё не отправлен на утверждение.
* При нажатии открывает форму с предзаполненными данными из сохранённого JSON, включая значение поля «Доп. эффекты».
* Скрыта после отправки документа.

#### 3.3 Кнопка «Отправить на утверждение»

* Скрыта или заблокирована с пояснением, пока документ не сохранён.
* Становится активной после сохранения документа.
* После отправки скрывается; повторная отправка невозможна.

#### 3.4 Кнопка «Перейти в SberDocs»

* Отображается только после успешной отправки документа на утверждение.
* Ссылка формируется на основе внешнего идентификатора документа, возвращённого backend.
* Открывает страницу документа в новой вкладке.
* Если backend не вернул внешний идентификатор — отображается сообщение об ошибке отправки.

#### 3.5 Номер документа

* Не вводится пользователем; поле ручного ввода реквизитов в форме отсутствует.
* Присваивается SberDocs в момент создания документа.
* После успешной отправки frontend получает номер от backend и отображает его как read-only атрибут рядом с кнопкой «Перейти в SberDocs».

#### 3.6 Поле «Доп. эффекты»

* Расположено под блоком скоркарты.
* Опциональное текстовое поле, доступное для ввода до отправки документа.
* После отправки документа становится недоступным для редактирования.
* **Не передаётся в документ**, отправляемый в SberDocs.
* Значение сохраняется в JSON документа и включается backend в предпросмотр письма и в итоговое письмо-уведомление.
* Если поле не заполнено — документ сохраняется и отправляется без ошибки; в письме секция доп. эффектов отсутствует или пуста.

#### 3.7 Раздел «Основные правила / изменения»

Раздел поддерживает два типа элементов:

* **Новое правило** — одно текстовое поле с описанием.
* **Изменение (AS IS / TO BE)** — два поля: текущее состояние и целевое состояние.

В режиме создания и редактирования пользователь может добавлять несколько элементов любых типов и удалять отдельные элементы. После отправки на утверждение раздел доступен только для просмотра.

#### 3.8 Предпросмотр письма

* Открывается в модальном окне при нажатии кнопки «Предпросмотр письма».
* Показывает письмо-уведомление — то, что получит утверждающий после перехода задачи в работу.
* Frontend передаёт backend текущие данные формы и значение поля «Доп. эффекты»; backend добавляет данные скоркарты с метриками и возвращает готовый HTML письма.
* Frontend отображает полученный HTML без дополнительной обработки.
* Доступен до и после сохранения документа; валидация не блокирует открытие.
* Модальное окно не содержит редактируемых элементов.
* Повторное открытие не приводит к задвоению контента.

#### 3.9 Скоркарта

* Отображается как read-only блок во всех состояниях страницы.
* Данные скоркарты и метрики не передаются в документ, отправляемый в SberDocs.
* Включаются backend в письмо-уведомление вместе с полем «Доп. эффекты».

### 4. Валидация полей

Клиентская валидация запускается при нажатии «Сохранить» или «Сохранить и отправить на утверждение».

* Незаполненные обязательные поля подсвечиваются с конкретным сообщением рядом с полем.
* Сообщения формулируются конкретно: «Укажите адресата», «Заполните наименование документа» и т. д.
* Валидация не блокирует открытие предпросмотра письма.

| Поле | Обязательность | Примечание |
| --- | --- | --- |
| Адресат | Обязательное | Текстовое поле |
| Наименование документа | Обязательное | Текстовое поле |
| Цель | Обязательное | Текстовое поле |
| Периметр применения | Хотя бы одно из полей должно быть заполнено | — |
| Риски | Необязательное | — |
| Заключение Антифрод | Необязательное | — |
| Доп. эффекты | Необязательное | Не входит в обязательную валидацию |

### 5. Письмо-уведомление утверждающему

#### 5.1 Когда отправляется

Письмо-уведомление не отправляется в момент нажатия кнопки «Отправить на утверждение».

Отправка происходит автоматически на стороне backend, когда при поллинге статуса в SberDocs для задачи с `taskType = "APPROVAL"` значение `taskState` становится `IN_WORK`. Frontend в этот процесс не вовлечён.

#### 5.2 Содержимое письма

Backend формирует письмо в формате HTML. Содержимое:

* данные из заполненной пользователем формы документа;
* скоркарта с метриками;
* значение поля «Доп. эффекты» (если заполнено).

Это тот же HTML, что отображается пользователю в предпросмотре письма.

#### 5.3 Что frontend обязан обеспечить

Frontend корректно передаёт backend все данные формы и значение поля «Доп. эффекты» при сохранении. Никаких дополнительных действий пользователя для отправки письма не требуется.

### 6. Взаимодействие с backend

#### 6.1 Загрузка страницы

Frontend запрашивает у backend:

* данные активной записи: наличие утверждающего, статус, наличие сохранённого JSON документа;
* данные активной скоркарты (через существующий метод);
* при наличии сохранённого JSON — его содержимое для предзаполнения формы, включая поле «Доп. эффекты»;
* внешний идентификатор и номер документа в SberDocs, если документ уже отправлен.

#### 6.2 Сохранение документа

Frontend передаёт backend JSON с данными формы, включая поле «Доп. эффекты». Backend сохраняет JSON и возвращает актуальное состояние записи с признаком наличия документа.

#### 6.3 Отправка на утверждение

Frontend инициирует отправку. Backend в ответ:

* сохраняет итоговый JSON документа;
* формирует документ (без скоркарты и поля «Доп. эффекты»), кодирует в Base64;
* вызывает метод создания документа в SberDocs;
* возвращает frontend внешний идентификатор, номер документа и обновлённый статус записи.

Frontend после успешного ответа скрывает форму и кнопки управления, отображает кнопку «Перейти в SberDocs» и номер документа (read-only).

#### 6.4 Предпросмотр письма

Frontend передаёт текущие данные формы и поле «Доп. эффекты» на endpoint предпросмотра. Backend добавляет данные скоркарты с метриками, формирует HTML письма и возвращает его. Frontend отображает HTML в модальном окне.

#### 6.5 Получение статуса

Frontend запрашивает актуальный статус записи у backend для отображения корректного состояния страницы. Поллинг статуса SberDocs, включая отслеживание `taskState = IN_WORK`, выполняет исключительно backend — frontend не обращается к SberDocs напрямую.

#### 6.6 Отзыв согласования

Отзыв выполняется только в интерфейсе SberDocs. Frontend не предоставляет кнопки или действия отзыва. Если backend вернул статус отзыва, frontend отображает актуальное состояние; редактирование документа не открывается автоматически без отдельного бизнес-правила.

---

### 7. Нефункциональные требования

* Переход в форму создания или редактирования не приводит к потере уже отображаемых данных страницы.
* Повторное открытие предпросмотра письма не приводит к задвоению контента.
* Страница корректно отображается при частично пустых данных.
* Frontend не реализует отдельную статусную модель документа — состояние определяется данными, полученными от backend.
* После отправки на утверждение frontend не позволяет переключить форму в режим редактирования ни при каком условии.
* Frontend не выполняет преобразование JSON документа в HTML — это делает backend.
* Frontend не формирует письмо-уведомление и не управляет его отправкой.

### 8. Матрица ответственности

| Область | Frontend | Backend |
| --- | --- | --- |
| Создание документа | Отображает форму | Формирует и сохраняет JSON |
| Редактирование документа | Открывает форму с данными из JSON | Сохраняет обновлённый JSON |
| Блокировка отправки без документа | Блокирует кнопку | Отклоняет запрос без JSON (HTTP 400) |
| Предпросмотр письма | Передаёт данные формы + доп. эффекты, отображает HTML | Формирует HTML письма (данные формы + скоркарта + доп. эффекты) |
| Отправка на утверждение | Инициирует действие | Формирует документ (без скоркарты и доп. эффектов), кодирует Base64, отправляет в SberDocs |
| Номер документа | Отображает как read-only после получения | Запрашивает у SberDocs и возвращает frontend |
| Кнопка «Перейти в SberDocs» | Отображает, открывает ссылку | Возвращает внешний ID документа |
| Base64-кодирование | — | Выполняет |
| Интеграция с SberDocs | Не управляет | Вызывает метод создания документа |
| Поллинг статуса SberDocs | Не выполняет | Поллит, отслеживает taskState |
| Отправка письма-уведомления | Не инициирует | Отправляет при обнаружении IN_WORK |
| Формирование письма-уведомления | Не участвует | Формирует HTML из JSON + скоркарты + доп. эффектов |
| Скоркарта | Отображает read-only блок | Возвращает данные скоркарты |
| Поле «Доп. эффекты» | Отображает, принимает ввод, передаёт backend | Сохраняет в JSON, включает в письмо, не включает в документ |
| Отзыв согласования | Действие недоступно | Получает внешний статус через поллинг SberDocs |

### 9. Граничные случаи

| Ситуация | Ожидаемое поведение |
| --- | --- |
| Утверждающий назначен, документ не создан | Доступна «Создать документ»; отправка заблокирована |
| Документ сохранён, не отправлен | Доступно редактирование; «Доп. эффекты» редактируемо |
| Документ отправлен | Только «Перейти в SberDocs» и номер документа; редактирование недоступно |
| Пользователь нажимает «Отправить» без сохранённого документа | Backend отклоняет (HTTP 400); frontend показывает ошибку |
| Пользователь открывает предпросмотр до сохранения | Backend формирует HTML письма по текущим данным формы + скоркарте + доп. эффектам |
| Backend не вернул внешний ID после отправки | Frontend показывает сообщение об ошибке отправки |
| Поле «Доп. эффекты» не заполнено | Документ сохраняется и отправляется без ошибки; в письме секция доп. эффектов отсутствует или пуста |
| Поллинг показал taskState = IN_WORK | Backend отправляет письмо-уведомление; состояние страницы не меняется |
| Согласование отозвано в SberDocs | Frontend отображает актуальный статус; редактирование не открывается автоматически |
| Пользователь хочет отозвать согласование | Действие недоступно; пользователь направляется в SberDocs |
