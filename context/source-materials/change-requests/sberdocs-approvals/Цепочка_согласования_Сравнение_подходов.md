## 1. Обзор подходов

|  | Подход 1 — Нативная реализация | Подход 2 — Интеграция с SberDocs |
|---|---|---|
| **Суть** | Вся логика согласования реализуется и хранится внутри АС КОДА | Маршрут настраивается в АС КОДА, а само согласование происходит в СберДокс |
| **Где нажимают «Согласовать»** | В интерфейсе АС КОДА | В интерфейсе АС SberDocs |
| **Откуда берётся статус** | Обновляется напрямую при действии пользователя в системе | Читается из СберДокс через периодические запросы к его API |

---

## 2. Подход 1 — Нативная реализация в АС КОДА

**Описание**: Пользователь задаёт маршрут в модальном окне АС КОДА. После подтверждения фронтенд выполняет два параллельных запроса: применяет действие и отдельно передаёт данные маршрута согласования на бэкенд.

**Пользовательский flow**:
1. Пользователь выбирает действие, требующее согласования
2. Открывается модальное окно настройки маршрута
3. Пользователь добавляет этапы с согласующими и выбирает утверждающего
4. Нажимает «Сохранить маршрут» — появляется диалог подтверждения
5. После подтверждения фронтенд одновременно применяет действие и отправляет маршрут согласования
6. Бэкенд сохраняет маршрут и инициализирует статусы всех участников как «ожидает»

**Поддерживаемые типы маршрутов**:
- Последовательное согласование
- Параллельное согласование
- Автоутверждение

**Хранение данных**: В базе данных АС КОДА фиксируется снапшот маршрута — все этапы, участники и их текущие статусы.

---

## 3. Подход 2 — Интеграция с АС SberDocs

**Описание**: Маршрут согласования настраивается в АС КОДА. После подтверждения фронтенд одновременно применяет действие и отправляет маршрут. Бэкенд фиксирует маршрут в своей базе данных и одновременно создаёт документ в СберДокс.

**Пользовательский flow**:
1. Пользователь выбирает действие, требующее согласования
2. Открывается модальное окно настройки маршрута
3. Пользователь задаёт маршрут — подтверждение
4. Фронтенд одновременно применяет действие и отправляет маршрут согласования
5. Бэкенд сохраняет маршрут и создаёт документ в СберДокс с переданным составом участников
6. Согласующий входит в **СберДокс** и согласовывает там
7. АС КОДА периодически запрашивает статус из СберДокс и отображает его в интерфейсе

**Поддерживаемые типы маршрутов**: Смешанные маршруты — последовательные этапы комбинируются с параллельными группами.

## 4. Детальное сравнение

| Параметр | Подход 1 — Нативная реализация | Подход 2 — Интеграция с SberDocs |
|---|---|---|
| **Параллельное согласование** | Поддерживается | Поддерживается |
| **Смешанные маршруты** | Поддерживается | Поддерживается |
| **Интерфейс согласования** | Единый UX — всё внутри АС КОДА | Разрыв UX — пользователи переходят в СберДокс |
| **Хранение маршрута и статусов** | Полностью в БД АС КОДА | В СберДокс, АС КОДА хранит только ссылку на документ |
| **Синхронизация статусов** | Мгновенная | Требует периодического опроса API СберДокс |
| **Email-бриф** | Полностью кастомный | Передаётся как вложение к документу |
| **Внешние зависимости** | Нет — система автономна | Зависимость от СберДокс |
| **Трудоёмкость разработки** | Выше | Ниже по логике, но нужна интеграция |
| **Юридическая значимость** | Не предусмотрена | Поддержка ЭП из коробки |
| **Гибкость при доработках** | Полная | Частичная |

## 5. Риски и ограничения

| Риск | Подход | Уровень |
|---|---|---|
| Зависимость от внешнего SLA | Подход 2 | Высокий |
| Разрыв UX | Подход 2 | Средний |
| Нагрузка от опроса статусов | Подход 2 | Средний |
| Маппинг идентификаторов сотрудников | Подход 2 | Средний |
| Атомарность двух параллельных запросов | Оба | Средний |
| Сложность реализации параллельного этапа | Подход 1 | Средний |
| Граничные сценарии | Подход 1 | Средний |
| Юридическая значимость | Подход 1 | Низкий |

---

## 6. Рекомендация

**Подход 1 — предпочтительнее, когда**:
- Важен единый UX без переключения между системами
- Нужна полная гибкость в кастомизации логики
- Юридически значимые ЭП не требуются на данном этапе
- Команда готова поддерживать flow согласования

**Подход 2 — предпочтительнее, когда**:
- Требуется юридически значимое согласование с ЭП
- Есть стабильное SLA и согласованный контракт интеграции
- Нужна единая платформа ЭДО для всех процессов согласования
- Приоритет — минимизировать объём собственной разработки

> **Открытый вопрос**: Необходимо уточнить: потребуется ли в будущем юридически значимое согласование с ЭП?
