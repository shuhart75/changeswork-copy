## Вкладка "Цепочка согласования"

### Режим создания или редактирования маршрута

**Ограничения**:
- Форма и все действия внутри неё доступны только пользователю с ролью METODOLOG
- Если backend вернул участника с неподходящей ролью для выпадающих списков — frontend не отображает такого пользователя
- Ограничение на frontend: не более 10 этапов согласования в одном маршруте
- Ограничение на frontend: не более 10 согласующих на одном этапе

### Диалог подтверждения сохранения маршрута

| Экран | Элемент | Действие / поведение |
|---|---|---|
| Диалог подтверждения сохранения маршрута | Кнопка «Сохранить» | Подтверждает сохранение маршрута и вызывает метод создания или редактирования цепочки |
| Кнопка «Вернуться к редактированию» | Закрывает диалог и возвращает пользователя к редактированию маршрута без потери введённых данных |

**Содержимое диалога**:
- Заголовок: «Сохранить маршрут согласования?»
- Текст: «Маршрут будет сохранён и применён к текущей сущности. **После сохранения процесс согласования будет запущен по указанной цепочке!**»
- Кнопки: «Сохранить», «Вернуться к редактированию»

### Диалог «Несохранённые изменения»

| Экран | Элемент | Действие / поведение |
|---|---|---|
| Диалог «Несохранённые изменения» | Кнопка «Выйти без сохранения» | Закрывает режим редактирования маршрута без сохранения |
| Кнопка «Остаться» | Закрывает диалог и возвращает пользователя к редактированию маршрута |

**Содержимое диалога**:
- Заголовок: «Есть несохранённые изменения»
- Текст: «Вы внесли изменения в маршрут согласования, но ещё не сохранили их. Если выйти сейчас, изменения будут потеряны.»
- Кнопки: «Выйти без сохранения», «Остаться»

### Вкладка «Цепочка согласования» (просмотр созданного маршрута)

| Экран | Элемент | Действие / поведение |
|---|---|---|
| Карточка сущности — вкладка «Цепочка согласования» | Блок этапа | Отображает номер этапа, состав участников этапа и их текущие статусы |
| Строка участника | Отображает ФИО участника. Если ФИО отсутствует — идентификатор или «Неизвестный» |
| Статус участника | «Согласовано» / «Отказано» / «Ожидает» для согласующих, «Утверждено» / «Отклонено» / «Ожидает» для утверждающего |
| Дата и время решения | Отображаются только если решение зафиксировано в системе |
| Комментарий | Отображается только если сохранён ранее |
| Кнопка «Редактировать» | Доступна только METODOLOG и только до статуса цепочки `ratified` |
| Кнопка «Отозвать» | Доступна EDITOR и METODOLOG при статусах `in_approval`, `awaiting_ratification`, `in_ratification` (НАХОДИТСЯ НА ВКЛАДКЕ "ОСНОВНАЯ ИНФОРМАЦИЯ") |

## 1. Статусная модель на UI

| Комбинация состояний | Отображаемый статус |
|---|---|
| Переходный бизнес-статус + нет активной цепочки | Ожидает выставления маршрута |
| Переходный бизнес-статус + `in_approval` | На согласовании |
| Переходный бизнес-статус + `in_ratification` | На утверждении |
| Конечный бизнес-статус / статус цепочки `ratified` | Конечный бизнес-статус сущности |
| Непереходный бизнес-статус + отсутствие активной цепочки | Бизнес-статус сущности |
| Статус цепочки `approval_rejected`, `ratification_rejected` или `cancelled` | Предыдущий непереходный бизнес-статус сущности |

## Взаимодействие с Backend

### Инициация действия

- Если согласование не требуется — вызывается только метод применения действия
- Если согласование требуется — вызывается метод применения действия, затем отображается состояние «Ожидает выставления маршрута»
- Редактирование сущности через PUT-метод в статусе ACTIVE приводит к переходному бизнес-статусу; дальнейший сценарий аналогичен
- Инициировать действие и редактировать сущность на UI может только пользователь с ролью EDITOR

### Выставление маршрута

- При открытии вкладки «Цепочка согласования» в состоянии создания или редактирования вызывается метод получения участников из справочника
- После подтверждения в диалоге вызывается метод создания или редактирования цепочки
- При ошибке создания цепочки метод применения действия повторно не вызывается

### Маппинг действий на методы Backend

| Когда | Что вызывает frontend | Результат |
|---|---|---|
| Загрузка карточки сущности | Метод получения бизнес-статуса + метод получения активной цепочки | Определяется отображаемый статус |
| Открытие вкладки «Цепочка согласования» | Метод получения активной цепочки | Полный маршрут: этапы, участники, результаты решений, дата/время, комментарии |
| Открытие истории изменений | Метод получения истории изменений | Общий поток истории, включая события согласования |

## Требования Backend

### Ролевая модель и проверки доступа

**Роли**: EDITOR, METODOLOG, SUPERVISOR

| Операция / метод | Допустимые роли |
|---|---|
| Получение списка участников | METODOLOG |
| Создание цепочки | METODOLOG |
| Редактирование цепочки | METODOLOG |
| Отзыв цепочки | EDITOR, METODOLOG |
| Получение активной цепочки | Пользователь с правом просмотра сущности |
| Получение истории изменений | Пользователь с правом просмотра сущности |

### Модель данных цепочки согласования

| Поле | Тип | Описание |
|---|---|---|
| Идентификатор | UUID | Уникальный идентификатор записи цепочки |
| Тип сущности | VARCHAR | Тип связанной сущности |
| Идентификатор сущности | UUID | Идентификатор версии сущности |
| Статус цепочки | VARCHAR | Текущий статус цепочки согласования |
| Снэпшот | JSONB | Полная структура цепочки на момент создания или редактирования |
| Признак активности | BOOLEAN | true — активная версия, false — архив |
| Дата создания | TIMESTAMP | Дата и время создания |
| Создатель | UUID | Идентификатор пользователя, создавшего запись |
| Дата обновления | TIMESTAMP | Дата и время последнего изменения |
| Кто обновил | UUID | Идентификатор пользователя, изменившего запись |

**Структура снэпшота**:
```json
{
  "approvalStages": [
    {
      "index": 1,
      "approvers": [
        {
          "employeeNumber": "111111111",
          "decision": null,
          "decisionTimestamp": null,
          "comment": null
        }
      ]
    }
  ],
  "ratificationStage": {
    "employeeNumber": "444444444",
    "decision": null,
    "decisionTimestamp": null,
    "comment": null
  }
}
```

**Важно**: Поле `ratificationStage` является **обязательным**.

### 3. Статусная модель цепочки

Используются статусы:
- `in_approval`
- `awaiting_ratification`
- `in_ratification`
- `approval_rejected`
- `ratification_rejected`
- `cancelled`
- `ratified`

**Статусы цепочки**:

| Статус | Возможные действия (триггеры) | Новый статус | Логика работы |
|---|---|---|---|
| — (нет активной цепочки) | Выставить маршрут (с этапами согласования) | `in_approval` | Backend фиксирует событие approvalChainCreated |
| — | Выставить маршрут (без этапов согласования) | `in_ratification` | Backend фиксирует approvalChainCreated + sentToRatification |
| `in_approval` | Редактировать маршрут | `in_approval` | Текущая версия деактивируется, создаётся новая |
| `in_approval` | Отозвать | `approval_cancelled` | Сущность возвращается в исходный бизнес-статус |
| `in_approval` | Все участники приняли положительные решения | `in_ratification` | Backend фиксирует approvalCompleted + sentToRatification |
| `in_approval` | Хотя бы один отказ | `approval_rejected` | Сущность возвращается в предыдущий бизнес-статус |
| `in_ratification` | Редактировать маршрут | `in_ratification` | Текущая версия деактивируется, создаётся новая |
| `in_ratification` | Отозвать | `ratification_cancelled` | Сущность возвращается в исходный бизнес-статус |
| `in_ratification` | Утверждающий принял положительное решение | `ratified` | Сущность переводится в конечный бизнес-статус |
| `in_ratification` | Утверждающий принял отрицательное решение | `ratification_rejected` | Сущность возвращается в предыдущий бизнес-статус |

## История изменений сущности

| eventCode | Сообщение | Статус цепочки после события |
|---|---|---|
| `approvalChainCreated` | '{users.name}' выставил(-а) маршрут согласования | `in_approval` / `in_ratification` |
| `approvalChainUpdated` | '{users.name}' изменил(-а) маршрут согласования | Активный статус цепочки |
| `approvalCompleted` | Согласование завершено | `in_ratification` |
| `approvalRequiresRework` | Требуется доработка | `approval_rejected` |
| `sentToRatification` | Отправлено на утверждение | `in_ratification` |
| `ratificationRejected` | Утверждение отклонено | `ratification_rejected` |
| `approvalRevoked` | '{users.name}' отозвал(-а) сущность с согласования | `approval_cancelled` |
| `ratificationRevoked` | '{users.name}' отозвал(-а) сущность с утверждения | `ratification_cancelled` |

## Нефункциональные требования

- На каждом этапе можно указать не более 10 согласующих
- Количество этапов согласования в одном маршруте не должно превышать 10
- Ограничения валидируются как на frontend, так и на backend
- При нарушении backend возвращает валидационную ошибку
