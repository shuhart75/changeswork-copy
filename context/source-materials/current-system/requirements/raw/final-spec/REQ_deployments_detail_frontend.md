## Бизнес-требования (контекст деталки)

### Контекст

Основание для UX-модели:
- прототип `prototypes/deployments.html`;
- жизненный цикл, заданный для MVP.

`Внедрение` для фронта рассматривается как доменная сущность с единственной рабочей карточкой и следующими состояниями:
- `draft`:
  - доступно редактирование;
  - доступно архивирование;
  - доступна отправка на согласование/утверждение;
- после отправки на утверждение:
  - UI не использует собственный промежуточные статусы ЖЦ;
  - UI отображает статус прикреплённого `approvalInstance`;
- `deployed`:
  - конечный статус;
- `archived`:
  - конечный статус.

Ключевое правило фронта:
- после отправки из `draft` состояние внедрения в списке, деталке и истории определяется по `approvalInstance.status`;
- фронт не дублирует и не переизобретает статусы процесса поверх `approvalInstance`.

### Определения

- `Внедрение`:
  доменная сущность, отображаемая пользователю как одна карточка со списком, детальной формой и формой редактирования.
- `Детальная форма внедрения`:
  экран просмотра текущего состояния внедрения и связанных данных без изменения полей.
- `Форма редактирования внедрения`:
  экран изменения полей внедрения и состава связанных скоркарт. Связанные с внедрением артефакты изменяются отдельно.
- `Блок "Связанные сущности"`:
  read-only блок, который показывает `Pilot` и `Simulation`, вычисленные backend через связанные скоркарты текущего внедрения.
- `Текущее состояние согласования`:
  состояние, определяемое по `approvalInstance.status` и отображаемое на русском языке в пользовательском интерфейсе.
- `Тип внедрения`:
  неизменяемая характеристика внедрения. В MVP доступны значения `Общий` и `Внедрение по результатам симуляции`.
- `Lineage симуляции`:
  отдельное поле в карточке внедрения, которое для типа `Внедрение по результатам симуляции` указывает на `Simulation`, результаты которой внедряются.

## Пользовательские требования

### ПТ-3. Детальная форма

**Описание:** Пользователь должен иметь отдельную страницу просмотра внедрения с возвратом к списку.

**Детали:**
- в шапке показываются:
  - идентификатор внедрения;
  - название;
  - кнопка возврата к списку;
- в блоке общей информации показываются:
  - название;
  - цель;
  - описание изменений;
  - периметр применения;
  - критичность;
  - статус;
  - автор;
  - дата создания;
  - дата внедрения;
- на деталке обязательно должен быть отдельный блок `Тип внедрения и внедряемая симуляция`:
  - показывает `Тип внедрения`;
  - для типа `Внедрение по результатам симуляции` показывает ссылку на `Simulation`, результаты которой внедряются;
  - если тип `Общий`, вместо симуляции ничего не показываем;
- статус должен отображаться единообразно с таблицей:
  - `Черновик`;
  - либо текущий статус прикреплённого `approvalInstance` в русскоязычном отображении;
  - либо конечный `Внедрено`;
  - либо конечный `Архив`;
- в деталке обязательно должен быть блок `Связанные скоркарты`;
- в деталке обязательно должен быть блок `Связанные сущности`;
- в деталке обязательно должен быть блок `Артефакты`;
- если связанных сущностей нет, должно отображаться явное пустое состояние;
- если артефактов нет, должно отображаться явное пустое состояние.

### ПТ-4. Блок `Связанные скоркарты`

**Описание:** Пользователь должен видеть, какие скоркарты участвуют во внедрении.

**Детали:**
- в режиме просмотра скоркарты отображаются в виде набора чипов/ссылок;
- каждая скоркарта показывает:
  - идентификатор;
  - название;
  - критичность;
- для внедрения типа `Внедрение по результатам симуляции` одна скоркарта должна быть помечена как обязательная lineage-скоркарта;
- цвет чипа должен кодировать критичность скоркарты:
  - красный для `high`;
  - зелёный для `low`;
- при наличии навигации по системе чип скоркарты должен вести на карточку скоркарты;
- если скоркарт больше, чем лимит визуализации, должен отображаться индикатор вида `+N ещё`.
- для обязательной lineage-скоркарты должен отображаться отдельный визуальный признак `Обязательная`.

### ПТ-5. Блок `Связанные сущности`

**Описание:** Пользователь должен видеть связанные доменные сущности текущего внедрения.

**Детали:**
- блок доступен только для просмотра;
- блок заполняется backend и вычисляется через связанные скоркарты текущего внедрения;
- допустимые типы связанных сущностей:
  - `Simulation`;
  - `Pilot`;
- внутри блока для каждой сущности показываются:
  - тип;
  - идентификатор;
  - название;
- элементы отображаются отдельными чипами/ссылками;
- если сущность является `lineage`-симуляцией текущего внедрения, её чип-ссылка должна быть визуально выделена и помечена как `Внедряемая симуляция`;
- блок не редактируется вручную на фронте;
- текущее внедрение не должно отображаться в этом блоке как связанная сущность, даже если backend-вычисление выполняется через usage-связи скоркарт;
- если у внедрения нет доменных элементов, связанных через скоркарты, показывается пустое состояние.

### ПТ-6. Блок `Артефакты`

**Описание:** Пользователь должен видеть прикреплённые артефакты и управлять ими в форме редактирования.

**Детали:**
- в режиме просмотра для каждого артефакта показываются на отдельной вкладке;
- открытие артефакта выполняется в новой вкладке браузера.


### ПТ-9. Жизненный цикл внедрения

**Описание:** Интерфейс должен поддерживать жизненный цикл, заданный для MVP, без собственных промежуточных статусов.

**Детали:**
- жизненный цикл внедрения:
  - `draft`;
  - затем текущий статус прикреплённого `approvalInstance`;
  - `deployed`;
  - `archived`;
- из `draft` доступны действия:
  - `Редактировать`;
  - `Архивировать`;
  - `Отправить на согласование/утверждение`;
- после отправки внедрения UI отображает статус прикреплённого `approvalInstance`;
- для `approvalInstance.status = in_approval` доступно только действие `Отозвать с согласования`;
- для `approvalInstance.status = in_ratification` доступно только действие `Отозвать с согласования`;
- при отзыве из `in_approval` или `in_ratification` внедрение для MVP возвращается в `draft`;
- для `approvalInstance.status = approved` доступны:
  - `Отправить на утверждение`;
  - `Архивировать`;
  - `Редактировать`;
- если пользователь начинает редактирование из состояния `approved`, статус `approved` сбрасывается, и внедрение переходит в `draft`;
- для `approvalInstance.status = ratified` доступны:
  - `Внедрить`;
  - `Архивировать`;
  - `Редактировать`;
- если пользователь начинает редактирование из состояния `ratified`, статус `ratified` сбрасывается, и внедрение переходит в `draft`;
- для `approvalInstance.status` из группы `rejected` и `cancelled` доступны:
  - `Архивировать`;
  - `Редактировать`;
  - `Повторно отправить на согласование/утверждение`;
- при редактировании после `rejected` или `cancelled` внедрение возвращается в `draft`;
- действие `Внедрить` становится доступным только если backend вернул `deploy` в списке `available_actions`, что для MVP происходит после перехода `approvalInstance` в `ratified`;
- `deployed` является конечным статусом;
- `archived` является конечным статусом.

### ПТ-10. Роли и доступность действий

**Описание:** Интерфейс должен ограничивать действия по ролям.

**Детали:**
- просмотр списка и деталки доступен ролям, имеющим доступ к доменной модели;
- `prm` и `methodologist` видят внедрения всех продуктов;
- создание, редактирование, отправка на согласование, архивирование и внедрение доступны:
  - `prm` своего продукта;
  - `admin`;
- `methodologist` видит данные по всем продуктам, но не видит кнопок создания и изменения жизненного цикла;
- `approver` и `ratifier` не управляют внедрением с этой страницы.

## Критерии приемки

### КП-2. Детальная форма

- [ ] Деталка открывается отдельным экраном
- [ ] В деталке есть блоки общей информации, блока `Тип внедрения и внедряемая симуляция`, связанных скоркарт, связанных сущностей и артефактов
- [ ] В блоке общей информации отображаются `Цель`, `Описание изменений`, `Периметр применения`
- [ ] В отдельном блоке отображается `Тип внедрения`
- [ ] Для типа `Внедрение по результатам симуляции` в отдельном блоке отображается ссылка на внедряемую `Simulation`
- [ ] Empty-state отображается для пустых связанных сущностей и артефактов
- [ ] Кнопка возврата из деталки в список работает корректно

### КП-4. Жизненный цикл

- [ ] Из `draft` доступны редактирование, архивирование и отправка на согласование/утверждение
- [ ] После отправки статус внедрения определяется только по `approvalInstance`
- [ ] В `in_approval` доступен только отзыв с согласования
- [ ] В `in_ratification` доступен только отзыв с согласования
- [ ] Отзыв из `in_approval` и `in_ratification` возвращает внедрение в `draft`
- [ ] В `approved` доступны отправка на утверждение, архивирование и редактирование
- [ ] При редактировании из `approved` внедрение возвращается в `draft`
- [ ] В `ratified` доступны внедрение, архивирование и редактирование
- [ ] При редактировании из `ratified` внедрение возвращается в `draft`
- [ ] При `approvalInstance.status = ratified` backend возвращает действие `deploy` в `available_actions`
- [ ] Кнопка `Внедрить` отображается только если backend вернул `deploy` в `available_actions`
- [ ] В `rejected` и `cancelled` доступны архивирование, редактирование и повторная отправка
- [ ] `deployed` отображается как конечный статус
- [ ] `archived` отображается как конечный статус
- [ ] Интерфейс не использует собственный промежуточный статус поверх `approvalInstance`

### КП-5. Критичность и связи

- [ ] Критичность рассчитывается автоматически по связанным скоркартам
- [ ] Блок `Связанные сущности` отображает только данные только для чтения
- [ ] Блок `Связанные сущности` отображает сущности, вычисленные backend через скоркарты
- [ ] Текущее внедрение не отображается в блоке `Связанные сущности` само себе
- [ ] Фронт не позволяет вручную редактировать блок `Связанные сущности`
- [ ] Для типа `Внедрение по результатам симуляции` фронт требует заполненный `Lineage` и обязательную lineage-скоркарту
- [ ] В блоке `Связанные сущности` чип внедряемой симуляции выделяется визуально


## Функциональные требования

### ФТ-1. Адреса страниц

В MVP должны поддерживаться следующие адреса страниц:
- `/deployments`;
- `/deployments/:id`.

Правила:
- `/deployments` открывает списочную форму;
- `/deployments/:id` открывает детальную форму;
- форма редактирования открывается на адресе страницы `/deployments/:id` переключением страницы в режим редактирования;
- если внедрение находится в `in_approval`, `in_ratification`, `deployed` или `archived`, переход в режим редактирования должен быть запрещён.

### ФТ-3. Компонент детальной формы

Компонент должен включать:
- кнопку возврата;
- заголовок с `display_id`;
- блок основных атрибутов;
- отдельный блок `Тип внедрения и внедряемая симуляция`;
- блок связанных скоркарт;
- блок связанных сущностей;
- блок артефактов;
- панель действий жизненного цикла.

Правило отображения действий:
- набор кнопок действий должен строиться по полю `available_actions`, полученному от backend;
- если в `available_actions` присутствует `deploy`, интерфейс должен показывать кнопку `Внедрить`;
- если `deploy` отсутствует в `available_actions`, кнопка `Внедрить` не показывается.

Правило отображения type/lineage:
- отдельный блок обязательно показывает `Тип внедрения`;
- для типа `Внедрение по результатам симуляции` этот же блок обязательно показывает ссылку на связанную `Simulation`;
- в блоке `Связанные сущности` симуляция с признаком `isLineage = true` должна выделяться визуально относительно остальных чипов.


### ФТ-5. Модель статусов

Frontend должен работать с такими отображаемыми состояниями:
Клиентская часть должна работать с такими отображаемыми состояниями:
- `draft`;
- любой допустимый `approvalInstance.status` после отправки;
- `deployed`;
- `archived`.

Правила:
- фронт не создаёт отдельный enum "статусов внедрения в процессе";
- для промежуточного состояния фронт просто отображает текущее состояние `approvalInstance`;
- во всех пользовательских местах интерфейса статус должен отображаться на русском языке;

### ФТ-6. Действия жизненного цикла

Поддерживаемые действия интерфейса:
- `submit_for_approval`;
- `start_ratification`;
- `recall`;
- `deploy`;
- `archive`.

Матрица доступности:

| Отображаемое состояние | Действия |
|---|---|
| `draft` | `edit`, `submit_for_approval`, `archive` |
| `in_approval` | `recall` |
| `in_ratification` | `recall` |
| `approved` | `edit`, `start_ratification`, `archive` |
| `ratified` | `edit`, `deploy`, `archive` |
| `rejected` | `edit`, `submit_for_approval`, `archive` |
| `cancelled` | `edit`, `submit_for_approval`, `archive` |
| `deployed` | нет |
| `archived` | нет |

Примечания:
- при `edit` из `approved`, `ratified`, `rejected` и `cancelled` внедрение должно быть возвращено в `draft`;
- для внедрения отзыв из `in_approval` и `in_ratification` возвращает состояние в `draft`;
- если `ApprovalInstance` различает `approval_rejected`/`ratification_rejected` и `approval_cancelled`/`ratification_cancelled`, фронт должен отображать их в группах `rejected` и `cancelled` по одинаковым правилам доступности.

## Интеграция с Backend API

### Используемые endpoint'ы

- `GET /api/v1/deployments/{id}`
- `POST /api/v1/deployments/{id}/action`

### Требования к контракту

- деталка должна возвращать:
  - основные поля внедрения;
  - `deployment_type`;
  - `lineage_simulation`, если применимо;
  - связанные скоркарты;
  - связанные `Simulation`;
  - связанные `Pilot`;
  - артефакты;
  - текущий `approvalInstance.status`, если внедрение уже отправлено;
  - `available_actions`;
- `lineage_simulation` должен содержать:
  - `id`;
  - `display_id`;
  - `name`;
  - `status`;
  - `url`;
- связанные `Simulation` и `Pilot` должны содержать:
  - `entity_type`;
  - `id`;
  - `display_id`;
  - `name`;
  - `status`;
  - `url`;
- связанные `Simulation` должны содержать признак `is_lineage` для визуального выделения нужного чипа в блоке `Связанные сущности`;
- действие `submit_for_approval` должно создавать или привязывать `approvalInstance`;
- фронт ожидает, что после отправки backend вернёт актуальный `approvalInstance.status`;
- фронт ожидает, что при `approvalInstance.status = ratified` backend вернёт действие `deploy` в `available_actions`;
- действие `start_ratification` переводит `approved` в `in_ratification` или иной согласованный ratification-статус процесса;
- действие `recall` переводит внедрение из `in_approval` или `in_ratification` обратно в `draft`;
- действие `deploy` переводит внедрение в `deployed`;
- действие `archive` переводит внедрение в `archived`.

## Обработка ошибок

- ошибки загрузки деталки должны блокировать отображение неполной карточки;
- ошибки действий жизненного цикла должны показываться в контексте карточки внедрения;
- если backend запрещает редактирование не-`draft` внедрения, фронт должен показать понятное сообщение и вернуть пользователя в режим просмотра.
