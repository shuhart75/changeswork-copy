## Бизнес-требования

### Цель

Реализовать клиентскую часть для работы со скоркартами в MVP, чтобы пользователь мог:
- открыть детальную страницу скоркарты;
- создать новую скоркарту из контекста доменного элемента;
- отредактировать скоркарту, если её текущий статус допускает редактирование;
- видеть источники и использование скоркарты;
- выбирать существующие скоркарты для привязки к доменным элементам;
- загружать поля, значения по умолчанию и пороги из актуальной версии шаблона продукта;
- автоматически рассчитывать критичность по мере ввода.

### Контекст

Скоркарта в текущей доменной модели является самостоятельной сущностью с собственным версионированием, источниками и жизненным циклом.  
Скоркарта не является самостоятельным объектом согласования.  
Статус скоркарты в MVP не повторяет статусы согласования связанных сущностей и определяется только наличием связей с доменными элементами.

В MVP создание скоркарты инициируется только из контекста одного доменного элемента:
- `Pilot`
- `Deployment`

`Simulation` в MVP не является контекстом создания скоркарты, но может использоваться как источник данных для частичного предзаполнения формы скоркарты.

Каждый доменный элемент принадлежит ровно одному продукту, поэтому продуктовый контекст заранее известен фронту до открытия формы.

### Определения

- `Скоркарта` — самостоятельная предметная сущность, описывающая набор настроек и финансовых эффектов для оценки изменения в продукте.
- `Версия скоркарты` — неизменяемый снимок содержимого скоркарты на конкретный момент времени. Именно версия скоркарты связывается с версиями доменных элементов.
- `Версия шаблона скоркарты` — зафиксированная структура полей, значений по умолчанию и порогов расчёта критичности, по которой строится форма.
- `Родительский доменный элемент` — доменный элемент, из формы которого пользователь открывает создание скоркарты. В MVP это только `Pilot` или `Deployment`.
- `Источник данных скоркарты` — сущность, данные которой используются для частичного предзаполнения формы. В MVP поддерживается только `Simulation`.
- `Использование скоркарты` — факт связи конкретной версии скоркарты с конкретной версией `Pilot` или `Deployment`.
- `Обязательная lineage-скоркарта внедрения` — скоркарта, которая для внедрения типа `simulation_based` фиксируется как обязательная связь и не может быть удалена из состава такого внедрения.

## Границы MVP

### Входит в MVP
- Детальная страница скоркарты.
- Форма создания скоркарты из контекста `Pilot` или `Deployment`.
- Форма редактирования скоркарты.
- Создание скоркарты из контекста `Pilot` или `Deployment`.
- Частичное предзаполнение формы скоркарты данными из `Simulation`.
- Переход на скоркарту из связанных сущностей.
- Отображение источников формирования скоркарты.
- Отображение использования в `Pilot` и `Deployment`.
- Загрузка структуры и значений по умолчанию шаблона.
- Автоматический расчёт критичности на фронте.
- Использование API списка выбора для выбора существующих скоркарт при редактировании связей с доменными элементами.

### Не входит в MVP
- Отдельная полноценная страница списка скоркарт с таблицей и панелью действий.
- Экспорт в Excel, настройка колонок, расширенные таблицы списка.
- Универсальный редактор JSON как основной интерфейс формы скоркарты.
- Отдельный интерфейс истории версий скоркарты.

## Пользовательские требования к АС КОДА

### ПТ-1. Просмотр деталки скоркарты

**Описание:** Пользователь должен иметь возможность открыть и просмотреть скоркарту.

**Детали:**
- Переход по прямой ссылке или по ссылке из `Pilot`/`Deployment`.
- Отображение основных полей, текущей версии и статуса.
- Отображение продукта.
- Отображение конфигурации и финансовых эффектов.
- Отображение автора и дат.

### ПТ-2. Просмотр источников скоркарты

**Описание:** Пользователь должен видеть, на основании каких данных была сформирована скоркарта.

**Детали:**
- Отдельный блок `Источники формирования`.
- В MVP должен поддерживаться источник типа `Simulation`, если скоркарта создавалась с частичным предзаполнением по результатам симуляции.
- Для каждой записи показываются ID, название и статус.
- Клик по записи ведёт на связанную сущность.
- Если источников нет, показывается пустое состояние.

### ПТ-3. Просмотр использования скоркарты

**Описание:** Пользователь должен видеть, где скоркарта используется.

**Детали:**
- Отдельный блок `Используется в`.
- Две секции:
  - `Пилоты`;
  - `Внедрения`.
- Для каждой записи показываются ID, название и статус.
- Клик по записи ведёт на связанную сущность.
- Если использования нет, показывается пустое состояние.

### ПТ-4. Создание скоркарты

**Описание:** Пользователь должен иметь возможность создать скоркарту.

**Детали:**
- Создание доступно только из контекста `Pilot` или `Deployment`.
- `Simulation` не является самостоятельным контекстом создания скоркарты в MVP.
- Продуктовый контекст берётся из родительского доменного элемента и не вводится пользователем вручную.
- Актуальная версия шаблона для продукта загружается автоматически без пользовательского выбора.
- Название скоркарты обязательно, но всегда формируется системой и не редактируется пользователем.
- Если скоркарта создаётся вручную из контекста `Pilot`, поле названия автоматически заполняется в формате `Скоркарта {scorecardId} для пилота {pilotId}: {pilotName}.`.
- Если скоркарта создаётся вручную из контекста `Deployment`, поле названия автоматически заполняется в формате `Скоркарта {scorecardId} для внедрения {deploymentId}: {deploymentName}.`.
- Если скоркарта создаётся со сценарием частичного предзаполнения по `Simulation`, поле названия заполняется автоматически в формате `Скоркарта {scorecardId} по симуляции {simulationId}: {simulationName}.`.
- Значение `{scorecardId}`, используемое в автоназвании, должно быть известно форме до отправки запроса на создание.
- Поля формы строятся по шаблону скоркарты.
- Критичность рассчитывается автоматически и не редактируется вручную.
- Значения по умолчанию загружаются из шаблона.
- Форма допускает ручное создание и сценарий частичного предзаполнения по данным `Simulation`.
- Перед сохранением пользователь должен видеть, что создание скоркарты приведёт к выпуску новой версии родительского доменного элемента.

### ПТ-5. Редактирование скоркарты

**Описание:** Пользователь должен иметь возможность отредактировать скоркарту, если статус и права это допускают.

**Детали:**
- Редактирование открывается с деталки или по прямой ссылке.
- Сохранение создаёт новую версию скоркарты.
- Для скоркарт в статусе `archive` редактирование недоступно.
- Пользователь должен видеть текущий номер версии как поле только для чтения.
- Пользователь должен видеть название скоркарты как поле только для чтения; ручное редактирование названия не допускается.
- Значения полей редактируются в структурированной форме, а не через сырой JSON.
- Критичность пересчитывается автоматически при изменении релевантных полей.
- Изменение состава связей скоркарты с доменными элементами из формы редактирования скоркарты не допускается.
- Перед сохранением пользователь должен видеть предупреждение, что будут созданы новые версии всех связанных доменных элементов, а их согласование при необходимости будет прервано по правилам этих сущностей.

### ПТ-6. Выбор существующей скоркарты в форме доменного элемента

**Описание:** Пользователь должен иметь возможность выбирать существующие скоркарты для привязки к текущему доменному элементу в тех формах `Pilot` или `Deployment`, где по правилам предметной области допускается управление такими связями.

**Детали:**
- Мультиселект с поиском.
- Отображение названия, ID, статуса и критичности.
- Пользователю показываются только скоркарты того же продукта, к которому относится текущий доменный элемент; отдельного ручного фильтра по продукту в MVP нет.
- Компонент используется в форме того доменного элемента, где разрешено добавление или удаление связей со скоркартами.
- Если для конкретной формы доменного элемента действует правило по минимальному количеству привязанных скоркарт, эта проверка выполняется правилами самой формы доменного элемента, а не внутренней логикой компонента выбора скоркарты.

### ПТ-7. Доступность действий по ролям

**Описание:** Интерфейс должен корректно показывать доступные действия в зависимости от ролей.

**Детали:**
- Просмотр доступен всем ролям с доступом к сущности.
- Создание и редактирование доступны:
  - `prm` своего продукта;
  - `methodologist` своего продукта;
  - `admin`.
- Для `approver` и `ratifier` кнопки редактирования и создания не показываются.

## Критерии приемки

### КП-1. Детальная страница
- [ ] Деталка открывается по ссылке
- [ ] Все ключевые поля отображаются корректно
- [ ] Статус и критичность имеют корректную визуальную индикацию
- [ ] Конфигурация и финансовые эффекты отображаются без потери структуры

### КП-2. Источники и использование
- [ ] Блок `Источники формирования` корректно показывает данные `Simulation`, если скоркарта создавалась с предзаполнением по симуляции
- [ ] Блок `Используется в` корректно показывает `Pilot` и `Deployment`
- [ ] Пустые состояния отображаются явно
- [ ] Ссылки на связанные сущности работают

### КП-3. Создание и редактирование
- [ ] Форма создания открывается корректно
- [ ] Форма редактирования открывается для допустимых статусов
- [ ] Сценарий создания скоркарты из контекста `Pilot` или `Deployment` работает корректно
- [ ] Сценарий частичного предзаполнения по данным `Simulation` работает корректно
- [ ] Продуктовый контекст корректно определяется из родительского доменного элемента
- [ ] Поля, значения по умолчанию и пороги автоматически загружаются из актуальной версии шаблона продукта
- [ ] Поле названия является обязательным
- [ ] При ручном создании скоркарты из контекста `Pilot` название автоматически заполняется в формате `Скоркарта {scorecardId} для пилота {pilotId}: {pilotName}.` и недоступно для редактирования
- [ ] При ручном создании скоркарты из контекста `Deployment` название автоматически заполняется в формате `Скоркарта {scorecardId} для внедрения {deploymentId}: {deploymentName}.` и недоступно для редактирования
- [ ] При создании скоркарты с частичным предзаполнением по `Simulation` название заполняется автоматически в формате `Скоркарта {scorecardId} по симуляции {simulationId}: {simulationName}.` и недоступно для редактирования
- [ ] В режиме редактирования название отображается как read-only и не может быть изменено пользователем
- [ ] Критичность пересчитывается автоматически по мере ввода
- [ ] Поле критичности недоступно для ручного редактирования
- [ ] Сохранение валидной формы приводит к переходу на деталку
- [ ] Перед сохранением новой скоркарты показывается предупреждение о выпуске новой версии родительского доменного элемента
- [ ] Перед сохранением новой версии скоркарты показывается предупреждение о перевыпуске всех связанных доменных элементов
- [ ] Если форма открыта из контекста `Deployment` без зафиксированного типа внедрения, в потоке создания скоркарты можно выбрать тип внедрения
- [ ] Если выбран тип внедрения `simulation_based`, форма требует указать `Simulation` и синхронизирует её с будущим `Lineage` внедрения
- [ ] Нельзя сохранить без обязательных полей
- [ ] Нельзя редактировать скоркарты в статусе `archive`

### КП-4. Изменение связей в формах доменных элементов
- [ ] При добавлении скоркарты к `Pilot` или `Deployment` интерфейс предупреждает о выпуске новой версии только этого доменного элемента
- [ ] При удалении скоркарты из `Pilot` или `Deployment` интерфейс предупреждает о выпуске новой версии только этого доменного элемента
- [ ] Если после удаления связи у текущего `Pilot` или `Deployment` не остаётся ни одной скоркарты, интерфейс показывает отдельное уведомление об автоматическом создании новой скоркарты из актуального шаблона со значениями по умолчанию
- [ ] Если удаляется последняя связь скоркарты, интерфейс корректно показывает, что скоркарта перейдёт в `archive`
- [ ] Если скоркарта помечена как обязательная lineage-скоркарта внедрения типа `simulation_based`, интерфейс не позволяет удалить её из состава этого внедрения

### КП-5. Работа с шаблоном
- [ ] Для продукта автоматически загружается актуальная версия шаблона
- [ ] Структура полей, значения по умолчанию и пороги приходят из шаблона без отдельного выбора в интерфейсе
- [ ] Пользователь не может менять шаблон или его версию через интерфейс

### КП-6. Интеграция с другими формами
- [ ] Компонент выбора скоркарт работает в формах доменных элементов, где допускается изменение связей со скоркартами
- [ ] Поиск по имени и ID работает
- [ ] Выбранные значения корректно сериализуются в `scorecard_ids`

## Функциональные требования

### ФТ-1. Маршруты и точки входа

В MVP должны поддерживаться следующие сценарии навигации:
- `/scorecards/:id` — деталка;
- `/scorecards/new` — создание при наличии контекста родительского доменного элемента `Pilot` или `Deployment`;
- `/scorecards/:id/edit` — редактирование;
- переходы из деталей связанных доменных сущностей;
- открытие по прямой ссылке без обязательной промежуточной навигации через отдельную страницу списка.

### ФТ-2. Компонент деталки скоркарты

Компонент `ScorecardDetailPage` должен включать:
- шапку с ID, названием, статусом, критичностью и действиями;
- блок основной информации;
- блок конфигурации;
- блок финансовых эффектов;
- блок `Источники`;
- блок `Используется в`.

Для блоков `Источники` и `Используется в` клиент должен получать по каждой записи:
- тип сущности;
- название;
- статус;
- ссылку для перехода.

#### Поля основной информации
- `display_id`
- `name`
- `version`
- `status`
- `criticality`
- `product`
- `description`
- `is_manual`
- `created_by`
- `created_at`
- `updated_at`

### ФТ-3. Компонент формы скоркарты

Компонент `ScorecardForm` должен поддерживать режимы:
- `create`
- `edit`

#### Поля формы
- Название (`required`; только для чтения во всех режимах)
- Номер текущей версии (только для чтения в режиме редактирования)
- Критичность (только для чтения, вычисляется автоматически)
- Структурированные поля шаблона
- Структурированные поля эффектов
- Описание

#### Поведение формы
- В `create` режиме форма открывается только из контекста родительского доменного элемента.
- В `create` режиме продуктовый контекст уже известен из родительского доменного элемента.
- В `create` режиме форма автоматически загружает актуальную версию шаблона для продукта родительского доменного элемента и при необходимости частично предзаполняется данными `Simulation`.
- Если в `create` режиме форма открыта из контекста `Pilot` без `Simulation`, поле `name` автоматически заполняется по формату `Скоркарта {scorecardId} для пилота {pilotId}: {pilotName}.`.
- Если в `create` режиме форма открыта из контекста `Deployment` без `Simulation`, поле `name` автоматически заполняется по формату `Скоркарта {scorecardId} для внедрения {deploymentId}: {deploymentName}.`.
- Если в `create` режиме форма частично предзаполняется данными `Simulation`, поле `name` автоматически заполняется по формату `Скоркарта {scorecardId} по симуляции {simulationId}: {simulationName}.`.
- Если в `create` режиме форма частично предзаполняется данными `Simulation`, значение `{scorecardId}` должно быть известно форме до отправки запроса.
- Если в `create` режиме форма открыта из контекста `Deployment`, а тип внедрения ещё не зафиксирован, форма должна позволять выбрать тип внедрения для родительского `Deployment`.
- Если в `create` режиме выбран тип родительского внедрения `simulation_based`, форма обязана работать в сценарии с `Simulation` и передать выбранную симуляцию обратно в поток создания внедрения как будущий `Lineage`.
- В `create` режиме поле `name` отображается как поле только для чтения и не может быть изменено пользователем.
- В `edit` режиме форма загружает текущую версию.
- В `edit` режиме поле `name` остаётся только для чтения и не может быть изменено пользователем.
- Критичность пересчитывается на каждом изменении релевантных полей.
- На время отправки запроса кнопка `Сохранить` блокируется, чтобы пользователь не отправил форму повторно.
- Перед подтверждением `create` фронт показывает, что будет создана новая версия родительского доменного элемента.
- Перед подтверждением `edit` фронт показывает, что будут созданы новые версии всех связанных доменных элементов.
- После успешного сохранения происходит переход на деталку созданной или обновлённой скоркарты.

### ФТ-4. Структурированная форма по шаблону

Основной интерфейс формы скоркарты в MVP должен быть структурированным и построенным по шаблону.

Требования:
- фронт автоматически получает актуальную версию шаблона для продукта без отдельного выбора в интерфейсе;
- булевы поля шаблона отображаются как `ДА/НЕТ`;
- числовые поля эффектов отображаются как числовые поля ввода;
- для метрик со статусом `Не применять` используется скрытое или неактивное состояние по согласованным правилам интерфейса;
- пороги расчёта критичности приходят в составе шаблона вместе со структурой полей и значениями по умолчанию;
- фронт хранит типизированное состояние формы и сериализует его в `configuration` и `financial_effects` для API.

### ФТ-5. Связи с доменными элементами

Для MVP скоркарта должна отображаться и редактироваться с учётом связи с версиями доменных элементов.

Правила:
- `Simulation` может быть только источником данных для скоркарты и не участвует в её статусе;
- `Pilot` и `Deployment` могут быть связаны со многими скоркартами;
- одна версия скоркарты может быть связана со многими версиями `Pilot` и `Deployment`;
- версия скоркарты всегда жёстко ассоциирована с конкретными версиями доменных элементов, с которыми она связана;
- создание новой версии скоркарты должно приводить к созданию новых версий всех связанных доменных элементов;
- удаление или добавление связи скоркарты к доменному элементу должно создавать новую версию только этого доменного элемента;
- изменение состава связей может инициироваться только из формы соответствующего доменного элемента;
- форма редактирования скоркарты не должна содержать контролов для добавления или удаления связей с доменными элементами, так как такие изменения могут потребовать отдельного согласования или утверждения по правилам этих сущностей;
- если после удаления связи текущий доменный элемент остаётся без единой скоркарты, backend автоматически создаёт для него новую скоркарту из актуальной версии шаблона продукта со значениями по умолчанию;
- если удаляется последняя связь скоркарты с доменными элементами, скоркарта автоматически получает статус `archive`.
- если скоркарта является обязательной lineage-скоркартой внедрения типа `simulation_based`, удаление этой связи из формы `Deployment` запрещено;

Требования к интерфейсу:
- В форме скоркарты должно отображаться уведомление о последствиях сохранения для связанных доменных элементов.
- В формах `Pilot` и `Deployment`, где меняется состав связанных скоркарт, должно отображаться отдельное подтверждение и уведомление о перевыпуске только текущего доменного элемента.
- В форме `Deployment` для обязательной lineage-скоркарты должен отображаться неизменяемый маркер `Обязательная`.
- Текст предупреждения должен явно различать сценарии `создание скоркарты`, `создание новой версии скоркарты` и `добавление или удаление связи`.
- Для сценария `создание новой версии скоркарты` должен использоваться текст:
  `После сохранения будет создана новая версия скоркарты и новые версии всех связанных сущностей Pilot и Deployment. Если по затронутым версиям уже запущены согласование или утверждение, эти процессы будут прерваны.`
- Если после удаления связи у текущего `Pilot` или `Deployment` не остаётся ни одной связанной скоркарты, интерфейс должен дополнительно показать уведомление:
  `После сохранения для этого элемента будет автоматически создана новая скоркарта из текущего шаблона со значениями по умолчанию.`
- Если при удалении связи удаляется последняя активная связь скоркарты с `Pilot` и `Deployment`, к предупреждению об удалении должен добавляться текст:
  `Эта скоркарта больше не будет связана ни с одним Pilot или Deployment и после сохранения перейдёт в статус archive.`

### ФТ-6. Автоматическая загрузка шаблона

В MVP отдельный компонент интерфейса для выбора шаблона не требуется.

Требования:
- фронт должен автоматически запрашивать актуальную версию шаблона для продукта родительского доменного элемента;
- пользователь не должен видеть селектор шаблона или версии шаблона;
- пользователь не должен иметь возможности менять шаблон в интерфейсе создания или редактирования;
- полученный шаблон должен содержать структуру полей, значения по умолчанию и продуктовые пороги для расчёта критичности.

### ФТ-7. Компонент выбора скоркарт

Переиспользуемый компонент `ScorecardPicker` должен:
- работать как поле выбора с поиском и множественным выбором;
- искать по `display_id` и `name`;
- показывать статус и критичность в опции;
- поддерживать асинхронную подгрузку списка;
- использоваться в формах доменных элементов, где допускается управление связями со скоркартами;
- возвращать массив `scorecard_ids`.

### ФТ-8. Доработка деталки пилота

Для интеграции скоркарт в интерфейс `Pilot` требуется изменить состав блоков на детальной странице и в форме редактирования пилота.

Требования:
- в детальной карточке `Pilot` существующий блок `Финансовые метрики` должен быть заменён на блок добавления и редактирования скоркарт;
- новый блок должен использовать компонент выбора скоркарт и отображать текущие привязанные скоркарты;
- в режиме просмотра пилота блок должен отображаться в disabled/read-only состоянии без возможности изменить состав привязанных скоркарт;
- в режиме редактирования пилота блок должен быть активен и позволять добавлять и удалять скоркарты по правилам формы пилота;
- при попытке изменить состав скоркарт в режиме редактирования пилота должны использоваться предупреждения, описанные в разделе требований к интерфейсу для изменения связей;
- если пользователь удаляет у пилота последнюю связанную скоркарту, интерфейс пилота должен показать дополнительное уведомление об автоматическом создании новой скоркарты из текущего шаблона со значениями по умолчанию;
- замена блока `Финансовые метрики` на блок скоркарт должна быть отражена как в обычной деталке пилота, так и в edit-сценарии, если они реализованы разными компонентами.

### ФТ-9. Доступные действия

В деталке должны отображаться действия:
- `Редактировать` — только если `can_edit = true`;
- `Создать скоркарту` — только при наличии прав;
- действия перехода к связанным сущностям.

Кнопки жизненного цикла самой скоркарты в MVP не требуются, так как её статус определяется наличием или отсутствием связей с доменными элементами.

### ФТ-10. Алгоритм расчёта критичности

Критичность должна вычисляться на фронте автоматически.

Правила:
- итог `Высокий`, если хотя бы один критичный индикатор или пороговый эффект равен `Высокий`;
- иначе итог `Низкий`;
- для текущего шаблона MVP допустимы только два результирующих значения: `high` и `low`;
- расчёт выполняется и при создании, и при редактировании версии;
- рассчитанное значение отправляется на сервер в теле запроса;
- продуктовый контекст родительского доменного элемента определяет, шаблон какого продукта должен быть автоматически загружен;
- пороги, состав полей и значения по умолчанию приходят из актуальной версии шаблона этого продукта;
- фронт сравнивает введённые пользователем значения только с `criticality_thresholds` из ответа `default-config` и не вычисляет пороги самостоятельно.

Пример расчёта:
- Из шаблона продукта `КК` пришли пороги:
  - `launch_new_credit_product` и `new_decision_model` дают `high`, если значение поля равно `true`;
  - `ar_delta_pp >= 5.0` даёт `high`;
  - `issuance_volume_pct >= 0.05` даёт `high`;
  - `rc_pct >= 0.1` даёт `high`;
  - `npv_pct >= 0.15` даёт `high`.
- Пользователь ввёл:
  - `launch_new_credit_product = false`;
  - `new_decision_model = false`;
  - `ar_delta_pp = 4.2`;
  - `issuance_volume_pct = 0.03`;
  - `rc_pct = 0.11`;
  - `npv_pct = 0.12`.
- Фронт последовательно проверяет каждое правило из `criticality_thresholds`.
- В этом примере:
  - булевы индикаторы не срабатывают;
  - `ar_delta_pp` не достигает порога;
  - `issuance_volume_pct` не достигает порога;
  - `rc_pct` достигает порога `0.1`, значит этот эффект получает `high`;
  - `npv_pct` не достигает порога.
- Так как хотя бы одно правило дало `high`, итоговая критичность всей скоркарты равна `high`.

Пример расчёта `low`:
- если все булевы индикаторы равны `false`, а все числовые эффекты ниже соответствующих порогов из `criticality_thresholds`, итоговая критичность равна `low`.

### Контракты данных клиентской части

Примечание:
- Серверный API использует `snake_case`.
- В клиентской части допускается преобразование в `camelCase` на уровне клиента API и слоя адаптации.
- Ниже приведены клиентские DTO после преобразования.
- Продуктовый контекст формы создания приходит из родительского доменного элемента.
- `parent_context` определяет контекст открытия формы, а `source_refs` описывает происхождение данных скоркарты и не обязан совпадать с ним один-в-один.
- `templateVersionId` в форме является техническим полем, которое фронт заполняет автоматически после загрузки актуального шаблона продукта.

```typescript
interface ScorecardRef {
  id: string;
  displayId: string;
  name: string;
  status: ScorecardStatus;
  criticality: Criticality;
}

interface ScorecardDetail {
  id: string;
  displayId: string;
  name: string;
  version: number;
  status: ScorecardStatus;
  criticality: Criticality;
  description?: string;
  isManual: boolean;
  product: EntityRef;
  configuration: Record<string, unknown>;
  financialEffects?: Record<string, unknown>;
  createdBy: UserRef;
  createdAt: string;
  updatedAt: string;
  relations: ScorecardRelations;
  canEdit: boolean;
}

interface ScorecardRelations {
  sources: ScorecardRelationItem[];
  usage: ScorecardRelationItem[];
}

interface ScorecardRelationItem {
  entityType: 'simulation' | 'pilot' | 'deployment';
  id: string;
  displayId: string;
  name: string;
  status: string;
  url: string;
}

interface ScorecardFormState {
  parentEntityType: 'pilot' | 'deployment';
  parentEntityId: string;
  pendingDeploymentType?: 'general' | 'simulation_based';
  pendingDeploymentLineageSimulationId?: string;
  prefillSimulationId?: string;
  productCode: 'PK' | 'KK' | 'ULKK' | 'POS' | 'AK' | 'ZHK' | 'BNPL' | 'RATINGS';
  templateVersionId: string;
  name: string;
  description?: string;
  criticality: 'high' | 'low';
  indicators: Record<string, boolean>;
  effects: {
    arDeltaPp: number;
    issuanceVolumePct: number;
    rcPct: number;
    npvPct: number;
    npl30Pct: number;
    fRorwaPct: number;
  };
}

interface ScorecardTemplateConfig {
  fields: {
    indicators: Array<{
      code: string;
      label: string;
      type: 'boolean';
      required: boolean;
    }>;
    financialEffects: Array<{
      code: string;
      label: string;
      type: 'number';
      required: boolean;
      unit?: string;
    }>;
  };
  defaultValues: {
    configuration: Record<string, boolean>;
    financialEffects: Record<string, number>;
  };
  criticalityThresholds: {
    booleanIndicatorsHighIfTrue: string[];
    effectThresholds: Record<
      string,
      {
        comparison: 'abs_gte' | 'not_applicable';
        value?: number;
        resultIfTrue?: 'high';
      }
    >;
    finalRule: 'high_if_any_indicator_or_effect_high_else_low';
  };
}
```

## Валидация

### Поля формы
- `name` — обязательное, непустое, до `255` символов
- при создании скоркарты с частичным предзаполнением по `Simulation` поле `name` должно быть автоматически заполнено значением формата `Скоркарта {scorecardId} по симуляции {simulationId}: {simulationName}.` и недоступно для ручного редактирования
- `criticality` — обязательное вычисляемое поле, недоступно для ручного ввода
- `template_version_id` — технически обязательное поле тела запроса, заполняется фронтом автоматически
- `configuration` — формируется фронтом из структурированной формы
- `financial_effects` — формируются фронтом из структурированной формы
- булевы поля шаблона — обязательны как явно заданные значения `ДА/НЕТ`
- числовые поля эффектов — обязательны значением `0` или иным введённым числом, если поле применимо продукту

### Пользовательские сообщения
- Для пустого названия: `Укажите название скоркарты`
- Для ошибки загрузки шаблона: `Не удалось загрузить актуальный шаблон скоркарты`
- Для недоступного продукту показателя: `Показатель не применяется для выбранного продукта`
- Для недоступного редактирования: `Редактирование недоступно для текущего статуса скоркарты`

## Обработка ошибок

### Ошибки API
- `400` — показать сообщение о некорректных данных формы
- `403` — показать сообщение о недостатке прав
- `404` — показать, что скоркарта или шаблон не найдены
- `422` — показать ошибки бизнес-валидации рядом с полями или общим сообщением
- `500` — показать общее сообщение об ошибке и дать возможность повторить запрос

### Состояния интерфейса
- Заглушка загрузки на деталке
- Индикатор загрузки для блока источников и использования
- Неактивное состояние кнопки сохранения
- Пустое состояние для пустых блоков источников и использования
- Состояние ошибки с возможностью повторить загрузку
