# Система управления внедрениями риск-стратегий
## Презентация для стейкхолдеров

---

## 1. Executive Summary

### Что это?
Комплексная система для управления жизненным циклом изменений кредитных риск-стратегий — от симуляций на исторических данных до продуктового внедрения.

### Зачем?
- **Снижение рисков** при внедрении изменений в кредитные стратегии
- **Прозрачность** процесса от идеи до продакшена
- **Контроль** на каждом этапе: симуляция → пилот → внедрение
- **Координация** между командами риск-менеджмента, аналитики и разработки

### Ключевые метрики эффективности
- Прогноз финансовых эффектов на основе исторических данных
- Валидация на реальных клиентах через контролируемые пилоты
- Отслеживание фактических результатов после внедрения

---

## 2. Бизнес-проблема

### Текущие вызовы
1. **Фрагментированность процесса** — симуляции, пилоты и внедрения управляются отдельно
2. **Потеря контекста** — связь между этапами теряется
3. **Сложность согласования** — нет единого процесса для утверждения изменений
4. **Отсутствие трейсабилити** — сложно отследить, откуда взялась скоркарта в продакшене

### Решение
Единая платформа с:
- **Сквозным управлением** всеми этапами внедрения
- **Автоматическим отслеживанием** связей между сущностями
- **Встроенным процессом согласования** с маршрутизацией
- **Полной историей** от симуляции до продакшена

## 4. Ключевые возможности

### 4.1 Управление внедрениями
- **Группировка** всей работы по стратегическому изменению
- **Автоматический расчет статуса** на основе вложенных сущностей
- **Жизненный цикл**: draft → active → deployed → archived
- **Портфельный view** всех активных внедрений

### 4.2 Симуляции на исторических данных
- Анализ влияния изменений на исторических данных
- Прогноз финансовых эффектов (AR, NPV, RC, Объём выдач)
- Выбор периода данных для анализа
- Создание скоркарт на основе результатов

### 4.3 Пилотирование
- Контролируемые эксперименты на реальных клиентах
- A/B/C тестирование (несколько скоркарт в одном пилоте)
- Мониторинг метрик в реальном времени
- Процесс активации и корректировки

### 4.4 Продуктовые внедрения
- Управление релизами скоркарт в продакшен
- Процесс согласования и утверждения
- Отслеживание фактических эффектов
- Механизм откатов (rollback)

### 4.5 Управление скоркартами
- Версионирование скоркарт
- Отслеживание источников (симуляция/пилот/ручное создание)
- Связь с продуктами и подпродуктами
- Оценка критичности изменений

### 4.6 Процесс согласования
- Двухэтапное согласование: Approval → Ratification
- Гибкая маршрутизация по ролям
- Пакетное согласование связанных изменений
- Отзыв и повторная отправка

## 5. Архитектура решения

### 5.1 Архитектурный подход
**Initiative-Centric Architecture** — Инициатива как агрегат верхнего уровня

```
Инициатива (Initiative)
├── Симуляции (0..*)
├── Пилоты (0..*)
├── Внедрения (0..*)
└── Скоркарты (0..*)
```

### 5.2 Концепция цепочки внедрения
Цепочка связывает этапы жизненного цикла:

```
Симуляции → Пилоты → Изменение
```

Допустимые варианты:
- Только симуляции (исследование)
- Симуляции + Изменение (пропуск пилота)
- Симуляции + Пилот + Изменение (полный цикл)
- Пилот + Изменение (пропуск симуляций)
- Только Изменение (прямое внедрение)

### 5.3 Ключевые бизнес-правила

**Автоматический расчет статуса Инициативы:**
- draft: пустое ИЛИ все вложенные в draft
- active: хотя бы одна вложенная сущность не в draft
- deployed: хотя бы одно Внедрение в deployed

**Связи через скоркарты:**
- Пилот может быть связан с несколькими Внедрениями (многие-ко-многим)
- Внедрение наследует связи с Симуляциями/Пилотами через источники Скоркарт

**Согласование:**
- Нельзя отправить на согласование на этапе симуляции
- Пилот: когда status = 'requires_activation' или 'requires_correction'
- Изменение: когда status = 'requires_approval'

## 6. Пользовательский интерфейс

### 6.1 Главная страница — Портфель внедрений
- Таблица всех внедрений с фильтрацией по продуктам
- Статусы, критичность, даты
- Быстрый переход к детальной информации

### 6.2 Детальная страница внедрения
**Вкладки:**
- **Обзор** — сводка, согласования, ключевые метрики
- **Симуляции** — список симуляций с результатами
- **Пилоты** — активные и завершенные пилоты
- **Изменения** — продуктовые внедрения
- **Скоркарты** — все связанные скоркарты

**Функции:**
- Создание новых сущностей
- Пакетная отправка на согласование
- Отзыв с согласования
- Просмотр брифа для утверждающих

### 6.3 Детальные страницы сущностей
Каждая сущность (Симуляция, Пилот, Изменение, Скоркарта) имеет:
- Полную информацию о статусе и параметрах
- Связи с другими сущностями (кликабельные)
- Историю изменений
- Результаты/эффекты (для завершенных)

### 6.4 Страницы списков
- **Симуляции** — все симуляции по продуктам
- **Пилоты** — активные и завершенные пилоты
- **Изменения** — история внедрений
- **Скоркарты** — каталог скоркарт с фильтрацией
- **Пакеты** — сгруппированные согласования

### 6.5 Дизайн-система
- Унифицированная цветовая схема для статусов
- Мягкие, приглушенные цвета для снижения когнитивной нагрузки
- Консистентное отображение чипов, карточек, таблиц
- Адаптивная верстка для разных устройств

## 7. Технические характеристики

### 7.1 Текущая реализация
- **Прототип UI**: Автономный HTML с React + Material-UI
- **Запуск**: Открытие в браузере, сборка не требуется
- **Данные**: Mock-данные для демонстрации концепции

### 7.2 Технологический стек (рекомендуемый)
**Frontend:**
- React 18+ с TypeScript
- Material-UI v5 для компонентов
- React Router для навигации
- React Query для управления состоянием

**Backend:**
- Node.js + Express или Python + FastAPI
- PostgreSQL для хранения данных
- Redis для кэширования
- REST API или GraphQL

**Инфраструктура:**
- Docker для контейнеризации
- Kubernetes для оркестрации
- CI/CD pipeline (GitLab CI / GitHub Actions)
- Мониторинг (Prometheus + Grafana)

### 7.3 Интеграции

**Уже существующие интеграции:**
- **Система симуляций** ✅ — запуск симуляций и получение результатов
- **Платформа пилотирования** ✅ — управление пилотами и получение метрик

**Планируемые интеграции (Фаза 2):**
- **Система деплоя** — внедрение скоркарт в продакшен
- **LDAP/AD** — аутентификация пользователей
- **Workflow engine** — процесс согласования (или собственная реализация)

  - Создание и редактирование внедрений

- Интеграция с системой деплоя скоркарт
- Механизм откатов (Rollback)
- Дашборды и расширенная аналитика
- Уведомления и алерты (Email, Slack/Teams)
- Расширенная отчетность

**Результат:** Промышленная эксплуатация на всех продуктах с полным функционалом

## 10. Риски и митигация

### Технические риски
**Риск:** Сложность интеграций с существующими системами
**Митигация:** Поэтапный подход, начало с mock-данных, постепенное подключение интеграций

**Риск:** Производительность при большом объеме данных
**Митигация:** Оптимизация запросов, кэширование, пагинация

### Организационные риски
**Риск:** Сопротивление изменениям со стороны пользователей
**Митигация:** Вовлечение пользователей на этапе проектирования, обучение, постепенное внедрение

**Риск:** Недостаточная поддержка руководства
**Митигация:** Регулярная демонстрация прогресса, фокус на бизнес-ценности

### Процессные риски
**Риск:** Изменение требований в процессе разработки
**Митигация:** Agile-подход, короткие итерации, регулярная обратная связь
