# Модель домена v3.3 - Архитектура на основе Инициатив

**Версия:** 3.3
**Дата:** 2026-03-31
**Архитектура:** Инициатива-центричная с явным управлением Скоркартами

---

## Краткое резюме

Версия 3.3 фиксирует актуальную модель домена и уточняет правила процесса согласования, пакетного утверждения и типизации внедрений:

- **Инициатива** (Initiative, ранее Deployment) - агрегат верхнего уровня, группирующий всю связанную работу
- **Внедрение** (Deployment, ранее Change) - фаза продуктового внедрения модификации стратегии
- Сущность **Stage** декомпозирована на три отдельных агрегата: **Симуляция**, **Пилот**, **Внедрение**
- **Скоркарта** - полноценный агрегат со связями многие-ко-многим
- **Deployment** типизируется как `general` или `simulation_based`
- Для `simulation_based` внедрений вводится ограниченный **Lineage**: ссылка на одну `Simulation`, результаты которой внедряются
- Явное отслеживание происхождения данных Скоркарт (в MVP через Симуляции)
- **Package** создаётся только в момент отправки на Ratification; решения и бизнес-статусы при этом по-прежнему принадлежат входящим `ApprovalInstance`
- Роли **Approver** и **Ratifier** разделены для этапов Approval и Ratification

## Ограниченные контексты

### 1. Контекст идентификации и доступа

**Назначение:** Управление пользователями, ролями и правами доступа.

**Сущности:**
- `User` - Пользователь системы
- `UserRole` - Роль пользователя с привязкой к продукту (опционально)

**Роли:**
- `prm` (Product Risk Manager) - ПРМ, управляет инициативами своего продукта
- `methodologist` - Методолог, управляет скоркартами и документами
- `approver` - Согласующий на этапах Approval
- `ratifier` - Утверждающий на этапе Ratification
- `admin` - Администратор, полный доступ

**Ответственность:**
- Управление пользователями и их ролями
- Разграничение доступа к сущностям
- Назначение согласующих на этапы согласования
- Поддержка множественных ролей для одного пользователя

**Бизнес-правила:**
- Пользователь может иметь несколько ролей
- Роли ПРМ и методолог привязаны к конкретному продукту (product_id NOT NULL)
- Роли approver, ratifier, админ глобальные (product_id IS NULL)
- ПРМ и методолог могут быть назначены approver или ratifier
- Для страницы `Пакеты` действует доменное исключение: `methodologist` может просматривать элементы любых продуктов и формировать один пакет из элементов разных продуктов
- Approver видит только те согласования на этапах Approval, где он назначен
- Ratifier видит только те согласования на этапе Ratification, где он назначен
- Правила совмещения ролей: ✅ (ПРМ ИЛИ Методолог ИЛИ Админ) + Approver + Ratifier; ❌ ПРМ + Методолог

---

### 2. Контекст каталога продуктов

**Назначение:** Управление иерархией и классификацией продуктов.

**Сущности:**
- `Product` - Кредитный продукт (например, "Кредитная карта")
- `SubProduct` - Вариант продукта (например, "Карта с кэшбэком")

**Ответственность:**
- Управление каталогом продуктов
- Связь продукт-стратегия

---

### 3. Контекст шаблонов скоркарт

**Назначение:** Управление версионированными шаблонами для создания скоркарт.

**Сущности:**
- `ScorecardTemplate` - Определение шаблона для продукта
- `ScorecardTemplateVersion` - Версионированный шаблон с описанием полей, значениями по умолчанию и порогами расчёта критичности

**Ответственность:**
- Версионирование шаблонов во времени
- Значения полей и структура по умолчанию
- Хранение порогов для вычисления критичности на фронте
- Управление шаблонами для конкретных продуктов

**Бизнес-правила:**
- Шаблоны версионируются по продуктам
- Каждая версия имеет период действия (valid_from, valid_to)
- `default_config` хранится как JSONB
- `default_config` должен содержать три логических части:
  - структуру полей шаблона;
  - значения по умолчанию;
  - `criticality_thresholds`, по которым фронт вычисляет `high/low`
- В MVP у каждого продукта есть своя версия шаблона, даже если фактическое содержимое шаблона у разных продуктов совпадает

---

### 4. Контекст скоркарт

**Назначение:** Управление фактическими скоркартами, используемыми в пилотах и внедрениях.

**Сущности:**
- `Scorecard` - Экземпляр скоркарты, созданный из шаблона
- `ScorecardVersion` - Версионированная конфигурация скоркарты
- `ScorecardSource` - Отслеживает происхождение данных для скоркарты

**Ответственность:**
- Создание скоркарт (вручную или из источников)
- Версионирование скоркарт при изменении
- Трассируемость источников
- Управление финансовыми эффектами
- Управление жизненным циклом скоркарты по составу связей

**Бизнес-правила:**
- Скоркарта создается из ScorecardTemplateVersion
- В MVP скоркарта создаётся только из контекста `Pilot` или `Deployment`
- Скоркарта может быть создана вручную (`is_manual = true`) или с частичным предзаполнением по данным `Simulation`
- Название скоркарты обязательно
- Если скоркарта создаётся с частичным предзаполнением по `Simulation`, её название автоматически задаётся в формате `Скоркарта {scorecardId} по симуляции {simulationId}: {simulationName}.`
- Каждое изменение создает новую ScorecardVersion
- financial_effects хранятся в JSONB внутри ScorecardVersion
- Критичность вычисляется на фронте по полям шаблона и передаётся в `ScorecardVersion`
- Использование скоркарты фиксируется связями многие-ко-многим с версиями `Pilot` и `Deployment`
- Источники отслеживаются в таблице `scorecard_source`; в MVP допустим только источник типа `simulation`
- Если скоркарта зафиксирована как обязательная lineage-скоркарта внедрения типа `simulation_based`, её нельзя удалить из состава этого внедрения
- В MVP подпродукты для скоркарт не используются
- Скоркарта не проходит согласование как самостоятельный target
- Жизненный цикл скоркарты в MVP включает только статусы `active` и `archive`
- Статус `active` означает, что скоркарта связана хотя бы с одной версией `Pilot` или `Deployment`
- Статус `archive` означает, что у скоркарты не осталось связей с версиями `Pilot` или `Deployment`
- Источники из `scorecard_source` не влияют на статус скоркарты

### 5. Контекст инициатив

**Назначение:** Группировка верхнего уровня всей работы, связанной со стратегическим изменением.

**Корневой агрегат:**
- `Initiative`

**Ответственность:**
- Группировка связанных Симуляций, Пилотов и Внедрений
- Предоставление контекста для всех активностей
- Отслеживание общего статуса инициативы

**Бизнес-правила:**
- Инициатива принадлежит одному Product
- Может быть связана с несколькими SubProduct в рамках Product (многие-ко-многим)
- Если подпродукты не указаны, Инициатива относится ко всем подпродуктам Product
- Может содержать множество Симуляций, Пилотов и Внедрений
- Жизненный цикл: draft → active → deployed → archived
- Статус рассчитывается автоматически на основе вложенных сущностей

---

### 6. Контекст исследований и выполнения

**Назначение:** Управление тремя фазами развертывания стратегии.

**Корневые агрегаты:**
- `Simulation` - Анализ исторических данных
- `Pilot` - Реальный эксперимент
- `Deployment` - Продуктовое внедрение

#### Симуляция

**Ответственность:**
- Запуск анализа на исторических данных
- Генерация прогнозов финансовых эффектов
- Служит источником для Скоркарт

**Бизнес-правила:**
- Принадлежит одной Инициативе
- Может быть источником для множества Скоркарт
- Может выступать `lineage`-источником для множества внедрений типа `simulation_based`
- Жизненный цикл: draft → running → completed/failed
- Результаты хранятся как JSONB

#### Пилот

**Ответственность:**
- Выполнение контролируемого эксперимента
- Сбор реальных метрик
- Валидация прогнозов скоркарты
- Использует скоркарты для проверки гипотез и валидации эффектов

**Бизнес-правила:**
- Принадлежит одной Инициативе
- Может быть связан максимум с ОДНИМ Внедрением через общие Скоркарты (0..1)
- Связан с одной или несколькими Скоркартами (многие-ко-многим)
- Если создан из Скоркарт, источником которых были Симуляции, наследует связи с Симуляциями
- Жизненный цикл: draft → requires_activation → active → completed → archived
- Может быть отправлен на согласование в статусе "requires_activation"

#### Внедрение

**Ответственность:**
- Развертывание стратегии в продакшн
- Отслеживание конфигурации развертывания
- Управление откатом

**Бизнес-правила:**
- Принадлежит одной Инициативе
- Может иметь несколько Пилотов через общие Скоркарты (0..*)
- Связано с одной или несколькими Скоркартами (многие-ко-многим)
- Имеет неизменяемый тип `general` или `simulation_based`
- Для `simulation_based` хранит ссылку `lineage_simulation_id` на одну `Simulation`
- Для `simulation_based` обязано иметь обязательную lineage-скоркарту, созданную по результатам `lineage_simulation_id`
- Поля версии внедрения включают `name`, `goal`, `change_description`, `application_perimeter`
- Жизненный цикл версии: `draft → in_approval / awaiting_ratification / in_ratification → ratified → deployed → rolled_back`, с reject/cancel-статусами по правилам `ApprovalInstance`
- Может быть отправлено на согласование из статусов `draft`, `approval_rejected`, `ratification_rejected`, `approval_cancelled`, `ratification_cancelled`
- Только одно активное развертывание на Внедрение (rollback_at = null)

---
### 7. Контекст отслеживания происхождения (Lineage)

**Назначение:** Ограниченное отслеживание происхождения для сущностей, где оно уже реализовано в MVP.

**Сущности:**
- В текущем MVP-срезе для внедрений используется агрегатное поле `Deployment.lineage_simulation_id`
- Более широкие таблицы `*_source` остаются целевой архитектурой следующих итераций

**Ответственность:**
- Хранение минимально необходимой связи происхождения для `simulation_based` внедрений
- Отображение этой связи в UI как отдельного поля `Lineage`
- Трассируемость происхождения внедряемой симуляции

**Бизнес-правила:**
- Lineage в текущем MVP не является отдельной CRUD-сущностью
- Lineage в текущем MVP реализован только для `Deployment` типа `simulation_based`
- Для такого внедрения допускается ровно одна `lineage`-симуляция
- Указанная `lineage`-симуляция должна совпадать с источником обязательной lineage-скоркарты внедрения
- Рекурсивная визуализация полной цепочки происхождения остаётся целевым состоянием следующих итераций

---

### 8. Контекст пакетов

**Назначение:** Группировка нескольких `ApprovalInstance` из очереди `awaiting_ratification` для совместного утверждения.

**Корневой агрегат:**
- `Package`

**Сущности:**
- Отдельные сущности элементов пакета в MVP не выделяются
- Активный состав пакета определяется по `ApprovalInstance.package_id`

**Ответственность:**
- Группировка связанных `ApprovalInstance` для совместного утверждения
- Хранение брифа пакета
- Хранение пакетного контекста в истории связанных `ApprovalInstance`

**Бизнес-правила:**
- Package создаётся только в момент отправки на утверждение
- Backend-черновики и отдельные состояния `draft` / `sent` в MVP отсутствуют
- Может содержать PilotVersion и/или DeploymentVersion из ЛЮБЫХ Initiatives
- Все элементы Package должны ссылаться на `ApprovalInstance` в состоянии `awaiting_ratification`
- После создания связанные `ApprovalInstance` переходят в `in_ratification`
- Поддерживает частичное утверждение: итоговое решение фиксируется по каждому связанному `ApprovalInstance`
- Критичность вычисляется на лету по входящим элементам:
  - если хотя бы один элемент имеет `criticality = high`, пакет имеет `criticality = high`
  - иначе, если хотя бы один элемент имеет `criticality = low`, пакет имеет `criticality = low`
  - иначе пакет имеет `criticality = null`
- Package не является согласуемым доменным элементом и не имеет собственного `ApprovalInstance`
- Пакет не имеет собственного статусного поля
- Если после индивидуальных решений в пакете остаётся меньше двух активных элементов, Package прекращает существование как активная группировка
- История пакетных действий и комментариев отображается через `process_snapshot` соответствующих `ApprovalInstance`

### 9. Контекст согласования

**Назначение:** Многоэтапный процесс согласования и утверждения для PilotVersion и DeploymentVersion без предзаданных маршрутов.

**Корневой агрегат:**
- `ApprovalInstance` - Экземпляр процесса согласования для конкретного PilotVersion или DeploymentVersion

**Ответственность:**
- Управление многоэтапным процессом согласования
- Хранение текущего состояния маршрута и участников в `ApprovalInstance`
- Хранение полной истории этапов, участников и решений в `ApprovalInstance.process_snapshot`
- Поддержка автоматического и ручного утверждения
- Поддержка частичного утверждения элементов, объединённых в Package

**Бизнес-правила:**
- Маршрут создаётся при отправке на согласование (не предзадан)
- Пользователь выбирает согласующих для каждого этапа при отправке
- ApprovalInstance имеет флаг `auto_ratification: boolean`
- **Этапы согласования (Approval stages):**
  - Количество: 0..* (переменное)
  - Внутри этапа: согласующие работают параллельно
  - Между этапами: последовательное выполнение
  - Минимум 1 согласующий на этап
  - Все согласующие этапа должны одобрить для перехода на следующий этап
  - Любое отклонение провалит весь процесс
- **Этап утверждения (Ratification stage):**
  - Количество: 1 (обязателен в MVP)
  - Ровно 1 утверждающий
    - если `auto_ratification=true`, утверждение запускается сразу после Approval или сразу после submit, если Approval-этапов нет
    - если `auto_ratification=false`, процесс сначала попадает в `awaiting_ratification`, а затем Ratification запускается вручную: индивидуально или через Package
- **Общие правила:**
  - Минимально допустимый маршрут: `0` этапов Approval и `1` этап Ratification
  - Один человек не может быть в нескольких этапах
- **После всех Approval этапов:**
  - `ApprovalInstance` получает статус `approved`
  - Если `auto_ratification = true` → автоматический переход в `in_ratification`
  - Если `auto_ratification = false` → переход в `awaiting_ratification`
- **Package для утверждения (Ratification):**
  - Создаётся вручную из `ApprovalInstance`, находящихся в состоянии `awaiting_ratification`
  - Может содержать PilotVersion/DeploymentVersion из ЛЮБЫХ Initiatives
  - После создания пакет сразу считается отправленным
  - Активный состав пакета определяется по `approval_instance.package_id`
  - При создании все связанные `ApprovalInstance` переводятся в `in_ratification`
  - Решение возможно по всей группе сразу или по каждому элементу отдельно, но фиксируется по каждому связанному `ApprovalInstance`
- Отдельные таблицы `approval_stage`, `approval_assignment` и `approval_decision` в MVP не используются
- В истории этапов отдельное поле `status` не используется

## Корневые агрегаты

### Инициатива

**Идентификатор:** UUID
**Жизненный цикл:** draft → active → deployed → archived

**Содержит:**
- Симуляции (0..*)
- Пилоты (0..*)
- Внедрения (0..*)
- Артефакты (0..*)

**Инварианты:**
- Должна принадлежать одному Product
- Может быть связана с несколькими SubProduct (многие-ко-многим через initiative_sub_product)
- Если подпродукты не указаны, относится ко всем подпродуктам Product
- Имя должно быть уникальным в рамках Product
- Статус рассчитывается автоматически:
  - draft: пустая ИЛИ все вложенные сущности в draft
  - active: хотя бы одна вложенная сущность не в draft
  - deployed: хотя бы одно Внедрение в deployed
  - archived: переведено вручную (НЕТ каскадного архивирования)
- Нельзя архивировать Инициативу, если есть deployed Внедрения или неархивированные вложенные сущности

---

### Симуляция

**Идентификатор:** UUID
**Жизненный цикл:** draft → running → completed/failed

**Содержит:**
- Результаты (JSONB)

**Инварианты:**
- Должна принадлежать одной Инициативе
- Период данных должен быть валидным (from < to)
- Не может быть изменена после завершения

**Может быть источником для:**
- Скоркарт (через scorecard_source)

---

### Пилот

**Идентификатор:** UUID (Pilot)

**Версионирование (MVP):**
- Пилот является логической сущностью и содержит версии `PilotVersion` (1..*)
- Любые изменения пилота выполняются через **новую версию**
- Пока новая версия не согласована/утверждена и не опубликована, **предыдущая active-версия остаётся активной**
- Согласование/утверждение и Package работают на уровне **версий** (target = `PilotVersion`)
- При публикации новой версии предыдущая версия становится **inactive** (не-current)

**Жизненный цикл версии (PilotVersion):** draft → requires_activation → active → inactive|completed → archived
**Отказы согласования/утверждения (MVP):** approval_rejected / ratification_rejected (редактируемо, можно переотправить или отправить в архив)

**Содержит:**
- Метрики (JSONB)

**Инварианты:**
- Должен принадлежать одной Инициативе
- Должен быть связан минимум с одной Скоркартой
- pilot_group_size + control_group_size > 0
- Не может быть удален, если находится в процессе согласования
- В один момент времени у Pilot может быть только одна **current** версия (показывается в списках как "пилот")

**Связи:**
- Многие-ко-многим с версиями Скоркарт (через `scorecard_version_pilot_version`)
- Связь с ApprovalInstance (полиморфная, создаётся при отправке на согласование версии PilotVersion)

**Процесс согласования:**
- Pilot проходит согласование индивидуально (не в составе Lineage)
- При отправке (submit) пользователь указывает:
  - Этапы согласования (0..* Approval stages)
  - Этап утверждения (1 Ratification stage, обязательно в MVP)
  - Флаг auto_ratification
- Статус requires_activation требует согласования (для PilotVersion)
- После всех Approval этапов версия получает `approved`, после чего `auto_ratification` определяет дальнейший путь: `awaiting_ratification` или `in_ratification`
- Можно отозвать (recall) с согласования, история сохраняется

**Связь со скоркартами:**
- В MVP `Pilot` не является источником для `scorecard_source`
- `PilotVersion` связывается со `ScorecardVersion` через `scorecard_version_pilot_version`

---

### Внедрение

**Идентификатор:** UUID (Deployment)

**Версионирование (MVP):**
- Внедрение является логической сущностью и содержит версии `DeploymentVersion` (1..*)
- Любые изменения внедрения выполняются через **новую версию**
- Пока новая версия не согласована/утверждена и не опубликована, **предыдущая deployed-версия остаётся deployed**
- Согласование/утверждение и Package работают на уровне **версий** (target = `DeploymentVersion`)

**Жизненный цикл версии (DeploymentVersion):**
- `draft`
- `in_approval`
- `approved`
- `awaiting_ratification`
- `in_ratification`
- `ratified`
- `approval_rejected`
- `ratification_rejected`
- `approval_cancelled`
- `ratification_cancelled`
- `deployed`
- `rolled_back`
- `archived`

**Содержит:**
- Конфигурация развертывания (JSONB)
- План отката (Text)

**Инварианты:**
- Должно принадлежать одной Инициативе
- Должно быть связано минимум с одной Скоркартой
- Должно иметь неизменяемый `deployment_type`
- Для `deployment_type = simulation_based` должно иметь:
  - `lineage_simulation_id`
  - `required_scorecard_id`
- Только одно активное развертывание (rollback_at = null)
- Не может быть удалено (hard delete), если развернуто
- В один момент времени у Deployment может быть только одна **current** версия (показывается в списках как "внедрение")
- Агрегат хранит ссылки на `latest_version` и `production_version`
- Критичность версии внедрения рассчитывается как максимум по связанным `ScorecardVersion`
- Поля версии внедрения вместо единого `description` содержат:
  - `goal`
  - `change_description`
  - `application_perimeter`

**Связи:**
- Многие-ко-многим с версиями Скоркарт (через `deployment_version_scorecard`)
- Связь с ApprovalInstance (полиморфная, создаётся при отправке на согласование версии DeploymentVersion)
- Ограниченный `Lineage` к одной `Simulation` на уровне агрегата `Deployment`

**Процесс согласования:**
- Deployment проходит согласование индивидуально (не в составе Lineage)
- При отправке (submit) пользователь указывает:
  - Этапы согласования (0..* Approval stages)
  - Этап утверждения (1 Ratification stage, обязательно в MVP)
  - Флаг auto_ratification
- Если есть Approval-этапы, версия стартует в `in_approval`
- Если Approval нет и `auto_ratification=true`, версия стартует в `in_ratification`
- Если Approval нет и `auto_ratification=false`, версия стартует в `awaiting_ratification`
- После всех Approval этапов `auto_ratification` определяет дальнейший путь
- Можно отозвать (recall) с согласования, история сохраняется
- После `ratified` версия может быть переведена в `deployed`
- При публикации новой версии предыдущая продуктивная версия переводится в `rolled_back`

---

### Скоркарта

**Идентификатор:** UUID
**Жизненный цикл:** active → archive

**Содержит:**
- Версии (1..*)
- Источники (0..*)
- Связи использования с версиями `Pilot` и `Deployment`

**Инварианты:**
- Должна быть создана из ScorecardTemplateVersion
- Каждое изменение создает новую версию
- Номера версий - последовательные целые числа
- Если не ручная, должна иметь минимум один источник
- Название скоркарты обязательно
- Если скоркарта создаётся с частичным предзаполнением по `Simulation`, её название автоматически задаётся в формате `Скоркарта {scorecardId} по симуляции {simulationId}: {simulationName}.`
- В MVP не использует SubProduct
- При создании или редактировании фронт вычисляет критичность и сохраняет её в `ScorecardVersion`
- Новая версия скоркарты всегда жёстко ассоциируется с конкретными версиями доменных элементов, к которым она привязана
- Создание новой версии скоркарты создаёт новые версии всех связанных доменных элементов
- Создание или удаление связи скоркарты с доменным элементом создаёт новую версию только этого доменного элемента
- Изменение состава связей скоркарты допускается только из формы или API соответствующего доменного элемента; редактирование скоркарты не управляет составом связей
- Если есть хотя бы одна связь с `PilotVersion` или `DeploymentVersion`, скоркарта находится в статусе `active`
- Если удаляется последняя связь с `PilotVersion` и `DeploymentVersion`, скоркарта автоматически переходит в `archive`
- Источники из `scorecard_source` не влияют на статус скоркарты
- Скоркарта не имеет собственного `ApprovalInstance`
- Если скоркарта является `required_scorecard_id` у внедрения типа `simulation_based`, связь с этим внедрением не может быть удалена

**Связи:**
- Многие-ко-многим с версиями Пилотов
- Многие-ко-многим с версиями Внедрений
- Многие-ко-многим с источниками типа `simulation` (через `scorecard_source`)

---
### Lineage (визуализация происхождения)

**НЕ является CRUD-сущностью** - это визуализация связей "создано из"

**Текущая реализация MVP:**
- `Deployment.lineage_simulation_id` - ссылка Внедрения типа `simulation_based` на одну `Simulation`
- более широкая сеть связей через `*_source` таблицы остаётся целевой архитектурой следующих итераций

**Правила:**
- Lineage доступен только для `Deployment` типа `simulation_based`
- Для одного внедрения допускается только одна lineage-симуляция
- lineage-симуляция должна совпадать с источником обязательной lineage-скоркарты внедрения
- Frontend визуализирует Lineage отдельным полем в карточке внедрения
- Полная рекурсивная цепочка `Simulation → Pilot → Deployment` не входит в текущий MVP-срез

**Назначение:**
- Показать, результаты какой симуляции внедряются в `simulation_based` deployment
- Обеспечить минимальную трассируемость происхождения внедрения
- НЕ является объектом согласования (PilotVersion и DeploymentVersion проходят согласование индивидуально)

### Пакет

**Идентификатор:** UUID
**Жизненный цикл:** отдельный статусный enum отсутствует

**Содержит:**
- `ApprovalInstance` (2..*) через поле `approval_instance.package_id`
- Бриф пакета

**Инварианты:**
- Создаётся как рабочая группировка элементов из очереди `awaiting_ratification`
- Создаётся и отправляется одной операцией
- После создания пакет доступен только для просмотра
- Может содержать `PilotVersion` и/или `DeploymentVersion` из ЛЮБЫХ Initiatives
- Может содержать элементы разных продуктов, если это разрешено RBAC
- Для `methodologist` в MVP допустим смешанный пакет из элементов разных продуктов
- Критичность пакета вычисляется на лету по связанным элементам:
  - если хотя бы один элемент имеет `criticality = high`, пакет имеет `criticality = high`
  - иначе, если хотя бы один элемент имеет `criticality = low`, пакет имеет `criticality = low`
  - иначе пакет имеет `criticality = null`
- Package не является target согласования и не имеет собственного `ApprovalInstance`
- Package не имеет собственных полей `status`, `name`, `description`, `ratifier_id`
- Если после индивидуальных решений в пакете остаётся меньше 2 активных элементов, пакет прекращает существование как активная группировка
- История пакетных действий сохраняется в историях входящих `ApprovalInstance`

**Назначение:**
- Группировка PilotVersion/DeploymentVersion для совместного утверждения (Ratification)
- Экономия времени утверждающих (утверждение за один приём)
- Операционная подготовка пакетного ratification и аудит пакетного контекста

**Частичное утверждение:**
- Утверждающий может принять решение по всей группе сразу или по каждому элементу индивидуально
- Каждое решение обновляет соответствующий `ApprovalInstance` и его `process_snapshot`
- Package может быть частично обработан; после сокращения активного состава меньше чем до 2 элементов он перестаёт существовать как активная группировка

## Основные концепции

### Источники скоркарт

Скоркарты могут быть созданы из нескольких источников:

1. **Ручное создание**
   - Пользователь создает скоркарту напрямую
   - is_manual = true
   - Нет записей в scorecard_source

2. **Из Симуляции**
   - Симуляция завершается с результатами
   - Пользователь создаёт Скоркарту из контекста `Pilot` или `Deployment` с частичным предзаполнением по результатам Симуляции
   - scorecard_source: (scorecard_version_id, 'simulation', simulation_id)

3. **Комбинированные источники**
   - Скоркарта может иметь множество источников типа `simulation`
   - Источники используются только для происхождения данных и не заменяют связь скоркарты с версиями доменных элементов
   - В MVP источники типа `pilot` для `scorecard_source` не используются

### Lineage инициативы

В текущем MVP Lineage инициативы представляет ограниченную трассировку для внедрений типа `simulation_based`:

```
Симуляция (0..1) → Внедрение (0..*)
```

**Допустимые пути Lineage:**
1. Только симуляции (фаза исследования, без lineage на внедрении)
2. Симуляция + Внедрение(я) типа `simulation_based`
3. Только Внедрение(я) типа `general`

**Правила Lineage:**
- Для одного внедрения допустима только одна lineage-симуляция
- Одна симуляция может быть lineage-источником для нескольких внедрений
- Все сущности должны принадлежать одной Инициативе
- Lineage НЕ является объектом согласования (только визуализация)
- PilotVersion и DeploymentVersion проходят согласование индивидуально

### Связи скоркарт

```
Симуляция (источник) → Скоркарта
Скоркарта (многие) ←→ (многие) Пилот
Скоркарта (многие) ←→ (многие) Внедрение
```

**Примеры:**

1. **Одна Скоркарта, Множество Пилотов (A/B/C тестирование)**
   - Скоркарта "Модель риска v2.3"
   - Пилот A: 10% трафика
   - Пилот B: 15% трафика
   - Пилот C: 20% трафика

2. **Множество Скоркарт, Один Пилот**
   - Пилот "Эксперимент Q1 2026"
   - Скоркарта "Application Score"
   - Скоркарта "Behavioral Score"
   - Скоркарта "Income Verification"

3. **Множество Скоркарт, Одно Внедрение**
   - Внедрение "Новая риск-стратегия"
   - Скоркарта "Application Score v3"
   - Скоркарта "Behavioral Score v2"

**Правила связей:**
- Версия скоркарты всегда связана с версиями доменных элементов, а не просто с агрегатами верхнего уровня
- Новая версия скоркарты перевыпускает новые версии всех связанных доменных элементов
- Добавление или удаление одной связи перевыпускает только тот доменный элемент, связь которого меняется

### Связь Симуляция-Пилот-Внедрение

Когда Пилот создается из Скоркарт, источником которых были Симуляции:

1. Пользователь запускает Симуляцию S1
2. Пользователь создает Скоркарту SC1 из результатов S1
   - scorecard_source: (SC1.version_id, 'simulation', S1.id)
3. Пользователь создает Пилот P1 и связывает SC1
   - scorecard_version_pilot_version: (SC1.version_id, P1.current_version_id)
4. Пилот P1 теперь неявно связан с Симуляцией S1 через Скоркарту SC1

Запрос для поиска Симуляций для Пилота:
```sql
SELECT DISTINCT s.*
FROM simulation s
JOIN scorecard_source ss ON ss.source_id = s.id AND ss.source_type = 'simulation'
JOIN scorecard_version_pilot_version spv ON spv.scorecard_version_id = ss.scorecard_version_id
JOIN pilot p ON p.current_version_id = spv.pilot_version_id
WHERE p.id = :pilot_id
```

## Бизнес-правила

### БП-1: Управление инициативой

- Статус Инициативы рассчитывается автоматически:
  - draft: Инициатива пустая ИЛИ все вложенные сущности в статусе draft
  - active: хотя бы одна вложенная сущность перешла в следующий статус (running, requires_activation, in_approval, awaiting_ratification, in_ratification, active, approved и т.д.)
  - deployed: хотя бы одно Внедрение в статусе deployed
  - archived: переведено вручную пользователем
- **НЕТ каскадного архивирования** - вложенные сущности НЕ архивируются автоматически
- Нельзя архивировать Инициативу, если:
  - Хотя бы одно Внедрение в статусе deployed
  - Есть неархивированные вложенные сущности (Симуляции, Пилоты, Внедрения)
- Архивированная Инициатива не может быть изменена
- Можно создавать Симуляции, Пилоты и Внедрения в Инициативе с любым статусом (кроме archived)

### БП-2: Выполнение симуляции

- Симуляция может быть запущена только из статуса 'draft'
- Выполняющаяся Симуляция не может быть изменена
- Результаты завершенной Симуляции неизменяемы
- Провалившаяся Симуляция может быть отредактирована и перезапущена

### БП-3: Создание и управление скоркартой

- Скоркарта должна быть создана из валидной ScorecardTemplateVersion
- В MVP скоркарта создаётся только из контекста `Pilot` или `Deployment`
- Ручная Скоркарта: is_manual = true, источники не требуются
- Скоркарта с предзаполнением: is_manual = false, требуется минимум один источник типа `simulation`
- Название Скоркарты обязательно
- При создании Скоркарты с предзаполнением по `Simulation` название автоматически задаётся в формате `Скоркарта {scorecardId} по симуляции {simulationId}: {simulationName}.`
- Фронт вычисляет критичность по полям шаблона и порогам `criticality_thresholds`, затем передаёт рассчитанное значение в `ScorecardVersion`
- Каждое редактирование создает новую ScorecardVersion с инкрементированным номером версии
- Новая версия скоркарты создаёт новые версии всех связанных доменных элементов
- Добавление или удаление связи скоркарты с доменным элементом создаёт новую версию только этого доменного элемента
- Добавление или удаление связи скоркарты допускается только в контексте самого доменного элемента, так как может требовать отдельного согласования или утверждения по правилам этой сущности
- Скоркарта не имеет собственного процесса согласования
- Статус Скоркарты определяется только составом связей:
  - если существует хотя бы одна связь с `PilotVersion` или `DeploymentVersion`, статус Скоркарты = `active`
  - если не осталось ни одной связи с `PilotVersion` или `DeploymentVersion`, статус Скоркарты = `archive`
- Источники из `scorecard_source` не влияют на статус Скоркарты
- Скоркарта в статусе `archive` не может быть изменена

### БП-4: Управление пилотом

- Пилот должен быть связан минимум с одной Скоркартой
- Пилот проходит согласование индивидуально (не в составе Lineage)
- При отправке на согласование пользователь выбирает согласующих для каждого этапа
- Пилот может быть отправлен на согласование когда status = 'requires_activation'
- После этапов Approval: флаг auto_ratification определяет дальнейший путь
  - Если auto_ratification = true → сразу на Ratification индивидуально
  - Если auto_ratification = false → попадает в awaiting_ratification (очередь ожидания утверждения), дальше по команде пользователя:
    - запустить Ratification индивидуально (без Package) или
    - создать Package и отправить на Ratification (групповое утверждение)
- Если PilotVersion находится в статусе `awaiting_ratification` и пользователь вносит изменения:
  - версия снимается из очереди ручного утверждения
  - требуется повторная отправка на согласование/утверждение
- Можно отозвать Пилот из согласования в любой момент (история сохраняется)
- Активный Пилот не может быть удален (hard delete)

### БП-5: Управление внедрением

- Внедрение должно быть связано минимум с одной Скоркартой
- Внедрение проходит согласование индивидуально (не в составе Lineage)
- При отправке на согласование пользователь выбирает согласующих для каждого этапа
- Внедрение может быть отправлено на согласование когда status = `draft`, `approval_rejected`, `ratification_rejected`, `approval_cancelled` или `ratification_cancelled`
- После этапов Approval: флаг auto_ratification определяет дальнейший путь
  - Если auto_ratification = true → сразу на Ratification индивидуально
  - Если auto_ratification = false → попадает в awaiting_ratification (очередь ожидания утверждения), дальше по команде пользователя:
    - запустить Ratification индивидуально (без Package) или
    - создать Package и отправить на Ratification (групповое утверждение)
- Если DeploymentVersion находится в статусе `awaiting_ratification` и пользователь вносит изменения:
  - версия снимается из очереди ручного утверждения
  - требуется повторная отправка на согласование/утверждение
- Можно отозвать Внедрение из согласования в любой момент (история сохраняется)
- Развернутое Внедрение не может быть удалено (hard delete)
- Только одно активное развертывание на Внедрение (rollback_at = null)
- После отката автоматическое создание новой версии не требуется; при необходимости пользователь создаёт новую версию отдельно

### БП-6: Lineage (визуализация происхождения)

- Lineage НЕ является CRUD-сущностью
- В текущем MVP Lineage реализован только как ссылка `deployment.lineage_simulation_id`
- Lineage доступен только для внедрений типа `simulation_based`
- Для одного внедрения допустима только одна lineage-симуляция
- Все сущности (Симуляции и Внедрения) должны принадлежать одной Инициативе
- Lineage НЕ является объектом согласования
- Lineage используется только для визуализации и трассировки происхождения внедрения
- Lineage-симуляция должна совпадать с источником обязательной lineage-скоркарты

### БП-7: Согласование Pilot и Deployment

- PilotVersion и DeploymentVersion проходят согласование индивидуально (не в составе Lineage)
- PilotVersion может быть отправлен на согласование когда status = 'requires_activation'
- DeploymentVersion может быть отправлен на согласование когда status ∈ {`draft`, `approval_rejected`, `ratification_rejected`, `approval_cancelled`, `ratification_cancelled`}
- При отправке (submit) пользователь указывает:
  - Этапы согласования (0..* Approval stages)
  - Для каждого этапа: список согласующих (минимум 1) - **только пользователи с ролью Approver**
  - Этап утверждения (1 Ratification stage, обязателен в MVP)
  - Флаг auto_ratification
  - Для Ratification:
    - если auto_ratification = true: 1 утверждающий выбирается сразу при submit (**только пользователь с ролью Ratifier**)
    - если auto_ratification = false: утверждающий выбирается при запуске ratification (индивидуально или через Package)
- Минимальный маршрут: `0` этапов Approval и `1` этап Ratification
- Один человек не может быть в нескольких этапах
- **Нельзя выбрать самого себя**
- Можно отозвать (recall) с согласования, история сохраняется:
  - на этапе Approval: `ApprovalInstance.status = approval_cancelled`
  - на этапе Ratification: `ApprovalInstance.status = ratification_cancelled`

### БП-8: Управление пакетом

- Package создаётся вручную из элементов очереди `awaiting_ratification` и сразу считается отправленным
- Backend-черновики, изменение состава после создания и отдельные статусы Package в MVP отсутствуют
- Package может содержать PilotVersion и/или DeploymentVersion из ЛЮБЫХ Initiatives
- Package может содержать элементы разных продуктов, если это разрешено RBAC
- Для `methodologist` в MVP допустимо формирование одного Package из элементов разных продуктов
- Package не может быть создан, если список PilotVersion/DeploymentVersion пуст
- В MVP: Package не может быть создан, если выбрано меньше 2 элементов (минимальный размер пакета = 2)
- Критичность Package вычисляется на лету по правилам:
  - если хотя бы один элемент имеет `criticality = high`, пакет имеет `criticality = high`
  - иначе, если хотя бы один элемент имеет `criticality = low`, пакет имеет `criticality = low`
  - иначе пакет имеет `criticality = null`
- Package это объединение позиций для удобства утверждающего: одно письмо/одна кнопка.
- Решение может быть принято как по всем позициям сразу, так и по каждой позиции отдельно; при этом итог сохраняется по каждому связанному `ApprovalInstance`.
- Активный состав Package определяется по `approval_instance.package_id`
- Package не имеет собственного ApprovalInstance; пакетный контекст хранится в историях связанных `ApprovalInstance`

### БП-9: Процесс согласования

**Отправка на согласование:**
- Пользователь выбирает согласующих при отправке (маршрут не предзадан)
- Создаётся ApprovalInstance с этапами и назначениями
- Email уведомления согласующим первого этапа

**Этапы согласования (Approval stages):**
- Количество: 0..* (переменное)
- Внутри этапа: согласующие работают параллельно
- Между этапами: последовательное выполнение
- Все согласующие этапа должны одобрить для перехода на следующий этап
- Любое отклонение завершает ApprovalInstance в статусе `approval_rejected` и переводит связанную версию в `approval_rejected`
- После прохождения всех Approval этапов: `ApprovalInstance.status = approved`

**Переход на Ratification:**
- Если `auto_ratification = true`:
  - после `approved` тот же `ApprovalInstance` автоматически переходит в `in_ratification`
  - Email уведомление утверждающему
  - PilotVersion/DeploymentVersion проходит утверждение индивидуально
- Если `auto_ratification = false`:
  - после `approved` `ApprovalInstance` попадает в состояние `awaiting_ratification`
  - Далее по команде пользователя:
    - запустить Ratification индивидуально (без Package) или
    - создать Package для совместного утверждения; после создания связанные `ApprovalInstance` переходят в `in_ratification`

**Этап утверждения (Ratification):**
- Для индивидуального PilotVersion/DeploymentVersion: 1 утверждающий принимает решение
- Для Package: утверждающий может принять решение по всей группе сразу или по каждому элементу индивидуально, но итог фиксируется по каждому `ApprovalInstance`
- После утверждения: статус = "ratified"
- После отклонения: статус = "ratification_rejected"

**Отзыв с согласования:**
- Можно отозвать PilotVersion/DeploymentVersion с любого этапа, а также прекратить пакетное Ratification для элементов, объединённых в Package
- На этапе Approval `ApprovalInstance.status = approval_cancelled`
- На этапе Ratification `ApprovalInstance.status = ratification_cancelled`
- История полностью сохраняется
- Сущность получает статус `approval_cancelled` или `ratification_cancelled` в зависимости от этапа
- Отзыв (recall) обязателен для MVP и должен работать как с Approval, так и с Ratification
- Reject не возвращает сущность в `draft`: после reject target переходит в `approval_rejected` или `ratification_rejected` (в зависимости от этапа); в этих статусах доступны редактирование, повторная отправка и архивирование

### БП-10: Управление доступом

**1. Scope доступа ПРМ:**
- ПРМ видит все сущности всех продуктов (Simulation/Pilot/Deployment/Scorecard)
- ПРМ управляет только Initiative своего Product (создание, редактирование, архивирование)
- При создании Initiative автоматически проверяется product_id из UserRole
- ПРМ может управлять Simulation/Pilot/Deployment только в рамках своих Initiative
- ПРМ может создавать/редактировать Scorecard своего продукта
- ПРМ может отправлять свои PilotVersion/DeploymentVersion на согласование
- ПРМ может формировать Package из своих PilotVersion/DeploymentVersion (в разделе "Отправка на утверждение")

**2. Scope доступа Методолога:**
- Методолог видит все сущности всех продуктов
- Методолог управляет Scorecard своего продукта и артефактами своих сущностей
- Методолог не может создавать Initiative/Simulation/Pilot/Deployment
- Методолог может отправлять свои PilotVersion/DeploymentVersion на согласование
- Методолог может формировать Package из PilotVersion/DeploymentVersion любых продуктов, доступных на странице `Пакеты`

**3. Scope доступа Approver/Ratifier:**
- Approver и Ratifier видят все сущности системы (Simulation/Pilot/Deployment/Scorecard)
- Approver и Ratifier не имеют прав на создание/редактирование сущностей
- Approver и Ratifier не имеют доступа к разделу "Отправка на утверждение" (формирование Package)
- В разделе "Согласования" Approver/Ratifier видят только те элементы, где они назначены, и могут принимать решения

**4. Назначение согласующих:**
- При отправке PilotVersion/DeploymentVersion/Package на согласование пользователь выбирает согласующих для каждого этапа:
  - Для этапов **Approval**: только пользователи с ролью **Approver**
  - Для этапа **Ratification**: только пользователи с ролью **Ratifier**
- Нельзя выбрать самого себя при отправке на согласование
- ПРМ/Методолог, назначенные Approver/Ratifier, видят свои назначения + свою обычную работу

**5. Множественные роли:**
- User может иметь несколько UserRole (например, prm для Product A + approver)
- При входе в раздел "Согласования" показываются все назначения пользователя
- Админ видит все согласования всех пользователей

**5. Матрица прав доступа:**

## Управление жизненным циклом

### Жизненный цикл инициативы

```
draft → active → deployed
  ↓       ↓
archived (только если нет deployed внедрений)
```

**Автоматический расчет статуса:**
- **draft**: Инициатива пустая ИЛИ все вложенные сущности в draft
- **active**: Хотя бы одна вложенная сущность перешла в следующий статус (running, requires_activation, in_approval, awaiting_ratification, in_ratification, active, approved и т.д.)
- **deployed**: Хотя бы одно Внедрение в статусе deployed

**Ручной переход:**
- **любой статус → archived**: Пользователь архивирует Инициативу
  - Все вложенные сущности также переводятся в archived
  - **Ограничение**: Нельзя архивировать, если есть Внедрения в deployed

**Восстановление:**
- archived → draft: Пользователь восстанавливает Инициативу (все вложенные сущности восстанавливаются)

### Жизненный цикл симуляции

```
draft → running → completed
                → failed
```

**Переходы:**
- draft → running: Пользователь запускает Симуляцию
- running → completed: Симуляция успешно завершена
- running → failed: Симуляция провалилась
- running → draft: Пользователь отменяет Симуляцию
- failed → draft: Пользователь редактирует и перезапускает

### Жизненный цикл версии пилота (PilotVersion)

```
draft → requires_activation → active → inactive → archived
                          ↘
                           completed → archived

requires_activation → approval_rejected → requires_activation
requires_activation → ratification_rejected → requires_activation

active → inactive (замещено более новой версией)
```

**Переходы:**
- draft → requires_activation: Пользователь отправляет версию на согласование/активацию
- requires_activation → active: Версия согласована/утверждена и опубликована как текущая
- active → inactive: Опубликована более новая версия (текущая сменилась)
- requires_activation → approval_rejected: Версия отклонена на этапе согласования (Approval)
- requires_activation → ratification_rejected: Версия отклонена на этапе утверждения (Ratification)
- approval_rejected → requires_activation: Исправлено и повторно отправлено
- ratification_rejected → requires_activation: Исправлено и повторно отправлено
- active → completed: Пилот завершён вручную (по текущей версии)
- completed → archived: Версия архивирована
- inactive → archived: Версия архивирована

### Жизненный цикл версии внедрения (DeploymentVersion)

```
draft → in_approval
draft → in_ratification
draft → awaiting_ratification

in_approval → approved
approved → in_ratification
approved → awaiting_ratification
awaiting_ratification → in_ratification
in_ratification → ratified

in_approval → approval_rejected
in_approval → approval_cancelled
approved → ratification_cancelled
awaiting_ratification → ratification_cancelled
in_ratification → ratification_rejected

ratified → deployed
deployed → rolled_back

approval_rejected → in_approval / in_ratification / awaiting_ratification
ratification_rejected → in_approval / in_ratification / awaiting_ratification
approval_cancelled → in_approval / in_ratification / awaiting_ratification
ratification_cancelled → in_approval / in_ratification / awaiting_ratification
rolled_back → archived
```

**Переходы:**
- draft → in_approval: Пользователь отправляет версию на согласование, и маршрут содержит Approval-этапы
- draft → in_ratification: Пользователь отправляет версию без Approval и с автоматическим Ratification
- draft → awaiting_ratification: Пользователь отправляет версию без Approval и с ручным Ratification
- in_approval → approved: Все этапы Approval завершены успешно
- approved → in_ratification: Ratification запускается автоматически
- approved → awaiting_ratification: Ratification ожидает ручного запуска
- awaiting_ratification → in_ratification: Пользователь или пакетный процесс запускает Ratification
- in_approval → approval_rejected: Версия отклонена на этапе Approval
- in_approval → approval_cancelled: Версия отозвана на этапе Approval
- approved → ratification_cancelled: Версия отозвана после Approval, но до старта Ratification
- awaiting_ratification → ratification_cancelled: Версия отозвана из очереди ручного утверждения
- in_ratification → ratification_rejected: Версия отклонена на этапе Ratification
- in_ratification → ratified: Версия утверждена
- ratified → deployed: Пользователь переводит версию в прод
- deployed → rolled_back: Пользователь откатывает текущую продуктивную версию или версия автоматически замещается новой продуктивной версией
- approval_rejected / ratification_rejected / approval_cancelled / ratification_cancelled → in_approval / in_ratification / awaiting_ratification: Исправленная версия повторно отправляется в процесс
- rolled_back → archived: Неактуальная версия архивируется

### Жизненный цикл скоркарты

```
active → archive
```

**Переходы:**
- создание скоркарты в контексте `Pilot` или `Deployment` сразу задаёт статус `active`, потому что в момент создания уже существует хотя бы одна связь использования
- active → archive: удалена последняя связь с `PilotVersion` или `DeploymentVersion`

**Важно:**
- Скоркарта не имеет собственного ApprovalInstance
- Статус Скоркарты не зависит от этапов согласования связанных `PilotVersion` и `DeploymentVersion`
- Одна Скоркарта может быть прикреплена к нескольким `PilotVersion` и `DeploymentVersion`
- Пока существует хотя бы одна такая связь, статус Скоркарты остаётся `active`

### Жизненный цикл пакета

У `Package` нет собственного enum статусов.

**Создание пакета:**
- Package создаётся из 2 и более `ApprovalInstance` в `awaiting_ratification`
- В `package` сохраняются бриф и служебные поля создания
- После создания связанные `ApprovalInstance` получают `package_id` и переходят в `in_ratification`

**Существование активной группировки:**
- После создания Package доступен только для просмотра
- Пока активных связанных элементов 2 и более, Package существует как активная группировка
- Если после индивидуальных решений активных элементов остаётся меньше 2, Package перестаёт возвращаться как активная группировка

**История решений:**
- Решения и комментарии по пакетному утверждению сохраняются в `process_snapshot` соответствующих `ApprovalInstance`
- Пакетный контекст остаётся доступен через истории связанных `ApprovalInstance` даже после распада активной группировки

## Процесс согласования

### Динамическое согласование

Система использует **динамическое создание маршрутов согласования** при отправке PilotVersion/DeploymentVersion на согласование.

**Ключевые принципы:**
- Нет предзаданных маршрутов согласования (ApprovalRoute)
- Пользователь выбирает согласующих при отправке на согласование
- Переменное количество этапов Approval (0..*)
- Обязательный этап Ratification (1)
- Минимально допустимый маршрут: `0` этапов Approval и `1` этап Ratification
- Один человек не может быть в нескольких этапах одного ApprovalInstance

**Этапы Approval (Согласование)**
- Переменное количество этапов (0..*)
- Каждый этап: минимум 1 согласующий
- Последовательное выполнение между этапами
- Параллельная работа согласующих внутри этапа
- Все согласующие этапа должны одобрить для перехода к следующему этапу
- Любое отклонение провалит процесс

**Этап Ratification (Утверждение)**
- Обязательный этап (1)
- Ровно 1 утверждающий
  - если `auto_ratification=true` выбирается при submit
  - если `auto_ratification=false` выбирается при запуске ratification (индивидуально или через Package)
- Финальное решение руководства
- Может запускаться автоматически или вручную

### Объекты согласования

- **PilotVersion**: Индивидуальное согласование версии пилота (статус: requires_activation)
- **DeploymentVersion**: Индивидуальное согласование версии внедрения (стартовые статусы процесса: `in_approval`, `in_ratification` или `awaiting_ratification`)
- **Package**: Группировка связанных `ApprovalInstance` для совместного Ratification
  - Может содержать PilotVersion/DeploymentVersion из разных Initiatives
  - Может содержать элементы разных продуктов, если это разрешено RBAC
  - Поддерживает частичное согласование, но итог фиксируется по каждому `ApprovalInstance`
  - Не является самостоятельным объектом согласования и не имеет собственного `ApprovalInstance`
- **Скоркарта**: Не является самостоятельным объектом согласования; может отображаться как связанная сущность у PilotVersion/DeploymentVersion

### Workflow согласования

**Шаг 1: Отправка на согласование**
- ПРМ отправляет PilotVersion (`requires_activation`) или DeploymentVersion (`draft` / reject / cancel-статусы)
- При отправке пользователь выбирает:
  - Этапы Approval (0..*): для каждого этапа указывает список согласующих (минимум 1)
    - **Можно выбрать только пользователей с ролью Approver**
  - Этап Ratification (1): обязателен
    - если auto_ratification = true: указывает ровно 1 утверждающего (**только роль Ratifier**)
    - если auto_ratification = false: утверждающий выбирается при запуске ratification (индивидуально или через Package)
- Минимальный маршрут: `0` этапов Approval и `1` этап Ratification
- Один человек не может быть в нескольких этапах
- **Нельзя выбрать самого себя**
- Создаётся ApprovalInstance с target_type/target_id (PilotVersion/DeploymentVersion)
- В `ApprovalInstance` сохраняются:
  - флаг `auto_ratification`;
  - бриф согласования;
  - полный `process_snapshot` маршрута и истории
- Email уведомления согласующим первого этапа

**Шаг 2: Этапы Approval (если есть)**
- Этапы выполняются последовательно (stage_order: 1, 2, 3...)
- Внутри этапа согласующие работают параллельно
- Каждый согласующий: approve или reject (комментарий опционален)
- Если хотя бы один reject → ApprovalInstance.status = `approval_rejected`, связанная версия переходит в `approval_rejected`, процесс завершён
- Если все в этапе approve → переход к следующему этапу Approval
- Если все этапы Approval пройдены → `ApprovalInstance.status = approved`, далее переход в `awaiting_ratification` или `in_ratification`

**Шаг 3: Этап Ratification**
- Проверяется флаг ApprovalInstance.auto_ratification

**Вариант A: auto_ratification = true (индивидуальное утверждение)**
- `ApprovalInstance` автоматически переходит в `in_ratification`
- Email уведомление утверждающему
- Утверждающий принимает решение: ratify или reject
- Если ratify → `ApprovalInstance` переходит в `ratified`, а target — в согласованный доменный статус
- Если reject → PilotVersion/DeploymentVersion переходит в `ratification_rejected`

**Вариант B: auto_ratification = false (очередь ожидания утверждения)**
- `ApprovalInstance` попадает в состояние `awaiting_ratification`
- Далее по команде пользователя:
  - **Индивидуально**: запустить Ratification для выбранного PilotVersion/DeploymentVersion, переводя текущий `ApprovalInstance` в `in_ratification`
  - **Пакетом**: создать Package из выбранных элементов `awaiting_ratification`; после создания пакет сразу считается отправленным, а входящие `ApprovalInstance` переходят в `in_ratification`

**Шаг 4: Частичное согласование Package**
- При утверждении Package утверждающий видит список всех PilotVersion/DeploymentVersion в Package
- Утверждающий может принять решение:
  - по всем элементам сразу (одной кнопкой) или
  - по каждому элементу индивидуально
- Решения сохраняются в `process_snapshot` соответствующих `ApprovalInstance` с пакетным контекстом
- Package считается обработанным, когда активная группировка перестаёт существовать или когда по всем связанным элементам принято итоговое решение

**Шаг 5: Отзыв из согласования (Recall)**
- ПРМ может отозвать PilotVersion/DeploymentVersion из согласования в любой момент
- История согласования сохраняется:
  - на этапе Approval `ApprovalInstance.status = approval_cancelled`
  - на этапе Ratification `ApprovalInstance.status = ratification_cancelled`
- Связанная версия получает статус `approval_cancelled` или `ratification_cancelled` в зависимости от этапа, после чего может быть повторно отправлена в процесс
- Можно переотправить на согласование с новыми согласующими

**Шаг 6: Завершение активной группировки Package**
- После создания Package на странице пакетов доступен только для просмотра.
- Если после индивидуальных решений активных элементов остаётся меньше 2, Package перестаёт отображаться как активная группировка.
- Запись `package` и пакетный контекст в историях связанных `ApprovalInstance` сохраняются.

**Шаг 7: После утверждения**
- PilotVersion: может стать текущей активной версией (status = active)
- DeploymentVersion: может быть развёрнут (status = deployed)

### Правила согласования

- PilotVersion может быть отправлен на согласование в статусах: requires_activation
- DeploymentVersion может быть отправлен на согласование в статусах: `draft`, `approval_rejected`, `ratification_rejected`, `approval_cancelled`, `ratification_cancelled`
- Lineage НЕ является объектом согласования (только визуализация связей)
- Package создаётся из элементов очереди `awaiting_ratification` и сразу считается отправленным
- Package может содержать PilotVersion/DeploymentVersion из разных Initiatives
- Package может содержать элементы разных продуктов, если это разрешено RBAC
- При отправке на согласование пользователь выбирает согласующих для каждого этапа
- Минимальный маршрут согласования: `0` этапов Approval и `1` этап Ratification
- Один человек не может быть в нескольких этапах одного ApprovalInstance
- Package поддерживает частичное согласование (каждый элемент решается индивидуально)
- Можно отозвать PilotVersion/DeploymentVersion из согласования в любой момент (история сохраняется)
- Активный состав Package определяется по `approval_instance.package_id`; отдельного draft-состава пакета в MVP нет
- Скоркарты, прикрепленные к PilotVersion/DeploymentVersion, не меняют свой статус по этапам согласования этих сущностей
- Статус Скоркарты определяется только наличием хотя бы одной связи с PilotVersion/DeploymentVersion

## Единый язык

### Основные термины

- **Инициатива** - Группировка верхнего уровня всей работы, связанной со стратегическим изменением
- **Симуляция** - Анализ исторических данных для прогноза эффектов
- **Пилот** - Контролируемый эксперимент с реальными клиентами
- **Внедрение** - Продуктовое внедрение модификации стратегии
- **Lineage** - Ограниченная для текущего MVP ссылка `simulation_based` внедрения на Симуляцию-источник
- **Скоркарта** - Скоркарта кредитного решения с конфигурацией и финансовыми эффектами
- **Шаблон скоркарты** - Версионированный шаблон для создания скоркарт с полями, значениями по умолчанию и порогами критичности
- **Пакет** - Группа `ApprovalInstance`, связанных с версиями PilotVersion/DeploymentVersion, для совместного Ratification; активный состав определяется по `package_id`

### Термины ролевой модели

- **Пользователь** - Участник системы с одной или несколькими ролями
- **ПРМ** (Product Risk Manager) - Управляет инициативами своего продукта, может отправлять на согласование и формировать пакеты
- **Методолог** - Управляет скоркартами и документами, может отправлять на согласование и формировать пакеты
- **Approver** - Согласующий на этапах Approval, видит только раздел "Согласования" и свои назначения
- **Ratifier** - Утверждающий на этапе Ratification, видит только раздел "Согласования" и свои назначения
- **Администратор** - Полный доступ ко всем функциям системы
- **Назначение на согласование** - Привязка пользователя к этапу согласования
- **Множественные роли** - Пользователь может иметь несколько ролей одновременно (например, ПРМ + Approver + Ratifier)
- **Scope доступа** - Область видимости и управления для роли; для `methodologist` действует отдельное исключение на странице `Пакеты`, где разрешена работа с элементами разных продуктов

### Термины скоркарт

- **Шаблон скоркарты** - Определение шаблона для продукта
- **Версия шаблона скоркарты** - Версионированный шаблон со структурой полей, значениями по умолчанию и порогами расчёта критичности
- **Скоркарта** - Экземпляр, созданный из шаблона
- **Версия скоркарты** - Версионированная конфигурация скоркарты
- **Источник скоркарты** - Происхождение данных скоркарты; в MVP источник для `scorecard_source` — только Симуляция
- **Ручная скоркарта** - Скоркарта, созданная напрямую пользователем
- **Скоркарта из источника** - Скоркарта, созданная в контексте `Pilot` или `Deployment` с частичным предзаполнением по данным Симуляции
- **Согласование скоркарты** - Скоркарта не согласуется самостоятельно и может только отображаться как связанная сущность в согласовании Пилота или Внедрения
- **Вычисление статуса скоркарты** - Статус Скоркарты определяется только наличием или отсутствием связей с `PilotVersion` и `DeploymentVersion`

### Термины жизненного цикла

- **Черновик** - Начальное состояние, может быть отредактировано
- **Активна** - В процессе, выполняется
- **Требует активации** - Пилот готов к согласованию для запуска
- **Требует корректировки** - Пилот нуждается в корректировке
- **Требует одобрения** - Внедрение готово к согласованию для развертывания
- **Одобрено** - Одобрено всеми согласующими, готово к развертыванию
- **Внедрено** - В продакшне
- **Завершено** - Успешно завершено
- **Откачено** - Возвращено к предыдущему состоянию
- **Архивировано** - Удалено из активного использования

### Термины согласования

- **Процесс согласования** - Многоэтапный процесс с динамическим выбором согласующих
- **Динамический маршрут** - Маршрут согласования создаётся при отправке, не предзадан
- **Этап Approval** - Этап согласования риск-менеджерами (0..* этапов)
- **Этап Ratification** - Обязательный этап утверждения руководством (1 этап в MVP)
- **Согласующий** - Лицо, принимающее решение на этапе Approval
- **Утверждающий** - Лицо, принимающее финальное решение на этапе Ratification
- **Решение по согласованию** - Индивидуальный выбор (approve/reject/ratify)
- **Частичное согласование** - Возможность принять решение по каждому элементу Package индивидуально
- **Отзыв из согласования** - Возврат PilotVersion/DeploymentVersion из процесса согласования с сохранением истории
- **Автоматическое утверждение** - PilotVersion/DeploymentVersion переходит на Ratification индивидуально (auto_ratification = true)
- **Ручное утверждение** - PilotVersion/DeploymentVersion попадает в `awaiting_ratification` (очередь ожидания утверждения), после чего по команде пользователя может быть утвержден индивидуально или в составе Package (auto_ratification = false)
- **Approval** - Этап согласования риск-менеджерами (может быть несколько этапов)
- **Ratification** - Этап утверждения руководством, обязательный для процесса в MVP

### Термины связей

- **Lineage** - Ссылка внедрения типа `simulation_based` на одну Симуляцию, результаты которой внедряются
- **Связь** - Ассоциация между сущностями (например, Пилот связан со Скоркартой)
- **Источник** - Происхождение данных (в MVP Симуляция как источник для Скоркарты)
- **Объект согласования** - Сущность, проходящая процесс согласования (`PilotVersion` или `DeploymentVersion`)

## Ключевые изменения от v3.0

### Терминологические изменения

1. **Deployment → Initiative (Внедрение → Инициатива)**
   - v3.0: Deployment - агрегат верхнего уровня
   - v3.1: Initiative - агрегат верхнего уровня
   - Причина: Улучшение понимания роли сущности как группировки стратегических изменений

2. **Change → Deployment (Изменение → Внедрение)**
   - v3.0: Change - фаза продуктового внедрения
   - v3.1: Deployment - фаза продуктового внедрения
   - Причина: Более точное отражение сути операции (развертывание в продакшн)

**Переименованные таблицы:**
