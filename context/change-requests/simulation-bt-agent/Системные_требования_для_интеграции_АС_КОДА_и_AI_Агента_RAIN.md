## Функциональные требования

### 1. Общее описание и цели интеграции

#### 1.1. Цель

Обеспечить возможность взаимодействия ПРМ аналитика КК с AI-агентом RAIN через диалоговый интерфейс в АС КОДА для:

* создания бизнес-требований (БТ) в Confluence (только для завершенных симуляций);
* получения консультаций (на любых этапах работы внутри раздела).

#### 1.2. Участники и зоны ответственности

| Участник | Роль | Зона ответственности |
| --- | --- | --- |
| **Продукт** | «Кредитные карты» | * область применения. |
| **ПРМ аналитик** | Пользователь (роль experiment_editor_CC) | * инициирует сценарий; * общается с агентом; * подтверждает создание БТ. |
| **АС КОДА** | Клиент | * отображение диалогового интерфейса; * передача контекста в агент; * хранить UI-сессию и состояние диалога (стандартная СУДИР сессия); * обработка технических ошибок (HTTP != 200). *АС КОДА не управляет сценарием диалога и не реализует бизнес-логику формирования БТ.* |
| **AI-агент RAIN** | Сервер | * ведение диалога; * управление сценарием общения; * бор недостающих данных для БТ; * формирование БТ; * создание страницы в Confluence; * возврат ссылки. |
| **Confluence** | Хранилище | * хранение итоговых страниц БТ по шаблону «ПРМ КК». |
| **СУДИР** | Источник данных | * предоставление ФИО пользователя (через АС КОДА) |

### 2. Режим работы агента

#### 2.1. Консультационный режим

**Условия:** симуляция НЕ завершена  
**Где:** главный экран со списком всех симуляций, экран с созданием/редактированием симуляции  
**Ограничения:** БТ не создается

#### 2.2. Режим создания БТ

**Условия:** симуляция имеет статус «Завершена»  
**Где:** экран конкретной завершенной симуляции  
**Возможности:** создание БТ разрешено

### 3. UI-Требования

#### 3.1. Диалоговое окно

Диалоговое окно должно отображаться в виде боковой панели, располагаться справа на экране и не блокировать основной интерфейс

#### 3.2. Работа с контентом

Пользователь должен иметь возможность: копировать текст из АС КОДА, писать текст в чат, копировать текст из чата, вставлять данные в чат

#### 3.3. Отображение данных

АС КОДА НЕ должна отображать: JSON, технические структуры

#### 3.4. Рендеринг сообщений

Чат должен поддерживать форматирование: структурированный текст (markdown)

#### 3.5. Кнопка вызова агента

Кнопка вызова диалога должна быть доступна: на главном экране со списком симуляций, на странице симуляции (любой статус), на странице завершенной симуляции

### 4. Форматирование контекста

#### 4.1. Общие требования

АС КОДА:

* формирует контекст запроса;
* передает его агенту;
* не валидирует и не интерпретирует данные.

#### 4.2. Информация для формирования БТ

| Поле | Заполнение | Логика заполнения | Источник |
| --- | --- | --- | --- |
| Дата создания | Текущая дата | Предоставляет АС КОДА, дата создания = дате запуска диалогового окна в UI | АС КОДА |
| Наименование запрошенных изменений | Предложить | Заглушка на Агенте | Вернет Агент |
| Бизнес-цель | Предложить | На основе запроса пользователя заполняется агентом | Вернет Агент |
| Бизнес-эффект |  | Агент запросит у пользователя | Вернет Агент |
| Кто заказчик | Авто/Предложить | Агент спрашивает, кто заказчик доработки (предлагать ПРМ.КК и другое) | Вернет Агент |
| Отв. аналитик | Авто/Предложить | АС КОДА предоставляет ФИО ПРМ аналитика из СУДИР (формат: «Фамилия Имя Отчество» (кириллица)) | АС КОДА |
| Ограничения | Авто | Пробуем нарастить логику в зависимости от параметра УЛКК или Оформление (уточнить у ПРМ) | Вернет Агент |
| Бизнес-требования | Предложить | АС КОДА берет из ФП Симуляции as is, to be на основе запроса пользователя с указанием simulationId (в АС КОДА поле "ID ФП Симуляции") Данные берутся из основной симуляции в рамках ID ФП Симуляции из вкладки "Изменение Риск-параметров" Название риск-параметра из "Риск-параметр" AS IS из "Текущее значение" TO BE из "Новое значение" Если было изменено несколько РП, то указать все Если значение РП не изменилось, все равно указывать AS IS/TO BE | АС КОДА |
| Риск-параметры | Предложить | В режиме консультирования аналитик может запросить инфо по РП у агента | Вернет Агент |

#### 4.3. Обязательные данные

**Для консультативного режима агента:**

* ФИО из СУДИРа ПРМ аналитика;
* дата = дате запуска диалогового окна.

**Для режима создания БТ:**

* риск-параметры (AS IS / TO BE), если их несколько, то РП, которые участвовали в симуляции;
* ID симуляции (в АС КОДА поле "ID ФП Симуляции");
* дата = дате запуска диалогового окна;
* ФИО из СУДИРа ПРМ аналитика.

#### 4.4. Требование к prompt

Контекст, формируемый АС КОДА:

* должен быть передан агенту в виде текста (prompt);
* не должен отображаться аналитику в виде JSON.

### 5. Сценарии работы

#### 5.1. Основной сценарий (создание БТ)

**Шаг 1**  
Симуляция завершается успешно

**Шаг 2**  
Аналитик нажимает кнопку диалогового окна с Агентом

**Шаг 3**  
АС КОДА формирует текстовый контекст, включающий данные симуляции и инструкцию для агента добавить ID симуляции в БТ

**Шаг 4**  
АС КОДА отображает: человекочитаемый текст, содержащий: ID симуляции, риск-параметры (AS IS / TO BE), ФИО, дату. Аналитик может: редактировать текст

**Шаг 5**  
Агент уточняет недостающие данные (бизнес-эффект), показывает превью перед созданием БТ (риск-параметры (AS IS / TO BE), подтверждает создание БТ

**Шаг 6**  
Аналитик отвечает:

* «Да» - Агент возвращает ссылку на БТ;
* «Нет» - завершение диалога.

Без явного подтверждения ("Да") Агент не создает БТ

**Шаг 7**  
АС КОДА показывает ссылку в диалоговом окне

**Шаг 8**  
Аналитик может скопировать ссылку и вставить в завершенную симуляцию в раздел "Итоги симуляции"

*Аналитик может создать несколько БТ для одной завершенной симуляции и сам выбрать, какое добавлять в раздел "Итоги симуляции"*  
*АС КОДА не проверяет заполнен ли раздел "Итоги симуляции"*

#### 5.2. Консультационный сценарий

**Шаг 1**  
Аналитик нажимает кнопку диалогового окна с Агентом

**Шаг 2**  
АС КОДА формирует контекст с ФИО и предлагаю запустить Агента, указывая в явном виде аналитику, что БТ в данном режим не создается (не редактируется, этот режим подразумевает общение с Агентом)

**Шаг 3**  
Аналитик отправляет запрос Агенту и общается с ним (БТ не создается)

### 6. Работа с ссылкой на БТ

**АС КОДА:**

* отображает ссылку в диалоговом окне;
* не сохранять ссылку;
* не записывает ее в раздел "Итоги симуляции".

**Аналитик:**

* самостоятельно копирует ссылку;
* самостоятельно сохраняет в раздел "Итоги симуляции".

### 7. Сессии

АС КОДА должна:

* хранить сессию (стандартная СУДИР сессия);
* хранить историю сообщений до сброса сессии в рамках одного экрана;
* при повторном открытии диалога в рамках СУДИР сессии диалог восстанавливается с историей общения с ПРМ аналитиком в рамках одного экрана;
* сессии не переносятся между экранами во 2Q;
* генерировать sessionId для каждого диалога и передавать его агенту.

*Агент не хранит сессию*

### 8. Обработка ошибок

#### 8.1. Ошибка Агента

При HTTP ≠ 200: АС КОДА должна: отобразить понятное сообщение об ошибке, предложить перезапустить сессию

Также должна быть дефолтная возможность перезапуска сессии в любой момент аналитиком в диалоговом окне

#### 8.2. Таймаут

Если агент долго не отвечает АС КОДА должна: отобразить понятное сообщение об ошибке, предложить перезапустить сессию

#### 8.3. Требование к агенту (в проработке)

Агент должен обеспечить уникальность названия БТ или запросить его у аналитика
