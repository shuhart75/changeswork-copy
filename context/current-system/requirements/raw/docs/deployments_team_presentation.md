## 1. Что такое `Внедрение` в текущем MVP

`Внедрение` в MVP - это не просто форма для перевода изменения в прод. Это самостоятельная доменная сущность `Deployment`, которая:
- принадлежит одному продукту;
- хранится как агрегат с версиями (`Deployment` + `DeploymentVersion`);
- связывает выбранные скоркарты;
- показывает артефакты;
- автоматически рассчитывает критичность;
- автоматически показывает связанные `Pilot` и `Simulation` через скоркарты;
- проходит согласование через связанный `ApprovalInstance`;
- завершается в одном из конечных состояний: `deployed` или `archived`.

Ключевая мысль для презентации команды:
`Deployment` выступает оркестратором продуктового внедрения. Само внедрение хранит состав и жизненный цикл, но часть данных о риске и происхождении приходит из связанных сущностей.

## 2. Что важно показать команде

### 2.1. Суть разработки

Разработка `Внедрений` состоит не только из CRUD-экранов. В ней есть четыре важные особенности:

1. Создание идёт в два шага через `draft-shell`.
Сначала создаётся минимальная оболочка внедрения, чтобы форма получила `deployment.id` и могла в этом контексте создавать скоркарты и артефакты.

2. Критичность не вводится вручную.
Она вычисляется backend по связанным скоркартам:
- `high`, если хотя бы одна скоркарта `high`;
- иначе `low`.

3. Блок `Связанные сущности` не редактируется.
Он вычисляется backend через связи выбранных скоркарт с `Pilot` и `Simulation`.

4. Промежуточные статусы не принадлежат самому внедрению.
После отправки в процесс интерфейс показывает не "свои" статусы, а текущий `approvalInstance.status`.

### 2.2. Что пользователь видит в интерфейсе

В MVP у внедрения есть три режима:
- список `/deployments`;
- детальная страница `/deployments/:id`;
- режим редактирования на той же странице `/deployments/:id`.

На карточке пользователь всегда видит:
- основные поля внедрения;
- статус;
- критичность;
- связанные скоркарты;
- связанные сущности;
- артефакты;
- доступные действия жизненного цикла.

Важно:
- набор кнопок строится по `available_actions`, которые вернул backend;
- если backend не вернул `deploy`, кнопка `Внедрить` не показывается;
- фронт не придумывает собственный промежуточный enum поверх `ApprovalInstance`.

## 3. Связи с другими сущностями

### 3.1. Главная схема

```text
Deployment
  -> связан с 1..* ScorecardVersion
  -> показывает 0..* Artifact
  -> получает process-state через ApprovalInstance
  -> вычисляет related Pilot через Scorecard usage
  -> вычисляет related Simulation через Scorecard source/usage
```

### 3.2. Что это значит practically

- `Scorecard` - главный носитель содержательных изменений.
- `Deployment` не хранит вручную связи с `Pilot` и `Simulation`.
- `Pilot` и `Simulation` попадают в карточку внедрения только потому, что они уже связаны с выбранными скоркартами.
- `Artifact` живёт отдельно и прикрепляется к `Deployment` как к родительской сущности.
- `ApprovalInstance` живёт отдельно и является источником правды для process-state.

Это хороший тезис для команды:
`Deployment` не дублирует чужие данные, а собирает текущую проекцию из нескольких bounded contexts.

## 4. Поведение интерфейса по статусам

| Что видит пользователь | Что является источником правды | Какие действия доступны |
|---|---|---|
| `draft` | статус версии внедрения | `edit`, `submit_for_approval`, `archive` |
| `in_approval` | `approvalInstance.status` | `recall` |
| `in_ratification` | `approvalInstance.status` | `recall` |
| `approved` и другие пред-deploy process states | `approvalInstance.status` | `start_ratification` или другой action, который вернул backend, плюс `edit`, `archive` по правилам карточки |
| `ratified` | `approvalInstance.status` | `deploy`, `edit`, `archive` |
| `rejected` / `cancelled` | `approvalInstance.status` | `edit`, `submit_for_approval`, `archive` |
| `deployed` | статус версии внедрения | действий нет |
| `archived` | статус версии внедрения | действий нет |

Правило, которое стоит отдельно проговорить:
если пользователь начинает редактирование из состояний после процесса (`approved`, `ratified`, `rejected`, `cancelled`), approval-state сбрасывается, а внедрение возвращается в `draft`.

## 5. Юзкейсы для презентации команды

Ниже сценарии, которые лучше всего показывают суть реализации.

### Юзкейс 1. Создание нового внедрения с новой скоркартой

**Зачем показывать**
Этот сценарий объясняет, что форма внедрения - это orchestration flow, а не обычный single-request create.

**Сюжет**
ПРМ создаёт новое внедрение для продукта "Кредитные карты". У него пока нет готовой скоркарты, поэтому он заводит внедрение и прямо из формы создаёт новую скоркарту вручную.

**Как это выглядит для пользователя**
1. Пользователь открывает раздел `Внедрения` и нажимает `Создать`.
2. Заполняет название и описание.
3. Система создаёт `draft-shell` и получает `deployment.id`.
4. В секции скоркарт пользователь выбирает режим `Создать вручную`.
5. В секции артефактов добавляет бизнес-кейс и ROV ссылками.
6. После финального сохранения открывается детальная карточка внедрения.

**Что здесь важно с точки зрения разработки**
- создание идёт через `POST /api/v1/deployments`, а затем через `PUT /api/v1/deployments/{id}`;
- скоркарты создаются своим API, а не nested-объектом внутри deployment;
- артефакты создаются своим API в контексте `DEPLOYMENT/{entityId}`;
- до финального `PUT` внедрение существует как технический черновик-оболочка;
- если второй шаг не завершён за 2 минуты, `draft-shell` и его артефакты удаляются автоматически.

**Что это иллюстрирует**
- отдельную доменную сущность `Deployment`;
- двухфазное создание;
- интеграцию со скоркартами и артефактами;
- устойчивость интерфейса к промежуточному состоянию.

### Юзкейс 2. Подключение существующей скоркарты и автоматическое заполнение блока `Связанные сущности`

**Зачем показывать**
Этот сценарий лучше всего объясняет, почему `Deployment` связан с другими сущностями не напрямую, а через скоркарты.

**Сюжет**
ПРМ создаёт внедрение и выбирает уже существующую скоркарту, которая раньше использовалась в пилоте и была частично предзаполнена из симуляции.

**Как это выглядит для пользователя**
1. Пользователь создаёт или открывает внедрение в `draft`.
2. В секции скоркарт выбирает режим `Использовать существующую`.
3. Выбирает скоркарту по ID или названию.
4. После сохранения в деталке автоматически появляются:
   - выбранная скоркарта;
   - связанный пилот;
   - связанная симуляция.

**Что здесь важно с точки зрения разработки**
- фронт не редактирует блок `Связанные сущности`;
- backend сам собирает `related_pilots` и `related_simulations`;
- если одна и та же сущность пришла через несколько скоркарт, backend убирает дубликаты;
- текущее внедрение не должно отображаться в related-блоке само себе.

**Что это иллюстрирует**
- read-only характер блока `Связанные сущности`;
- наследование бизнес-контекста от выбранных скоркарт;
- связь `Deployment -> Scorecard -> Pilot/Simulation`.

### Юзкейс 3. Автоматический расчёт критичности внедрения

**Зачем показывать**
Это хороший пример бизнес-логики, в которой интерфейс показывает итог, но не даёт пользователю сломать правило ручным вводом.

**Сюжет**
Во внедрение добавляются две скоркарты: одна `low`, другая `high`.

**Как это выглядит для пользователя**
1. Пользователь добавляет в форму несколько скоркарт.
2. Поле `Критичность` отображается как read-only.
3. После сохранения в списке и в деталке внедрение получает чип `high`.

**Что здесь важно с точки зрения разработки**
- критичность внедрения не передаётся пользователем;
- backend пересчитывает её на create и update;
- правило простое и прозрачное: хотя бы одна `high` скоркарта делает всё внедрение `high`.

**Что это иллюстрирует**
- separation of concerns между UI и доменной логикой;
- зависимость агрегированного признака `Deployment` от состава связанных скоркарт.

### Юзкейс 4. Отправка на согласование: интерфейс живёт по `ApprovalInstance`

**Зачем показывать**
Этот сценарий объясняет, что у внедрения есть собственный жизненный цикл, но процессные состояния не моделируются вторым, параллельным enum на фронте.

**Сюжет**
Черновик внедрения готов. ПРМ отправляет его в процесс согласования.

**Как это выглядит для пользователя**
1. На детальной карточке в `draft` доступна кнопка `Отправить на согласование/утверждение`.
2. После действия карточка обновляется.
3. Вместо локального "внутреннего" статуса пользователь видит текущий статус процесса из `approvalInstance`.
4. В зависимости от ответа backend меняется набор кнопок:
   - при `in_approval` доступен только `Отозвать`;
   - при достижении ratification-ready состояния появляется действие запуска/продолжения ratification;
   - после `ratified` появляется `Внедрить`, но только если backend вернул `deploy` в `available_actions`.

**Что здесь важно с точки зрения разработки**
- deployment page не хранит собственную сложную process-state machine;
- источник правды для процесса - `ApprovalInstance`;
- источник правды для кнопок - `available_actions`;
- одна и та же карточка должна одинаково корректно работать в списке, деталке и после action-refresh.

**Что это иллюстрирует**
- интеграцию `Deployment` с approval core;
- принцип "backend decides, frontend renders";
- зависимость UI-поведения от контракта, а не от локальных догадок.

### Юзкейс 5. Редактирование после согласования сбрасывает процесс в `draft`

**Зачем показывать**
Это один из самых важных product-behaviour сценариев. Он объясняет, что редактирование после согласования - это не косметика, а новая содержательная версия.

**Сюжет**
После прохождения этапа согласования пользователь замечает, что нужно заменить одну из скоркарт или поправить описание внедрения.

**Как это выглядит для пользователя**
1. Пользователь открывает карточку в статусе после процесса, где редактирование ещё разрешено.
2. Нажимает `Редактировать`.
3. Меняет описание, состав скоркарт или артефакты.
4. Сохраняет форму.
5. Карточка возвращается в `draft`.

**Что здесь важно с точки зрения разработки**
- редактирование разрешено не только в `draft`, но и в части post-approval состояний;
- backend при сохранении должен сбросить approval-state;
- фронт должен честно показать возврат в `draft`, а не сохранять старый процессный статус;
- таким образом внедрение ведёт себя как versioned domain aggregate, а не как обычная запись.

**Что это иллюстрирует**
- версионную природу `Deployment`;
- связь между бизнес-изменением и повторным прохождением approval flow.

### Юзкейс 6. Финальный переход в `deployed`

**Зачем показывать**
Это сценарий, который закрывает весь путь и показывает, что `deploy` - не просто кнопка, а разрешённое backend-действие только после успешного процесса.

**Сюжет**
Внедрение прошло все этапы согласования и готово к раскатке.

**Как это выглядит для пользователя**
1. Пользователь открывает карточку и видит process-state, при котором backend вернул `deploy`.
2. На карточке появляется кнопка `Внедрить`.
3. После нажатия карточка получает конечный статус `Внедрено`, а в данных фиксируется `deployed_at`.

**Что здесь важно с точки зрения разработки**
- фронт не должен показывать кнопку заранее;
- backend обязан сам проверить допустимость действия;
- `deployed` - конечный статус, после него редактирование недоступно.

**Что это иллюстрирует**
- финализацию жизненного цикла;
- жёсткую связку UI и backend-правил;
- отличие process-state от terminal state.

## 6. Короткая структура выступления команды

Если нужен компактный рассказ на 10-15 минут, можно идти так:

1. Что такое `Deployment` в MVP.
Это самостоятельная сущность продуктового внедрения, а не просто шаг внутри инициативы.

2. Почему реализация нетривиальна.
Потому что создание двухфазное, статус процесса приходит извне, а связи с `Pilot` и `Simulation` вычисляются через скоркарты.

3. Как выглядит интерфейс.
Список, карточка, режим редактирования на той же странице, кнопки по `available_actions`.

4. Три главных сценария.
- создание с `draft-shell`;
- подключение существующей скоркарты и автосборка related-блока;
- отправка в approval flow и появление `deploy` только по сигналу backend.

5. Что это даёт бизнесу и разработке.
- прозрачный жизненный цикл внедрения;
- отсутствие ручного расхождения между сущностями;
- предсказуемое UI-поведение;
- хорошую трассируемость через скоркарты и approval process.

## 7. Тезисы, которые стоит проговорить отдельно

- В MVP нет ручного редактирования блока `Связанные сущности`.
- В MVP нет ручного ввода критичности внедрения.
- В MVP нет отдельного экрана истории версий внедрения, хотя модель под капотом версионная.
- Скоркарта не проходит согласование отдельно; согласуется версия `Deployment`.
- `methodologist` видит внедрения всех продуктов, но не управляет их жизненным циклом с этой страницы.

## 8. Один слайд в одной фразе

`Внедрение` в MVP - это карточка продуктового rollout'а, которая собирает состав скоркарт, артефакты и вычисленные связи, а весь процесс согласования и доступные действия получает от backend через `ApprovalInstance` и `available_actions`.
