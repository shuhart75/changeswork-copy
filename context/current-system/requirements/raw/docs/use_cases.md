# Система управления внедрениями риск-стратегий
## Сценарии использования

---

## Сценарий 1: Полный цикл внедрения новой скоринговой модели

### Контекст
Команда риск-аналитики разработала новую скоринговую модель для Premium сегмента кредитных карт. Модель показывает потенциал улучшения approval rate на 5% при сохранении уровня риска.

### Участники
- **Анна** — старший риск-аналитик
- **Борис** — риск-менеджер продукта
- **Виктор** — Head of Risk (утверждающий)

---

### Шаг 1: Создание внедрения и симуляции

**Анна создает новое Внедрение:**
1. Открывает страницу "Внедрения"
2. Нажимает "Создать внедрение"
3. Заполняет форму:
   - Название: "Новая скоринговая модель Premium v3.0"
   - Продукт: Кредитные карты → Premium
   - Описание: "Улучшенная модель с новыми факторами..."
   - Автор: Анна Иванова
4. Сохраняет внедрение (статус: draft)

**Анна создает симуляцию:**
1. На детальной странице внедрения переходит на вкладку "Симуляции"
2. Нажимает "Новая симуляция"
3. Заполняет параметры:
   - Название: "Симуляция модели v3.0 на данных 2025"
   - Период данных: 01.01.2025 - 31.12.2025
   - Режим: Полная симуляция
   - Конфигурация модели: [загружает файл]
4. Запускает симуляцию

**Система:**
- Отправляет запрос в систему симуляций через API
- Обновляет статус: running
- Статус внедрения автоматически меняется на active

**Через 2 дня симуляция завершается:**
- Система получает результаты через webhook
- Статус симуляции: completed
- Результаты отображаются на карточке:
  - AR(абс): +2.3M
  - Объём выдач: +15%
  - NPV: +5.2M
  - RC: -0.3%

---

### Шаг 2: Создание скоркарты и пилота

**Анна создает скоркарту на основе результатов:**
1. На детальной симуляции нажимает "Создать скоркарту"
2. Система автоматически заполняет:
   - Источник: Симуляция #1
   - Продукт: Кредитные карты → Premium
   - Финансовые эффекты: данные из симуляции
3. Анна дополняет:
   - Название: "Скоркарта Premium v3.0"
   - Критичность: High (влияет на большой объем клиентов)
   - Конфигурация: [детали модели]
4. Сохраняет скоркарту (статус: draft)

**Борис создает пилот:**
1. На детальной внедрения переходит на вкладку "Пилоты"
2. Нажимает "Новый пилот"
3. Заполняет форму:
   - Название: "A/B тест модели v3.0"
   - Связанные симуляции: Симуляция #1
   - Скоркарты: Скоркарта Premium v3.0 (тест), Текущая модель (контроль)
   - Параметры:
     - Доля трафика: 10% на тест, 90% на контроль
     - Длительность: 30 дней
     - Целевые метрики: Approval rate, Default rate, Revenue
4. Сохраняет пилот (статус: draft)

---

### Шаг 3: Согласование пилота

**Борис отправляет пилот на согласование:**
1. На детальной пилота нажимает "Отправить на согласование"
2. Система меняет статус: requires_activation
3. Пользователь выбирает маршрут согласования вручную:
   - Этап(ы) Approval: Старший риск-менеджер, Аналитик данных
   - Ratifier: Виктор (Head of Risk)

**Согласующие проверяют пилот:**
- Получают уведомление
- Проверяют методологию, параметры, риски
- Одобряют согласование

**Виктор утверждает пилот:**
1. Получает уведомление о необходимости утверждения
2. Открывает детальную пилота
3. Просматривает:
   - Результаты симуляции
   - Параметры пилота
   - Оценку рисков
4. Нажимает "Утвердить"
5. Система меняет статус: approved

**Борис активирует пилот:**
1. Получает уведомление об утверждении
2. На детальной пилота нажимает "Активировать"
3. Система отправляет запрос в платформу экспериментов
4. Статус меняется: active
5. Начинается сбор метрик

---

### Шаг 4: Мониторинг пилота

**В течение 30 дней:**
- Система получает метрики из платформы экспериментов
- Борис мониторит результаты на дашборде
- Промежуточные результаты показывают:
  - Approval rate: +4.8% (близко к прогнозу)
  - Default rate: без изменений
  - Revenue: +2.1M (соответствует прогнозу)

**Пилот завершается:**
- Система автоматически меняет статус: completed
- Финальные результаты фиксируются
- Борис получает уведомление

---

### Шаг 5: Внедрение в продакшен

**Борис создает Изменение:**
1. На детальной внедрения переходит на вкладку "Изменения"
2. Нажимает "Новое изменение"
3. Заполняет форму:
   - Название: "Внедрение модели Premium v3.0"
   - Связанные пилоты: Пилот #1
   - Скоркарты: Скоркарта Premium v3.0
   - Тип: Полное внедрение (100% трафика)
   - Плановая дата: 15.03.2026
4. Сохраняет изменение (статус: draft)

**Борис отправляет на согласование:**
1. На детальной изменения нажимает "Отправить на согласование"
2. Статус меняется: requires_approval
3. Маршрут согласования:
   - Согласующие: Риск-комитет (3 человека)
   - Утверждающий: Виктор (Head of Risk)

**После согласования и утверждения:**
- Статус: approved
- Борис планирует дату внедрения

**День внедрения:**
1. Борис нажимает "Внедрить"
2. Система отправляет запрос в систему деплоя
3. Скоркарта внедряется в продакшен
4. Статус изменения: deployed
5. Статус внедрения: deployed

**Результат:**
- Полная трейсабилити от симуляции до продакшена
- Все решения задокументированы
- История изменений сохранена
- Фактические эффекты отслеживаются

## Сценарий 2: Срочное внедрение регуляторного требования

### Контекст
Регулятор выпустил новое требование, которое необходимо внедрить в течение 2 недель. Нет времени на симуляции и пилоты.

### Участники
- **Дмитрий** — риск-менеджер
- **Елена** — Head of Compliance
- **Виктор** — Head of Risk

---

### Быстрый путь

**Дмитрий создает внедрение:**
1. Название: "Регуляторное требование #2026-15"
2. Продукт: Все кредитные продукты
3. Критичность: Critical
4. Обоснование: Регуляторное требование

**Дмитрий создает скоркарту вручную:**
1. Без связи с симуляцией/пилотом
2. Источник: Ручное создание (регуляторное требование)
3. Конфигурация: согласно требованиям регулятора

**Дмитрий создает изменение:**
1. Связывает со скоркартой
2. Тип: Срочное внедрение
3. Плановая дата: через 10 дней

**Ускоренное согласование:**
1. Дмитрий отправляет на согласование с пометкой "Срочно"
2. Елена (Compliance) подтверждает соответствие требованиям
3. Виктор утверждает в течение дня
4. Внедрение происходит в срок

**Преимущества системы:**
- Даже срочное внедрение задокументировано
- Обоснование зафиксировано
- Процесс согласования соблюден (но ускорен)
- История сохранена для аудита

## Сценарий 3: A/B/C тестирование нескольких вариантов

### Контекст
Команда разработала 3 варианта лимитной политики для Standard сегмента. Нужно выбрать лучший через пилот.

### Участники
- **Федор** — продуктовый аналитик
- **Галина** — риск-менеджер

---

### Процесс

**Федор создает внедрение:**
- "Оптимизация лимитной политики Standard"

**Федор создает 3 симуляции:**
1. Симуляция варианта A (консервативный)
2. Симуляция варианта B (сбалансированный)
3. Симуляция варианта C (агрессивный)

**После завершения симуляций:**
- Создает 3 скоркарты на основе результатов
- Все 3 показывают положительные эффекты, но разные

**Галина создает пилот с 3 скоркартами:**
1. Название: "A/B/C тест лимитной политики"
2. Скоркарты:
   - Вариант A (25% трафика)
   - Вариант B (25% трафика)
   - Вариант C (25% трафика)
   - Контроль (25% трафика)
3. Длительность: 45 дней

**После завершения пилота:**
- Вариант B показал лучшие результаты
- Галина создает Изменение только с вариантом B
- Варианты A и C не внедряются, но история сохранена

**Ценность:**
- Система поддерживает множественные варианты
- Все эксперименты задокументированы
- Можно вернуться к неиспользованным вариантам позже

## Сценарий 4: Откат проблемного внедрения

### Контекст
После внедрения новой модели для автокредитов обнаружена проблема: default rate вырос на 2% выше прогноза.

### Участники
- **Игорь** — риск-менеджер автокредитов
- **Виктор** — Head of Risk

---

### Процесс отката

**Игорь обнаруживает проблему:**
1. Мониторит метрики внедренного изменения
2. Видит отклонение от прогноза
3. Анализирует причины

**Игорь инициирует откат:**
1. Открывает детальную изменения
2. Нажимает "Откатить"
3. Указывает причину: "Default rate выше прогноза на 2%"
4. Подтверждает откат

**Система:**
1. Отправляет запрос в систему деплоя
2. Возвращает предыдущую версию скоркарты
3. Меняет статус изменения: rolled_back
4. Фиксирует дату и причину отката
5. Уведомляет всех заинтересованных

**Игорь создает новую симуляцию:**
1. Для анализа причин проблемы
2. С корректировками модели
3. Новый цикл: Симуляция → Пилот → Изменение

**Ценность:**
- Быстрый откат при проблемах
- История отката сохранена
- Можно проанализировать причины
- Процесс улучшения продолжается

## Сценарий 5: Пакетное согласование связанных изменений

### Контекст
Команда подготовила комплексное изменение стратегии, затрагивающее 3 продукта одновременно. Все изменения должны быть внедрены синхронно.

### Участники
- **Кирилл** — руководитель отдела риск-стратегий
- **Виктор** — Head of Risk

---

### Процесс

**Кирилл создает 3 внедрения:**
1. Внедрение для кредитных карт
2. Внедрение для потребительских кредитов
3. Внедрение для автокредитов

**В каждом внедрении:**
- Симуляции завершены
- Пилоты успешны
- Изменения готовы к внедрению

**Кирилл создает Пакет:**
1. Открывает страницу "Пакеты"
2. Нажимает "Создать пакет"
3. Заполняет:
   - Название: "Комплексное изменение стратегии Q1 2026"
   - Добавляет 3 цепочки (по одной из каждого внедрения)
   - Маршрут согласования: единый для всех
   - Критичность: High (максимальная из всех цепочек)
4. Сохраняет пакет

**Кирилл отправляет пакет на согласование:**
1. Все 3 изменения отправляются одновременно
2. Согласующие видят их как единый пакет
3. Могут одобрить/отклонить все вместе или по отдельности

**Виктор утверждает пакет:**
1. Видит все 3 изменения в едином брифе
2. Оценивает комплексное влияние
3. Утверждает все изменения одной кнопкой

**Кирилл внедряет синхронно:**
1. Планирует единую дату внедрения
2. Все 3 изменения внедряются одновременно
3. Система координирует процесс

**Ценность:**
- Координация связанных изменений
- Единый процесс согласования
- Снижение административной нагрузки
- Синхронное внедрение

## Сценарий 6: Исследовательская симуляция без внедрения

### Контекст
Аналитик хочет проверить гипотезу о влиянии нового фактора, но пока не планирует внедрение.

### Участники
- **Лариса** — младший аналитик

---

### Процесс

**Лариса создает внедрение:**
1. Название: "Исследование: влияние фактора X"
2. Тип: Исследовательское (без планов внедрения)

**Лариса создает симуляцию:**
1. Настраивает параметры
2. Запускает симуляцию
3. Получает результаты

**Анализ результатов:**
- Результаты не показывают значимого улучшения
- Лариса документирует выводы
- Закрывает внедрение без создания пилота/изменения

**Лариса архивирует внедрение:**
1. Статус: archived
2. Причина: "Гипотеза не подтвердилась"
3. Выводы сохранены для будущих исследований

**Ценность:**
- Даже неудачные эксперименты задокументированы
- Команда не повторяет одни и те же ошибки
- База знаний накапливается
- Прозрачность исследовательской работы

---

## Сценарий 7: Корректировка пилота на основе промежуточных результатов

### Контекст
Пилот показывает неожиданные результаты через 2 недели. Нужно скорректировать параметры.

### Участники
- **Максим** — риск-менеджер
- **Нина** — старший аналитик

---

### Процесс

**Максим мониторит пилот:**
1. Видит отклонение метрик от прогноза
2. Анализирует причины с Ниной
3. Принимает решение о корректировке

**Максим останавливает пилот:**
1. На детальной пилота нажимает "Приостановить"
2. Статус меняется: requires_correction
3. Указывает причину: "Необходима корректировка параметров"

**Нина создает новую симуляцию:**
1. С скорректированными параметрами
2. Быстрая симуляция на коротком периоде
3. Результаты показывают улучшение

**Максим обновляет пилот:**
1. Изменяет параметры на основе новой симуляции
2. Отправляет на повторное согласование
3. После утверждения возобновляет пилот

**Пилот продолжается:**
- С новыми параметрами
- Метрики приходят в норму
- Пилот успешно завершается

**Ценность:**
- Гибкость процесса
- Возможность корректировки на лету
- История корректировок сохранена
- Снижение рисков неудачного пилота

---

## Выводы

### Система поддерживает разные сценарии:

1. **Полный цикл** — для критичных изменений с высоким риском
2. **Быстрый путь** — для срочных внедрений
3. **Множественные варианты** — для A/B/C тестирования
4. **Откаты** — для управления проблемами
5. **Пакетное согласование** — для координации связанных изменений
6. **Исследования** — для проверки гипотез
7. **Корректировки** — для адаптации в процессе

### Ключевые преимущества:

- **Гибкость** — система адаптируется под разные ситуации
- **Прозрачность** — вся история решений сохранена
- **Контроль** — процесс согласования на каждом этапе
- **Трейсабилити** — от идеи до продакшена
- **Снижение рисков** — валидация на каждом этапе
