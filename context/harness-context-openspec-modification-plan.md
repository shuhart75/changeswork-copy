# План модификации обвязки: малое окно контекста + заимствования из `openspec-skills`

Статус: черновик для согласования
Дата: 2026-06-09

## 1. Цель

Сделать обвязку пригодной для уверенной работы LLM с малым контекстным окном на больших объёмах требований.

Ожидаемое поведение:

- пользователь работает в терминах своей роли и не управляет контекстом вручную;
- обвязка сама решает, когда собирать контекст, обновлять сводки, делать чекпойнты, запускать исследование и проверять полноту;
- все принятые решения фиксируются в markdown-артефактах репозитория;
- временная память, research-файлы и внешние memory-инструменты являются вспомогательными, а не источником истины.

## 2. Что заимствуем

### Из `agentmemory`

Берём архитектурный принцип, а не обязательную зависимость:

- короткие структурированные воспоминания вместо полного перечитывания;
- чекпойнты хода работы;
- извлечение только релевантного контекста;
- разделение фактов, выводов, предположений и открытых вопросов;
- ограниченный бюджет контекста.

Внешняя память может быть ускорителем, но источник истины остаётся в репозитории.

### Из `/home/reutov/Downloads/coda_docs/openspec-skills`

Берём:

- двухэтапное интервью: сначала "что и зачем", затем "как устроено и какие ограничения";
- исследование по ролям;
- `.research/*.yaml` и агрегированную сводку как вспомогательный слой;
- проверочный список полноты по данным, API, интеграциям, миграциям, ошибкам, безопасности, логам, метрикам, настройкам и критериям проверки;
- идею технической передачи в разработку по готовому срезу;
- идею чернового плана реализации;
- дисциплину пошагового выполнения с обязательной проверкой.

Не берём:

- сервисную спецификацию как главный источник аналитики;
- отдельный `change.md` как параллельную правду;
- `apply-change` и `archive-change`.

## 3. Главный принцип пользовательского слоя

Пользователь не должен знать, что LLM нужна контекстная капсула, research, чекпойнт или проверка полноты. Эти действия выполняются "под капотом".

Публичные команды должны соответствовать рабочему намерению пользователя:

- аналитику - завести фичу, собрать требования, сделать прототип, разложить на срезы;
- планирующему аналитику или руководителю - подготовить план, оценки и план-факт;
- разработчику - взять срез в разработку, разобрать код, реализовать, проверить, подготовить к ревью;
- тестировщику - подготовить проверки, негативные сценарии, покрытие и зафиксировать пробелы.

## 4. Роль: аналитик

### Публичные команды

| Команда | Когда используется | Что получает аналитик |
|---|---|---|
| `новая фича` / `по этим требованиям заведи новую фичу` | Есть папка или набор требований, нужно начать работу | Предварительный разбор: что уже есть, что новая дельта, какие фичи и текущее описание системы затронуты |
| `создай общий кликабельный прототип` | Нужно согласовать концепт с заказчиком | Один общий прототип фичи и заметки по прототипу |
| `делаем требования` | Нужно подготовить требования по фиче | Корневой файл требований по фиче |
| `актуализируй требования` | Появились новые материалы или уточнения | Обновлённые требования и список затронутых артефактов |
| `разложи требования на срезы` | Корневые требования готовы к декомпозиции | Список срезов, карточки срезов, обновлённая структура требований |
| `подготовь детальные требования по срезам` | Нужно разложить root requirements в детали | Детальные требования к интерфейсу и серверной части по каждому срезу |
| `создай прототип среза для фронта` | Общий прототип согласован, нужен handoff для интерфейса | Прототип среза для разработки интерфейса |
| `общий прототип согласован` | Пользователь подтвердил общий прототип | Разрешение на подготовку прототипов срезов |

### Внутренние автоматические действия

- сбор и обновление контекста фичи;
- сбор и обновление контекста среза;
- чекпойнты перед и после длинных проходов;
- исследование среза по ролям;
- проверка полноты среза;
- сверка требований, срезов и прототипов;
- актуализация контекста прототипа;
- фиксация открытых вопросов и предположений.

### Когда спрашивать аналитика

- материалы противоречат друг другу;
- непонятно, новая это фича или изменение существующей;
- есть несколько вариантов границ среза;
- найдено поведение, которого нет в требованиях, но оно влияет на пользователя;
- прототип и требования расходятся;
- исследование выявило необходимость изменить корневые требования;
- изменение затрагивает соседнюю фичу или текущее описание системы;
- нужно принять бизнес-решение.

## 5. Роль: планирование

`openspec-skills` почти не покрывает планирование: `tasks.md` у коллег является исполнительским планом реализации, а не планом фичи, оценки, квартала и план-факта. Для нашего планирования полезнее принципы `agentmemory`: краткий контекст, допущения, решения, связи и чекпойнты.

### Публичные команды

| Команда | Когда используется | Что получает пользователь |
|---|---|---|
| `новая фича` | Есть новая инициатива или папка материалов | Предварительный разбор: текущее состояние, новая дельта, затронутые фичи, спорные места |
| `занимаемся планированием` / `спланируй фичу` | Нужно разложить фичу для квартального планирования | Плановые истории, черновые оценки, риски и допущения |
| `подготовь HLE` | Нужно дать верхнеуровневую оценку | Оценки по ролям: аналитик, разработчик интерфейса, разработчик серверной части, тестировщик; основания оценки |
| `собери квартальный план` | Нужно собрать план работ без управленческого буфера | Квартальный план по плановым историям |
| `собери командирский план` | Нужно добавить управленческий буфер | План с буфером и сохранённой связью с исходным квартальным планом |
| `сравни план и факт` | Нужно понять отклонения | Сопоставление плановых историй и фактических задач |

### Внутренние автоматические действия

- сбор контекста исходных материалов;
- разделение текущего состояния системы, новой дельты и спорных пунктов;
- поиск похожих или затронутых фич;
- фиксация планировочных допущений;
- фиксация причин оценки и рисков;
- связь "источник -> дельта -> плановая история -> срез";
- проверка, что плановые истории не смешаны с задачами реализации;
- чекпойнты планирования после intake, после оценки и после сборки плана.

### Предлагаемые планировочные артефакты

- расширение `planning/intake/*.md` блоком "Контекст для LLM";
- `features/<feature>/planning/planning-context.md`;
- `features/<feature>/planning/assumptions.md`;
- `features/<feature>/planning/risk-register.md`;
- `features/<feature>/planning/story-map.md`.

Если отдельные файлы окажутся избыточными, их можно объединить в один `planning-context.md` с разделами.

### Когда спрашивать пользователя

- неясно, это новая дельта или уже существующее поведение;
- фича может пересекаться с другой фичей или другим кварталом;
- оценка зависит от бизнес-решения или варианта реализации;
- нет достаточных материалов для оценки;
- есть конфликт между планом, требованиями и прототипом;
- нужно подтвердить границы квартального объёма.

## 6. Роль: актуализация реального положения дел

Это отдельный контур от планирования. Планирование отвечает за baseline плана, а актуализация факта отвечает за реальные задачи, исполнителей, даты, прогресс, замену плановых историй фактическими задачами и план-факт.

### Публичные команды

| Команда | Когда используется | Что получает пользователь |
|---|---|---|
| `обнови реальный прогресс` / `обновляем прогресс` | Известны новые факты по задачам | Обновлённые задачи, прогресс и диаграмма план-факт |
| `задача X завершена` | Нужно отметить факт завершения | Обновлённый статус, даты, прогресс, связанные плановые истории |
| `задачу X взял Y` | Изменился исполнитель или старт задачи | Обновлённый исполнитель, старт, план-факт и ресурсная раскладка |
| `добавь реальные задачи вместо story X` | Плановая история заменяется фактическими задачами | Связь story -> tasks, задачи реализации, обновлённый план-факт |
| `сравни план и факт` | Нужно понять отклонения | Объяснение отклонений по срокам, исполнителям, объёму и зависимостям |
| `собери puml без инклюдов` | Нужно отдать диаграмму во внешний инструмент | Standalone PlantUML без include |

### Внутренние автоматические действия

- сбор контекста плановых историй, задач реализации, actualization map и командного roster;
- проверка активного режима: если нужно менять факт, работа должна идти в `execution-update`;
- сохранение видимости плановых историй для сравнения план-факт;
- проверка, что реальные задачи не дублируются при many-to-many связях;
- обновление связей story -> tasks;
- перенос не начатых задач в допустимое будущее без изменения markdown-источника дат, если это правило генератора;
- проверка перегрузки исполнителей;
- регенерация actual-progress и standalone-экспорта;
- чекпойнт фактического состояния после обновления.

### Предлагаемые артефакты факта

- `features/<feature>/planning/actualization.md` как источник связи плановых историй и фактических задач;
- `features/<feature>/slices/<slice>/execution/tasks.md` как источник фактических задач;
- `features/<feature>/execution-context.md` или раздел в существующем planning context для краткой сводки факта;
- обновлённые gantt-артефакты actual-progress;
- standalone PlantUML export без include.

### Когда спрашивать пользователя

- факт противоречит уже записанным датам или статусам;
- задача относится сразу к нескольким историям и связь неочевидна;
- исполнитель отсутствует в roster или роль неясна;
- изменение факта затрагивает плановый baseline, а не только actual-progress;
- задача выглядит новой scope-добавкой, а не фактическим исполнением существующего среза;
- генератор не может корректно разложить ресурсы без выбора пользователя.

## 7. Роль: разработчик

### Публичные команды

| Команда | Когда используется | Что получает разработчик |
|---|---|---|
| `возьми срез в разработку` | Разработчик получил готовый срез | Краткую сводку: что реализовать, где требования, какие прототипы и проверки учитывать |
| `разбери срез по коду` | Нужно понять, где в коде реализовывать | Технический разбор затронутых модулей, точек входа, данных, интеграций, тестов и рисков |
| `предложи план реализации` | Нужен черновой порядок работ | План реализации, где задачи связаны с требованиями и проверками |
| `начни реализацию` | План понятен и можно выполнять | Кодовую правку по первой небольшой задаче, список файлов и результат проверок |
| `продолжи реализацию` | Работа была прервана или план частично выполнен | Продолжение с правильного места по чекпойнту |
| `проверь реализацию среза` | Нужно сверить результат с требованиями | Отчёт: что выполнено, что не покрыто, какие проверки прошли, какие риски остались |
| `подготовь к ревью` | Перед pull request / merge request | Сводку изменений, проверок, рисков и ссылок на требования |

### Внутренние автоматические действия

- сбор и обновление контекста среза;
- чтение требований, прототипов, критериев проверки и технической передачи;
- исследование кода по ролям;
- сверка плана реализации с требованиями;
- проверка, что каждая задача связана с требованием или критерием проверки;
- проверка, что реализация не вышла за рамки среза;
- запуск доступных проверок: сборка, тесты, статический анализ;
- обновление технического чекпойнта после изменения кода.

### Когда спрашивать разработчика или техлида

- найдено несколько возможных мест реализации;
- требования противоречат текущей реализации;
- требуется архитектурное решение, которого нет в требованиях;
- изменение затрагивает соседний срез или фичу;
- невозможно надёжно определить команду сборки или тестов;
- проверка упала, и причина неясна;
- реализация требует изменения требований.

## 8. Роль: тестировщик

### Публичные команды

| Команда | Когда используется | Что получает тестировщик |
|---|---|---|
| `подготовь проверки по срезу` | Срез готов к тестированию или почти готов | Черновой набор пользовательских, серверных, интеграционных и технических проверок |
| `собери негативные сценарии` | Нужно усилить тест-дизайн | Негативные и граничные сценарии с привязкой к требованиям |
| `сверь проверки с требованиями` | Перед финализацией тест-дизайна | Матрицу покрытия: требование -> проверки |
| `проверь прототип по срезу` | Нужно проверить ожидаемое поведение интерфейса | Список UX/поведенческих проверок и расхождений |
| `проверь реализацию по срезу` | Есть реализованный код или стендовый результат | Отчёт: что проверять вручную, что автоматизировать, какие риски остались |
| `зафиксируй найденные пробелы` | Найдена недосказанность или противоречие | Список вопросов или дефектов требований для аналитика |

### Внутренние автоматические действия

- сбор и обновление контекста среза;
- сверка требований с прототипом;
- исследование ошибок, валидации, ролей, интеграций, миграций и граничных случаев;
- построение матрицы покрытия;
- отделение пользовательских проверок от технических;
- пометка предположений как предположений;
- обновление чекпойнта тест-дизайна.

### Когда спрашивать тестировщика или аналитика

- проверка основана на предположении, которого нет в требованиях;
- требование невозможно проверить в текущем виде;
- прототип и требования расходятся;
- критерии проверки слишком общие;
- неясно, обязательный сценарий или опциональный;
- найдено влияние на соседний срез;
- нужны тестовые данные, но правила их подготовки не описаны;
- реализация отличается от требований, но возможно требования устарели.

## 9. Внутренние research-роли

Эти роли не являются пользовательскими ролями. Они используются обвязкой автоматически.

| Research-роль | Что ищет | Выход |
|---|---|---|
| `interface` | Экраны, состояния, действия, видимость, ошибки интерфейса, связь с прототипом | `frontend.yaml` |
| `backend` | API, обработчики, бизнес-правила серверной части, ограничения | `backend.yaml` |
| `data` | Сущности, поля, таблицы, миграции, справочники | `data.yaml` |
| `integrations` | Внешние системы, события, документы, очереди, файлы, статусы обмена | `integrations.yaml` |
| `errors-validation` | Валидация, ошибки, ручное разрешение, повторы, пограничные случаи | `errors-validation.yaml` |
| `roles-access` | Роли, права, доступность экранов и действий | `roles-access.yaml` |
| `observability-config` | Логи, метрики, аудит, настройки, эксплуатационные признаки | `observability-config.yaml` |

Внутренние команды типа `собери контекст фичи`, `исследуй срез`, `обнови чекпойнт`, `сверь прототип со срезом` не добавлять как пользовательские команды аналитика. Их описать в policy как автоматические этапы.

## 12. Проверки и ограничения

Нужно обеспечить:

- research-артефакты не становятся источником истины;
- любое принятое решение переносится в требования, карту влияния, прототип, handoff, test-plan или релизный пакет;
- задачи реализации не заменяют плановые истории;
- planning context не заменяет planning stories и estimates;
- актуализация факта не меняет quarter/commander baseline без явного режима и команды;
- actual-progress сохраняет видимость baseline-плана для сравнения;
- прототипы срезов создаются только после согласованного общего прототипа;
- пользователь видит вопросы только при реальной неопределённости;
- после длинных проходов обновляется чекпойнт;
- LLM с малым окном может продолжить работу по context-summary и artifact-map.

## 13. Предлагаемый порядок реализации

1. Добавить политики контекста и research.
2. Добавить шаблоны context-summary и artifact-map.
3. Добавить планировочные шаблоны: planning-context, assumptions, risk-register, story-map.
4. Обновить контракт LLM и mode-файлы: автоматические этапы вместо пользовательских контекстных команд.
5. Добавить команды для аналитика, планирования, actual-progress, разработчика и тестировщика в command catalog.
6. Добавить шаблоны research, handoff, implementation-plan и test-plan.
7. Добавить валидатор контекста.
8. Проверить интеграцию с gantt/actual-progress генераторами.
9. Провести пробный проход на одной существующей фиче и одном срезе.
10. После проверки включить правила в основной workflow.
